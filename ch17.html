JavaScript & TypeScript 从入门到精通

第 17 章 · 高级类型：条件、映射、模板字面量与 infer

本章目标

上一章我们把**泛型（generics）**从头讲透——它让类型能像参数一样被“传进来“。但泛型本身还比较“直“：你传进什么类型，T 就是什么类型，中间不做加工。

真正让 TypeScript 的类型系统变成一门“小型编程语言“的，是在类型层面做计算的能力：

  • 根据条件选择类型（if / else）；
  • 遍历一个类型的每个属性并逐个改造；
  • 从一个复杂类型里反推 / 提取出某个部分；
  • 从字符串模板里拼接 / 解析出新的字符串字面量类型。

这四件事，正好对应本章四大主角：条件类型（conditional types）、映射类型（mapped types）、infer 关键字、模板字面量类型（template literal types）。社区把这类“用类型算类型“的技巧戏称为类型体操（type gymnastics）。学完本章，你将能够：

  • 写出条件类型 T extends U ? X : Y，并理解它和“泛型约束“的区别。
  • 讲清分布式条件类型（distributive conditional types）：联合类型在条件类型里会“分发“，以及如何用 [T] 阻止分发。
  • 用 infer 从数组、函数、Promise、元组里提取类型。
  • 写出映射类型 { [K in keyof T]: ... }，用 +/- 修饰符增删 readonly 和 ?，用 as 做键重映射（key remapping）。
  • 用模板字面量类型拼接字符串，配合内置的 Uppercase / Lowercase / Capitalize / Uncapitalize。
  • 亲手实现内置工具类型：Partial、Required、Readonly、Pick、Omit、Record、ReturnType、Exclude、Extract、NonNullable、Awaited。
  • 写出递归类型：DeepReadonly、DeepPartial。
  • 避开类型体操的常见坑：never 的分发、递归深度限制、以及“别过度炫技“。
  • 做出一个类型级工具库，用自制的 Equal / Expect 在类型层面写单元测试。

本章多为“类型层代码“。我们的做法是：用类型断言 + console.log 演示，用 npx tsc --noEmit 验证类型正确。运行示例：npx tsx ch17/01-conditional.ts；整体类型检查：在 code/ 目录执行 npx tsc --noEmit。

📌 一句话记住本章：泛型是“类型的参数“，本章是“类型的函数体“——条件类型是 if，映射类型是 for...in，infer 是“模式匹配里的变量绑定”，模板字面量是“字符串拼接“。

条件类型：类型世界的 if / else

先从最像“编程“的一个特性开始。条件类型的语法是：

T extends U ? X : Y

读作：“如果 T 能赋值给（是……的子类型）U，就取 X，否则取 Y。“注意这里的 extends 不是“继承”，而是“能不能赋值 / 是不是子类型“的判断。

// 文件：code/ch17/01-conditional.ts

// 最简单的条件类型：判断 T 是不是 string
type IsString<T> = T extends string ? true : false;

type A = IsString<string>; // true
type B = IsString<number>; // false
type C = IsString<"hello">; // true（字面量 "hello" 也是 string 的子类型）

逐段拆解：

  • IsString<T> 是一个泛型类型别名，它的“函数体“是一个条件类型。
  • IsString<string>：string extends string 成立，取 true。
  • IsString<number>：number extends string 不成立，取 false。
  • IsString<"hello">：字面量类型 "hello" 是 string 的子类型，所以 "hello" extends string 成立，取 true。

💡 结果类型 true / false 是布尔字面量类型，不是值 true / false。你可以把 IsString<string> 当成一个“类型层的布尔值“，后面能拿去做进一步的类型判断。

条件类型可以嵌套，就像 if / else if / else：

// 文件：code/ch17/01-conditional.ts（续）
type TypeName<T> = T extends string
  ? "string"
  : T extends number
    ? "number"
    : T extends boolean
      ? "boolean"
      : T extends undefined
        ? "undefined"
        : T extends Function
          ? "function"
          : "object";

type T1 = TypeName<"abc">; // "string"
type T2 = TypeName<42>; // "number"
type T3 = TypeName<true>; // "boolean"
type T4 = TypeName<() => void>; // "function"
type T5 = TypeName<{ x: number }>; // "object"

逐段拆解：这就是一串“三元表达式套三元表达式“。从上往下依次判断：是 string 吗？是 number 吗？……全都不是，就落到最后的 "object"。TypeName<() => void> 走到 T extends Function 这一支得到 "function"。

📌 条件类型 vs 泛型约束——两个都用 extends，但意思完全不同：

  • 约束（第 16 章）：function f<T extends U>(x: T) 里的 extends 是“门槛“，T 不满足 U 就直接编译报错。
  • 条件类型：T extends U ? X : Y 里的 extends 是“分支“，T 不满足 U 不报错，只是走 else 分支取 Y。

看这个“数组就取元素类型、不是数组就原样返回“的例子（infer 下一节详讲，这里先感受条件分支）：

// 文件：code/ch17/01-conditional.ts（续）
type Flatten<T> = T extends Array<infer Item> ? Item : T;

type F1 = Flatten<number[]>; // number
type F2 = Flatten<string>; // string（不是数组，走 else，原样返回）

Flatten<number[]>：number[] 匹配 Array<...>，提取出元素 number。Flatten<string>：string 不是数组，走 else 得到 string 本身。如果这里用的是约束而非条件类型，Flatten<string> 就会直接报错，而不是优雅地返回 string。

运行验证：

npx tsx ch17/01-conditional.ts
# 类型检查通过：IsString<string> 推导为 true，得到 true
# 类型检查通过：IsString<number> 推导为 false，得到 false

文件末尾我们写了 const r1: IsString<string> = true;——如果 IsString<string> 不是 true，这行会编译报错。这是本章反复用到的技巧：用一个精确类型的变量声明，把“类型是否符合预期“变成编译期检查。

分布式条件类型：联合类型会“分发“

这是本章第一个“反直觉但极其重要“的知识点，务必吃透。

📌 规则：当条件类型作用在一个裸类型参数（naked type parameter）上——也就是形如 T extends U ? ...，且 T 是直接的类型参数、没被别的东西包住——如果传进来的 T 是联合类型，那么条件类型会对联合的每个成员分别求值，最后把结果合并成新的联合。这叫“分发（distribute）“。

// 文件：code/ch17/02-distributive.ts
type ToArray<T> = T extends unknown ? T[] : never;

type A = ToArray<string | number>; // string[] | number[]

逐段拆解——这里发生了分发：

ToArray<string | number>
= ToArray<string> | ToArray<number>          // 联合被拆开，逐个代入
= (string extends unknown ? string[] : never)
  | (number extends unknown ? number[] : never)
= string[] | number[]

💡 注意结果是 string[] | number[]（“字符串数组 或 数字数组”），不是 (string | number)[]（“元素可能是字符串或数字的数组”）。这个区别很关键。

如果你想要 (string | number)[] 呢？用一对方括号把类型参数包起来，让它不再是“裸“的，就能阻止分发：

// 文件：code/ch17/02-distributive.ts（续）
type ToArrayNonDist<T> = [T] extends [unknown] ? T[] : never;

type B = ToArrayNonDist<string | number>; // (string | number)[]

逐段拆解：[T] 把 T 包进了一个元组类型，T 不再是“裸类型参数“，于是整个联合作为一个整体参与判断：[string | number] extends [unknown] 成立，取 (string | number)[]。

⚠️ [T] 是关闭分发的标准手法，记住它——很多时候分发帮了大忙，但也有很多时候它是 bug 的来源。

分发的经典应用，是实现 Exclude（从联合里剔除某些成员）：

// 文件：code/ch17/02-distributive.ts（续）
type MyExclude<T, U> = T extends U ? never : T;

type C = MyExclude<"a" | "b" | "c", "a">; // "b" | "c"

逐段拆解——分发让它逐个成员判断“该不该被剔除“：

MyExclude<"a" | "b" | "c", "a">
= ("a" extends "a" ? never : "a")   // 命中，变 never
  | ("b" extends "a" ? never : "b") // 不命中，保留 "b"
  | ("c" extends "a" ? never : "c") // 不命中，保留 "c"
= never | "b" | "c"
= "b" | "c"

💡 最后一步 never | "b" | "c" 化简成了 "b" | "c"。这是因为 never 是联合类型的“零元“——never | X 永远等于 X，就像 0 + x = x。正是这个性质，让“把不想要的成员映射成 never“成为过滤联合的通用套路。

never 的分发陷阱

never 本身有个坑，很多人栽过：

// 文件：code/ch17/02-distributive.ts（续）
type Wrap<T> = T extends unknown ? T[] : never;
type D = Wrap<never>; // never（不是 never[]！）

逐段拆解：直觉会以为 Wrap<never> 得到 never[]，但实际是 never。原因是 never 就是“空联合“——里面一个成员都没有。分发是“对每个成员循环“,既然没有任何成员，循环体一次都不执行，直接返回 never。就像对空数组做 map，得到的还是空数组。

⚠️ 这个坑在写工具类型时经常暗中生效：只要你的条件类型作用在裸参数上，传入 never 就会“短路“成 never。想让 never 也走一次分支，还是用 [T] 阻止分发：

// 文件：code/ch17/02-distributive.ts（续）
type WrapSafe<T> = [T] extends [never] ? "空类型" : "非空";
type E = WrapSafe<never>; // "空类型"
type G = WrapSafe<string>; // "非空"

[never] extends [never] 是把 never 当整体判断，成立，得到 "空类型"。这也是社区检测“某类型是不是 never“的标准写法。

infer：从类型里“提取“出一部分

前面 Flatten 里出现过 infer，现在正式讲。

📌 infer X 只能用在条件类型的 extends 右侧，作用是“声明一个临时类型变量 X，让 TypeScript 在匹配的过程中自动把对应位置的类型填给它“。你可以把它理解成”类型层的模式匹配 + 变量绑定“：写一个“模板”,在想提取的位置放 infer X，匹配成功后 X 就绑定到那个位置的实际类型。

提取数组元素类型

// 文件：code/ch17/03-infer.ts
type ElementType<T> = T extends Array<infer E> ? E : never;

type E1 = ElementType<number[]>; // number
type E2 = ElementType<string[]>; // string
type E3 = ElementType<Array<{ id: number }>>; // { id: number }
type E4 = ElementType<number>; // never（不是数组）

逐段拆解：T extends Array<infer E> 的意思是“如果 T 长得像 Array<某个东西>，就把那个’某个东西’捕获到 E 里“。ElementType<number[]>：number[] 就是 Array<number>，匹配成功，E 绑定到 number。ElementType<number>：number 不是数组，匹配失败，走 else 得到 never。

提取函数返回值类型（手写 ReturnType）

// 文件：code/ch17/03-infer.ts（续）
type MyReturnType<T> = T extends (...args: any[]) => infer R ? R : never;

type R1 = MyReturnType<() => string>; // string
type R2 = MyReturnType<(a: number) => boolean>; // boolean
type R3 = MyReturnType<() => void>; // void

逐段拆解：模板是“一个函数 (...args: any[]) => infer R“，我们不关心参数（用 ...args: any[] 通配），只在返回值位置放 infer R。匹配成功后 R 就是返回值类型。这就是标准库 ReturnType 的核心实现。

提取参数类型

// 文件：code/ch17/03-infer.ts（续）
type FirstArg<T> = T extends (first: infer P, ...rest: any[]) => any ? P : never;
type P1 = FirstArg<(name: string, age: number) => void>; // string

type MyParameters<T> = T extends (...args: infer A) => any ? A : never;
type PA = MyParameters<(a: string, b: number) => void>; // [a: string, b: number]

逐段拆解：FirstArg 把 infer P 放在第一个参数的位置，...rest: any[] 吃掉剩下的，于是只提取到第一个参数类型。MyParameters 把 infer A 放在整个参数列表位置，捕获到的是一个元组类型 [a: string, b: number]（TypeScript 会保留参数名做标签，这是“具名元组“）。

Promise 解包与递归

// 文件：code/ch17/03-infer.ts（续）
type UnwrapPromise<T> = T extends Promise<infer V> ? V : T;
type U1 = UnwrapPromise<Promise<number>>; // number
type U2 = UnwrapPromise<string>; // string（不是 Promise，原样返回）

// 递归解包：多层 Promise 一直剥到底（近似标准库 Awaited）
type MyAwaited<T> = T extends Promise<infer V> ? MyAwaited<V> : T;
type W1 = MyAwaited<Promise<Promise<number>>>; // number
type W2 = MyAwaited<Promise<Promise<Promise<string>>>>; // string

逐段拆解：UnwrapPromise 只剥一层。而 MyAwaited 在提取出内层类型 V 后，又把 V 交给自己——这就是递归类型。MyAwaited<Promise<Promise<number>>>：第一次剥出 Promise<number>，递归；第二次剥出 number，number 不是 Promise，停下来返回 number。这正是标准库 Awaited<T> 的思路。

一个条件类型里用多个 infer

// 文件：code/ch17/03-infer.ts（续）
type HeadTail<T> = T extends [infer H, ...infer Rest] ? [H, Rest] : never;
type HT = HeadTail<[1, 2, 3]>; // [1, [2, 3]]

逐段拆解：模板 [infer H, ...infer Rest] 同时捕获元组的头（H）和尾（Rest，用 ... 收集剩余）。HeadTail<[1, 2, 3]> 得到 [1, [2, 3]]。这种“头 + 尾“拆解是元组类型递归处理的基础套路。

运行验证：

npx tsx ch17/03-infer.ts
# 类型检查通过：MyAwaited 解出的类型是 number，值为 42

文件里用了 const n: MyAwaited<ReturnType<typeof fetchCount>> = await fetchCount();——把类型层的 MyAwaited 和运行时真实的 await 对上，编译通过就说明我们的类型推导和 TS 的运行时语义一致。

映射类型：遍历并改造每个属性

条件类型是 if，映射类型就是 for...in。它的基本形状是：

{ [K in keyof T]: 对 T[K] 做点什么 }

读作：“对 T 的每一个键 K，生成一个新属性。”

// 文件：code/ch17/04-mapped.ts
interface User {
  id: number;
  name: string;
  email: string;
}

// 原样复制，等价于 T 本身
type Identity<T> = { [K in keyof T]: T[K] };
type SameAsUser = Identity<User>; // { id: number; name: string; email: string }

逐段拆解：keyof User 是联合 "id" | "name" | "email"。[K in keyof T] 让 K 依次取这三个键，每次生成属性 K: T[K]（T[K] 是“索引访问类型“，取键 K 对应的值类型）。这里没做任何改造，所以结果和 User 一模一样。真正有用的是在 T[K] 或键上“做手脚“。

修饰符：readonly 和 ? 的增删

// 文件：code/ch17/04-mapped.ts（续）
// 手写 Partial：给每个属性加 ?
type MyPartial<T> = { [K in keyof T]?: T[K] };
type PU = MyPartial<User>; // { id?: number; name?: string; email?: string }

// 手写 Readonly：给每个属性加 readonly
type MyReadonly<T> = { readonly [K in keyof T]: T[K] };
type RU = MyReadonly<User>; // { readonly id: number; ... }

逐段拆解：在 [K in keyof T] 后面加 ?，就给每个属性加上“可选“修饰符；在前面加 readonly，就给每个属性加上“只读“。这就是标准库 Partial 和 Readonly 的真身。

📌 修饰符还能减：+ 是加（默认可省略），- 是减。

// 文件：code/ch17/04-mapped.ts（续）
// 手写 Required：用 -? 去掉可选
type MyRequired<T> = { [K in keyof T]-?: T[K] };
type MaybeUser = { id?: number; name?: string };
type RequiredUser = MyRequired<MaybeUser>; // { id: number; name: string }

// 用 -readonly 去掉只读
type Mutable<T> = { -readonly [K in keyof T]: T[K] };
type MutableUser = Mutable<MyReadonly<User>>; // { id: number; name: string; email: string }

逐段拆解：-? 表示“移除可选修饰符“，把 id? 变回必填的 id——这就是 Required。-readonly 表示“移除只读修饰符“，把 MyReadonly<User> 又变回可写。有了 +/-，你就能在类型层面自由地增删这两个修饰符。

key remapping：用 as 改造键名

📌 TypeScript 4.1 起，映射类型支持在键上用 as 重新命名——这叫键重映射（key remapping）。配合模板字面量类型，能批量改键名。

// 文件：code/ch17/04-mapped.ts（续）
// 生成 getter：id -> getId, name -> getName
type Getters<T> = {
  [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K];
};
type UserGetters = Getters<User>;
// { getId: () => number; getName: () => string; getEmail: () => string }

逐段拆解：[K in keyof T as 新键名] 里的 as 后面是新的键名类型。这里用模板字面量 `get${Capitalize<string & K>}`：Capitalize 把首字母大写，string & K 是因为 K 可能不是纯 string（keyof 理论上还含 number | symbol），用 string & K 收窄到字符串。于是 id 变 getId，值类型变成 () => T[K]（一个返回该属性的函数）。

💡 key remapping 还能过滤键：把某个键映射成 never，它就会被删除。这正是手写 Omit 的关键：

// 文件：code/ch17/04-mapped.ts（续）
// 手写 Omit：删掉键 K
type MyOmit<T, K extends keyof any> = {
  [P in keyof T as P extends K ? never : P]: T[P];
};
type WithoutEmail = MyOmit<User, "email">; // { id: number; name: string }

逐段拆解：[P in keyof T as P extends K ? never : P]——遍历 T 的每个键 P，如果 P 是要删的键（P extends K），就把键名映射成 never（该属性消失）；否则保留原键名 P。MyOmit<User, "email">：遍历到 "email" 时映射成 never 被删掉，其余保留。

运行验证：

npx tsx ch17/04-mapped.ts
# 类型检查通过：getName() = Ada

模板字面量类型：在类型层拼字符串

模板字面量类型让你能像写模板字符串一样，在类型层拼接字符串字面量。

// 文件：code/ch17/05-template-literal.ts
type Greeting = `Hello, ${string}`;
const g1: Greeting = "Hello, world"; // ✅
// const g2: Greeting = "Hi";        // ❌ 不匹配模板

逐段拆解：`Hello, ${string}` 描述“以 Hello, 开头、后面接任意字符串“的类型。"Hello, world" 符合，"Hi" 不符合会报错。

📌 和联合类型结合会分发，做出笛卡尔积：

// 文件：code/ch17/05-template-literal.ts（续）
type Lang = "zh" | "en";
type Kind = "title" | "body";
type Key = `${Lang}_${Kind}`;
// "zh_title" | "zh_body" | "en_title" | "en_body"

逐段拆解：模板里出现联合类型时，TypeScript 会对每一种组合都生成一个字面量——2 个 Lang × 2 个 Kind = 4 个结果。这在生成“所有合法 key“时非常有用。

内置字符串工具类型

TypeScript 内置了四个字符串变换类型：

// 文件：code/ch17/05-template-literal.ts（续）
type U = Uppercase<"hello">; // "HELLO"
type L = Lowercase<"HELLO">; // "hello"
type Cap = Capitalize<"hello">; // "Hello"
type Unc = Uncapitalize<"Hello">; // "hello"

Uppercase / Lowercase 全大写 / 全小写；Capitalize / Uncapitalize 只改首字母。它们常和 key remapping 配合，比如从事件名生成处理器名：

// 文件：code/ch17/05-template-literal.ts（续）
type EventName = "click" | "focus" | "blur";
type Handlers = {
  [E in EventName as `on${Capitalize<E>}`]: () => void;
};
// { onClick: () => void; onFocus: () => void; onBlur: () => void }

逐段拆解：遍历联合 EventName 的每个成员 E，键名映射成 `on${Capitalize<E>}`——click → onClick。这就是 React 里那种“事件名到 props 名“约定的类型化表达。

用 infer 从字符串里提取

模板字面量还能出现在 extends 右侧，配合 infer 从字符串里解析出部分：

// 文件：code/ch17/05-template-literal.ts（续）
type ExtractId<S> = S extends `user/${infer Id}` ? Id : never;
type Id1 = ExtractId<"user/123">; // "123"
type Id2 = ExtractId<"post/1">; // never（前缀不匹配）

逐段拆解：模板 `user/${infer Id}` 里，user/ 是固定前缀，infer Id 捕获后面的部分。"user/123" 匹配，Id = "123"；"post/1" 前缀不对，走 else 得 never。

再看一个递归 + 模板 + infer 的综合例子——把 snake_case 转成 camelCase：

// 文件：code/ch17/05-template-literal.ts（续）
type SnakeToCamel<S extends string> =
  S extends `${infer Head}_${infer Tail}`
    ? `${Head}${Capitalize<SnakeToCamel<Tail>>}`
    : S;
type C1 = SnakeToCamel<"user_name">; // "userName"
type C2 = SnakeToCamel<"first_name_last">; // "firstNameLast"
type C3 = SnakeToCamel<"id">; // "id"（没有下划线，原样返回）

逐段拆解，以 SnakeToCamel<"user_name"> 为例：

  1. "user_name" 匹配 `${infer Head}_${infer Tail}`：Head = "user"，Tail = "name"。
  2. 结果 = `${Head}${Capitalize<SnakeToCamel<Tail>>}` = `user${Capitalize<SnakeToCamel<"name">>}`。
  3. 递归 SnakeToCamel<"name">："name" 里没有 _，走 else 返回 "name"。
  4. Capitalize<"name"> = "Name"，拼起来 = "userName"。

C2 里有两个下划线，会递归两次，把每段的下划线后首字母都大写。C3 没有下划线，第一步就匹配失败，原样返回。

运行验证：

npx tsx ch17/05-template-literal.ts

手写内置工具类型：把原理彻底讲透

第 16 章你学会了用 Partial / Pick / Omit 这些工具类型。现在有了条件类型、映射类型、infer，我们把它们亲手造一遍，你就再也不会觉得它们“神秘“了。

// 文件：code/ch17/06-rebuild-utility-types.ts

// —— 映射类型家族 ——
type MyPartial<T> = { [K in keyof T]?: T[K] };            // 全可选
type MyRequired<T> = { [K in keyof T]-?: T[K] };          // 全必填（-?）
type MyReadonly<T> = { readonly [K in keyof T]: T[K] };   // 全只读
type MyPick<T, K extends keyof T> = { [P in K]: T[P] };   // 只留 K 里的键
type MyRecord<K extends keyof any, T> = { [P in K]: T };  // 以 K 为键，值都是 T

逐段拆解：

  • MyPartial / MyRequired / MyReadonly：前面讲过的修饰符增删，不再赘述。
  • MyPick<T, K extends keyof T>：约束 K extends keyof T 保证只能挑真实存在的键，然后 [P in K] 只遍历 K（而不是 keyof T），所以只保留这些键。
  • MyRecord<K, T>：[P in K] 遍历 K 的每个成员当键，值统一是 T。MyRecord<"a" | "b", number> = { a: number; b: number }。注意 K extends keyof any——keyof any 是 string | number | symbol，即“任何能当键的类型“。
// 文件：code/ch17/06-rebuild-utility-types.ts（续）
// —— 条件类型 / 分发家族 ——
type MyExclude<T, U> = T extends U ? never : T;           // 剔除
type MyExtract<T, U> = T extends U ? T : never;           // 保留
type MyNonNullable<T> = T extends null | undefined ? never : T;

// —— Omit = Pick + Exclude ——
type MyOmit<T, K extends keyof any> = MyPick<T, MyExclude<keyof T, K>>;

逐段拆解：

  • MyExclude 前面讲过——靠分发逐个成员判断，命中的变 never 被吸收。MyExtract 是它的反面：命中的保留，其余变 never。
  • MyNonNullable<T>：把 null | undefined 剔除。也可以直接写成 MyExclude<T, null | undefined>，原理一样。
  • MyOmit 的经典实现是组合：先用 MyExclude<keyof T, K> 从“所有键“里剔除要删的 K，得到“剩下的键“，再 MyPick 出这些键。这比上一节键重映射版更贴近标准库老实现，也展示了工具类型如何互相搭积木。
// 文件：code/ch17/06-rebuild-utility-types.ts（续）
// —— infer 家族 ——
type MyReturnType<T extends (...args: any[]) => any> =
  T extends (...args: any[]) => infer R ? R : never;

type MyParameters<T extends (...args: any[]) => any> =
  T extends (...args: infer P) => any ? P : never;

type MyAwaited<T> = T extends Promise<infer V> ? MyAwaited<V> : T;

逐段拆解：这三个前面 infer 一节都出现过。这里给 MyReturnType / MyParameters 加了约束 T extends (...args: any[]) => any，和标准库一致——这样传入非函数会在调用处就报错，而不是默默返回 never，错误信息更友好。

我们用一个含可选、只读属性的接口验证全部实现：

// 文件：code/ch17/06-rebuild-utility-types.ts（续）
interface User {
  id: number;
  name?: string;
  readonly role: "admin" | "user";
}

type _P = MyPartial<User>;   // 全可选
type _R = MyRequired<User>;  // name 变必填
type _Pick = MyPick<User, "id" | "role">;   // { id; role }
type _Omit = MyOmit<User, "role">;          // { id; name? }
type _Exc = MyExclude<"a" | "b" | "c", "b">; // "a" | "c"
type _NN = MyNonNullable<string | null | undefined>; // string
type _RT = MyReturnType<() => number>;      // number
type _AW = MyAwaited<Promise<Promise<boolean>>>; // boolean

递归类型：DeepReadonly 与 DeepPartial

标准库的 Readonly / Partial 只作用于顶层属性——嵌套对象里面还是可变的。要“深“下去，需要递归的映射类型。

// 文件：code/ch17/type-utils.ts（节选）
type DeepReadonly<T> = T extends (...args: any[]) => any
  ? T // 函数原样返回，别去 readonly 它的属性
  : T extends object
    ? { readonly [K in keyof T]: DeepReadonly<T[K]> }
    : T; // 基本类型到底，原样返回

逐段拆解，这是“三层判断“：

  1. T 是函数吗？是就原样返回（函数也是对象，但我们不想把它的属性变只读，那没意义）。
  2. T 是对象吗？是就映射它的每个属性，加 readonly，并对属性值 T[K] 递归调用 DeepReadonly——这一步是“深“的关键。
  3. 都不是（基本类型如 string / number），到底了，原样返回。

DeepReadonly<{ a: { b: number } }> 的推导：外层是对象 → { readonly a: DeepReadonly<{ b: number }> } → 内层还是对象 → { readonly a: { readonly b: DeepReadonly<number> } } → number 是基本类型 → { readonly a: { readonly b: number } }。每一层都被加上了 readonly。

DeepPartial 同理，只是把 readonly 换成给属性加 ?：

// 文件：code/ch17/type-utils.ts（节选）
type DeepPartial<T> = T extends (...args: any[]) => any
  ? T
  : T extends object
    ? { [K in keyof T]?: DeepPartial<T[K]> }
    : T;

⚠️ 递归深度有上限。TypeScript 对类型递归有大约 50 层的展开限制，太深会报 Type instantiation is excessively deep and possibly infinite。日常对象嵌套远达不到这个深度，但如果你写“对元组逐元素累加计数“之类的极限体操，就可能撞墙。届时的信号很明确：编译器直接报错提示“递归过深“。

常见坑与最佳实践

📌 1. 分清“约束 extends“和“条件 extends“。 同一个 extends，在 <T extends U> 里是“门槛“（不满足报错），在 T extends U ? X : Y 里是“分支“（不满足走 else）。看它在不在三元问号里。

⚠️ 2. 分发是把双刃剑。 裸类型参数 + 联合 = 自动分发。想要“整体判断“就用 [T] extends [U] 包起来。写工具类型时时刻问自己：“这里我要不要分发？”

⚠️ 3. never 会短路分发。 条件类型作用在裸参数上，传入 never 直接返回 never（空联合没有成员可循环）。检测“是不是 never“必须用 [T] extends [never]。

⚠️ 4. boolean 也会分发。 boolean 其实是 true | false 的联合，所以 SomeCond<boolean> 会分别对 true 和 false 求值。偶尔会得到意外的联合结果，记住这一点能省下调试时间。

💡 5. Equal 判等要用“函数兼容“写法，别用双向 extends。 朴素的 A extends B ? B extends A ? true : false : false 在多数情况下够用，但区分不了 any、也区分不了 { a: 1 } 与 { readonly a: 1 }。社区标准的 Equal 用两个泛型函数是否互相兼容来判断，更严格（下面小项目会用到）。

⚠️ 6. 递归有深度上限（约 50 层）。 撞到 excessively deep 报错时，考虑用尾递归优化的写法，或干脆承认“这个不该用类型来算“。

💡 7. 别过度炫技（这是最重要的一条）。 类型体操很好玩，但类型的目的是帮人，不是难为人。一个需要盯着看五分钟才懂的类型，往往不如一个直白的 interface 加几行注释。真实项目里，DeepReadonly、PickByValueType 这种有明确用途的很值得写；而“用类型实现斐波那契数列“只适合练手。能让调用方少犯错、类型提示更准，才是好的类型编程。

本章小项目：类型级工具库

我们把本章所有武器集中起来，做一个 type-utils.ts——一个类型级工具库，并用自制的 Equal / Expect 在类型层面写单元测试。

第一步：断言工具 Equal / Expect

// 文件：code/ch17/type-utils.ts
type Equal<A, B> =
  (<T>() => T extends A ? 1 : 2) extends (<T>() => T extends B ? 1 : 2)
    ? true
    : false;

type Expect<T extends true> = T;
type NotEqual<A, B> = Equal<A, B> extends true ? false : true;

逐段拆解：

  • Equal<A, B>：构造两个泛型函数类型 <T>() => T extends A ? 1 : 2 和 <T>() => T extends B ? 1 : 2，判断前者能不能赋给后者。只有当 A 和 B 完全一致时，这两个函数类型才互相兼容，结果 true。这是 type-challenges 社区公认最可靠的类型判等写法。
  • Expect<T extends true>：约束 T 必须是 true。如果你把一个结果为 false 的 Equal 塞进来，false extends true 不成立，立刻编译报错。它就是我们的“断言“。
  • NotEqual：Equal 取反，用来表达“应当不相等“。

📌 用法：写 type _ = Expect<Equal<实际, 期望>>。编译通过 = 断言成立；编译报错 = 断言失败。这就是在类型层面写单元测试。

第二步：DeepReadonly / DeepPartial / PickByValueType

DeepReadonly / DeepPartial 上一节讲过了。新增 PickByValueType——挑出“值类型匹配“的属性：

// 文件：code/ch17/type-utils.ts
type PickByValueType<T, V> = {
  [K in keyof T as T[K] extends V ? K : never]: T[K];
};

逐段拆解：又是“键重映射做过滤“。遍历每个键 K，如果它的值类型 T[K] 能赋给 V（T[K] extends V），就保留键名 K；否则映射成 never 删掉。PickByValueType<Mixed, number> 只保留值是 number 的属性。

第三步：CamelCaseKeys / PathKeys / Paths

// 文件：code/ch17/type-utils.ts
type SnakeToCamel<S extends string> =
  S extends `${infer Head}_${infer Tail}`
    ? `${Head}${Capitalize<SnakeToCamel<Tail>>}`
    : S;

type CamelCaseKeys<T> = {
  [K in keyof T as SnakeToCamel<string & K>]: T[K];
};

type PathKeys<T> = keyof T & string; // 顶层键（string 化）

逐段拆解：SnakeToCamel 是前面那个字符串转换器。CamelCaseKeys 用 key remapping，把每个键名交给 SnakeToCamel 改写（string & K 把键收窄到字符串），值类型不变。PathKeys 简单地取顶层键并交给 string 过滤掉 number | symbol。

重头戏是 Paths——递归生成所有嵌套路径 "a" | "a.b" | "a.b.c"：

// 文件：code/ch17/type-utils.ts
type IsRecord<T> = T extends readonly any[]
  ? false
  : T extends (...args: any[]) => any
    ? false
    : T extends object
      ? true
      : false;

type Paths<T> = IsRecord<T> extends true
  ? {
      [K in keyof T & string]: IsRecord<T[K]> extends true
        ? K | `${K}.${Paths<T[K]>}`
        : K;
    }[keyof T & string]
  : never;

逐段拆解：

  • ⚠️ IsRecord 是关键的“护栏“。 数组和函数在 TS 里都是 object，如果直接对它们递归，会冒出 "tags.length"、"tags.0"、数组方法名等一堆噪声。所以我们先判断：数组 → false，函数 → false，其余对象 → true，基本类型 → false。只有“真正的记录型对象“才往里钻。
  • Paths<T>：如果 T 是记录型对象，就遍历它的每个键 K——若 T[K] 还是记录型对象，产出 K（当前这层的路径）和 `${K}.${Paths<T[K]>}`（拼上子路径）；否则只产出 K。
  • 结尾的 [keyof T & string]：这是索引访问。前面 { [K in ...]: ... } 得到的是一个“键 → 路径联合“的映射对象，用 [keyof T & string] 把所有值（各个路径联合）取出来合并成一个大联合。这是“把映射对象压平成联合“的经典技巧。

Paths<{ id: number; profile: { name: string; address: { city: string } } }> 会产出 "id" | "profile" | "profile.name" | "profile.address" | "profile.address.city"。

第四步：类型层单元测试

// 文件：code/ch17/type-utils.ts
type _tests = [
  Expect<Equal<
    DeepReadonly<{ a: { b: number } }>,
    { readonly a: { readonly b: number } }
  >>,
  Expect<Equal<
    DeepPartial<{ a: { b: number } }>,
    { a?: { b?: number } }
  >>,
  Expect<Equal<PickByValueType<Mixed, number>, { id: number; score: number }>>,
  Expect<Equal<PickByValueType<Mixed, boolean>, { active: boolean }>>,
  Expect<Equal<SnakeToCamel<"user_name">, "userName">>,
  Expect<Equal<
    CamelCaseKeys<SnakeShape>,
    { userName: string; createdAt: number; isAdmin: boolean }
  >>,
  Expect<Equal<PathKeys<Nested>, "id" | "profile" | "tags">>,
  Expect<Equal<
    Paths<Nested>,
    | "id" | "tags" | "profile"
    | "profile.name" | "profile.address" | "profile.address.city"
  >>,
  Expect<NotEqual<string, number>>,
];

逐段拆解：_tests 是一个元组类型，每个元素都是一个 Expect<Equal<...>>。只要任何一个 Equal 结果是 false，对应的 Expect 就会报错，整个文件编译失败。编译通过，就等于这一整套断言全部成立。 注意最后 Paths<Nested> 的期望里没有 "tags.length" 之类——因为 tags 是数组，被 IsRecord 挡在了外面。

扩展练习（加分项）

  1. DeepRequired<T>：仿照 DeepPartial，递归地去掉所有可选（-?），让嵌套里所有属性都必填。
  2. OmitByValueType<T, V>：PickByValueType 的反面——删掉值类型匹配 V 的属性。提示：把 T[K] extends V ? never : K 反过来。
  3. CamelToSnake<S>：SnakeToCamel 的逆运算，把 userName 转回 user_name。提示：用模板匹配大写字母，需要配合联合 "A" | "B" | ... | "Z" 或递归逐字符处理。
  4. Get<T, P>：给一个对象类型 T 和一条路径字符串 P（如 "profile.address.city"），返回该路径处的类型。提示：用 P extends `${infer Head}.${infer Rest}` 递归下钻。
  5. 给 Paths 加数组下标支持：让数组路径产出 "list.0"、"list.1" 这种（可以用 number 索引）。想清楚要不要这么做——很多时候不做才是对的。

小结与自测

本章我们进入了 TypeScript 类型系统最强大、也最像“编程“的一块——类型层计算：

  • 条件类型 T extends U ? X : Y：类型世界的 if / else，可嵌套。这里的 extends 是“分支“（不满足走 else），不同于泛型约束的“门槛“。
  • 分布式条件类型：裸类型参数 + 联合 = 自动分发，对每个成员分别求值再合并。用 [T] extends [U] 阻止分发。never（空联合）会短路成 never。
  • infer：条件类型 extends 右侧的“模式匹配变量绑定“，用来从数组、函数返回值、参数、Promise、元组里提取类型。配合递归能解开多层 Promise、拆解元组。
  • 映射类型 { [K in keyof T]: ... }：类型世界的 for…in。+/- 增删 readonly 和 ?；as 做键重映射（改名 / 用 never 过滤）。
  • 模板字面量类型：类型层拼字符串，和联合结合做笛卡尔积；内置 Uppercase / Lowercase / Capitalize / Uncapitalize；配合 infer 能从字符串解析。
  • 手写工具类型：Partial / Required / Readonly / Pick / Omit / Record / Exclude / Extract / NonNullable / ReturnType / Parameters / Awaited 全部亲手实现了一遍——它们不再神秘。
  • 递归类型：DeepReadonly / DeepPartial 深入每一层；注意约 50 层的递归深度上限。
  • 小项目：type-utils.ts 用自制的 Equal / Expect 在类型层面写单元测试，做出 DeepReadonly、PickByValueType、CamelCaseKeys、Paths 等实用工具。
  • 别过度炫技：类型是帮人的，不是难为人的。

类型体操让我们能在编译期表达极其精细的约束。但类型系统还有另一面同样重要：在运行时，一个值的类型往往是“联合“或“宽泛“的（比如 string | number、unknown、A | B | C），我们需要在代码里逐步缩小它的范围，才能安全地使用。

第 18 章 · 类型收窄、类型守卫与断言将带你掌握这套“从宽到窄“的功夫：typeof / instanceof / in 收窄、真值收窄、可辨识联合（discriminated unions）、用户自定义类型守卫（type guard） x is T、断言函数 asserts x is T，以及 as 断言和 ! 非空断言的正确用法与滥用陷阱。类型体操负责“设计精密的类型“，类型收窄负责“在运行时安全地落地它们“——两者合起来，才是完整的 TypeScript 类型思维。
