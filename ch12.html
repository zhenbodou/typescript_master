JavaScript & TypeScript 从入门到精通

第 12 章 · 正则表达式与文本处理

本章目标

前面几章我们一直在跟数据打交道：数组、对象、文件、网络请求。可有一类数据格外顽固——文本（text）。校验一个邮箱格式对不对、从一段日志里揪出所有 IP、把 2026-07-02 拆成年月日、把连续的空格压成一个……这些活儿如果全靠 indexOf、slice、split 一点点手搓，代码会又长又脆。

正则表达式（regular expression，简称 regex 或 regexp）就是专门对付文本的“模式语言“。你写一小段“模式“，它就能在字符串里帮你查找、校验、提取、替换。它非常强大，但对新手确实不友好——满屏的 \d+、(?<=...) 看着像天书。所以本章我会从零开始、一个符号一个符号地讲，每讲一点就配一个能跑的小例子。

学完本章，你将能够：

  • 说清楚正则是什么、它擅长解决什么问题、又不该拿它干什么。
  • 用字面量和 new RegExp 两种方式创建正则，理解 g i m s u y 这些 flags（标志）。
  • 掌握元字符（. \d \w \s 等）、字符组 [...]、量词 * + ? {n,m}、贪婪与非贪婪。
  • 掌握锚点 ^ $ 和单词边界 \b。
  • 掌握分组与捕获：(...)、非捕获 (?:...)、命名捕获 (?<name>...)、反向引用、|。
  • 认识前瞻/后顾（lookahead/lookbehind）。
  • 熟练在 JS 里用 test、match、matchAll、replace、split、exec，并绕开 g 标志的 lastIndex 陷阱。
  • 写出几个常用实用正则（邮箱、URL、手机号等），并知道它们的边界。
  • 亲手做一个 Markdown → HTML 迷你解析器。

本章示例仍用 Node.js 运行。别急着背符号，跟着例子敲一遍，感觉自然就来了。

正则表达式是什么

先看一个“不用正则“的场景。假设要判断一个字符串是不是“纯数字“：

function isAllDigits(str) {
  for (const ch of str) {
    if (ch < "0" || ch > "9") return false;
  }
  return str.length > 0;
}

能用，但啰嗦。换成正则：

const isAllDigits = (str) => /^\d+$/.test(str);

一行搞定。/^\d+$/ 读作：“从头（^）到尾（$），全是数字（\d），至少一个（+）”。

📌 正则的本质：用一串符号描述“一类字符串长什么样“，也就是一个模式（pattern）。引擎拿这个模式去一段文本里“扫描“，找出符合模式的片段。它最擅长四件事：校验（这段文本合法吗）、查找（有没有 / 在哪）、提取（把符合的部分抠出来）、替换（把符合的部分换掉）。

⚠️ 正则也有它不擅长的：处理有嵌套结构的东西（HTML、JSON、编程语言源码）。这些应该交给专门的解析器（parser）。本章末尾会反复强调这一点。

创建正则：字面量 vs 构造函数

JS 里有两种方式造一个正则对象（RegExp）。

// 文件：code/ch12/01-basics.js

// 1) 字面量方式：两个斜杠之间写模式，斜杠后面写 flags。
const re1 = /hello/i; // i = 忽略大小写（ignore case）
console.log("1) test:", re1.test("Hello, world")); // true

// 2) 构造函数方式：模式是字符串，适合"模式在运行时才拼出来"的场景。
const word = "hello";
const re2 = new RegExp(word, "i");
console.log("2) test:", re2.test("HELLO")); // true

逐段看：

  • 字面量 /hello/i：两个斜杠 / / 之间是模式，第二个斜杠后面是 flags。这是最常用的写法，简洁，编辑器还能高亮。当你写模式时就已经确定内容，就用它。
  • 构造函数 new RegExp("hello", "i")：模式是一个字符串，flags 是第二个字符串参数。当模式要在运行时动态拼出来（比如从用户输入或变量组装）才用它。
// 文件：code/ch12/01-basics.js

// ⚠️ 用字符串写模式时，反斜杠要写两个：JS 字符串先吃掉一层。
const digitLiteral = /\d+/; // 字面量里一个反斜杠就够
const digitCtor = new RegExp("\\d+"); // 字符串里必须写两个
console.log("3) 字面量匹配:", digitLiteral.test("abc123")); // true
console.log("4) 构造匹配:", digitCtor.test("abc123")); // true

⚠️ 反斜杠陷阱：在普通字符串里，\ 本身是转义符（比如 \n 是换行）。所以 "\d" 会先被 JS 字符串解释一次，\d 并不是合法转义，结果可能不是你想的。用 new RegExp 时，正则里的每个 \ 都要写成 \\。这也是“能用字面量就用字面量“的一个理由。

flags：一串小写字母，改变匹配行为

flags 写在字面量的第二个斜杠之后，或作为 RegExp 的第二个参数。常用的有 6 个：

flag名字作用
gglobal全局匹配，找出所有，而不是找到第一个就停
iignore case忽略大小写
mmultiline让 ^ $ 匹配每一行的开头/结尾
sdotAll让 . 也能匹配换行符
uunicode正确处理 Unicode（如 emoji、\u{...}）
ysticky粘性匹配，必须从 lastIndex 那个位置开始匹配

可以组合，比如 /foo/gi 就是“全局 + 忽略大小写“。这些 flag 后面用到时会再演示，现在有个印象即可。

先认识 test 和 exec

RegExp 对象有两个自带方法，先见个面：

// 文件：code/ch12/01-basics.js

// test：返回 true/false，"有没有匹配到"。
// exec：返回一个"匹配结果数组"，没匹配到返回 null。
const m = /(\d+)-(\d+)/.exec("范围 12-34 结束");
console.log("5) exec:", m[0], m[1], m[2], "index=", m.index);
// m[0] 是整段匹配 "12-34"，m[1]/m[2] 是两个分组
  • test(str)：只回答“有没有“，返回布尔值。校验场景最常用。
  • exec(str)：找到一次匹配，返回一个数组：m[0] 是整段匹配到的字符串，m[1]、m[2]……是各个分组（后面讲），m.index 是匹配开始的位置。没找到就返回 null。

运行 node code/ch12/01-basics.js 看看结果。下面我们正式拆解模式里的符号。

元字符：用一个符号代表“一类字符“

正则里，大部分字符就是它字面的意思——a 匹配字母 a，5 匹配数字 5。但有些字符有“魔力“，叫元字符（metacharacter）。最有用的是这几个“字符类简写“：

// 文件：code/ch12/02-metachars.js

const s = "价格 A1_ b2? 空 格";

// \d 数字，\D 非数字
console.log("\\d 找数字:", s.match(/\d/g)); // ['1','2']
// \w 单词字符（字母、数字、下划线），\W 取反
console.log("\\w 单词字符:", "a_1 中!".match(/\w/g)); // ['a','_','1']  中文和 ! 不算
// \s 空白（空格、制表符、换行），\S 非空白
console.log("\\s 空白数量:", ("a b\tc\nd".match(/\s/g) || []).length); // 3

逐个记：

  • \d = digit，一个数字 0-9；大写 \D 取反，一个非数字。
  • \w = word，一个单词字符：字母、数字、下划线（[A-Za-z0-9_]）；大写 \W 取反。⚠️ 注意：\w 不包含中文，中文的 中 不算单词字符。
  • \s = space，一个空白字符：空格、制表符 \t、换行 \n 等；大写 \S 取反。

📌 规律：小写是“某一类“，对应的大写就是“这类之外的所有“。

点 .：几乎任意一个字符

// 文件：code/ch12/02-metachars.js

// . 匹配"除换行外任意一个字符"
console.log(".  任意字符:", "cat cut cot".match(/c.t/g)); // ['cat','cut','cot']

. 匹配除换行符外的任意一个字符。c.t 就是“c、任意一个字符、t“。⚠️ 想让 . 连换行也匹配，加 s（dotAll）flag。

字符组 [...]：自己定义“一类字符“

// 文件：code/ch12/02-metachars.js

// [...] 字符组：括号内任选一个
console.log("[aeiou] 元音:", "regex".match(/[aeiou]/g)); // ['e','e']
// 范围：a-z、0-9
console.log("[a-f0-3]:", "a3g9c".match(/[a-f0-3]/g)); // ['a','3','c']
// [^...] 取反：不是这些字符
console.log("[^0-9] 非数字:", "a1b2".match(/[^0-9]/g)); // ['a','b']
  • [aeiou]：方括号里的任意一个字符（这里是任一元音）。
  • [a-f0-3]：用连字符 - 表示范围，等于 [abcdef0123]。
  • [^...]：方括号里开头加 ^ 表示取反——“不是这些字符里的任何一个”。比如 [^0-9] 就是“非数字“。

⚠️ 组内元字符会失去魔力：

// 文件：code/ch12/02-metachars.js

// 在字符组里，大多数元字符会"失去魔力"变成普通字符。
console.log("组内的 . + *:", "1.2+3*4".match(/[.+*]/g)); // ['.','+','*']

在 [...] 里面，.、+、* 这些平时有特殊含义的符号，大多变回它们字面本身。所以 [.+*] 匹配的就是真正的点、加号、星号。这是个好消息：想匹配这些符号本身时，放进字符组是最省心的写法之一。

量词：控制“重复多少次“

前面 \d 只匹配一个数字。想匹配“一串数字“就得用量词（quantifier），写在某个模式后面，表示它要重复几次。

// 文件：code/ch12/03-quantifiers.js

// * 零或多，+ 一或多，? 零或一
console.log("ab* :", "a ab abbb".match(/ab*/g)); // ['a','ab','abbb']
console.log("ab+ :", "a ab abbb".match(/ab+/g)); // ['ab','abbb']（单独的 a 不匹配）
console.log("colou?r:", "color colour".match(/colou?r/g)); // ['color','colour']

三个基础量词：

  • *：零次或多次。ab* = 一个 a，后面跟零个或多个 b。所以单独的 a 也算。
  • +：一次或多次。ab+ 要求至少一个 b，所以单独的 a 不匹配。
  • ?：零次或一次（可有可无）。colou?r 里 u 可有可无，于是 color 和 colour 都匹配。

⚠️ 注意量词只作用于紧挨着它前面的那一个东西（一个字符、一个字符组、或一个分组）。ab* 里 * 只管 b，不管 a。

精确次数：{n} {n,} {n,m}

// 文件：code/ch12/03-quantifiers.js

// {n} 恰好 n 次，{n,} 至少 n 次，{n,m} n 到 m 次
console.log("\\d{4}:", "年份 2026 和 12".match(/\d{4}/g)); // ['2026']
console.log("\\d{2,3}:", "1 22 333 4444".match(/\d{2,3}/g)); // ['22','333','444']
  • {n}：恰好 n 次。\d{4} = 恰好 4 个数字。上例里 12 只有两位，被排除。
  • {n,}：至少 n 次（上不封顶）。
  • {n,m}：n 到 m 次。\d{2,3} 匹配 2~3 位数字。注意 4444 会先吃掉 444（尽量多），剩下的单个 4 不足 2 位就被丢弃了。

贪婪 vs 非贪婪：吃多还是吃少

这是新手最容易栽的坑。默认情况下，量词是**贪婪（greedy）**的——尽可能多吃字符。

// 文件：code/ch12/03-quantifiers.js

const html = "<b>粗</b><i>斜</i>";
// 贪婪：默认尽可能多吃
console.log("贪婪 <.+>:", html.match(/<.+>/)[0]); // 整个字符串！
// 非贪婪：量词后加 ?，尽可能少吃
console.log("非贪婪 <.+?>:", html.match(/<.+?>/g)); // ['<b>','</b>','<i>','</i>']

拆解：

  • <.+> 想匹配“一个标签“，但 .+ 贪婪，它从第一个 < 一路吃到最后一个 >，结果把整个字符串当成一个匹配。
  • <.+?> 在 + 后面加了个 ?，把它变成非贪婪（lazy / non-greedy）——尽可能少吃。于是每遇到最近的 > 就停下，正确地一个个抠出标签。

📌 记忆法：量词后加 ? = “见好就收”。凡是用 .+ 或 .* 去“框住一段内容“时，几乎都要加 ? 改成非贪婪，否则很容易一口吞太多。

锚点与边界：匹配“位置“而不是字符

有些符号不匹配任何字符，而是匹配一个位置。

// 文件：code/ch12/04-anchors.js

// ^ 匹配开头，$ 匹配结尾
console.log("^\\d:", /^\d/.test("1abc"), /^\d/.test("abc1")); // true false
console.log("\\d$:", /\d$/.test("abc1"), /\d$/.test("1abc")); // true false

// 完整匹配一个整数：从头到尾都必须是数字
const isInt = /^\d+$/;
console.log("整数校验:", isInt.test("12345"), isInt.test("12a45")); // true false
  • ^：字符串开头。^\d = “开头是一个数字”。
  • $：字符串结尾。\d$ = “结尾是一个数字”。
  • ^...$ 一起用，表示“整个字符串都要符合中间的模式“，这是做校验的标准套路。^\d+$ 就是“从头到尾全是数字“。

⚠️ 别混淆两个 ^：在字符组开头 [^...] 是“取反“；在模式开头 ^ 是“字符串开头“。位置不同，含义不同。

单词边界 \b

// 文件：code/ch12/04-anchors.js

// \b 单词边界：单词字符与非单词字符之间的"缝隙"
console.log("\\bcat\\b:", "cat category scatter".match(/\bcat\b/g)); // ['cat']

\b 匹配一个单词边界（word boundary）——就是单词字符（\w）和非单词字符之间的那道“缝“。\bcat\b 要求 cat 两侧都是边界，所以只匹配独立的单词 cat，而 category（右边还是字母）和 scatter（cat 嵌在中间）都不算。做“整词搜索/替换“时特别有用。

m 标志：多行模式

// 文件：code/ch12/04-anchors.js

const text = "第一行\n第二行\n第三行";
console.log("不带 m:", text.match(/^第.行/g)); // ['第一行']
console.log("带 m:", text.match(/^第.行/gm)); // ['第一行','第二行','第三行']

默认 ^ $ 只认整段文本的开头和结尾。加上 m（multiline）后，它们会匹配每一行的开头和结尾。处理多行文本（日志、配置文件）时常用。

分组与捕获：把关心的部分抠出来

用一对圆括号 (...) 把一段模式括起来，就成了一个分组（group）。分组有两个作用：让量词作用于一整段，以及把匹配到的内容单独“捕获“下来。

// 文件：code/ch12/05-groups.js

// 捕获分组：括号里的内容会被单独"抓"出来
const date = "2026-07-02";
const m1 = date.match(/(\d{4})-(\d{2})-(\d{2})/);
console.log("捕获:", m1[1], m1[2], m1[3]); // 2026 07 02

(\d{4})-(\d{2})-(\d{2}) 有三个分组。匹配后，m1[0] 是整段 2026-07-02，m1[1]/m1[2]/m1[3] 分别是年、月、日。这就是捕获（capture）——把一次匹配里的“部件“分别取出来。

| 或

// 文件：code/ch12/05-groups.js

// | 或：匹配多个备选之一。用分组限定"或"的范围。
console.log("cat|dog:", "I have a cat and a dog".match(/cat|dog/g)); // ['cat','dog']
console.log("分组限定或:", "grey gray".match(/gr(a|e)y/g)); // ['grey','gray']

| 是“或“，cat|dog 匹配 cat 或 dog。⚠️ | 的“势力范围“很大，会一路延伸到两端或最近的括号边界。想只在局部选择，就用分组框起来：gr(a|e)y 表示 gr 后面是 a 或 e，再接 y。

非捕获分组 (?:...)

有时你只想用括号来“分组“（配合量词或 |），并不需要把内容捕获下来。这时用 (?:...)：

// 文件：code/ch12/05-groups.js

// 非捕获分组 (?:...)：只想分组、不想留捕获结果时用
const m2 = "grey".match(/gr(?:a|e)y/);
console.log("非捕获:", m2[0], "没有 m2[1]:", m2[1]); // grey undefined

(?:a|e) 只做分组，不占用捕获编号，所以 m2[1] 是 undefined。💡 好处：捕获编号更清爽，性能也略好。不需要取出内容时，优先用非捕获分组。

命名捕获 (?<name>...)

用数字编号取分组，多了容易数错。可以给分组起名字：

// 文件：code/ch12/05-groups.js

// 命名捕获 (?<name>...)：给分组起名字，通过 groups 访问
const m3 = date.match(/(?<year>\d{4})-(?<month>\d{2})-(?<day>\d{2})/);
console.log("命名捕获:", m3.groups.year, m3.groups.month, m3.groups.day);

(?<year>...) 给分组起名叫 year，之后通过 m3.groups.year 访问。💡 分组一多，命名捕获比 m[3]、m[7] 可读得多。

反向引用 \1

分组还能在同一个模式内被“回引“：\1 表示“这里要重复第 1 个分组刚匹配到的内容“。

// 文件：code/ch12/05-groups.js

// 反向引用 \1：要求后面重复前面第 1 个分组匹配到的内容——可用来找叠词
console.log("叠词 (\\w)\\1:", "book cool tree hi".match(/(\w)\1/g)); // ['oo','oo','ee']

(\w)\1 的意思是：“一个单词字符，紧接着同一个字符”。于是它匹配 book 的 oo、cool 的 oo、tree 的 ee——都是“某个字符连着出现两次“。注意它要求的是“内容相同“，不是“再来一个任意字符“。

前瞻与后顾（lookaround）

有时你想说：“匹配 X，但前提是它左边/右边是（或不是）某个东西，而这个东西本身不算进匹配结果”。这就是环视（lookaround），分为前瞻和后顾。它们是“条件判断“，不消耗字符。

// 文件：code/ch12/06-lookaround.js

// 正向前瞻 (?=...)：要求右边"跟着"某模式，但不把它算进匹配
console.log("前瞻:", "10px 20em 30px".match(/\d+(?=px)/g)); // ['10','30']

// 负向前瞻 (?!...)：要求右边"不跟着"某模式
console.log("负前瞻:", "10px 20em 30px".match(/\d+(?!px|\d)/g)); // ['20']

// 正向后顾 (?<=...)：要求左边"前面是"某模式
console.log("后顾:", "$100 ￥200 $300".match(/(?<=\$)\d+/g)); // ['100','300']

// 负向后顾 (?<!...)：要求左边"前面不是"某模式
console.log("负后顾:", "$100 ￥200".match(/(?<!\$)\b\d+/g)); // ['200']

四种写法：

  • (?=...) 正向前瞻：右边必须跟着某模式。\d+(?=px) 匹配“后面跟着 px 的数字“，但匹配结果里不含 px。
  • (?!...) 负向前瞻：右边必须不跟着某模式。
  • (?<=...) 正向后顾：左边必须是某模式。(?<=\$)\d+ 匹配“前面是 $ 的数字“，结果不含 $。
  • (?<!...) 负向后顾：左边必须不是某模式。

一个经典应用——给数字加千分位逗号：

// 文件：code/ch12/06-lookaround.js

const n = "1234567";
const withCommas = n.replace(/\B(?=(\d{3})+(?!\d))/g, ",");
console.log("千分位:", withCommas); // 1,234,567

这个模式在“每往右数正好是 3 的倍数个数字、且后面没有更多数字“的位置插逗号。看不懂没关系——它就是想说明环视能表达相当精巧的条件。💡 环视是进阶特性，初学阶段能看懂 (?=...) 和 (?<=...) 就够了。

在 JavaScript 里使用正则

模式会写了，现在看怎么把它用在字符串上。核心是 RegExp 的 test/exec 和 String 的 match/matchAll/replace/split。

// 文件：code/ch12/07-methods.js

const s = "订单 A-12, B-34, C-56";

// str.match：不带 g 返回带分组的详细结果；带 g 只返回所有整段匹配的数组
console.log("match 无 g:", s.match(/([A-Z])-(\d+)/)); // 含分组、index
console.log("match 有 g:", s.match(/[A-Z]-\d+/g)); // ['A-12','B-34','C-56']

⚠️ match 的行为取决于有没有 g：

  • 不带 g：像 exec 一样，返回第一次匹配的详细数组（含分组、index）。
  • 带 g：返回所有整段匹配组成的数组，但丢掉分组信息（没有 m[1]）。

想“既要全部匹配、又要分组“，用 matchAll：

// 文件：code/ch12/07-methods.js

// str.matchAll：返回迭代器，每项都带分组，最适合"全部匹配且要分组"
for (const m of s.matchAll(/([A-Z])-(\d+)/g)) {
  console.log("matchAll:", m[1], "=>", m[2]);
}

matchAll 返回一个迭代器（iterator），用 for...of 遍历，每一项都是完整的匹配结果（带分组）。⚠️ matchAll 的正则必须带 g，否则报错。这是现代代码里遍历全部匹配的首选。

replace：查找并替换

// 文件：code/ch12/07-methods.js

// str.replace：$1 引用分组，$<name> 引用命名分组
console.log("replace $:", s.replace(/([A-Z])-(\d+)/g, "$1$2"));
// 函数替换：拿到每次匹配，动态计算替换内容
console.log(
  "replace 函数:",
  s.replace(/(\d+)/g, (whole) => Number(whole) * 2)
); // 数字翻倍

replace 有两种“替换值“：

  • 字符串：里面可用 $1、$2 引用分组，$<name> 引用命名分组，$& 代表整段匹配。上例 "$1$2" 把 A-12 换成 A12（去掉中间的横杠）。
  • 函数：每匹配一次就调用一次，参数依次是“整段匹配、各分组、位置、原串“，返回值作为替换内容。上例把每个数字翻倍。函数替换非常灵活，能做字符串模板做不到的动态计算。

⚠️ 想替换所有匹配，正则要带 g（或用更新的 replaceAll）。不带 g 只换第一处。

split：用正则切分

// 文件：code/ch12/07-methods.js

// str.split：用正则切分
console.log("split:", "a1b22c333d".split(/\d+/)); // ['a','b','c','d']

split 的分隔符可以是正则。split(/\d+/) 表示“按一段或多段数字切开“，很适合分隔符不固定的场景。

exec + lastIndex：g 标志的坑

这是本章最重要的陷阱，务必看懂。带 g 的正则对象是有状态的：它会记住一个 lastIndex 属性，下次匹配从这个位置继续。

// 文件：code/ch12/07-methods.js

// exec + lastIndex：带 g 的正则每次从 lastIndex 继续，配合 while 遍历全部匹配。
const re = /[A-Z]-\d+/g;
let match;
while ((match = re.exec(s)) !== null) {
  console.log("exec:", match[0], "下次从", re.lastIndex, "开始");
}

这段能正确遍历所有匹配：每次 exec 找到下一个，并把 lastIndex 前移；找不到时返回 null，循环结束。这是“逐个处理匹配“的经典写法。

但同一个“有状态“的特性，如果误用，就会咬人：

// 文件：code/ch12/07-methods.js

// ⚠️ 陷阱：把带 g 的正则放进循环反复 test，lastIndex 会乱跳。
const bad = /\d/g;
console.log("坑演示:", bad.test("1"), bad.test("1"), bad.test("1"));
// true false true —— 状态残留导致结果不稳定！

对同一个带 g 的正则连续调 test（或 exec），lastIndex 会一直变，导致对相同输入给出不一样的结果。

📌 避坑口诀：

  1. 做校验（test）的正则，别加 g。
  2. 要复用带 g 的正则做全局操作时，优先用 matchAll / replace，它们不依赖你手动管理状态。
  3. 用 exec 循环时，正则对象声明在循环外，让 lastIndex 自然推进；别每次循环都新建。

常见实用正则

下面是几个日常够用的正则。⚠️ 先说重点：邮箱、URL 的“完全正确“规范极其复杂，这里的目标是“表单校验够用“，不是覆盖所有合法情况。

// 文件：code/ch12/08-practical.js

// 邮箱（简化版）：用户名@域名.后缀
const emailRe = /^[\w.+-]+@[\w-]+(\.[\w-]+)+$/;
console.log("邮箱:", emailRe.test("a.b+tag@mail.example.com"), emailRe.test("bad@@x"));

// URL（简化版）：http(s):// 开头
const urlRe = /^https?:\/\/[\w.-]+(:\d+)?(\/[^\s]*)?$/i;
console.log("URL:", urlRe.test("https://example.com/path?q=1"), urlRe.test("ftp://x"));

// 中国大陆手机号：1 开头，第二位 3-9，共 11 位
const phoneRe = /^1[3-9]\d{9}$/;
console.log("手机号:", phoneRe.test("13812345678"), phoneRe.test("12345"));
  • 邮箱 ^[\w.+-]+@[\w-]+(\.[\w-]+)+$：@ 前是用户名（允许字母数字和 . + -），@ 后是“域名段（点 段）+“，保证至少有一个点分后缀。
  • URL ^https?:\/\/...：https? 里 ? 让 s 可有可无（http 或 https）；\/\/ 是转义的双斜杠；后面是主机、可选端口、可选路径。
  • 手机号 ^1[3-9]\d{9}$：1 开头，第二位 3~9，再加 9 位数字，共 11 位。

两个常用文本清洗：

// 文件：code/ch12/08-practical.js

// 去除多余空白：把连续空白压成一个空格，并去掉首尾
const messy = "  多个   空格   和\t制表符  ";
console.log("去空白:", JSON.stringify(messy.replace(/\s+/g, " ").trim()));

// 提取所有数字（含小数）
const text = "苹果 3.5 元，香蕉 12 元，一共 15.5";
console.log("提取数字:", text.match(/\d+(\.\d+)?/g)); // ['3.5','12','15.5']
  • 去多余空白：replace(/\s+/g, " ") 把任意连续空白压成一个空格，再 trim() 去首尾。
  • 提取数字：\d+(\.\d+)? 匹配整数或小数（小数部分可选）。

💡 别过度依赖正则：3.5 好提取，可一旦遇到千分位 1,234.5、科学计数法 1e3、负号，正则会越写越长越易错。结构复杂时，先分割成小块，再用 Number()/parseFloat 转换，往往比一个巨型正则清晰得多。

常见坑与最佳实践

  • ⚠️ . 不匹配换行：想跨行匹配，加 s（dotAll）flag。
  • ⚠️ 贪婪吞太多：凡是 .+ / .* 想“框住一段“，几乎都要改成非贪婪 .+? / .*?。
  • ⚠️ g 的 lastIndex 状态：校验用的正则不要加 g；遍历全部匹配优先 matchAll。别对同一个带 g 的正则反复 test。
  • ⚠️ 在字符串里写模式要双反斜杠：new RegExp("\\d")。能用字面量就用字面量。
  • ⚠️ 忘了转义元字符：想匹配真正的点 .、问号 ?、括号 (，要写成 \.、\?、\(，或放进字符组 [.?()]。
  • 💡 用命名捕获和非捕获分组提高可读性：需要取值就 (?<name>...)，只分组就 (?:...)。
  • 💡 复杂正则加注释和测试：正则是“写起来一时爽、半年后看不懂“的重灾区。旁边写一行注释说明它匹配什么，并留几个测试用例。
  • 💡 别用正则解析嵌套结构：HTML、JSON、编程语言……交给专门的解析库。正则适合“局部、扁平、规则简单“的文本。

ReDoS：灾难性回溯

⚠️ 有一类正则会在特定输入下卡死，叫 ReDoS（Regular expression Denial of Service，正则拒绝服务）。根源是灾难性回溯（catastrophic backtracking）：当模式里出现“嵌套的量词“且能以指数级方式尝试组合时，引擎会陷入海量回溯。

典型危险模式长这样：(a+)+$、(\w+\s?)*$——量词套量词，且允许同一段文本被多种方式切分。喂给它一个“差一点就匹配“的长字符串，CPU 就会飙满、页面/服务卡住。

怎么防：

  • 避免“量词套量词“：如 (x+)+、(x*)*、(x+)*。能改成 (?:x)+ 这种单层就改。
  • 让分支互斥：别让引擎有“多种切法都可能“的空间。比如把 (\d+)* 改成 \d+。
  • 限制输入长度：对用户输入先卡个合理上限，再跑正则。
  • 拿不准就上工具：用现成的校验库，或用能检测危险模式的工具（如 safe-regex 思路）体检你的正则。

📌 记住：正则的强大来自“回溯尝试“，而灾难性回溯正是这份强大的阴暗面。写校验正则时，多想一句“最坏输入会让它试多少种可能“。

本章小项目：Markdown → HTML 迷你解析器

我们来把学到的正则串成一个能跑的小工具：把一小撮 Markdown 语法转成 HTML。

⚠️ 郑重声明：这是一个教学玩具，只为体会正则怎么协作。真实项目请用成熟库（如 marked、markdown-it），它们处理了海量边界情况，也做过 ReDoS 加固。别把这个玩具搬进生产环境。

支持的语法：标题 #/##/###、粗体 **x**、斜体 *x*、行内代码 `x`、链接 [文字](地址)、无序列表 - item、以及普通段落。

思路分两层：行内元素（在一行内部替换，如粗体、链接）和块级元素（按行扫描，如标题、列表、段落）。

行内替换

// 文件：code/ch12/md-to-html.js

function renderInline(text) {
  // 行内代码要最先处理：先把 `x` 抠出来，避免里面的 * 被误当成粗体。
  text = text.replace(/`([^`]+)`/g, "<code>$1</code>");

  // 链接 [文字](地址)：\[ \] \( \) 都要转义，因为它们是元字符。
  text = text.replace(/\[([^\]]+)\]\(([^)]+)\)/g, '<a href="$2">$1</a>');

  // 粗体 **x**：必须放在斜体之前，否则单个 * 会先把 ** 拆掉。
  text = text.replace(/\*\*(.+?)\*\*/g, "<strong>$1</strong>");

  // 斜体 *x*：单个星号包裹。此时 ** 已经被换成 <strong>，不会再干扰。
  text = text.replace(/\*(.+?)\*/g, "<em>$1</em>");

  return text;
}

逐条拆解正则，顺序很关键：

  • /`([^`]+)`/g —— 行内代码。两个反引号之间，[^]+抓"不是反引号的一段字符"。**最先处理**，这样代码块内部的*、[` 不会被后面的规则误伤。
  • /\[([^\]]+)\]\(([^)]+)\)/g —— 链接。\[ \] \( \) 全都转义（它们是元字符）。第 1 组 [^\]]+ 抓方括号里的文字，第 2 组 [^)]+ 抓圆括号里的地址，替换成 <a href="$2">$1</a>。
  • /\*\*(.+?)\*\*/g —— 粗体。\*\* 是两个转义星号，(.+?) 非贪婪抓中间最短内容。⚠️ 必须在斜体之前，否则单星号规则会先把 ** 拆成一对 *，粗体就毁了。
  • /\*(.+?)\*/g —— 斜体。单星号包裹。此时 ** 已经变成 <strong>，剩下的单星号才是真斜体。

📌 这里体现了正则处理的一个通用原则：规则之间会互相影响，处理顺序需要精心设计（先内层/先具体，再外层/再宽泛）。

按行扫描块级结构

// 文件：code/ch12/md-to-html.js

function mdToHtml(md) {
  const lines = md.split(/\r?\n/); // 兼容 Windows 的 \r\n 和 Unix 的 \n
  const out = []; // 收集输出的 HTML 片段
  let inList = false; // 当前是否正处在一个 <ul> 里

  const closeListIfOpen = () => {
    if (inList) {
      out.push("</ul>");
      inList = false;
    }
  };

  for (const rawLine of lines) {
    const line = rawLine.trimEnd();

    // 空行：结束段落/列表，不产生标签
    if (/^\s*$/.test(line)) {
      closeListIfOpen();
      continue;
    }

    // 标题：(#{1,3}) 抓 1~3 个井号，井号个数决定 h 几
    const heading = line.match(/^(#{1,3})\s+(.+)$/);
    if (heading) {
      closeListIfOpen();
      const level = heading[1].length;
      out.push(`<h${level}>${renderInline(heading[2])}</h${level}>`);
      continue;
    }

    // 无序列表项：行首可有空格，然后是 "- " 再跟内容
    const item = line.match(/^\s*-\s+(.+)$/);
    if (item) {
      if (!inList) {
        out.push("<ul>");
        inList = true;
      }
      out.push(`<li>${renderInline(item[1])}</li>`);
      continue;
    }

    // 其它非空行：当作段落
    closeListIfOpen();
    out.push(`<p>${renderInline(line.trim())}</p>`);
  }

  closeListIfOpen(); // 文档结尾若列表未关，补上
  return out.join("\n");
}

逐段讲：

  • md.split(/\r?\n/) —— 按行拆。\r? 让回车符可有可无，同时兼容 Windows（\r\n）和 Unix（\n）换行。
  • inList 是一个状态开关：Markdown 的列表是“连续多行 - “合成一个 <ul>，所以我们要记住“当前是否在列表里”。遇到非列表行时用 closeListIfOpen() 补 </ul>。这类“跨行状态“是正则单独搞不定的，需要配合普通代码——这正说明正则是工具，不是全部。
  • /^\s*$/ —— 判断空行（全是空白）。空行结束当前列表。
  • /^(#{1,3})\s+(.+)$/ —— 标题。(#{1,3}) 抓 1~3 个井号，它的长度就是标题级别（h1~h3）；\s+ 至少一个空格；(.+) 是标题文字，再交给 renderInline 处理行内样式。
  • /^\s*-\s+(.+)$/ —— 列表项。行首允许缩进空格，然后 - ，再抓内容。第一次遇到就开 <ul>。
  • 都不匹配的非空行 → 段落 <p>。

小结与自测

本章我们从零把正则表达式过了一遍，核心要点：

  • 正则是描述“一类字符串“的模式语言，擅长校验、查找、提取、替换扁平文本；不擅长嵌套结构（那是解析器的活）。
  • 创建：字面量 /.../flags（首选）vs new RegExp("...", "flags")（模式动态拼装时用，反斜杠要双写）。
  • flags：g 全局、i 忽略大小写、m 多行、s 点匹配换行、u Unicode、y 粘性。
  • 元字符：.（任意非换行）、\d \w \s（数字/单词字符/空白）及大写取反；字符组 [...]、取反 [^...]、范围 a-z。
  • 量词：* + ?、{n} {n,} {n,m}；默认贪婪，加 ? 变非贪婪。
  • 锚点：^ $（开头/结尾，^...$ 做整串校验）、\b（单词边界）。
  • 分组：捕获 (...)、非捕获 (?:...)、命名 (?<name>...)、反向引用 \1、或 |。
  • 环视：前瞻 (?=)/(?!)、后顾 (?<=)/(?<!)，是“条件“不消耗字符。
  • JS 方法：test（校验，别加 g）、match（有无 g 行为不同）、matchAll（全部+分组，必带 g）、replace（$1/命名/函数替换）、split、exec+lastIndex（g 的状态陷阱）。
  • 安全：警惕 ReDoS，避免“量词套量词“，限制输入长度。
