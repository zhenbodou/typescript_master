JavaScript & TypeScript 从入门到精通

第 21 章 · 工具链：ESLint、Prettier 与打包器

本章目标

到目前为止，我们写代码基本是“能跑就行“：随手敲，随手运行。但真实项目里，代码不是给自己一个人看的——它要和同事协作、要长期维护、要发布出去给别人用。这时候光“能跑“远远不够，你还需要一整套**工具链（toolchain）**来保证：

  • 团队里每个人写出来的代码长得一样（风格统一，不会因为“你用双引号我用单引号“吵起来）。
  • 一些低级错误、可疑写法能被自动发现，而不是等上线炸了才知道。
  • 源码（尤其是 TypeScript）能被编译、打包成可以真正运行或发布的构建产物（build artifact）。

本章我们就来搭这套工具链。学完之后，你将能够：

  • 说清楚工具链里每个工具的分工，不再把它们搅成一团。
  • 用 Prettier 做代码格式化，并让它和你的编辑器联动。
  • 用 ESLint 9 的扁平配置（flat config） 做代码检查（lint），集成 typescript-eslint，并让它和 Prettier 和平共处。
  • 用 tsc 做类型检查、用 tsup（基于 esbuild）把 TypeScript 打包成 dist/，并理解“打包器“和“编译器“的区别。
  • 把这些操作组织成一组清爽的 npm scripts。
  • 了解 EditorConfig 和 Git hooks（husky / lint-staged），让规范“自动执行“。

最后，我们会亲手搭一个生产级项目脚手架，以后开新项目直接照抄。

本章的示例脚手架放在 code/ch21/starter/，它是一个独立子项目，有自己的 package.json。它需要单独安装依赖，稍后会说明。

为什么需要工具链

先想象一个没有工具链的团队协作场景。

三个同事一起写一个项目。A 习惯用 2 个空格缩进、单引号、行尾不加分号；B 喜欢 4 个空格、双引号、每行都加分号；C 干脆 tab 和空格混着来。于是同一个文件里三种风格并存，git diff 里全是“某人重排了缩进“的噪音，真正的逻辑改动淹没其中。更糟的是，A 写了个 if (user = admin)（把 == 写成了 =），代码照样能跑，直到某天线上出了怪 bug 才发现。

这三类问题，正好对应工具链要解决的三件事：

  1. 一致的代码风格（consistent style） → 交给格式化工具（formatter），代表是 Prettier。
  2. 自动发现问题（catch bugs / bad patterns） → 交给代码检查工具（linter），代表是 ESLint。
  3. 产出可运行 / 可发布的构建产物 → 交给编译器（compiler，tsc） 和打包器（bundler，如 tsup、Vite）。

📌 记住这个分工表，本章后面所有内容都是它的展开：

工具管什么一句话
Prettier代码“长什么样“格式（缩进、引号、换行）
ESLint代码“有没有问题“质量（可疑写法、潜在 bug）
tsc类型对不对类型检查（可只查不产出）
bundler（tsup 等）打包成产物把源码变成 dist

它们各管一段，互不越界。理解这一点，你就不会再纠结“ESLint 能不能格式化““tsc 和打包器是不是重复了“这类问题了。

Prettier：把格式交给机器

Prettier 是一个“有主见的（opinionated）“代码格式化工具。它的核心理念很激进：不要在代码风格上浪费任何讨论时间。你把代码丢给它，它按一套规则重新排版输出，团队里所有人的代码从此长得一模一样。

它能格式化的不只是 JS/TS，还有 JSON、CSS、HTML、Markdown、YAML 等。

先装上，跑一次

在一个项目里安装 Prettier（作为开发依赖）：

npm install --save-dev prettier

假设我们有一段“很丑“的代码：

// 格式化前
const   user={name:"Alice",age:30,roles:['admin',"editor"]}
function greet(person){return "hi "+person.name}

在命令行运行：

npx prettier --write src/index.ts

--write 表示“就地改写文件“。运行后它会变成：

// 格式化后
const user = { name: 'Alice', age: 30, roles: ['admin', 'editor'] };
function greet(person) {
  return 'hi ' + person.name;
}

逐点拆解发生了什么：

  • 多余的空格被压平（const user → const user）。
  • 对象字面量补上了空格和结尾分号。
  • 字符串引号被统一（这里统一成了单引号）。
  • 函数体从“挤在一行“被拆成了规范的多行缩进。

注意：Prettier 只动“排版“，不动“逻辑“。它不会帮你把 "hi "+person.name 改成模板字符串，也不会重命名变量。它只关心“长什么样“。

⚠️ --write 会直接改文件，第一次在一个已有项目里跑之前，请确保代码已经提交到 git，方便你查看它改了什么、必要时回退。

配置：.prettierrc.json

Prettier 号称“零配置“，但它确实开放了少量选项，让团队定下自己的偏好。在项目根目录建一个配置文件：

// 文件：code/ch21/starter/.prettierrc.json
{
  "printWidth": 80,
  "semi": true,
  "singleQuote": true,
  "trailingComma": "all",
  "tabWidth": 2,
  "arrowParens": "always",
  "endOfLine": "lf"
}

逐项解释这些常用选项：

  • printWidth: 80——每行最多 80 个字符，超过就尝试换行。这是 Prettier 做换行决策的“标尺“。
  • semi: true——语句结尾加分号。设为 false 则不加。
  • singleQuote: true——字符串用单引号。这是很多项目的偏好。
  • trailingComma: "all"——多行的数组、对象、函数参数，末尾也加逗号。好处是以后新增一项时，git diff 只显示“新增一行“，而不是“上一行末尾多了个逗号“。
  • tabWidth: 2——缩进用 2 个空格。
  • arrowParens: "always"——箭头函数参数总是加括号，即 (x) => x 而不是 x => x。统一形式，减少心智负担。
  • endOfLine: "lf"——行尾用 \n（Unix 风格）。跨 Windows/Mac 协作时，统一行尾能避免“整个文件都变了“的假 diff。

💡 Prettier 故意没有“要不要 2 空格还是 4 空格该不该激烈争论“之外的一大堆选项。它的哲学就是“选项越少，争论越少”。别去找它没有的选项，那是设计使然。

.prettierignore：哪些不要碰

有些文件不该被格式化：依赖目录、构建产物、锁文件等。用 .prettierignore（语法和 .gitignore 一样）排除它们：

# 文件：code/ch21/starter/.prettierignore
# 依赖与构建产物不需要格式化
node_modules
dist
coverage

# 锁文件由包管理器维护
package-lock.json
pnpm-lock.yaml

与编辑器集成（强烈推荐）

命令行手动跑 Prettier 太累了。真正舒服的用法是让编辑器保存时自动格式化。以 VS Code 为例：

  1. 安装官方扩展 Prettier - Code formatter（发布者 esbenp）。
  2. 在项目里建 .vscode/settings.json：
// 文件：.vscode/settings.json（可选，团队共享编辑器配置）
{
  "editor.defaultFormatter": "esbenp.prettier-vscode",
  "editor.formatOnSave": true
}

配好之后，你每次 Ctrl/Cmd + S 保存，代码就自动排整齐了——你甚至不用再关心格式，闭着眼睛写，保存即工整。这是 Prettier 体验最好的地方。

用 npm scripts 包起来

除了编辑器，我们也希望能在命令行一键格式化整个项目（CI 里尤其需要）。在 package.json 里加两条脚本：

{
  "scripts": {
    "format": "prettier --write .",
    "format:check": "prettier --check ."
  }
}
  • npm run format——格式化整个项目（. 表示当前目录，Prettier 会递归处理，并自动遵守 .prettierignore）。
  • npm run format:check——只检查不改写，如果有文件不符合规范就以非 0 退出码失败。这个特别适合放进 CI（持续集成）：谁提交了没格式化的代码，流水线就报红。

ESLint：自动发现代码问题

Prettier 管“好不好看“，ESLint 管“对不对、好不好“。

Linting 这个词来自“lint“（衣服上的绒毛/棉絮）——linter 就是帮你挑出代码里那些“绒毛“：没用到的变量、可能是笔误的 ==、永远不会执行的分支、await 忘了写……这些代码可能照样能跑，但都是隐患或坏味道。

📌 关键区别：Prettier 重排代码但不改变含义；ESLint 关心代码的含义与质量，很多规则甚至能提示或修复潜在 bug。 两者互补，不是竞争。

关于版本：ESLint 9 与扁平配置

这里要特别说明版本，因为 ESLint 近年经历了一次重要变化。

老版本（ESLint 8 及更早）用一个叫 .eslintrc.js（或 .eslintrc.json）的配置文件，格式比较“魔法“：靠 extends、env、plugins 这些字段的字符串来层层继承，新手很难搞清楚规则到底从哪来。

从 ESLint 9（2024 年发布并成为默认）开始，官方推行扁平配置（flat config）：配置文件改叫 eslint.config.js，本质就是导出一个数组，数组里每一项是一个普通的配置对象。它是标准的 JS 模块，import 什么、用什么规则一目了然，不再有隐藏的继承魔法。

📌 本书用 ESLint 9 + flat config，这是当前（2024–2025）的现代默认写法。 如果你在网上看到 .eslintrc 开头的教程，那是旧写法，概念相通但文件格式不同，注意分辨。

安装

我们要检查的是 TypeScript，所以除了 ESLint 本体，还需要 typescript-eslint（让 ESLint 看懂 TS 语法的官方集成包）：

npm install --save-dev eslint typescript-eslint eslint-config-prettier
  • eslint——本体。
  • typescript-eslint——一个“套装“包，同时提供解析器（parser，让 ESLint 能读懂 TS）和一套 TS 规则。
  • eslint-config-prettier——一个“关闭冲突规则“的配置，稍后解释它为什么必不可少。

编写 eslint.config.js

来看我们脚手架里的完整配置，然后逐块拆解：

// 文件：code/ch21/starter/eslint.config.js
// ESLint 9 扁平配置（flat config）
import js from '@eslint/js';
import tseslint from 'typescript-eslint';
import prettier from 'eslint-config-prettier';

export default tseslint.config(
  // 1) 全局忽略：构建产物、依赖目录不参与 lint
  {
    ignores: ['dist/**', 'node_modules/**', 'coverage/**'],
  },

  // 2) ESLint 官方推荐规则（针对普通 JS）
  js.configs.recommended,

  // 3) typescript-eslint 推荐规则（针对 TS）
  ...tseslint.configs.recommended,

  // 4) 项目自定义规则（放在推荐规则之后才能覆盖它们）
  {
    files: ['**/*.ts'],
    rules: {
      // 允许以下划线开头的参数/变量表示“故意不用”
      '@typescript-eslint/no-unused-vars': [
        'error',
        { argsIgnorePattern: '^_', varsIgnorePattern: '^_' },
      ],
      // 禁止 console.log，但允许 console.warn / console.error
      'no-console': ['warn', { allow: ['warn', 'error'] }],
      eqeqeq: ['error', 'always'],
    },
  },

  // 5) 关闭所有与 Prettier 冲突的“格式类”规则，必须放在最后
  prettier,
);

逐块拆解：

  • 顶部三行 import：因为这是标准 JS 模块，配置里用到的东西都是显式导入进来的，没有任何隐藏魔法。@eslint/js 是 ESLint 自带的（随 eslint 一起装好），提供官方推荐规则集。

  • tseslint.config(...)：这是 typescript-eslint 提供的辅助函数。你完全可以手写一个数组，但用这个函数能获得更好的类型提示，且能正确展开各种预设。它接收若干个配置对象，最终返回那个扁平数组。

  • 第 1 块 ignores：单独一个只有 ignores 字段的对象，会被当作全局忽略。这里排除构建产物、依赖等目录——它们不是我们写的代码，不该被检查。

  • 第 2 块 js.configs.recommended：ESLint 官方对普通 JavaScript 的推荐规则，比如“禁止重复声明变量““禁止不可达代码”。

  • 第 3 块 ...tseslint.configs.recommended：注意前面的 ...（展开运算符）——因为它本身是一个数组（包含解析器设置 + 一组 TS 规则），必须展开后铺进外层数组。这一块让 ESLint 具备理解和检查 TypeScript 的能力。

  • 第 4 块自定义规则：files: ['**/*.ts'] 把这块规则限定在 .ts 文件上。rules 里我们调整了三条：

    • @typescript-eslint/no-unused-vars：报告“声明了却没用到“的变量。我们加了 argsIgnorePattern: '^_'，表示“名字以 _ 开头的参数是故意不用的，别报错“——这是一个常见约定。
    • no-console：把 console.log 标为警告（warn），但通过 allow 放行 console.warn 和 console.error（这两个通常是有意的日志）。
    • eqeqeq：强制使用 === / !==，禁止 == / !=，避免 JS 那套令人头疼的隐式类型转换。

    每条规则的第一个值是级别："off"（关）、"warn"（警告，不影响退出码成功）、"error"（错误，会让命令失败）。

  • 第 5 块 prettier：把 eslint-config-prettier 放在最后。它的作用是关闭 ESLint 里所有“管格式“的规则（比如缩进、引号、分号）。为什么需要它？见下一节。

⚠️ 顺序很重要。flat config 是数组，后面的配置会覆盖前面的。所以：推荐规则在前，你的自定义覆盖在中间，eslint-config-prettier（负责关闭格式规则）必须在最后，否则它的关闭会被前面的规则又打开。

为什么需要 eslint-config-prettier：避免打架

ESLint 历史上也内置了一些“格式类“规则（比如 indent、quotes）。如果你同时开着这些规则又用 Prettier，两者就会互相打架：Prettier 想把代码排成一样，ESLint 又跳出来报格式错误，你陷入“改也不是不改也不是“的循环。

eslint-config-prettier 的唯一职责就是把 ESLint 里所有和格式相关的规则统统关掉，从而划清界限：

  • 格式 100% 交给 Prettier。
  • 代码质量 100% 交给 ESLint。

这就是为什么它必须放最后、且不含任何“开启“规则——它只做“关闭“这一件事。

💡 记住这句话就够了：用了 Prettier，就一定要配 eslint-config-prettier，否则两个工具会打架。

运行 lint，以及 --fix

在 package.json 里加脚本：

{
  "scripts": {
    "lint": "eslint .",
    "lint:fix": "eslint . --fix"
  }
}
  • npm run lint——检查整个项目，把发现的问题打印出来。有 error 时命令以非 0 退出（CI 会报红）。
  • npm run lint:fix——加上 --fix，ESLint 会自动修复能自动修的那部分问题（比如把 != 改成 !==）。注意：并不是所有问题都能自动修，比如“未使用的变量“就得你自己判断该删还是该用。

举个 --fix 的例子。检查前：

if (status != 'done') {
  const unused = 42;
  doSomething();
}

npm run lint 会报两个问题：!= 违反 eqeqeq，unused 违反 no-unused-vars。运行 npm run lint:fix 后，!= 被自动改成 !==；但 unused 不会被自动删除（ESLint 不敢替你决定删代码），它会继续作为错误提示，等你手动处理。

lint 与 format 的分工（再强调一次）

新手最容易混淆这两者，我们用一张表彻底厘清：

Prettier（format）ESLint（lint）
关心代码长什么样代码有没有问题
例子引号、缩进、换行、分号未使用变量、==、await 漏写
会改逻辑吗不会--fix 可能改（如 !=→!==）
典型命令prettier --writeeslint . --fix

一个健康的项目两者都要有，各管一段。它们通过 eslint-config-prettier 划清边界，配合无间。

TypeScript 编译：tsc 的三种用法

TypeScript 代码不能被 Node.js 或浏览器直接运行——运行前必须先**编译（compile）**成 JavaScript。做这件事的是 TypeScript 官方编译器 tsc。前面第 20 章我们已经详细讲过 tsconfig.json，这里聚焦 tsc 作为工具链一环的三种典型跑法。

用法一：编译产出（生成 dist）

最基本的用法，直接运行：

npx tsc

它会读取 tsconfig.json，把 src/ 下的 TS 编译成 JS，输出到配置里 outDir 指定的目录（我们脚手架里是 dist/）。如果开了 "declaration": true，还会同时生成 .d.ts 类型声明文件。

用法二：监听模式（watch）

开发时你不想每改一次就手动编译一次，加 --watch（简写 -w）：

npx tsc --watch

tsc 会常驻在终端里，监视文件变化，你一保存它就自动重新编译，并实时报告类型错误。适合开发过程中随时看到反馈。

用法三：只检查不产出（--noEmit）

这是工具链里最常用的一种。有时你不需要 tsc 生成文件（因为产物交给了打包器，下一节讲），只想让它扮演一个纯粹的类型检查器：

npx tsc --noEmit

--noEmit 意思是“不要发出（emit）任何文件“，只做类型检查，有类型错误就报错退出。我们把它包成脚本：

{
  "scripts": {
    "typecheck": "tsc --noEmit"
  }
}

📌 为什么单独要一个 typecheck？因为很多现代打包器（包括我们下面用的 tsup）出于速度考虑，打包时并不做完整的类型检查——它们只管把 TS 语法去掉、转成 JS，速度飞快，但你写错了类型它可能视而不见。所以我们需要独立的一步 tsc --noEmit 来专门守住类型正确性。这就引出了下一节：打包器和编译器到底什么关系。

打包器（bundler）：为什么、有哪些、怎么用

为什么要打包

你可能会问：既然 tsc 就能把 TS 变成 JS，为什么还需要“打包器“？

因为一个真实项目往往由几十上百个模块组成，还依赖了一堆 node_modules 里的第三方包。如果只用 tsc，你会得到几十个零散的 .js 文件，它们之间靠 import 互相引用。这在某些场景（尤其是要发布给别人用、或要在浏览器里加载）会带来问题：文件太多、加载慢、依赖关系混乱。

打包器（bundler） 做的事情是：从一个入口文件出发，顺着 import 把所有相关模块“串“起来，处理依赖、转译语法，最终产出少量、优化过的文件（常常就一两个），可以直接运行或发布。

顺带它通常还做这些优化：

  • Tree shaking（摇树）：把没被用到的代码摇掉，减小体积。
  • Minify（压缩）：删空格、缩短变量名，让文件更小。
  • 打包依赖：把用到的第三方代码一并纳入（或按需处理）。

bundler vs tsc：一句话区别

📌 tsc 是“编译器“，逐个文件把 TS 翻译成 JS，并做完整类型检查，但不合并、不优化。打包器是“打包器“，把多文件合并优化成产物，速度极快，但通常不做类型检查。

所以现代项目的常见组合是各取所长：

  • 类型检查 → tsc --noEmit（慢一点，但类型查得准）。
  • 产物构建 → 打包器（快，产物优化好，但不查类型）。

两步分工，互不替代。

主流打包器速览（了解定位即可）

生态里打包/构建工具很多，简单认识一下它们的定位：

  • esbuild——用 Go 写的超快打包/转译器，是很多上层工具的“引擎“。底层、极快，但配置相对偏底层。
  • Vite——面向前端应用（尤其配合 React/Vue）的开发+构建工具，开发服务器体验极佳。做网页应用时的首选，但对“打包一个库“来说偏重。
  • Rollup——老牌打包器，尤其擅长打包库（library），tree shaking 做得早做得好。Vite 的生产构建底层就用它。
  • tsup——一个零配置的库打包工具，底层用 esbuild（快），但帮你把“生成 .d.ts 类型声明““同时输出多种模块格式“这些库作者常需要的事情都封装好了。对“把一个 TS 库打包成 dist“这个需求，它最省心。

💡 怎么选？做网页应用用 Vite；做要发布的库/CLI 工具用 tsup（省心）或 Rollup（更可控）；只想要底层极速转译用 esbuild。本章脚手架的目标是“一个标准 TS 项目/库“，所以我们选 tsup——它几乎不用配置就能干净利落地产出 dist。

用 tsup 打包：最小示例

安装：

npm install --save-dev tsup

最简单的用法连配置文件都不用，直接命令行指定入口：

npx tsup src/index.ts

它会把 src/index.ts（及其 import 的所有模块）打包，默认输出到 dist/。不过为了清晰可控，我们还是写一个配置文件：

// 文件：code/ch21/starter/tsup.config.ts
import { defineConfig } from 'tsup';

export default defineConfig({
  entry: ['src/index.ts'], // 入口文件
  format: ['esm'], // 输出 ES Module
  target: 'es2022', // 目标语法版本
  dts: true, // 生成 .d.ts 类型声明文件
  sourcemap: true, // 生成 source map，便于调试
  clean: true, // 每次构建前清空 dist
  minify: false, // 库通常不压缩，交给使用方按需处理
});

逐项拆解：

  • entry——从哪个（些）文件开始打包。tsup 会顺着它的 import 自动收集依赖。
  • format: ['esm']——输出 ES Module 格式（import/export）。如果你还想兼容老的 CommonJS，可以写成 ['esm', 'cjs']，tsup 会同时产出两份。
  • target: 'es2022'——目标 JS 语法版本，决定要不要把新语法降级。
  • dts: true——生成类型声明文件 .d.ts。这是 tsup 相比裸 esbuild 的一大便利：库的使用者需要这些 .d.ts 才能获得类型提示。
  • sourcemap: true——生成 source map，调试打包后代码时能映射回原始 TS。
  • clean: true——每次构建前先清空 dist/，避免遗留旧文件。
  • minify: false——不压缩。库通常保持可读，压缩留给最终应用去做。

把它包进脚本：

{
  "scripts": {
    "build": "tsup src/index.ts"
  }
}

运行 npm run build，你会看到类似输出：

ESM dist/index.js     347.00 B
DTS dist/index.d.ts   237.00 B
⚡️ Build success

dist/ 里就有了打包好的 index.js、类型声明 index.d.ts 和 source map。这就是可以拿去运行或发布的构建产物。

⚠️ 再强调：tsup 打包时不会帮你做完整类型检查。所以正确的构建流程是先 npm run typecheck（tsc 查类型），再 npm run build（tsup 出产物）。很多项目会把它们串起来，比如 "build": "tsc --noEmit && tsup src/index.ts"，确保类型没问题才产出。

npm scripts：把一切组织起来

前面每个工具我们都写了一条脚本，现在把它们汇总，形成一套标准命令。这是脚手架 package.json 里 scripts 部分的样子：

{
  "scripts": {
    "dev": "tsup src/index.ts --watch --onSuccess \"node dist/index.js\"",
    "build": "tsup src/index.ts",
    "typecheck": "tsc --noEmit",
    "lint": "eslint .",
    "lint:fix": "eslint . --fix",
    "format": "prettier --write .",
    "format:check": "prettier --check ."
  }
}

逐条说明：

  • dev——开发模式。--watch 让 tsup 监听改动自动重建，--onSuccess "node dist/index.js" 让每次构建成功后自动运行产物。改代码 → 自动重打包 → 自动重跑，反馈很快。
  • build——一次性打包产出 dist/。
  • typecheck——只做类型检查（tsc --noEmit）。
  • lint / lint:fix——检查 / 自动修复代码质量问题。
  • format / format:check——格式化 / 只检查格式。

💡 记住这套约定俗成的脚本名（dev/build/lint/format/typecheck），几乎所有现代 JS/TS 项目都长这样。以后你接手别人的项目，cat package.json 看一眼 scripts，就知道怎么跑起来了。

📌 一个团队里典型的“提交前检查“通常是：npm run typecheck && npm run lint && npm run format:check——类型对、没坏味道、格式规范，三关全过才允许提交。下一节我们看怎么让它自动执行。

EditorConfig 与 Git hooks（简介）

工具配好了，但还有个现实问题：怎么保证团队里每个人真的会去跑这些命令？ 靠自觉是不够的。这里介绍两个“让规范自动生效“的机制。

EditorConfig：跨编辑器的基础风格

不是所有人都用 VS Code。有人用 WebStorm，有人用 Vim。EditorConfig 是一个被几乎所有编辑器支持的标准，用一个 .editorconfig 文件统一最基础的编辑行为（缩进、行尾、编码），这样无论用什么编辑器，打开这个项目时的基础风格都一致：

# 文件：code/ch21/starter/.editorconfig
root = true

[*]
charset = utf-8
end_of_line = lf
insert_final_newline = true
trim_trailing_whitespace = true
indent_style = space
indent_size = 2

[*.md]
trim_trailing_whitespace = false

拆解：

  • root = true——告诉编辑器“到此为止，别再往上层目录找 .editorconfig 了“。
  • [*]——对所有文件生效的通用规则：UTF-8 编码、LF 行尾、文件末尾留一个空行、去掉行尾多余空格、缩进用 2 空格。
  • [*.md]——针对 Markdown 的例外：Markdown 里行尾的两个空格是“换行“的意思，所以不能去掉行尾空格。

EditorConfig 和 Prettier 有部分重叠（都管缩进），但它更底层、更通用，在编辑器打字的当下就生效，是一道“第一层防线“。

Git hooks：husky + lint-staged

Git hooks 是 git 提供的“钩子“——在特定操作（比如 git commit）发生前后自动执行脚本。我们可以利用它，在每次提交前自动跑 lint 和 format，把没规范的代码挡在提交之外。

手动配置 git hooks 比较繁琐，社区有两个常用工具让它变简单：

  • husky——让 git hooks 的配置变得简单、可提交进仓库（这样团队每个人克隆下来都自动生效）。
  • lint-staged——只对本次提交暂存（staged）的文件跑检查，而不是整个项目。这样提交快，且不会因为别人留下的老问题连累你。

典型搭配是：git commit 时，husky 触发 pre-commit 钩子，钩子里调用 lint-staged，lint-staged 对你即将提交的那几个文件跑 eslint --fix 和 prettier --write。配置大致长这样（了解即可，本章脚手架未强制引入，避免增加安装负担）：

// package.json 片段（示意）
{
  "lint-staged": {
    "*.{ts,js}": ["eslint --fix", "prettier --write"],
    "*.{json,md}": ["prettier --write"]
  }
}

💡 这套东西的价值在于：规范不再依赖人的自觉，而是被工具强制执行。 忘了格式化？提交时自动帮你格式化。写了 ==？提交前就被拦下。这是成熟团队的标配。本书为了让脚手架安装轻量，没有默认引入 husky，你在真实项目里可以按需加上。

常见坑与最佳实践

  • ⚠️ ESLint 8 的旧配置（.eslintrc）和 ESLint 9 的新配置（eslint.config.js）不通用。 网上很多教程还是旧格式，照抄会在 ESLint 9 上报错。认准文件名：eslint.config.js 才是本书用的 flat config。

  • ⚠️ 忘了装或忘了配 eslint-config-prettier，会导致 ESLint 和 Prettier 在格式上互相打架。用 Prettier 就一定要配它，且放在 flat config 数组的最后。

  • ⚠️ flat config 是数组，顺序即优先级，后面覆盖前面。 自定义规则要放在推荐规则之后才能生效；eslint-config-prettier 要放在最末尾。

  • ⚠️ 别指望打包器帮你查类型。 tsup / esbuild 为了快，只做语法转译，不做类型检查。类型正确性必须靠独立的 tsc --noEmit 守住。CI 里一定要单独有 typecheck 这一步。

  • ⚠️ prettier --write 和 eslint --fix 会改文件。 第一次在已有项目里跑之前先提交 git，方便查看和回退。

  • 💡 让编辑器保存即格式化（Prettier 插件 + formatOnSave），你就再也不用手动跑 format 了，体验天差地别。

  • 💡 把 dist、node_modules、coverage 同时写进 .gitignore、.prettierignore 和 ESLint 的 ignores。 这三个地方都要排除构建产物和依赖，漏一个就会有奇怪的报错或噪音。

  • 💡 脚本命名遵循社区约定（dev/build/lint/format/typecheck）。别自创名字，方便协作者和 CI 直接套用。

  • 💡 区分 format（写）和 format:check（只查）、lint（查）和 lint:fix（修）。 本地开发用“写/修“版本，CI 用“只查“版本（CI 不应该改代码，只该报告问题）。

本章小项目：标准项目脚手架

现在我们把本章所有工具攒成一个可以直接照抄的生产级项目模板，放在 code/ch21/starter/。以后你开任何新的 TS 项目，复制这个目录、改改名字就能开工。

目录结构

拆解几个关键字段：

第 20 章已详细讲过这些选项，这里只点出与工具链相关的几个："strict": true 打开全部严格类型检查（配合 typecheck 脚本）；"declaration": true 让 tsc 也能生成 .d.ts（虽然我们主要靠 tsup 的 dts 出声明）；"outDir": "dist" 与打包产物目录一致；"verbatimModuleSyntax": true 强制 import/export 语法清晰（配合 ESM 更稳）。

源码入口

// 文件：code/ch21/starter/src/index.ts

/** 计算两数之和。 */
export function add(a: number, b: number): number {
  return a + b;
}

/** 计算两数之差。 */
export function subtract(a: number, b: number): number {
  return a - b;
}

/** 一个简单的问候函数。 */
export function greet(name: string): string {
  return `你好，${name}！`;
}

// 直接用 `node dist/index.js` 运行时的演示入口。
// 作为库被 import 时，下面这段不会打印（因为没有顶层副作用调用它）。
function main(): void {
  console.warn(greet('世界'));
  console.warn(`1 + 2 = ${add(1, 2)}`);
}

main();

这个入口既导出了几个可复用的函数（供别人 import），又有一个 main() 演示直接运行的效果。注意里面故意用了 console.warn 而不是 console.log——因为我们的 ESLint 规则把 console.log 标为警告，而放行 console.warn。这正好演示了 lint 规则在实际代码里的约束。

小结与自测

本章我们搭起了一套现代 JS/TS 项目的工具链，核心是理解每个工具的分工：

  • Prettier 管格式（长什么样），配置放在 .prettierrc.json，最佳体验是“编辑器保存即格式化“。
  • ESLint 9（flat config，eslint.config.js） 管代码质量（有没有问题），通过 typescript-eslint 支持 TS，用 eslint-config-prettier 与 Prettier 划清边界。--fix 能自动修一部分问题。
  • tsc 管类型检查，tsc --noEmit 是“只查不产出“的常用姿势。
  • 打包器（本章用 tsup） 管构建产物，快速把 TS 打包成 dist/，但不查类型——所以类型检查要靠独立的 tsc --noEmit。
  • 这些操作用一组约定俗成的 npm scripts（dev/build/lint/format/typecheck）组织起来。
  • EditorConfig 和 Git hooks（husky + lint-staged） 让规范“自动生效“，不依赖人的自觉。

  3. flat config 是个数组，为什么 eslint-config-prettier 必须放在最后？

  6. tsup 的 dts: true 选项有什么用？对“发布一个库“为什么重要？

工具链保证了代码“写得规范、构建得出来“，但它无法保证代码逻辑是对的——add 函数会不会在某个边界值上算错？重构之后功能会不会悄悄坏掉？这些要靠测试来守护。下一章 第 22 章 · 测试与 TDD，我们就来给代码加上“逻辑正确性“的安全网，学习单元测试、断言、以及“测试驱动开发（TDD）“这种先写测试再写实现的开发方式。届时你会发现，测试脚本会自然地融入本章搭好的这套 npm scripts 里，成为工具链的又一环。
