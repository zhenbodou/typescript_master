JavaScript & TypeScript 从入门到精通

第 7 章 · 迭代器、生成器、Symbol 与集合类型

本章目标

前六章你已经掌握了变量、控制流、函数、闭包、面向对象。你会用 for...of 遍历数组、会用 ... 展开、会解构。但你可能从没想过一个问题：

for...of 凭什么能遍历数组？它能遍历字符串、能遍历 Map，为什么就不能直接遍历一个普通对象 {a:1}？

答案藏在一套叫**迭代协议（iteration protocol）**的“暗号”里。数组、字符串、Map、Set 都对上了这个暗号，普通对象没对上，所以 for...of 认它们、不认普通对象。这一章我们就把这套暗号彻底讲清，然后你会亲手让一个普通对象“对上暗号”，变得可迭代。

学完本章你会：

  • 理解 Symbol：一种“绝对唯一”的值，以及它为什么是迭代协议的基石。
  • 讲透可迭代协议与迭代器协议：for...of、扩展运算符、解构背后都是同一套机制，并手写一个可迭代对象。
  • 掌握生成器（generator）：用 function* 和 yield 轻松造出迭代器，实现惰性求值和无限序列，还能双向通信。
  • 学会 Map 与 Set 两种集合类型，知道它们和普通对象、数组的区别，以及何时该用哪个。
  • 认识 WeakMap / WeakSet：什么是弱引用，它们为什么不可迭代，能用来做私有数据和缓存。
  • 本章小项目：用生成器实现一个惰性分页数据流，再配合 Map / Set 做分组统计与去重。

这一章是通往下一章“异步编程”的桥梁——因为 async/await 的底层，正是本章的生成器。我们从 Symbol 讲起。

7.1 Symbol：一种“绝对唯一”的值

我们已经学过 7 种数据类型里的几种：字符串、数字、布尔、null、undefined。ES6 又加了一种原始类型（primitive type）：Symbol。

一句话概括：每个 Symbol 都是独一无二的，创造出来之后，世界上再没有第二个和它相等。

// 文件：code/ch07/01-symbol.js
const a = Symbol("id");
const b = Symbol("id");
console.log("a === b ?", a === b); // false，两个不同的值
console.log("a.description =", a.description); // "id"，只是给人看的标签

逐段拆解：

  • Symbol("id") 创建一个 Symbol。注意没有 new（new Symbol() 会报错）。
  • 括号里的 "id" 叫描述（description），纯粹是给人调试时看的标签，不影响唯一性。
  • 所以 a 和 b 描述一样，却 a === b 为 false——它们是两个不同的值。

你可能会想：一个“永远不等于别人”的值有什么用？它最经典的用途是作为对象的键（key），做一把“别人配不出来的钥匙”。

Symbol 作为对象的键

// 文件：code/ch07/01-symbol.js（续）
const ID = Symbol("id");
const user = {
  name: "小明",
  [ID]: 1001, // 计算属性名：用变量的值作键
};
console.log("user.name =", user.name);
console.log("user[ID] =", user[ID]); // 1001

console.log("Object.keys:", Object.keys(user)); // 只有 "name"
for (const k in user) console.log("for-in:", k); // 只有 "name"
console.log("symbol keys:", Object.getOwnPropertySymbols(user)); // [ Symbol(id) ]

逐段拆解：

  • [ID]: 1001 用到了第 3 章学的计算属性名语法：中括号里放一个表达式，它的值（这里是 Symbol ID）成为键。
  • 关键特性：Symbol 键不会出现在 Object.keys()、for...in、JSON.stringify 里。它像一个“隐藏抽屉”。
  • 想专门取出 Symbol 键，得用 Object.getOwnPropertySymbols(user)。

💡 为什么有用：想象你在给别人的对象“挂”一点自己的元数据，又怕和对方已有的属性名（比如 id）撞车、怕被对方的遍历逻辑扫到。用一个 Symbol 当键，就绝对不会冲突，也不会被常规遍历干扰。库作者经常这么干。

Symbol.for：全局注册表

普通 Symbol() 每次都造新的。但有时你想“按名字取回同一个 Symbol”，跨文件共享。这时用 Symbol.for：

// 文件：code/ch07/01-symbol.js（续）
const s1 = Symbol.for("app.token");
const s2 = Symbol.for("app.token");
console.log("s1 === s2 ?", s1 === s2); // true！注册表里同名返回同一个
console.log("keyFor:", Symbol.keyFor(s1)); // "app.token"

逐段拆解： Symbol.for("app.token") 会去一个全局的“Symbol 注册表”里查：有叫 app.token 的就返回它，没有就新建一个登记进去。所以两次调用拿到的是同一个 Symbol。Symbol.keyFor 则反过来，从 Symbol 查回它登记的名字。

📌 记住区别：Symbol("x") 每次都新建、永不相等；Symbol.for("x") 同名共享、可跨模块取回。

知名 Symbol（well-known symbol）

JavaScript 内置了一批特殊的 Symbol，用来当作语言各种行为的“钩子”。它们挂在 Symbol 上，比如 Symbol.iterator、Symbol.toPrimitive 等，统称知名 Symbol。

它们的作用是：当你在对象上实现了某个知名 Symbol 对应的方法，就等于告诉语言“我支持某种能力”。 下面看一个 Symbol.toPrimitive 的例子，它控制“对象被转成原始值时”的行为：

// 文件：code/ch07/01-symbol.js（续）
const range = {
  from: 1,
  to: 3,
  [Symbol.toPrimitive](hint) {
    if (hint === "string") return `范围 ${this.from}..${this.to}`;
    return this.to - this.from; // number / default
  },
};
console.log(`${range}`); // "范围 1..3"（模板字符串要 string）
console.log(+range); // 2（+ 要 number）

逐段拆解： 当 JS 需要把 range 变成原始值时，会去找它有没有 [Symbol.toPrimitive] 方法，有就调用它，并传入一个 hint（“string” / “number” / “default”）说明想要什么类型。这样我们就“劫持”了对象的类型转换行为。

而所有知名 Symbol 里，本章的绝对主角是 Symbol.iterator——它决定一个对象能不能被 for...of 遍历。下一节登场。

7.2 迭代协议：for...of 背后的暗号

现在回答开头的问题：for...of 到底认什么？答案是两个协议。别被“协议”吓到，它其实就是“约定好的方法名和返回格式”。

迭代器协议（iterator protocol）

一个对象是迭代器（iterator），当且仅当它有一个 next() 方法，每次调用返回一个形如 { value, done } 的对象：

  • value：本次产出的值。
  • done：布尔值。false 表示“还有”，true 表示“到头了”。

你可以把迭代器想象成一台取号机：每按一次 next()，吐出一个号（value）；号发完了，就一直吐 { value: undefined, done: true }。

可迭代协议（iterable protocol）

一个对象是可迭代对象（iterable），当且仅当它有一个 [Symbol.iterator]() 方法，调用后返回一个迭代器。

用取号机类比：可迭代对象是“取号机所在的大厅”，[Symbol.iterator]() 就是“领一台全新取号机”的动作。每次 for...of 开始遍历，都会先调用 [Symbol.iterator]() 领一台崭新的取号机，从头开始取。

我们先手动“开一台取号机”，看清底层：

// 文件：code/ch07/02-iterable.js
const arr = ["a", "b"];
const it = arr[Symbol.iterator](); // 数组内置了 Symbol.iterator
console.log(it.next()); // { value: 'a', done: false }
console.log(it.next()); // { value: 'b', done: false }
console.log(it.next()); // { value: undefined, done: true }

逐段拆解： 数组本身是可迭代对象（内置了 [Symbol.iterator]）。我们手动调用它，拿到迭代器 it，然后一次次 next()——这正是 for...of 在背后帮你做的事。for (const x of arr) 约等于：反复 it.next()，只要 done 不是 true，就把 value 交给 x。

手写一个可迭代对象

理论讲完，动手。我们造一个“数字区间”对象，让它能被 for...of 遍历：

// 文件：code/ch07/02-iterable.js（续）
const range = {
  from: 1,
  to: 5,
  [Symbol.iterator]() {
    let current = this.from;
    const last = this.to;
    return {
      next() {
        if (current <= last) {
          return { value: current++, done: false };
        }
        return { value: undefined, done: true };
      },
    };
  },
};

逐段拆解：

  • range 有一个 [Symbol.iterator]() 方法 —— 这一步让它成为可迭代对象。
  • 该方法内部用 current / last 记住进度，然后返回一个迭代器：一个带 next() 的对象。
  • next() 每次被调用：只要 current <= last，就返回 { value: current++, done: false }（先用 current 当 value，再自增）；否则返回 { done: true } 宣告结束。

📌 为什么把 current 定义在 [Symbol.iterator]() 内部、而不是对象上？ 因为这样每次 for...of 领到的都是独立的迭代器，各自有自己的进度，互不干扰。这正是“大厅可以同时开好几台取号机”的设计。

现在，这个对象“对上暗号”了，一堆语法立刻免费可用：

// 文件：code/ch07/02-iterable.js（续）
for (const n of range) process.stdout.write(n + " "); // 1 2 3 4 5
console.log([...range]); // [1,2,3,4,5]        —— 扩展运算符
const [first, second] = range; // 1 2           —— 解构
console.log(Array.from(range)); // [1,2,3,4,5]  —— Array.from

📌 一个重大认知：for...of、扩展运算符 ...、数组解构、Array.from、new Set(...)、Promise.all(...)……它们全都建立在迭代协议之上。你实现了一个 [Symbol.iterator]，等于一次性解锁了这一整套语法。这就是“协议”的威力：一处约定，处处通用。

⚠️ 陷阱：普通对象 {a:1, b:2} 没有 [Symbol.iterator]，所以 for...of {a:1} 会报错 is not iterable。遍历对象请用 for...in（遍历键）或 Object.entries(obj)（它返回数组，是可迭代的）。别把 for...of 和 for...in 搞混。

7.3 生成器：写迭代器的“简单模式”

上一节手写迭代器，要维护 current、要拼 {value, done}、要判断边界——有点繁琐。ES6 给了我们一个“外挂”：生成器（generator），让你几乎不用管这些细节。

生成器是一种特殊的函数，用 function*（星号）声明，内部用 yield 关键字“交出”一个值。

// 文件：code/ch07/03-generator.js
function* greet() {
  yield "你好";
  yield "世界";
  return "结束";
}

const g = greet();
console.log(g.next()); // { value: '你好', done: false }
console.log(g.next()); // { value: '世界', done: false }
console.log(g.next()); // { value: '结束', done: true }
console.log(g.next()); // { value: undefined, done: true }

逐段拆解——这里有个颠覆认知的点：

  • 调用 greet() 并不会执行函数体！它只是返回一个生成器对象（它同时是迭代器，也是可迭代对象）。
  • 只有当你调用 .next()，函数体才开始跑，跑到第一个 yield 就暂停，把 yield 后面的值作为 value 交出来（done: false）。
  • 再 .next()，函数从上次暂停处继续，跑到下一个 yield 再停。
  • 遇到 return（或函数体结束），产出最后一个 value 并 done: true。

💡 这就是生成器的魔法：函数可以“暂停”和“恢复”。普通函数一旦调用就一口气跑完，生成器却能走走停停，把控制权在 yield 处交还给调用方。这个“暂停/恢复”能力，正是后面异步编程的基石。

用生成器实现迭代器

还记得上一节那个繁琐的 range 吗？用生成器重写，简直不要太清爽：

// 文件：code/ch07/03-generator.js（续）
function* rangeGen(from, to) {
  for (let i = from; i <= to; i++) {
    yield i;
  }
}
console.log([...rangeGen(1, 5)]); // [1,2,3,4,5]

逐段拆解： 我们不用再手写 next()、不用拼 {value, done}、不用管 done —— 生成器全帮你搞定了。因为生成器对象天生可迭代，所以能直接 [...] 展开、for...of。以后要造迭代器，首选生成器。

yield*：把迭代委托出去

yield*（yield 加星号）可以把迭代“转交”给另一个可迭代对象——它会把对方的每一项依次 yield 出来：

// 文件：code/ch07/03-generator.js（续）
function* abc() {
  yield "a";
  yield "b";
}
function* combined() {
  yield 1;
  yield* abc(); // 展开 abc 的所有 yield
  yield* [8, 9]; // 数组也可迭代，同样能委托
  yield 2;
}
console.log([...combined()]); // [1,'a','b',8,9,2]

逐段拆解： yield* abc() 相当于“把 abc() 产出的每一项，都当成我自己的产出交出去”。yield* 后面可以跟任何可迭代对象（数组、字符串、其他生成器）。它常用于把小生成器组合成大生成器，或递归遍历树结构。

惰性求值与无限序列

生成器最迷人的特性是惰性（lazy）：值是“按需生产”的，你不 next，它就不算下一个。这让我们能表达无限序列而不会死循环：

// 文件：code/ch07/03-generator.js（续）
function* naturals() {
  let n = 1;
  while (true) {
    yield n++; // 无限，但每次 next 只走一步
  }
}
function take(iterable, count) {
  const out = [];
  for (const x of iterable) {
    if (out.length >= count) break;
    out.push(x);
  }
  return out;
}
console.log(take(naturals(), 5)); // [1,2,3,4,5]

逐段拆解：

  • naturals() 里是 while(true)——如果这是普通函数，直接卡死。但生成器不会：yield 会暂停，while 只有在你要下一个时才转一圈。
  • take 拿够 5 个就 break，于是 naturals 永远只被推进了 5 步。无限的定义、有限的执行，这就是惰性的价值：你可以描述一个巨大甚至无限的数据流，却只在真正需要时才付出计算代价。本章小项目正是靠这一点省下“多余的分页请求”。

双向通信：next(value)

yield 不只是“出”，还能“进”。yield 是一个表达式，它的值由下一次 next(value) 传入的参数决定——这让调用方能把数据“送回”生成器内部：

// 文件：code/ch07/03-generator.js（续）
function* calculator() {
  const x = yield "请给我第一个数";
  const y = yield "请给我第二个数";
  return x + y;
}
const c = calculator();
console.log(c.next().value); // "请给我第一个数"（启动，忽略传入值）
console.log(c.next(10).value); // 把 10 赋给 x，返回 "请给我第二个数"
console.log(c.next(5).value); // 把 5 赋给 y，返回 15

逐段拆解：

  • 第一次 c.next()：启动生成器，跑到第一个 yield，交出 "请给我第一个数"。此时函数停在 yield 处，x 还没有值。⚠️ 第一次 next() 传的参数会被忽略（因为还没有哪个 yield 在等它）。
  • c.next(10)：10 成为上一个 yield 表达式的值，于是 x = 10；函数继续到第二个 yield，交出 "请给我第二个数"。
  • c.next(5)：y = 5；函数 return x + y，即 15。

📌 这种“出一个值、等一个值回来”的双向管道，正是 async/await 的思想原型：把“暂停等待结果、拿到结果再继续”变成看起来同步的代码。第 8 章你会看到它开花结果。

7.4 Map：更强大的键值集合

第 3 章我们用普通对象 {} 存键值对。但对象有几个恼人的限制：键只能是字符串或 Symbol、混着原型链上的属性、拿数量还得 Object.keys().length。ES6 的 Map 专门解决这些问题。

Map 就是一张“键 → 值”的表，但比对象强在四点：键可以是任意类型、严格保持插入顺序、有 size、天生可迭代。

// 文件：code/ch07/04-map.js
const m = new Map();
const objKey = { id: 1 };
m.set("name", "小明"); // 字符串键
m.set(42, "数字键"); // 数字键，不会被转成字符串
m.set(objKey, "对象键"); // 对象也能当键！
m.set(NaN, "NaN 也能当键");

console.log(m.get(42)); // "数字键"
console.log(m.get("42")); // undefined —— 42 和 "42" 是不同的键
console.log(m.get(objKey)); // "对象键"
console.log(m.get(NaN)); // "NaN 也能当键"

逐段拆解：

  • new Map() 创建空 Map。set(键, 值) 存，get(键) 取。
  • 键可以是任意类型：数字 42 和字符串 "42" 是两个不同的键（对象会把它们都变成 "42"）；对象、函数也能当键；连 NaN 都能当键（Map 内部认为 NaN 等于 NaN，这点比 === 友好）。
// 文件：code/ch07/04-map.js（续）
console.log(m.size); // 直接拿数量
console.log(m.has(42)); // true —— 判断有没有某个键
m.delete(42); // 删除某个键
console.log(m.has(42)); // false

size 直接给数量；has 判断存在；delete 删除。这些比对象方便太多（对象得 key in obj、delete obj.key、Object.keys().length）。

Map 的有序迭代

Map 严格按插入顺序迭代，而且本身可迭代：

// 文件：code/ch07/04-map.js（续）
const scores = new Map([
  ["语文", 90],
  ["数学", 85],
  ["英语", 88],
]);
for (const [subject, score] of scores) {
  console.log(`${subject} => ${score}`);
}
console.log([...scores.keys()]); // ['语文','数学','英语']
console.log([...scores.values()]); // [90,85,88]
console.log([...scores.entries()]); // [['语文',90],...]

逐段拆解：

  • new Map([[k,v],[k,v]])：可以用“键值对数组”一次性初始化。
  • for (const [subject, score] of scores)：直接遍历 Map，每项是 [键, 值]，顺手解构。
  • keys() / values() / entries() 分别返回键、值、键值对的迭代器。

与普通对象互转

// 文件：code/ch07/04-map.js（续）
const obj = Object.fromEntries(scores); // Map -> 普通对象
const back = new Map(Object.entries(obj)); // 普通对象 -> Map

Object.fromEntries 把键值对序列变成对象；Object.entries 反过来。两者是 Map 和对象之间的桥。

⚠️ 对象的坑演示：

// 文件：code/ch07/04-map.js（续）
const plain = {};
plain[1] = "a";
console.log(Object.keys(plain)); // ['1'] —— 数字键 1 被悄悄转成字符串 '1'
console.log("toString" in {}); // true —— 来自原型链，容易误判

对象把数字键 1 转成了字符串 '1'；而 "toString" in {} 是 true（继承自原型），如果你用对象当“字典”，很容易被这些继承来的键坑到。Map 没有这些问题——它只装你放进去的东西。

7.5 Set：自动去重的值集合

Set 是“值的集合”，最大特点是每个值唯一，重复的自动忽略。它最常见的用途就是去重。

// 文件：code/ch07/05-set.js
const nums = [1, 2, 2, 3, 3, 3, 4];
const unique = new Set(nums);
console.log(unique.size); // 4
console.log([...unique]); // [1,2,3,4]
console.log([...new Set(nums)]); // 一行去重的经典写法

逐段拆解： new Set(可迭代对象) 会遍历传入的值逐个加入，重复的被丢弃。[...new Set(数组)] 是数组去重的最优雅写法，请记住它。注意这里又用到了迭代协议：new Set(...) 接受任何可迭代对象，[...set] 也依赖 Set 的可迭代性。

// 文件：code/ch07/05-set.js（续）
const s = new Set();
s.add("a").add("b").add("a"); // add 返回 Set 自身，可链式；重复 "a" 被忽略
console.log(s.has("a")); // true
s.delete("b");
console.log([...s]); // ['a']

add / has / delete 对应“加/查/删”，add 返回自身所以能链式调用。

集合运算：并、交、差

配合扩展运算符和 filter，Set 能优雅地做数学上的集合运算：

// 文件：code/ch07/05-set.js（续）
const A = new Set([1, 2, 3, 4]);
const B = new Set([3, 4, 5, 6]);

const union = new Set([...A, ...B]); // 并集
console.log([...union]); // [1,2,3,4,5,6]

const intersection = new Set([...A].filter((x) => B.has(x))); // 交集
console.log([...intersection]); // [3,4]

const difference = new Set([...A].filter((x) => !B.has(x))); // 差集 A-B
console.log([...difference]); // [1,2]

逐段拆解：

  • 并集：把 A、B 全展开塞进新 Set，重复的自然被去掉。
  • 交集：把 A 变成数组，筛出“也在 B 里”（B.has(x)）的。
  • 差集 A−B：筛出“不在 B 里”（!B.has(x)）的。

💡 用 Set.has 做“是否存在”判断是 O(1)（常数时间），比 数组.includes（要从头找，O(n)）快得多。当你要频繁判断“某个值在不在集合里”，Set 是首选。

7.6 WeakMap 与 WeakSet：弱引用

Map 和 Set 会牢牢抓住里面的键/值——只要它们在 Map 里，就永远不会被垃圾回收（garbage collection，GC）。这有时会造成内存泄漏。WeakMap / WeakSet 是它们的“弱引用（weak reference）”版本，专治这个问题。

先理解“弱引用”：普通引用像用绳子拴住一个对象，只要绳子在，对象就不会被回收；弱引用像“只是看一眼、记个位置”，如果这个对象在别的地方已经没人用了，垃圾回收器可以随时把它回收掉，连同 WeakMap 里的记录一起清除。

// 文件：code/ch07/06-weak.js
const wm = new WeakMap();
let key = { id: 1 };
wm.set(key, "关联数据");
console.log(wm.get(key)); // "关联数据"
console.log(wm.has(key)); // true

WeakMap 用法和 Map 类似（set/get/has/delete），但有三条硬限制：

  1. 键必须是对象（不能是字符串、数字等原始值）。
  2. 是弱引用：键对象在别处没人用了，就会被 GC 连带清掉。
  3. 不可迭代：没有 size，没有 keys()/values()，不能 for...of。

📌 为什么不可迭代？ 因为它的内容随时可能被垃圾回收清掉，遍历结果无法确定——你刚数到一半，几个键可能就被回收没了。为了避免这种不确定行为，语言干脆禁止遍历 WeakMap。这不是缺陷，是弱引用的必然结果。

用途一：给对象存私有数据

// 文件：code/ch07/06-weak.js（续）
const privateData = new WeakMap();
class BankAccount {
  constructor(balance) {
    privateData.set(this, { balance }); // 余额藏在模块级 WeakMap 里
  }
  deposit(amount) {
    privateData.get(this).balance += amount;
  }
  getBalance() {
    return privateData.get(this).balance;
  }
}
const acc = new BankAccount(100);
acc.deposit(50);
console.log(acc.getBalance()); // 150
console.log(acc.balance); // undefined —— 外部拿不到真正的余额

逐段拆解： 真正的余额存在模块级的 privateData 这张 WeakMap 里，以 this（实例）为键。外部代码拿不到 privateData，所以拿不到余额（acc.balance 是 undefined），实现了真正的私有。而且——当某个账户实例被销毁、没人再引用它时，它在 privateData 里的记录会被自动清除，不会造成内存泄漏。这正是必须用 WeakMap 而不是普通 Map 的原因。

用途二：缓存

// 文件：code/ch07/06-weak.js（续）
const cache = new WeakMap();
function compute(objInput) {
  if (cache.has(objInput)) return cache.get(objInput) + "（来自缓存）";
  const result = `对 ${objInput.name} 的计算结果`;
  cache.set(objInput, result);
  return result;
}
const config = { name: "配置A" };
console.log(compute(config)); // 首次计算
console.log(compute(config)); // 命中缓存

逐段拆解： 以对象为 key 缓存计算结果，第二次直接命中。妙处在于：当 config 这个对象在程序里彻底没人用了，缓存条目会被自动清理，你不用手动 delete，缓存不会无限膨胀。

WeakSet

WeakSet 同理：只能装对象、弱引用、不可迭代，常用于给对象“打标记”：

// 文件：code/ch07/06-weak.js（续）
const visited = new WeakSet();
const node = { value: 1 };
visited.add(node);
console.log(visited.has(node)); // true —— 标记“这个节点访问过了”

比如遍历图结构时，用 WeakSet 记录“访问过的节点”，避免重复处理，又不担心遍历完后这些节点无法被回收。

常见坑与最佳实践

1. for...of vs for...in 别搞混。

  • for...of 走迭代协议，遍历值，用于数组、字符串、Map、Set 等可迭代对象；不能直接用于普通对象。
  • for...in 遍历对象可枚举的键（包括继承来的），用于对象。遍历数组时它给的是字符串下标且会扫到自定义属性，通常不该用它遍历数组。

2. Symbol 不能被隐式转成字符串。 "" + Symbol() 会 TypeError。要转得显式：String(sym) 或 sym.description。

3. 生成器的第一次 next(v) 传参无效。 因为此时还没有任何 yield 在等待接收，传入的值被丢弃。要给生成器喂初值，得在创建时用参数，或先空跑一次 next()。

4. 一个迭代器只能用一次。 迭代器走到 done: true 就“用尽”了，再遍历得重新领一个。所以下面是坑：

const gen = rangeGen(1, 3);
console.log([...gen]); // [1,2,3]
console.log([...gen]); // [] —— 已经用尽了！

如果需要反复遍历，遍历可迭代对象（每次 for...of 会重新领迭代器），而不是缓存单个迭代器。

5. 何时用 Map / Set，何时用对象 / 数组？

  • 键不是字符串/Symbol（比如用对象当键），或键是运行时动态、数量频繁增删，或需要保证顺序和 size → 用 Map。
  • 只是固定结构的一条记录、需要 JSON 序列化、需要方法 → 用对象。
  • 需要去重、需要频繁判断“存在与否” → 用 Set。
  • 需要按下标、需要顺序、允许重复、要用 map/filter/reduce → 用数组。

6. 需要“对象存活时自动清理关联数据”就用 WeakMap/WeakSet（私有数据、缓存、标记）；需要遍历、需要 size、键是原始值，就用 Map/Set。

本章小项目：分页数据流

现在把生成器、Map、Set 揉进一个实用场景：分页数据流。

真实世界里数据常常是“分页”的：接口一次只给你一页（比如 10 条），要更多得翻页。调用方却往往只想“像遍历一个普通列表一样”把数据一条条拿出来，不想关心翻页细节；有时甚至只想看前几条。生成器的惰性天生适合这个需求。

完整代码见 code/ch07/paginated-stream.js，运行：node paginated-stream.js。

第一步：模拟数据源与“按页拉取”

// 文件：code/ch07/paginated-stream.js
const PRODUCTS = [
  { id: 1, name: "苹果", category: "水果", tags: ["生鲜", "热销"] },
  // ... 共 10 条，见完整文件
];

function fetchPage(pageIndex, pageSize) {
  const start = pageIndex * pageSize;
  const items = PRODUCTS.slice(start, start + pageSize);
  const hasNext = start + pageSize < PRODUCTS.length;
  return { items, hasNext };
}

逐段拆解： fetchPage 模拟“拉取第 N 页”：从大数组里切出这一页，并告诉调用方 hasNext（后面还有没有页）。真实项目里这一步通常是一次网络请求（返回 Promise），下一章学了异步后，你可以把它换成真正的 fetch——结构一模一样。

第二步：核心生成器，把“分页”变成“一项项 yield”

// 文件：code/ch07/paginated-stream.js（续）
function* paginatedStream(pageSize = 3) {
  let pageIndex = 0;
  let hasNext = true;
  while (hasNext) {
    const page = fetchPage(pageIndex, pageSize);
    console.log(`  [拉取第 ${pageIndex + 1} 页，共 ${page.items.length} 项]`);
    for (const item of page.items) {
      yield item; // 逐项交出去，把控制权还给调用方
    }
    hasNext = page.hasNext;
    pageIndex++;
  }
}

逐段拆解：

  • 外层 while (hasNext)：一页页地拉，直到没有下一页。
  • 里面 for...of page.items + yield item：把当前页的每一项逐个 yield 出去。
  • 那句 console.log("[拉取第 N 页...]") 是演示用的探针：它只在“真正去拉某一页”时才打印。看它什么时候打印，就能直观感受到惰性。
  • 关键：yield 会暂停。调用方拿到一项、处理完、再要下一项时，生成器才继续；只有当前页 yield 完了，才会绕回 while 顶部去拉下一页。没人要，就不拉。

第三步：只取前 N 项（惰性工具）

// 文件：code/ch07/paginated-stream.js（续）
function* take(iterable, count) {
  let i = 0;
  for (const item of iterable) {
    if (i >= count) return;
    yield item;
    i++;
  }
}

take 从任意可迭代对象里“惰性地”取前 count 项，取够就 return 停止遍历。因为它一停，上游的 paginatedStream 也就跟着停了。

第四步：Map 分组统计 + Set 去重

// 文件：code/ch07/paginated-stream.js（续）
function groupByCategory(iterable) {
  const groups = new Map();
  for (const product of iterable) {
    if (!groups.has(product.category)) groups.set(product.category, []);
    groups.get(product.category).push(product.name);
  }
  return groups;
}

function collectTags(iterable) {
  const tags = new Set();
  for (const product of iterable) {
    for (const tag of product.tags) tags.add(tag);
  }
  return tags;
}

逐段拆解：

  • groupByCategory 用 Map 把商品按分类归拢：键是分类名，值是该类商品名的数组。“没这个键就先建个空数组，再往里 push”是用 Map 做分组统计的经典模板。
  • collectTags 用 Set 收集所有出现过的标签，重复的自动去重。

运行效果：亲眼看到惰性

// 文件：code/ch07/paginated-stream.js（续，节选自 main）
console.log("=== A. 遍历全部 ===");
for (const product of paginatedStream(3)) {
  console.log("  ->", product.id, product.name);
}

console.log("=== B. 只取前 2 项（惰性：只拉取第 1 页！） ===");
for (const product of take(paginatedStream(3), 2)) {
  console.log("  ->", product.id, product.name);
}

运行 node paginated-stream.js，重点看 B 部分的输出：

=== B. 只取前 2 项（惰性：只拉取第 1 页！） ===
  [拉取第 1 页，共 3 项]
  -> 1 苹果
  -> 2 香蕉

📌 这就是本项目的高光时刻：明明数据有 4 页，但因为我们只要前 2 项，take 拿够就停，于是只有第 1 页被拉取，第 2、3、4 页的 fetchPage 根本没被调用！如果这是真实的网络请求，你就实实在在省下了 3 次请求。这正是生成器惰性的实用价值——描述一个可能很大的数据流，却只为你真正用到的部分付出代价。

C、D 部分则展示了用 Map 分组、用 Set 去重的结果（它们需要看完全部数据，所以会拉完 4 页）。

小结与自测

这一章我们打通了 JavaScript 的“迭代”体系和现代集合类型：

  • Symbol 是绝对唯一的原始值，常作对象键避免冲突；Symbol.for 走全局注册表可跨模块共享；知名 Symbol（尤其 Symbol.iterator）是语言的协议钩子。
  • 迭代协议是 for...of、扩展、解构、Array.from 的共同底座：可迭代对象有 [Symbol.iterator]() 返回迭代器，迭代器有 next() 返回 {value, done}。我们手写了一个可迭代对象。
  • 生成器 function* + yield 是写迭代器的简单模式，能暂停/恢复，实现惰性求值与无限序列，还能通过 next(v) 双向通信；yield* 委托给另一个可迭代对象。
  • Map：任意类型键、保序、有 size、可迭代，是比对象更纯粹的字典。Set：自动去重、O(1) 查存在、能做并交差。
  • WeakMap / WeakSet：弱引用、键必须是对象、不可迭代（因内容随时可能被 GC 清除），适合私有数据、缓存、打标记。
  • 小项目用生成器实现了惰性分页数据流，亲眼见证“只取前 N 项时不多拉一页”，并用 Map/Set 完成分组与去重。

  5. 为什么 paginatedStream 里“只取前 2 项”时，后面几页的 fetchPage 不会被调用？
  6. 什么时候该用 Map 而不是普通对象？什么时候该用 Set 而不是数组？

（参考答案：1. 前者 false——每个 Symbol() 都新建、永不相等；后者 true——Symbol.for 按名字从全局注册表返回同一个。2. 必须实现 [Symbol.iterator]()，它返回一个迭代器；迭代器要有 next() 方法，每次返回 {value, done}。3. 不会立刻执行，只返回一个生成器对象；直到第一次 .next()（或被 for...of/展开触发）才开始执行，且执行到第一个 yield 就暂停。4. [1,2] 和 []——迭代器一次性用尽，第二次已无剩余。5. 因为生成器惰性：take 拿够 2 项就 return，上游 for...of 随之停止，while 循环没绕回去，后面页的 fetchPage 根本没被调用。6. 键非字符串/Symbol、需保序与 size、动态频繁增删时用 Map；需去重或频繁判断存在性时用 Set。7. 因为内容是弱引用、随时可能被垃圾回收清除，遍历结果不确定，所以禁止遍历、无 size；弱引用只对“可被回收的对象”有意义，原始值不参与 GC，故键必须是对象。）
