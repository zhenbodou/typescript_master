JavaScript & TypeScript 从入门到精通

第 8 章 · 异步编程：事件循环、Promise 与 async/await

本章目标

前七章我们写的代码几乎都是**同步（synchronous）**的：一行执行完，才轮到下一行。但真实世界里，程序常常要“等“——等网络请求返回、等文件读完、等定时器到点、等用户点击。如果“等“的时候整个程序卡死，那体验将糟糕透顶。

**异步编程（asynchronous programming）**就是解决“如何优雅地等待“的技术，它也是 JavaScript 公认最烧脑、面试最爱问、初学者最容易踩坑的一块。这一章我们把它讲透。

学完本章你会：

  • 分清同步/异步、阻塞/非阻塞，理解为什么“单线程“的 JS 还能“同时“处理很多事。
  • 彻底搞懂 事件循环（event loop）：调用栈、宏任务（macrotask）队列、微任务（microtask）队列到底谁先谁后。经典输出顺序题一次讲清。
  • 认识**回调（callback）**和它带来的“回调地狱（callback hell）“。
  • 掌握 Promise：三种状态、then/catch/finally、链式调用、错误传播，以及 all / allSettled / race / any 四大组合器。
  • 用 async/await 把异步代码写得像同步一样清爽，用 try/catch 处理异步错误，分清串行与并行。
  • 避开四大常见坑：忘记 await、未捕获的 rejection、把可并行的请求写成串行、在 forEach 里用 async。
  • 写出一个可复用的 delay(ms) 工具，并动手实现本章小项目——并发请求限流器 asyncPool。

我们从最基础的“同步和异步到底差在哪“讲起。

8.1 同步 vs 异步、阻塞 vs 非阻塞

想象你去一家餐厅吃饭。

  • 同步阻塞（synchronous & blocking）：你在柜台点了一份现做的牛排，然后站在柜台前一动不动地等，直到牛排做好递给你，你才离开柜台去做别的事。这期间柜台被你霸占，后面的人干瞪眼。
  • 异步非阻塞（asynchronous & non-blocking）：你点完餐，服务员给你一个取餐号，你回座位刷手机、聊天，牛排好了会叫号通知你去取。这期间柜台没被占用，别人也能点餐。

代码里也是同一回事：

// 文件：code/ch08/01-sync-vs-async.js
// 1) 同步（阻塞）：下面这行会"卡住" 1 秒，期间程序什么也干不了
function sleepSyncBlocking(ms) {
  const end = Date.now() + ms;
  while (Date.now() < end) {
    // 空转，霸占 CPU，谁也别想动
  }
}

console.log("同步版本开始");
console.log("A");
sleepSyncBlocking(1000); // 阻塞 1 秒
console.log("B（1 秒后才打印，期间程序完全卡死）");
console.log("同步版本结束");

逐段拆解：

  • sleepSyncBlocking 用一个 while 空循环硬生生地占着 CPU 一秒。这一秒里，JavaScript 什么别的都做不了——它是“同步阻塞“的典型。
  • 所以 A 打印后，要苦等一秒，B 才出来。

再看异步版本：

// 文件：code/ch08/01-sync-vs-async.js（续）
console.log("异步版本开始");
console.log("C");
setTimeout(() => {
  console.log("D（1 秒后打印，但没有卡住主线程）");
}, 1000);
console.log("E（立刻打印，不用等 D）");
console.log("异步版本结束（注意 D 还没打印，它在未来）");

逐段拆解：

  • setTimeout(fn, 1000) 的意思是“1 秒后再执行 fn“，但它不会卡住当前代码。它把 fn 登记给“计时器”，然后立刻返回，代码继续往下跑。
  • 所以打印顺序是 C → E → （一秒后）D。注意 D 排在“异步版本结束“后面——因为它属于“未来“。

运行 node 01-sync-vs-async.js，实测输出正是：

C
E（立刻打印，不用等 D）
异步版本结束（注意 D 还没打印，它在未来）
D（1 秒后打印，但没有卡住主线程）

📌 同步：代码按书写顺序一行行执行，前一件事没完成，后面等着。异步：把“耗时的事“交给别人（定时器、系统 I/O、网络），登记好“完成后做什么“，当前代码继续往下跑，不傻等。

💡 术语区分：阻塞/非阻塞描述“发起调用的一方要不要停下来等“；同步/异步描述“结果是当场返回还是稍后通知“。日常里我们常把“同步阻塞“和“异步非阻塞“当成两套组合来记，够用了。

8.2 单线程为什么还能并发？——事件循环登场

新手最大的困惑是：JavaScript 是单线程（single-threaded）的——同一时刻只有一条“执行线“在跑代码。那它凭什么能“同时“处理定时器、网络请求、用户点击？

答案是：耗时的等待不是 JS 自己干的。

  • 你调 setTimeout，计时是由运行环境（浏览器/Node）在后台的计时器完成的，不占用 JS 线程。
  • 你发网络请求，实际的网络收发是操作系统 / 底层线程池在做的，不占用 JS 线程。
  • 等这些后台工作完成，环境会把你登记的“回调函数“放进一个队列，排队等 JS 线程有空了来执行。

那“JS 线程有空了来取队列“这个循环，就叫 事件循环（event loop）。它是这样一个不知疲倦的循环：

  1. 调用栈（call stack）：执行当前的同步代码。函数调用进栈，返回出栈。
  2. 栈空了（当前这批同步代码跑完了），就去任务队列里取一个排队的回调，压进栈里执行。
  3. 执行完再看队列，如此往复——“循环“由此得名。

用餐厅比喻：JS 线程是唯一的一个厨师。他不会站着等牛排烤熟（那样太浪费），而是把牛排丢进烤箱（交给环境），转身去处理下一张订单。烤箱“叮“一声（回调入队），厨师忙完手头的活（栈空）后，再回来处理这份烤好的牛排。一个厨师，靠“不傻等“，服务了一整个餐厅——这就是单线程并发的秘密。

⚠️ 正因为只有一个厨师，你千万别让他站着空等（比如 8.1 里的 while 空转）。一旦某段同步代码跑很久，整个页面/服务就“卡死“，所有回调都得排队干瞪眼。这叫“阻塞事件循环“，是性能杀手。

8.3 宏任务与微任务：谁先执行？

事件循环里其实有两种队列，优先级不同，这是那道经典面试题的核心。

  • 宏任务队列（macrotask queue，也叫 task queue）：setTimeout、setInterval、I/O 回调、setImmediate（Node）等属于这里。
  • 微任务队列（microtask queue）：Promise.then/catch/finally 的回调、queueMicrotask、Node 的 process.nextTick 属于这里。

关键规则，背下来：

📌 每执行完一个宏任务（包括最开始那段主脚本，它本身就是第一个宏任务），事件循环会立刻把微任务队列全部清空，然后才去取下一个宏任务。

换句话说：微任务永远排在下一个宏任务之前。而且微任务清空过程中若又产生新微任务，也会在本轮一并清完，才轮到宏任务。

看这段经典代码：

// 文件：code/ch08/02-event-loop.js
console.log("1 —— 同步：脚本一开始就执行");

setTimeout(() => {
  console.log("5 —— 宏任务：setTimeout 回调（任务队列）");
}, 0);

Promise.resolve().then(() => {
  console.log("3 —— 微任务：第一个 Promise.then");
});

queueMicrotask(() => {
  console.log("4 —— 微任务：queueMicrotask（和 then 同属微任务队列）");
});

console.log("2 —— 同步：脚本最后一行");

运行 node 02-event-loop.js，实测输出：

1 —— 同步：脚本一开始就执行
2 —— 同步：脚本最后一行
3 —— 微任务：第一个 Promise.then
4 —— 微任务：queueMicrotask（和 then 同属微任务队列）
5 —— 宏任务：setTimeout 回调（任务队列）

逐段拆解，跟着事件循环走一遍：

  1. 主脚本作为第一个宏任务开始执行（同步部分）。
  2. console.log("1...") → 立刻打印 1。
  3. 遇到 setTimeout(..., 0)：把回调登记到宏任务队列（哪怕是 0 毫秒，也得排队）。
  4. 遇到 Promise.resolve().then(...)：把回调登记到微任务队列。
  5. 遇到 queueMicrotask(...)：同样登记到微任务队列（排在上面那个 then 后面）。
  6. console.log("2...") → 立刻打印 2。主脚本这个宏任务执行完了，栈空。
  7. 事件循环先清空微任务队列：依次执行，打印 3、4。
  8. 微任务清空后，才去取宏任务队列：执行 setTimeout 回调，打印 5。

💡 记忆口诀：同步 → 微任务 → 宏任务。同一批同步代码里，Promise.then 一定比 setTimeout 先执行，哪怕后者是 0 毫秒。

Node 里的 process.nextTick

Node 还有一个 process.nextTick，它属于一个独立于 Promise 微任务的队列。

// 文件：code/ch08/03-microtask-order.js
console.log("start"); // 同步

setTimeout(() => console.log("setTimeout（宏任务）"), 0);

Promise.resolve().then(() => {
  console.log("promise 1（微任务）");
  // 在一个微任务里再注册微任务，会插到微任务队列末尾，仍在宏任务之前被清空
  Promise.resolve().then(() => console.log("promise 1.1（嵌套微任务）"));
});

process.nextTick(() => console.log("nextTick（Node 独有队列）"));

Promise.resolve().then(() => console.log("promise 2（微任务）"));

console.log("end"); // 同步

运行 node 03-microtask-order.js，本机（Node 24）实测输出：

start
end
promise 1（微任务）
promise 2（微任务）
promise 1.1（嵌套微任务）
nextTick（Node 独有队列）
setTimeout（宏任务）

逐段拆解：

  • start、end 是同步代码，最先打印。
  • 然后清空微任务：promise 1、promise 2。注意 promise 1 内部又注册了 promise 1.1——这个新微任务会追加到本轮微任务队列末尾，所以在 promise 2 之后、但仍在宏任务之前执行。
  • nextTick 也在所有宏任务之前跑完。
  • 最后才是 setTimeout 这个宏任务。

⚠️ 陷阱：网上流传“process.nextTick 永远优先于 Promise“。这只在“从同一次回调内部同时排队“时严格成立；在顶层脚本里同时排队时，不同 Node 版本表现可能不同（本机实测就是 Promise 先、nextTick 后）。所以不要依赖 nextTick 和 Promise 之间的相对顺序。你能可靠依赖的只有一条：它们都是微任务，都在任何 setTimeout 宏任务之前执行。

8.4 回调与回调地狱

在 Promise 出现之前，异步的“完成后做什么“全靠回调函数（callback）：你把一个函数传给异步操作，它做完了就调用你这个函数。

Node 早期约定了一种“错误优先（error-first）“回调风格：回调的第一个参数是错误，成功时它为 null，第二个参数才是数据。

// 文件：code/ch08/04-callback-hell.js
// 模拟一个"异步读取"：ms 毫秒后，用回调把结果交出去
function fakeRead(name, ms, cb) {
  setTimeout(() => {
    if (!name) {
      cb(new Error("名字不能为空")); // 出错时把 error 传回去
      return;
    }
    cb(null, `【${name} 的内容】`); // 成功：error 为 null
  }, ms);
}

单个回调还好，可一旦多个异步步骤有先后依赖（第二步要用第一步的结果），就只能一层层往里嵌：

// 文件：code/ch08/04-callback-hell.js（续）
fakeRead("a.txt", 100, (err1, data1) => {
  if (err1) return console.error(err1);
  console.log("读到：", data1);
  fakeRead("b.txt", 100, (err2, data2) => {
    if (err2) return console.error(err2);
    console.log("读到：", data2);
    fakeRead("c.txt", 100, (err3, data3) => {
      if (err3) return console.error(err3);
      console.log("读到：", data3);
      console.log("三步都完成了——但代码已经缩进成金字塔（callback hell）");
    });
  });
});

逐段拆解，看看它烂在哪：

  • 层层缩进：每多一步就多一层嵌套，代码向右倾斜成“厄运金字塔（pyramid of doom）“，读起来痛苦。
  • 错误处理重复：每一层都要写一遍 if (err) return ...，又啰嗦又容易漏。
  • 无法用返回值：结果只能靠回调层层往里传，没法像同步代码那样 const x = ...。
  • 组合困难：想“三个一起并行然后汇总“？用纯回调写会更乱。

这就是 回调地狱（callback hell）。Promise 和 async/await 就是来拯救它的。接下来我们用它们把这段金字塔拉平。

8.5 Promise：异步的“承诺“

Promise（承诺） 是一个代表“未来某个值“的对象。可以把它想成餐厅那张取餐号：你现在拿到的不是牛排本身，而是一张“将来能换牛排“的凭证。这张凭证有三种状态：

  • pending（进行中）：还没出结果（牛排在做）。
  • fulfilled（已成功）：拿到了值（牛排做好了）。通过 resolve(值) 进入此状态。
  • rejected（已失败）：出错了（牛排烤糊了）。通过 reject(错误) 进入此状态。

📌 Promise 的状态只能从 pending 变一次，要么变 fulfilled，要么变 rejected，之后永不再变。这个“一锤定音“的特性让它可预测。

创建与消费 Promise

// 文件：code/ch08/05-promise-basics.js
function fakeRead(name, ms) {
  return new Promise((resolve, reject) => {
    // executor 会立即同步执行；把异步结果通过 resolve/reject 交出去
    setTimeout(() => {
      if (!name) reject(new Error("名字不能为空"));
      else resolve(`【${name} 的内容】`);
    }, ms);
  });
}

fakeRead("a.txt", 100)
  .then((data) => {
    console.log("① then 拿到数据：", data);
  })
  .catch((err) => {
    console.log("① catch：", err.message);
  })
  .finally(() => {
    console.log("① finally：无论成败都会执行（收尾）");
  });

逐段拆解：

  • new Promise((resolve, reject) => {...})：传给它的函数叫 executor（执行器），它会立即同步执行。你在里面启动异步操作，完成时调用 resolve(值)（成功）或 reject(错误)（失败）。
  • .then(onFulfilled)：注册“成功后做什么“，参数就是 resolve 传的值。
  • .catch(onRejected)：注册“失败后做什么“，参数就是 reject 传的错误。
  • .finally(fn)：无论成功失败都会执行，适合做收尾（关加载动画、清理资源）。它不接收结果值。

链式调用：拉平金字塔

then 的返回值大有讲究：你在 then 里 return 什么，就成为下一个 then 的输入。特别地，如果你 return 一个 Promise，链条会等它完成再往下走。这正是拉平回调地狱的关键：

// 文件：code/ch08/05-promise-basics.js（续）
fakeRead("a.txt", 100)
  .then((d1) => {
    console.log("链 → 读到：", d1);
    return fakeRead("b.txt", 100); // 返回新 Promise，链会等它
  })
  .then((d2) => {
    console.log("链 → 读到：", d2);
    return fakeRead("c.txt", 100);
  })
  .then((d3) => {
    console.log("链 → 读到：", d3);
    console.log("三步完成——注意这次是'平的'，不再是金字塔");
  })
  .catch((err) => {
    // 链条上任何一环 reject 或 throw，都会"跳"到这里
    console.log("链 → 出错：", err.message);
  });

逐段拆解：

  • 三个 then 平级排列，不再层层嵌套——同样的三步依赖，代码从金字塔变成了一条直线。
  • 每个 then 里 return 下一个 fakeRead(...)，链条会自动等待，顺序执行。

错误传播：一个 catch 兜底

Promise 链的错误处理比回调优雅得多：链条上任何一环出错（reject 或 throw），都会跳过后面所有 then，直接进入最近的 catch：

// 文件：code/ch08/05-promise-basics.js（续）
fakeRead("", 50) // 空名字 → reject
  .then((d) => console.log("这行不会执行", d))
  .then((d) => console.log("这行也不会执行", d))
  .catch((err) => console.log("④ 错误一路传播到 catch：", err.message));

逐段拆解： fakeRead("") 因名字为空而 reject。于是中间两个 then 全被跳过，错误“冒泡“到末尾的 catch。一个 catch 就能兜住整条链的错误——回想 8.4 里每层都要写 if (err)，高下立判。

⚠️ 运行 node 05-promise-basics.js 时，由于各条链的定时器时长不同，它们的输出是交错的（比如那条最先 reject 的 ④ 会较早出现）。这正好印证了：多条 Promise 链是“并发“推进的，谁的定时器先到谁先打印。别被交错的顺序搞晕——每条链内部的顺序始终是对的。

📌 别忘了给每条 Promise 链都接上 .catch。没有 .catch 的 reject 会变成“未捕获的 rejection“（后面 8.8 会讲）。

8.6 Promise 的四大组合器

当你有多个 Promise 要一起处理时，Promise 提供了四个静态方法。假设有这么个任务生成器：

// 文件：code/ch08/06-promise-combinators.js
function task(name, ms, ok = true) {
  return new Promise((resolve, reject) => {
    setTimeout(() => {
      if (ok) resolve(`${name} 成功(${ms}ms)`);
      else reject(new Error(`${name} 失败(${ms}ms)`));
    }, ms);
  });
}

Promise.all：全部成功才成功，任一失败即整体失败。

// 文件：code/ch08/06-promise-combinators.js（续）
const allResult = await Promise.all([task("A", 100), task("B", 200), task("C", 50)]);
console.log("① all（全部成功）：", allResult);
// → [ 'A 成功(100ms)', 'B 成功(200ms)', 'C 成功(50ms)' ]
  • 返回一个结果数组，顺序对应输入（哪怕 C 最先完成，它仍排在结果第 3 位）。
  • 只要有一个 reject，all 会立即整体失败（短路），不等其他的。适合“缺一不可“的场景。

Promise.allSettled：等所有都“落定“，永不失败，逐个报告状态。

// 文件：code/ch08/06-promise-combinators.js（续）
const settled = await Promise.allSettled([task("F", 80), task("G", 40, false)]);
for (const r of settled) {
  if (r.status === "fulfilled") console.log("   ✅", r.value);
  else console.log("   ❌", r.reason.message);
}
  • 它永远成功，返回每个任务的 { status, value } 或 { status, reason }。适合“我要知道每个的结果，成功失败都要“的场景（如批量上传，要汇报哪些成了哪些败了）。

Promise.race：谁先“落定“（无论成败）就用谁。

// 文件：code/ch08/06-promise-combinators.js（续）
const raced = await Promise.race([task("慢", 200), task("快", 30)]);
console.log("④ race（最先落定者胜出）：", raced); // → 快 成功(30ms)
  • “赛跑”：第一个落定的结果（成功或失败）就是 race 的结果。常用于加超时（把真实任务和一个“超时就 reject 的 Promise“一起赛跑）。

Promise.any：第一个“成功“者胜出；全失败才失败。

// 文件：code/ch08/06-promise-combinators.js（续）
const anyResult = await Promise.any([
  task("H", 100, false),
  task("I", 60), // 第一个成功的
  task("J", 30, false),
]);
console.log("⑤ any（第一个成功者胜出）：", anyResult); // → I 成功(60ms)
  • 和 race 的区别：any 会忽略失败，只要有一个成功就返回它；只有全部失败时才失败（抛 AggregateError）。适合“多个备用源，谁先成功用谁“。

运行 node 06-promise-combinators.js 可验证以上全部输出。

📌 一句话记忆：all=全成才成、allSettled=全等到底、race=先到者（不论成败）、any=先成功者。

8.7 async/await：像写同步一样写异步

Promise 已经很好，但链式 then 多了还是有点绕。async/await 是建立在 Promise 之上的语法糖，让异步代码读起来像同步代码。

两条规则：

  1. 在函数前加 async，它就变成 async 函数，一定返回 Promise（return 的值被自动包进 resolve，throw 的错误变成 reject）。
  2. 在 async 函数内部，可以用 await 后跟一个 Promise，它会**“暂停“这个函数**，直到 Promise 落定，然后把结果值“解包“出来。失败则当场“抛错”。
// 文件：code/ch08/07-async-await.js
import { delay } from "./08-delay.js";

async function readFile(name, ms) {
  await delay(ms); // await 会"暂停"这个函数，直到 delay 的 Promise 落定
  if (!name) throw new Error("名字不能为空"); // throw 让返回的 Promise 变成 reject
  return `【${name} 的内容】`;
}

逐段拆解：

  • readFile 是 async 函数，它返回的是 Promise。
  • await delay(ms)：暂停在这里等 delay 完成。注意这个“暂停“不阻塞事件循环——函数被挂起后，JS 线程会去处理别的任务，等 delay 好了再回来接着往下跑。
  • return 的字符串会成为这个 Promise 的成功值；throw 会成为失败原因。

用 try/catch 处理异步错误

async/await 最爽的一点：可以用你早就熟悉的 try/catch 抓异步错误（回想第 6 章说过普通 try/catch 抓不住 setTimeout 里的错误，而 await 打通了这一层）。

// 文件：code/ch08/07-async-await.js（续）
async function demoErrorHandling() {
  try {
    const data = await readFile("a.txt", 100);
    console.log("① 拿到：", data);
    await readFile("", 50); // 会抛错
    console.log("这行不会执行");
  } catch (e) {
    console.log("① try/catch 抓到异步错误：", e.message);
  }
}

逐段拆解： await readFile("", 50) 内部 throw 了。因为 await 会把 Promise 的 reject 转成一次真正的抛错，所以外层的 catch 能抓到它，就像同步代码一样自然。抓到后，try 里剩下的“这行不会执行“被跳过。

串行 vs 并行——最容易踩坑的地方

⚠️ 重点中的重点：await 会暂停等待。如果你一个接一个地 await 多个互不依赖的任务，它们会被串行化，白白浪费时间。

// 文件：code/ch08/07-async-await.js（续）
// 串行：一个 await 接一个，后一个必须等前一个完成
async function demoSerial() {
  const t0 = Date.now();
  const a = await readFile("a.txt", 300);
  const b = await readFile("b.txt", 300);
  const c = await readFile("c.txt", 300);
  console.log(`② 串行耗时：约 ${Date.now() - t0}ms（300+300+300≈900）`);
}

三个任务各 300ms，但因为串行 await，总耗时约 900ms（实测约 904ms）。可它们明明互不依赖，本该同时进行！正确做法是先一起发起，再用 Promise.all 一起等：

// 文件：code/ch08/07-async-await.js（续）
// 并行：三个任务同时发起，再用 Promise.all 一起等
async function demoParallel() {
  const t0 = Date.now();
  const [a, b, c] = await Promise.all([
    readFile("a.txt", 300),
    readFile("b.txt", 300),
    readFile("c.txt", 300),
  ]);
  console.log(`③ 并行耗时：约 ${Date.now() - t0}ms（三者同时跑≈300）`);
}

逐段拆解： 关键在于——Promise.all([...]) 里的三个 readFile(...) 是先被调用的，调用的那一刻它们的定时器就都启动了、三个任务已经在并行跑了。然后 await Promise.all 只是“一起等它们全好“。总耗时约 300ms（实测约 302ms），比串行快了三倍。

📌 判断准则：任务之间有依赖（后一个要用前一个的结果）就串行 await；互不依赖就并行 Promise.all。 运行 node 07-async-await.js 亲眼看这 900ms vs 300ms 的差距。

8.8 常见坑与最佳实践

这一节把初学者最常踩的四个坑集中演示。对应文件 code/ch08/09-pitfalls.js，运行可复现全部现象。

前置一个模拟异步函数：

// 文件：code/ch08/09-pitfalls.js
import { delay } from "./08-delay.js";

async function getUser(id) {
  await delay(50);
  return { id, name: `用户${id}` };
}

坑 1：忘记 await。 拿到的会是一个 Promise 对象，而不是你要的数据。

// 文件：code/ch08/09-pitfalls.js（续）
const user = getUser(1); // ⚠️ 忘了 await
console.log("坑1 忘记 await：", user); // → Promise { <pending> }，不是用户！
const userOk = await getUser(1); // ✅
console.log("坑1 正确：", userOk); // → { id: 1, name: '用户1' }

💡 如果你发现变量打印出来是 Promise { <pending> }，八成是漏了 await。

坑 2：在 forEach 里用 async。 Array.prototype.forEach 不会等待里面的 async 回调，导致“后续代码“提前跑，且无法收集结果。

// 文件：code/ch08/09-pitfalls.js（续）
const ids = [1, 2, 3];
ids.forEach(async (id) => {
  const u = await getUser(id);
  console.log("坑2 forEach 里拿到（会在'完成'之后才打印）：", u.name);
});
console.log("坑2 forEach 之后：这行会立刻执行，根本没等上面三个！");

实测里“forEach 之后“这行会先于三个用户打印出来——因为 forEach 发起了三个 async 回调就立刻返回了，压根没等。正确写法有两种：

// 文件：code/ch08/09-pitfalls.js（续）
// ✅ 正确 A：要串行、要按顺序等，用 for...of
for (const id of ids) {
  const u = await getUser(id);
  console.log("   拿到：", u.name);
}
// ✅ 正确 B：要并行、要收集结果，用 map + Promise.all
const users = await Promise.all(ids.map((id) => getUser(id)));
console.log("坑2 正确（map+Promise.all 并行）：", users.map((u) => u.name));

📌 口诀：要顺序等，用 for...of；要并行拿结果，用 map + Promise.all。永远别用 forEach 配 async。

坑 3：把可并行的请求写成了串行（就是 8.7 的问题，这里再强调）。

// 文件：code/ch08/09-pitfalls.js（续）
const a = await getUser(1); // 串行：等 50ms
const b = await getUser(2); // 再等 50ms → 一共约 100ms
// 改成并行：一共约 50ms
const [c, d] = await Promise.all([getUser(1), getUser(2)]);

坑 4：未捕获的 rejection（unhandled rejection）。 一个 Promise 失败了却没有任何 .catch 或 try/catch 接住，Node 会报 UnhandledPromiseRejection 甚至让进程退出。

// 文件：code/ch08/09-pitfalls.js（续）
getUser(1)
  .then(() => {
    throw new Error("链里抛了错但没人接");
  })
  .catch((e) => console.log("坑4 正确：用 .catch 兜住了：", e.message));

💡 最佳实践清单：

  • 每条 Promise 链都要有 .catch；每个 await 要么在 try/catch 里，要么调用方负责兜错。
  • 互不依赖的任务用 Promise.all 并行，别串行。
  • 需要“成功失败都收集“用 Promise.allSettled。
  • 不要在 forEach/map 的回调里指望 await 能“等住“外层——forEach 不等，map 要配 Promise.all。
  • 别在主线程写 CPU 密集的同步长循环，那会阻塞事件循环。

一个可复用的 delay 工具

前面示例反复用到的 delay，值得单独封装成工具：

// 文件：code/ch08/08-delay.js
// delay(ms)：返回一个"ms 毫秒后自动 resolve"的 Promise
export function delay(ms, value) {
  return new Promise((resolve) => {
    setTimeout(() => resolve(value), ms);
  });
}

export const sleep = delay; // 别名，很多人习惯叫 sleep

// 可拒绝变体：ms 毫秒后 reject，常用来给别的 Promise 加超时
export function delayReject(ms, reason = new Error("超时")) {
  return new Promise((_, reject) => {
    setTimeout(() => reject(reason), ms);
  });
}

逐段拆解：

  • delay(ms) 把 setTimeout 包成 Promise，于是能在 async 函数里写 await delay(1000) 优雅地“睡一觉“。第二个参数 value 让它还能顺便携带一个返回值。
  • delayReject 是“到点就失败“的版本，配合 Promise.race 就能给任意任务加超时（本章练习会用到）。

8.9 本章小项目：并发请求限流器

场景：你要请求 100 个接口，但如果一口气 100 个并发发出去，可能会把服务器打爆、或触发对方的限流。你希望同时最多只有 limit 个在跑，完成一个就补一个进来，像流水线一样，直到全部做完——并且最终结果顺序要对应输入。

我们来实现 asyncPool(limit, items, iteratorFn)。

核心思路：固定数量的“工人“抢任务

不要去想“怎么控制并发数“这种抽象问题，换个具体画面：雇 limit 个工人（worker），面前放一排任务。每个工人的行为很简单——做完手头的，就回头认领队列里下一个还没人做的任务，直到没任务了才收工。任意时刻正在干活的工人不超过 limit 个，并发数自然就被限制住了。

// 文件：code/ch08/async-pool.js
export async function asyncPool(limit, items, iteratorFn) {
  const results = new Array(items.length); // 预分配，保证结果顺序 = 输入顺序
  let nextIndex = 0; // 下一个待认领的任务下标（共享游标）

  // 一个 worker：不断从队列里"认领"下一个任务来做，直到没活了
  async function worker() {
    while (nextIndex < items.length) {
      const current = nextIndex++; // 认领当前任务，并把游标推进
      results[current] = await iteratorFn(items[current], current);
    }
  }

  // 启动 min(limit, 任务数) 个 worker，让它们并行地抢任务
  const workerCount = Math.min(limit, items.length);
  const workers = [];
  for (let i = 0; i < workerCount; i++) {
    workers.push(worker());
  }

  await Promise.all(workers); // 等所有 worker 都干完
  return results;
}

逐段拆解——并发控制的原理：

  • results = new Array(items.length)：预先分配结果数组。每个任务把结果写回自己的下标 results[current]，所以无论完成顺序如何，最终结果顺序永远对应输入。这是“顺序对应“的关键。
  • nextIndex 是所有 worker 共享的游标，指向“下一个还没被认领的任务“。
  • worker 里的 while 循环：只要还有任务（nextIndex < items.length），就 nextIndex++ 认领一个任务并推进游标，然后 await iteratorFn 去做它。做完继续 while，认领下一个。没任务了，while 退出，这个 worker 收工。
  • 为什么这样就限流了？因为我们只启动了 limit 个 worker。同一时刻，正卡在 await iteratorFn(...) 上（即正在执行任务）的，最多就是这 limit 个 worker。完成一个，那个 worker 立刻回到 while 认领下一个——“完成一个补一个“自然发生。
  • Math.min(limit, items.length)：任务比 limit 少时，没必要启动多余的 worker。limit = Infinity 时就等于“任务数个 worker“，即全并发。
  • 最后 await Promise.all(workers) 等所有 worker 的 while 都跑完，返回 results。

💡 关于“共享游标“为什么安全：const current = nextIndex++ 是一段同步代码，中间不会有 await，所以不会被别的 worker “插队“打断。JS 单线程在这里反而成了优势——你不必担心多线程那种“两个 worker 抢到同一个下标“的竞态问题。

演示：limit=1 / 3 / Infinity 的耗时差异

// 文件：code/ch08/async-pool.js（演示部分）
const items = [1, 2, 3, 4, 5, 6];
const iteratorFn = async (n, i) => {
  await delay(200); // 假装在发网络请求，每个耗时 200ms
  console.log(`  任务#${i}（值=${n}）完成`);
  return n * n; // 返回平方
};
// 分别用 limit = 1 / 3 / Infinity 跑同一批任务

运行 node async-pool.js，实测（6 个任务，每个 200ms）：

=== limit = 1 ===         总耗时：约 1210ms   （串行：6 × 200）
=== limit = 3 ===         总耗时：约 404ms    （分 2 批：⌈6/3⌉ × 200）
=== limit = Infinity ===  总耗时：约 202ms    （全并发：1 批）
结果始终是 [1,4,9,16,25,36]，顺序对应输入

逐段拆解这组数字：

  • limit = 1：只有一个 worker，纯串行，6 个任务排队做，约 6 × 200 = 1200ms。
  • limit = 3：三个 worker 同时抢。第一批做 0/1/2，各 200ms；完成后立刻补 3/4/5，再 200ms。共 2 批 ≈ 400ms。
  • limit = Infinity：6 个 worker 全上，一批全部 200ms 内做完 ≈ 200ms。
  • 无论哪种并发度，结果都是 [1,4,9,16,25,36]，顺序纹丝不乱——这正是 results[current] = ... 按下标回填的功劳。

扩展练习

这个 asyncPool 是生产级思路的雏形，你可以继续增强（提示在括号里）：

  1. 加超时：让每个任务最多跑 timeout 毫秒，超了就算失败。（提示：Promise.race([iteratorFn(...), delayReject(timeout)])，用上 8.8 的 delayReject。）
  2. 加重试：任务失败后自动重试最多 n 次，仍失败才真失败。（提示：在 worker 里包一个 for 循环，try/catch 到最后一次才 throw。）
  3. 不因一个失败而全崩：目前某个任务抛错会让 Promise.all(workers) 整体失败。改成像 allSettled 那样，results[current] 存 { status, value/reason }。
  4. 进度回调：每完成一个任务就调用一次 onProgress(done, total)，方便显示进度条。

试着动手实现其中一两个——它们几乎覆盖了真实项目里“批量请求“要考虑的所有问题。

小结与自测

这一章我们把 JavaScript 异步的三层地基一次打牢：

  • 同步/异步、阻塞/非阻塞：异步的本质是“把耗时的等待交给环境，登记好回调，当前代码不傻等“。
  • 事件循环：单线程靠“栈空了就取队列回调“来实现并发。微任务（Promise.then / queueMicrotask / nextTick）在每个宏任务之后被整队清空，永远早于下一个宏任务（setTimeout）。口诀：同步 → 微任务 → 宏任务。
  • 回调地狱：多步依赖的回调会嵌成金字塔，错误处理啰嗦。
  • Promise：三态一锤定音，then/catch/finally + 链式 + 一个 catch 兜全链错误。组合器 all（全成才成）/ allSettled（全等到底）/ race（先到者）/ any（先成功者）。
  • async/await：async 函数返回 Promise，await 暂停等待、用 try/catch 抓错。互不依赖的任务要用 Promise.all 并行，别用连续 await 串行化。
  • 四大坑：忘 await、forEach 配 async、该并行却串行、未捕获 rejection。
  • 我们封装了 delay/sleep 工具，并实现了并发限流器 asyncPool——用“固定数量 worker 抢共享游标“控制并发，用“按下标回填“保证结果顺序。
