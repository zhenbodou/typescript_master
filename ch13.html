JavaScript & TypeScript 从入门到精通

第 13 章 · TypeScript 起步与类型注解

本章目标

恭喜你！走到这里，你已经学完了 JavaScript 的全部核心：变量与类型、函数、对象与数组、闭包、原型、异步、模块、Node……这十二章的分量，足以让你写出真正能用的程序。

但你可能也隐隐感觉到一件事：JavaScript 写着写着，就容易“翻车“。一个函数本来该收数字，你不小心传了字符串，它不吭声，直到运行到某一行才突然出错——甚至根本不出错，只是默默算出一个荒谬的结果。程序越大，这种“暗雷“越多。

还记得第 1 章结尾我们埋下的那句伏笔吗？

初步体会 JavaScript “动态类型“的特点，为后面学 TypeScript 埋下伏笔。

现在，伏笔要收了。从本章开始，我们进入本书的第二大板块：TypeScript（简称 TS）。它不是另一门语言，而是给 JavaScript “加了一层类型防护网”。学完本章，你将能够：

  • 说清楚为什么需要 TypeScript，理解动态类型的痛点和 TS 带来的价值。
  • 理解三个关键事实：TS 是 JS 的超集（superset）、TS 会被编译（compile）成 JS、类型在运行时会被擦除（type erasure）。
  • 会用两种方式运行 TS：tsx 直接跑、tsc 编译；并知道 tsconfig.json 是干嘛的。
  • 熟练写基础类型注解：number、string、boolean、null、undefined、数组、元组（tuple）。
  • 理解类型推断（type inference），知道“能推断就别写“、以及何时必须显式注解。
  • 给函数加类型：参数、返回值、可选参数、默认参数、rest 参数、void、函数类型表达式。
  • 给对象加类型：内联对象类型、可选属性、readonly。
  • 分清几个特殊类型：any（危险）、unknown（安全的 any）、never、object。
  • 入门字面量类型（literal type）、as const、类型断言 as。
  • 理解严格模式（strict） 的意义，尤其是 strictNullChecks。

本章所有示例都是 .ts 文件，用 npx tsx 文件.ts 运行。code 目录已经装好了 typescript 和 tsx，tsconfig.json 也已经开启了严格模式（strict）。开始吧。

13.1 为什么需要 TypeScript

JavaScript 的“动态类型“到底带来了什么麻烦

JavaScript 是动态类型（dynamically typed） 语言：一个变量今天装数字、明天装字符串都行，类型在运行时才确定，写代码时没人拦着你。这在写小脚本时很爽，但项目一大就成了灾难。看一个具体例子：

// 文件：code/ch13/02-pain.ts

// 想象这个函数从某个"配置"里取端口号，本意是数字相加。
function nextPort(port) {
  return port + 1;
}

假设 port 来自环境变量。在 Node 里，环境变量永远是字符串——哪怕你在 .env 里写的是 3000，读出来也是字符串 "3000"。于是：

  • 你以为 nextPort(3000) 会得到 3001。
  • 但如果传进来的是 "3000"，"3000" + 1 在 JS 里会走字符串拼接，结果是 "30001"——一个错得离谱的值。
  • 而且全程不报错。程序带着这个错误值继续跑，可能在几十行之后才以一种莫名其妙的方式崩溃，让你查半天。

这就是动态类型的核心痛点：错误发生的地方，和你发现错误的地方，隔着十万八千里。 再叠加几个常见场景，你就更能体会了：

  • 调一个函数，忘了它到底要几个参数、什么顺序、哪个可选——只能翻文档或翻源码。
  • 一个对象，你以为它有 user.name，实际字段叫 user.userName，写错了 undefined，运行时才发现。
  • 重构时改了个函数签名，结果三十个调用点里有五个没改到，你根本不知道漏了哪几个。

📌 核心痛点一句话：动态类型把“类型错误“从“写代码时“推迟到了“运行时“，而运行时的错误又晚又隐蔽，项目越大越难维护。

TypeScript 的价值

TypeScript 由微软开发，它做的事情很简单：让你给代码标注类型，然后在你写代码时（编译期）就把类型错误找出来。 它带来三个实打实的好处：

  1. 编译期报错（compile-time errors）：上面那个 nextPort，如果你声明了 port: number，那么传字符串进去时，TS 会在你敲代码的那一刻就用红色波浪线告诉你“类型不对“，根本轮不到运行时出问题。
  2. 编辑器智能提示（IntelliSense）：因为编辑器知道每个变量的类型，它能自动补全属性、方法，能告诉你函数要什么参数。写代码像开了自动挡，还顺带当文档。
  3. 可维护性（maintainability）：类型就是一份“永远不会过时的文档“。重构时改了签名，所有对不上的调用点会立刻标红，一个都跑不掉。团队协作、大型项目，靠的就是这个。

💡 一句话理解：TypeScript = JavaScript + 类型系统。 它不改变 JS 的运行逻辑，只是在旁边帮你“审稿“。

三个必须先记住的事实

在写任何代码之前，先把 TS 的三条“世界观“刻进脑子，后面很多困惑都能提前化解：

事实一：TS 是 JS 的超集（superset）。 意思是——任何合法的 JavaScript 代码，都是合法的 TypeScript 代码。你前十二章写的所有 JS，直接改成 .ts 后缀就能用。TS 只是额外允许你写类型注解。所以学 TS 不是推翻重来，而是“在你已有的 JS 功力上加装备“。

事实二：TS 要被编译（compile）成 JS 才能运行。 浏览器和 Node 不认识 TypeScript，它们只认识 JavaScript。所以 TS 代码要先经过一个叫编译器（compiler） 的工具，把类型注解去掉、转成普通 JS，才能真正跑起来。这个编译器就是 tsc（TypeScript Compiler）。

事实三：类型在运行时被擦除（type erasure）。 这是最关键、也最容易搞错的一点。你写的所有类型注解——: number、: string、interface、type……在编译成 JS 之后全部消失了。它们只在“编译期“存在，用来给编译器检查用；一旦变成 JS，运行时完全不存在任何类型信息。

举个例子，这段 TS：

const age: number = 30;

编译成 JS 后，就是：

const age = 30;

那个 : number 蒸发了。

⚠️ 类型擦除是新手最大的思维误区来源。 因为类型会被擦除，所以你不能在运行时用类型做判断。比如你不能写“如果 x 的类型是 AppConfig 就怎样“——运行时根本没有 AppConfig 这个东西。运行时想判断类型，还得靠老朋友 typeof、Array.isArray、in 这些JS 原生手段（第 1 章学过）。这一点本章后面会反复强调。

13.2 如何运行 TypeScript

方式一：tsx —— 直接跑，最省事（本书主用）

开发时我们不想每改一次就手动编译。工具 tsx（TypeScript Execute）能“一步到位“：它在背后帮你即时编译再运行，你只管写、只管跑。

// 文件：code/ch13/01-hello.ts

// message 的类型被显式注解为 string。
const message: string = "你好，TypeScript！";

console.log(message);

// 下面这行如果取消注释，tsx / tsc 会在「编译期」直接报错：
//   Type 'number' is not assignable to type 'string'.
// const wrong: string = 123;

export {};

在 code 目录下运行：

npx tsx ch13/01-hello.ts

输出：

你好，TypeScript！

逐段拆解：

  • const message: string = "..."：这里的 : string 就是类型注解（type annotation）——冒号后面跟类型，写在变量名和等号之间。它告诉 TS：“message 只能是字符串。”
  • 被注释掉的 const wrong: string = 123：如果你把它放开，TS 会立刻报错，因为 123 是数字，不能赋给声明为 string 的变量。这就是“编译期报错“的第一次亲身体验。
  • 末尾的 export {};：这行看着奇怪，作用是把这个文件标记成一个“模块“（第 9 章的 ESM），让它拥有自己独立的作用域，避免和其他示例文件的顶层变量重名。本书每个独立示例文件都会加这一行，你照做即可，第 14 章后会更自然。

💡 npx 是第 9 章学过的：它临时找到并运行本地装好的 tsx，不用全局安装。

方式二：tsc —— 正式编译成 JS

真正上线部署时，我们要把 .ts 编译成 .js。这就用 tsc：

# 把 hello.ts 编译成 hello.js（生成在同目录）
npx tsc ch13/01-hello.ts

（注意：本书的 tsconfig.json 开了 noEmit，不会真的生成文件；这里只是说明 tsc 的用途。）你会得到一个 hello.js，里面的 : string 已经被擦除——这正好印证了“类型擦除“。

tsc 还有一个开发中天天用的姿势：只检查类型、不生成文件：

# --noEmit：只做类型检查，不产出 .js
npx tsc --noEmit

这会把整个项目按 tsconfig.json 的规则检查一遍，报告所有类型错误。它常被放进 CI 流程，作为“代码合格线“。

tsconfig.json：TypeScript 的项目配置

一个 TS 项目通常有个 tsconfig.json，告诉编译器“用什么规则、检查哪些文件“。它可以用一条命令生成：

npx tsc --init

code 目录里已经有一份配好的了，我们看它最关键的一项：

{
  "compilerOptions": {
    "strict": true
  }
}

📌 "strict": true 打开了 TypeScript 的严格模式——一整套更严的检查（本章 13.9 详解）。本书全程开启严格模式，因为这才是 TS 真正发挥威力的方式。tsconfig.json 还有很多选项（target、module、lib……），我们放到第 20 章深入，本章只需知道它的存在和 strict 这一项。

13.3 基础类型注解

现在正式学怎么写类型。最基础的一批，直接对应第 1 章那七种 JS 类型。

// 文件：code/ch13/03-basic-types.ts

// —— 原始类型 ——
const age: number = 30;
const name: string = "小明";
const isAdmin: boolean = false;
const nothing: null = null;
const notAssigned: undefined = undefined;

console.log(age, name, isAdmin, nothing, notAssigned);

// —— 数组：两种等价写法 ——
const scores: number[] = [90, 85, 100];
const words: Array<string> = ["a", "b", "c"];

console.log("scores 之和：", scores.reduce((a, b) => a + b, 0));
console.log("words：", words.join(","));

// —— 元组（tuple）：定长、每个位置类型固定 ——
// 第 0 位必须是 string，第 1 位必须是 number。
const person: [string, number] = ["小红", 25];
console.log("姓名：", person[0], "年龄：", person[1]);

// person[0] 被 TS 记住是 string，所以可以直接用字符串方法：
console.log("名字长度：", person[0].length);

// 下面几行如果取消注释都会报错，正是元组的价值：
// const bad1: [string, number] = [25, "小红"]; // ❌ 顺序反了
// const bad2: [string, number] = ["小红"];      // ❌ 少了一个

export {};

运行 npx tsx ch13/03-basic-types.ts，输出：

30 小明 false null undefined
scores 之和： 275
words： a,b,c
姓名： 小红 年龄： 25
名字长度： 2

逐段拆解：

  • 原始类型：number、string、boolean、null、undefined 这几个类型名，全是小写。注意区分：小写的 number 是 TS 的类型，别写成大写的 Number（那是 JS 的包装对象构造函数，几乎永远不该用作类型）。
  • 数组的两种写法：number[]（“数字的数组”）和 Array<number>（用泛型语法，第 16 章讲泛型）完全等价。本书更常用 number[]，简洁。数组里每个元素都必须是声明的那个类型，往 number[] 里 push("4") 会报错。
  • 元组（tuple）：[string, number] 表示一个长度固定、每个位置类型固定的数组。第 0 位必须是 string，第 1 位必须是 number。它和普通数组的区别是“位置有意义“——常用来表达“一对固定搭配的值“，比如坐标 [x, y]、键值对 [key, value]。
  • person[0].length：因为 TS 记得第 0 位是 string，所以你点出来的全是字符串的方法/属性（.length、.toUpperCase()……），编辑器自动提示。这就是类型的日常好处。
  • 被注释的 bad1、bad2：元组会严格检查顺序和个数，放开任意一行都会报错。这正是你想要的保护。

13.4 变量注解与类型推断

写到这你可能会想：每个变量都要写 : 类型，也太啰嗦了吧？好消息是——大多数时候你根本不用写。TypeScript 有强大的类型推断（type inference）：它能根据你赋的值，自动推断出类型。

// 文件：code/ch13/04-inference.ts

// 没写类型，但 TS 通过右边的值推断出 count 是 number。
let count = 0;
count = 10; // OK
// count = "十"; // ❌ 报错：不能把 string 赋给 number

// 推断出 title 是 string。
const title = "TypeScript";
console.log(title.toUpperCase());

// 数组字面量也能推断：这是 number[]。
const nums = [1, 2, 3];
// nums.push("4"); // ❌ 报错：number[] 里不能放 string

// 函数返回值也能推断：TS 知道 double 返回 number。
function double(x: number) {
  return x * 2;
}
const result = double(21); // result 被推断为 number
console.log("result =", result);

// —— 何时必须显式注解？——
// 1) 函数参数：TS 无从推断调用者会传什么，必须自己写。
// 2) 先声明、后赋值的变量：
let later: string;
later = "稍后赋值";
console.log(later);

// 3) 你希望类型比推断结果更宽或更窄时（后面章节会遇到）。

export {};

逐段拆解：

  • let count = 0：虽然没写 : number，但 TS 看到你赋了 0，就推断 count 是 number。之后 count = "十" 会报错——推断出来的类型和显式写的一样有约束力。
  • const title = "TypeScript"：推断为字符串，title.toUpperCase() 有提示。（顺带一提：const 的推断比 let 更“窄“，这点 13.8 讲字面量类型时会揭晓。）
  • const nums = [1, 2, 3]：推断为 number[]，push("4") 会报错。
  • 函数返回值也能推断：double 的参数写了 x: number，TS 就知道 x * 2 是 number，于是推断返回值也是 number，进而 result 也是 number。整条链自动打通。

📌 黄金法则：能推断就别写。 变量、const、函数返回值，绝大多数交给推断即可，代码更干净。什么时候必须显式注解？ 主要三种：

  1. 函数参数——这是重中之重。TS 无法知道未来的调用者会传什么，所以参数几乎总是要你自己写类型（下一节详谈）。
  2. 先声明、后赋值的变量：像 let later: string; 这样声明时没给值，TS 推断不出来（会当成 any，严格模式下更麻烦），得手动注解。
  3. 当你故意想让类型比推断结果更宽或更窄时（后面章节会遇到具体场景）。

💡 别走极端：不是“永远不写类型“，也不是“处处都写“。原则是——在类型无法被推断、或推断结果不是你想要的地方，才动手写。

13.5 函数类型

函数是程序的主角，也是类型注解最能发挥价值的地方——因为函数是“接口契约“，说清楚它要什么、给什么，调用者才不会用错。

// 文件：code/ch13/05-functions.ts

// 参数与返回值都注解。
function add(a: number, b: number): number {
  return a + b;
}
console.log(add(2, 3));

// 可选参数用 ?，它的类型其实是 string | undefined。
function greet(name: string, title?: string): string {
  // 因为 title 可能是 undefined，这里要判断。
  return title ? `${title} ${name}` : `你好，${name}`;
}
console.log(greet("小明"));
console.log(greet("小明", "王老师"));

// 默认参数：不用写 ?，TS 从默认值推断类型为 number。
function power(base: number, exp = 2): number {
  return base ** exp;
}
console.log(power(5)); // 25
console.log(power(2, 10)); // 1024

// rest 参数：类型是数组。
function sum(...values: number[]): number {
  return values.reduce((a, b) => a + b, 0);
}
console.log(sum(1, 2, 3, 4)); // 10

// void：表示"不返回有意义的值"（只做副作用）。
function log(msg: string): void {
  console.log("[log]", msg);
}
log("这是一条日志");

// 函数类型表达式：把"函数的形状"作为一种类型。
// BinaryOp 表示"接受两个 number、返回 number"的函数。
type BinaryOp = (a: number, b: number) => number;

const multiply: BinaryOp = (a, b) => a * b; // 参数类型可省略，由 BinaryOp 推断出
console.log(multiply(6, 7)); // 42

export {};

运行输出：

5
你好，小明
王老师 小明
25
1024
10
[log] 这是一条日志
42

逐段拆解：

  • 参数与返回值注解：function add(a: number, b: number): number。参数在括号里各自注解，返回值类型写在参数括号后面、函数体前面（): number {）。虽然返回值常能被推断，但对外的重要函数显式写返回值是好习惯——能防止你不小心 return 了个错东西。
  • 可选参数 ?：title?: string 表示这个参数可以不传。⚠️ 关键点：可选参数的真实类型是 string | undefined（第 14 章讲的联合类型），所以在函数体里用它之前，必须先判断它是否存在（这里用 title ? ... : ...）。严格模式下，直接用一个可能为 undefined 的值会报错——这是好事，逼你处理“没传“的情况。
  • 默认参数：exp = 2。给了默认值就不用写 ?，而且 TS 会从默认值 2 推断出 exp 是 number。调用时不传，就用默认值。
  • rest 参数：...values: number[]。收集所有剩余实参成一个数组，所以类型是数组类型 number[]。（第 2 章学过 rest 语法，这里加了类型。）
  • void：log 只负责打印、不返回有意义的值，它的返回类型是 void。void 专门表示“这个函数不返回东西（或返回值不该被使用）“。别和 undefined 搞混——void 是给“函数返回值“用的语义标签。
  • 函数类型表达式：type BinaryOp = (a: number, b: number) => number 把“一个函数长什么样“本身定义成了一个类型。语法是 (参数: 类型) => 返回类型（注意这个 => 是类型层面的，不是箭头函数的 =>）。然后 const multiply: BinaryOp = (a, b) => ... 时，因为类型已由 BinaryOp 给定，箭头函数的参数 a、b 不用再写类型，TS 会反过来推断——这叫上下文类型（contextual typing）。

13.6 对象类型

对象是 JS 里最常见的数据结构。给对象加类型，就是描述“它有哪些属性、每个属性什么类型“。

// 文件：code/ch13/06-objects.ts

// 内联对象类型注解：直接在冒号后面描述对象的"形状"。
const user: { id: number; name: string; email?: string } = {
  id: 1,
  name: "小明",
  // email 是可选属性（?），可以不写。
};
console.log(user.name);

// 因为 email 可能不存在，直接用会被 TS 提醒它是 string | undefined。
if (user.email) {
  console.log(user.email.toLowerCase());
} else {
  console.log("这个用户没有邮箱");
}

// readonly：只读属性，赋值后不能再改（编译期检查）。
type Point = { readonly x: number; readonly y: number };
const origin: Point = { x: 0, y: 0 };
console.log(origin.x, origin.y);
// origin.x = 10; // ❌ 报错：Cannot assign to 'x' because it is a read-only property.

// 函数参数也能用内联对象类型，常用于"配置对象"。
function describe(p: { x: number; y: number }): string {
  return `(${p.x}, ${p.y})`;
}
console.log(describe({ x: 3, y: 4 }));

export {};

运行输出：

小明
这个用户没有邮箱
0 0
(3, 4)

逐段拆解：

  • 内联对象类型注解：{ id: number; name: string; email?: string }——用花括号，里面列出每个属性名和它的类型，属性之间用分号（也可用逗号）分隔。这就描述了对象的形状（shape）。
  • 可选属性 email?：属性名后加 ?，表示可有可无。上面的对象没给 email 也不报错。和可选参数一样，email 的类型实际是 string | undefined，用之前要判断——所以有了那段 if (user.email)。
  • readonly：readonly x: number 表示 x 一旦初始化就不能再被赋值。origin.x = 10 会在编译期报错。⚠️ 再次提醒类型擦除：readonly 只是编译期检查，运行时 origin.x 还是个普通属性，硬要用别的手段改它 JS 层面拦不住——readonly 是“给写代码的人看的护栏“，不是运行时的锁。
  • 函数参数用对象类型：describe(p: { x: number; y: number })。这种“接一个配置对象“的写法在真实项目里极其常见。

💡 内联对象类型写多了会又长又重复。第 14 章会教你用 interface 和 type 给对象形状起个名字、反复复用——那才是日常主流写法。本章先理解“对象类型“这个概念本身。

13.7 特殊类型：any、unknown、never、object

TS 有几个“特殊“的类型，理解它们能帮你避开大坑，尤其是 any 和 unknown 的区别。

// 文件：code/ch13/07-any-unknown.ts

// —— any：放弃类型检查，什么都能做，等于回到裸 JS ——
let loose: any = "字符串";
loose = 42; // 不报错
// loose.foo.bar.baz; // TS 不报错，但运行时会炸！any 关掉了所有保护（故注释掉）
console.log("any 会关闭检查，很危险，尽量别用。loose =", loose);

// —— unknown：也能装任何值，但用之前必须先"缩小类型" ——
let safe: unknown = "可能是任何东西";
safe = 42;

// safe.toFixed(2); // ❌ 报错：unknown 不允许直接操作
// 必须先判断类型（type narrowing），TS 才放行：
if (typeof safe === "number") {
  console.log("是数字：", safe.toFixed(2)); // 这里 safe 已被收窄为 number
}

// —— never：表示"永远不会有值"，比如永远抛错或死循环的函数 ——
function fail(msg: string): never {
  throw new Error(msg);
}
try {
  fail("演示 never");
} catch (e) {
  console.log("捕获到：", (e as Error).message);
}

// —— object：表示"非原始值"（数组、对象、函数都算），但很少直接用 ——
const obj: object = { a: 1 };
console.log(obj);

export {};

运行输出：

any 会关闭检查，很危险，尽量别用。loose = 42
是数字： 42.00
捕获到： 演示 never
{ a: 1 }

逐段拆解：

  • any——核弹级的“关闭检查“：声明为 any 的值，TS 对它完全放弃检查：你可以给它赋任何值、在它上面点任何属性、当任何类型用，编译器一律放行。上面注释掉的 loose.foo.bar.baz 编译期不报错，但运行时因为 loose 是数字 42，访问 .foo.bar 会直接崩溃。⚠️ any 是“传染“的、危险的：它会让类型防护网破一个大洞，你以为有 TS 保护，其实一路裸奔。尽量不要用 any。
  • unknown——“安全版的 any”：unknown 同样能装任何值（赋值这一步和 any 一样宽松），但关键区别在于“取用“：你不能直接在 unknown 值上做任何操作。safe.toFixed(2) 会直接报错。你必须先通过 typeof、instanceof、in 等手段收窄类型（type narrowing）——一旦 if (typeof safe === "number") 成立，在这个分支里 TS 就知道 safe 是 number，才允许你调 .toFixed()。
    • 📌 any vs unknown 一句话：两者都能“装任何东西“，但 any 让你随便用（危险），unknown 逼你先检查再用（安全）。当你确实不知道一个值的类型时（比如 JSON.parse 的结果、来自外部的数据），首选 unknown，别用 any。
  • never——“永远不会有值”：fail 函数永远抛异常，正常路径上它不会返回，所以返回类型是 never。never 表示“这里的代码走不到“或“这个值不可能存在“。你现在只需认识它；它在高级类型和穷尽检查里会更常出现（第 14 章讨论联合类型时会照面）。
  • object：表示“任何非原始值“（对象、数组、函数都算，但 number/string 这些原始值不算）。它太宽泛，实践中很少直接用——想描述对象通常会用具体的对象类型或 interface。这里认识即可。
  • 注意 (e as Error).message：catch 的 e 在 TS 里默认是 unknown（严格模式下），所以要用类型断言 as 告诉编译器“它是个 Error“——这正好引出下一节。

13.8 字面量类型与 as const

前面 number 表示“所有数字“，string 表示“所有字符串“。但有时你想更精确：这个值只能是某几个特定的字符串/数字。这就是字面量类型（literal type）。

// 文件：code/ch13/08-literal-const.ts

// 字面量类型：类型不是"所有 string"，而是"只能是这几个具体的字符串"。
let direction: "left" | "right" | "up" | "down";
direction = "left"; // OK
// direction = "forward"; // ❌ 报错：不在允许的字面量集合里
console.log("方向：", direction);

// 为什么有用？它把"魔法字符串"变成编译器能检查的东西。
function move(dir: "left" | "right"): string {
  return `向${dir === "left" ? "左" : "右"}移动`;
}
console.log(move("left"));

// —— 类型推断里的字面量 ——
// const 声明的原始值，会被推断为"字面量类型"：
const exact = "hello"; // 类型是 "hello"，不是 string
// let 声明会被放宽为 string，因为 let 可以再赋值：
let loose = "hello"; // 类型是 string
console.log(exact, loose);

// —— as const：把整个对象/数组"冻结"成最窄的只读字面量类型 ——
const config = {
  env: "production",
  retries: 3,
} as const;
// 此时 config.env 的类型是 "production"（而不是 string），
// 且所有属性变成 readonly，不能再改。
console.log(config.env, config.retries);
// config.retries = 5; // ❌ 报错：as const 后是只读的

export {};

运行输出：

方向： left
向左移动
hello hello
production 3

逐段拆解：

  • 字面量类型："left" | "right" | "up" | "down"（用 | 连接的是联合类型，第 14 章详讲）表示 direction 的值只能是这四个字符串之一，写 "forward" 会报错。字符串、数字、布尔都能当字面量类型。它的价值是把过去那种“靠约定的魔法字符串“变成编译器能强制检查的东西——写错一个字母立刻标红。
  • 推断中的字面量：这里揭晓 13.4 留的悬念——const exact = "hello" 被推断为字面量类型 "hello"（因为 const 不可能再变），而 let loose = "hello" 被放宽为 string（因为 let 以后可能赋别的字符串）。这个“const 更窄、let 更宽“的规则，理解了会少很多困惑。
  • as const：给对象/数组加 as const，会把它整个冻结成“最窄的只读字面量类型“——config.env 的类型不再是 string 而是精确的 "production"，且所有属性变 readonly，改不了。它常用于定义“常量配置表“、枚举替代品等。本章先认识；深入用法后面会再见。

13.9 类型断言 as，以及严格模式

类型断言 as：谨慎使用

有时你比 TS 更清楚一个值的真实类型。比如 JSON.parse 的返回类型是 any（更好的做法是当 unknown），但你知道解析出来是什么形状。这时可以用类型断言（type assertion） as 来“告诉“编译器。

// 文件：code/ch13/09-assertion.ts

// 场景：你比 TS 更清楚某个值的类型。
// 典型例子是 DOM / JSON 解析，返回值类型很宽（这里用模拟）。
const raw: unknown = JSON.parse('{"port": 8080}');

// 用 as 告诉编译器"我知道它是这个形状"。
const parsed = raw as { port: number };
console.log("端口：", parsed.port);

// ⚠️ 断言不做任何运行时检查！它只是"你对编译器的承诺"。
// 如果承诺错了，运行时照样出问题（下面这段故意注释掉，避免程序崩溃）：
// const wrong = "not a number" as unknown as number;
// console.log(wrong.toFixed(2)); // 运行时报错：wrong.toFixed is not a function
console.log("断言只影响编译期，不改变运行时的值，用错了运行时照样炸。");

export {};

运行输出：

端口： 8080
断言只影响编译期，不改变运行时的值，用错了运行时照样炸。

逐段拆解：

  • raw as { port: number }：as 类型 就是断言。它告诉编译器“把 raw 当作 { port: number } 处理“，之后 parsed.port 就有类型了。
  • ⚠️ 断言不是转换，更不是检查！ 这是全章最重要的警告之一。as 只影响编译期，它不会在运行时做任何验证或转换。你说它是什么，编译器就信什么——如果你说错了（比如把一个字符串断言成 number），编译期一片祥和，运行时照样炸（"...".toFixed 不存在）。
  • 那个 "..." as unknown as number 的双重断言，是“强行绕过检查“的写法，能不用就别用。

📌 用 as 的心态：断言是“你替编译器背书“。用得越多，说明你越是在绕开类型系统的保护，风险越大。优先考虑用 typeof/in 等收窄，或（下一章的）类型守卫，把 as 留给“确实无法用其他方式表达、且你 100% 确定“的少数场景。

严格模式（strict）与 strictNullChecks

本书 tsconfig.json 里那句 "strict": true，其实是一组严格检查的“总开关“。其中对新手影响最大、也最有价值的一项是 strictNullChecks。

在没开严格空检查的世界里，null 和 undefined 可以偷偷赋给任何类型——这正是无数 “Cannot read properties of undefined” 报错的根源。开了 strictNullChecks 后：

  • null 和 undefined 不再能随便赋给别的类型。一个 string 类型的变量，不能是 null。
  • 如果一个值可能是 null/undefined（比如可选属性、可选参数），你必须先判断、再使用，否则报错。

这听起来更麻烦，但它把 JS 里最常见的一类运行时崩溃，提前挡在了编译期。回想 13.5、13.6 里那些“因为可能是 undefined，所以要先 if 判断“的地方——正是 strictNullChecks 在保护你。

📌 本书全程开启 strict。 它一开始会让你多写几个判断，但养成的习惯会让你的代码健壮得多。请把“严格模式下的报错“当成朋友，而不是障碍——它每拦你一次，就是帮你省下一次线上事故。

常见坑与最佳实践

  • ⚠️ 别忘了类型会被擦除。 运行时没有任何类型信息。想在运行时判断类型，用 typeof、Array.isArray、instanceof、in，不是用你定义的 type/interface。这是新手最容易犯的根本性错误。
  • ⚠️ any 是防护网上的洞，尽量别用。 一个 any 会顺着数据流“传染“，让一大片代码失去检查。实在不知道类型时用 unknown，它逼你先检查再用，安全得多。
  • ⚠️ as 断言不做运行时检查。 它只是你对编译器的口头承诺，说错了运行时照样崩。能用类型收窄就别用断言；断言用得多，说明你在和类型系统对着干。
  • 💡 能推断就别写。 别给每个变量都手动标类型，那既啰嗦又容易和推断打架。该写类型的核心位置是：函数参数、对外函数的返回值、以及推断不出来的先声明后赋值变量。
  • ⚠️ 类型名是小写： number/string/boolean，不是 Number/String/Boolean。大写的是包装对象，几乎永远不该用作类型。
  • 💡 可选属性/参数（?）用之前一定先判断。 严格模式下不判断直接用会报错——这是好事，说明它在提醒你处理“没有值“的情况。
  • 💡 开发用 tsx 跑得快，提交前用 npx tsc --noEmit 全量类型检查。 把后者当作代码的“合格线“。
  • 💡 一个独立示例文件如果没有任何 import/export，加一行 export {}; 让它成为模块、拥有独立作用域，避免顶层变量和别的文件重名。

本章小项目：类型化配置读取器

真实项目里，配置常常来自“外部“——环境变量、JSON 文件、命令行参数。这些数据有两个共同特点让人头疼：（1）类型很宽松（环境变量全是字符串），（2）可能缺项（用户没配某项）。我们要做一个类型化配置读取器（typed config reader）：把这种“脏乱差“的原始输入，安全地转换成一个强类型的配置对象，带默认值和基本校验。

// 文件：code/ch13/config.ts
// 本章小项目：类型化配置读取器（typed config reader）。
// 运行：npx tsx ch13/config.ts

// ============================================================
// 1) 定义"目标类型"：我们的应用最终想要的强类型配置形状。
// ============================================================
type AppConfig = {
  port: number;
  host: string;
  debug: boolean;
  tags: string[];
};

// 一份默认值。当原始输入缺某项时，就用它兜底。
// readonly 保证默认值不会被手滑改掉。
const DEFAULTS: Readonly<AppConfig> = {
  port: 3000,
  host: "localhost",
  debug: false,
  tags: [],
};

// ============================================================
// 2) 描述"原始输入"：模拟从环境变量 / JSON 读来的数据。
//    真实世界里这种数据的键值全是字符串，而且可能缺项。
//    我们用 Record<string, string> 表达"键是字符串、值也是字符串"。
// ============================================================
type RawInput = Record<string, string>;

// ============================================================
// 3) 一组类型安全的"解析 + 兜底"小工具。
//    每个都接收 "可能为 undefined 的字符串" 和一个默认值，
//    返回强类型结果。
// ============================================================

function parseNumber(value: string | undefined, fallback: number): number {
  if (value === undefined) return fallback;
  const n = Number(value);
  // NaN 检查：Number("abc") 会得到 NaN，这时回退到默认值。
  return Number.isNaN(n) ? fallback : n;
}

function parseBoolean(value: string | undefined, fallback: boolean): boolean {
  if (value === undefined) return fallback;
  // 约定："true" / "1" 视为 true，其余视为 false。
  return value === "true" || value === "1";
}

function parseTags(value: string | undefined, fallback: string[]): string[] {
  if (value === undefined || value.trim() === "") return fallback;
  // 逗号分隔，去掉首尾空白，过滤空串。
  return value
    .split(",")
    .map((s) => s.trim())
    .filter((s) => s.length > 0);
}

// ============================================================
// 4) 核心函数：把宽松的 RawInput 转换成强类型 AppConfig。
//    返回类型标成 AppConfig，编译器会强制我们"每个字段都补齐"。
// ============================================================
function loadConfig(raw: RawInput): AppConfig {
  return {
    port: parseNumber(raw.port, DEFAULTS.port),
    host: raw.host ?? DEFAULTS.host, // 字符串直接用 ?? 兜底
    debug: parseBoolean(raw.debug, DEFAULTS.debug),
    tags: parseTags(raw.tags, DEFAULTS.tags),
  };
  // 如果这里少写一个字段（比如忘了 tags），TS 会报错：
  //   Property 'tags' is missing in type ... 这正是类型注解在帮你。
}

// ============================================================
// 5) 演示：模拟两份不同的原始输入。
// ============================================================

// 一份"完整且部分带脏数据"的输入。
const fromEnv: RawInput = {
  port: "8080",
  host: "0.0.0.0",
  debug: "1",
  tags: "web, api ,,cache",
};

// 一份"残缺"的输入：只给了 port，其余走默认值。
const partial: RawInput = {
  port: "not-a-number", // 脏数据，会被 NaN 检查挡下、回退默认
};

const configA = loadConfig(fromEnv);
const configB = loadConfig(partial);

console.log("configA =", configA);
console.log("configB =", configB);

// 因为返回值是强类型 AppConfig，下面这些用法都有编辑器提示和检查：
console.log(`服务将监听 ${configA.host}:${configA.port}`);
console.log("标签数量：", configA.tags.length);

// 下面这行如果取消注释会报错，证明类型确实在保护我们：
// configA.port.toUpperCase(); // ❌ number 上没有 toUpperCase

export {};

运行 npx tsx ch13/config.ts，输出：

configA = {
  port: 8080,
  host: '0.0.0.0',
  debug: true,
  tags: [ 'web', 'api', 'cache' ]
}
configB = { port: 3000, host: 'localhost', debug: false, tags: [] }
服务将监听 0.0.0.0:8080
标签数量： 3

逐段拆解，重点看类型注解如何帮我们避免错误：

  • ①定义目标类型 AppConfig：这是我们的“契约“——应用最终想要的形状。port: number、host: string、debug: boolean、tags: string[]，每个字段的类型都是我们运行逻辑真正需要的类型（比如 port 要能做数学运算，就必须是 number 而不是 "8080" 这种字符串）。
  • 默认值 DEFAULTS：类型是 Readonly<AppConfig>——Readonly<T> 是 TS 内置的工具类型，把 T 的所有属性变成 readonly。这样谁都不能手滑改掉默认值。同时它必须完整满足 AppConfig：少写一个字段，编译期就报错，保证默认值不残缺。
  • ②原始输入类型 RawInput = Record<string, string>：Record<K, V> 也是内置工具类型，表示“键是 K、值是 V 的对象“。这里精准刻画了“从 env/JSON 来的数据：键值都是字符串“。注意：从这种对象里取 raw.port，在严格模式下 TS 会认为它可能是 undefined（因为你不能保证这个键一定存在）——这正好逼我们处理“缺项“。
  • ③三个解析工具：注意每个的第一个参数类型都是 string | undefined，明确表达“这个值可能没有“。
    • parseNumber 里先判 undefined 兜底，再做 Number.isNaN 检查（第 1 章的坑！Number("abc") 得到 NaN），脏数据就回退默认值——永远返回一个真正的 number。
    • parseBoolean 把字符串 "true"/"1" 映射成布尔值——因为环境变量里没有真正的布尔，只有字符串。
    • parseTags 把 "web, api ,,cache" 这种逗号串，切分、去空白、过滤空项，得到干净的 string[]。
  • ④核心 loadConfig：返回类型标成 AppConfig，这是关键保护——编译器会强制这个返回对象把四个字段全部补齐、且类型正确。如果你漏写 tags，或者把 port 写成了字符串，编译期立刻报错。host 用了 ??（第 1 章的空值合并）直接兜底，因为它本来就是字符串。
  • ⑤演示：fromEnv 里 port: "8080"（字符串）被转成数字 8080，debug: "1" 被转成 true，脏乱的 tags 被清洗干净。partial 里 port 是脏数据 "not-a-number"，被 NaN 检查挡下、回退成默认 3000，其余字段全走默认值——程序没有崩，输出的是一份完整、干净、强类型的配置。
  • 最后那行注释 configA.port.toUpperCase()：因为 configA 是 AppConfig、port 是 number，在 number 上调字符串方法会被编译器拦下。这就是我们费这么大劲做类型化的回报：拿到手的配置，怎么用都有编辑器提示和检查，用错立刻知道。

📌 体会这个小项目的精髓：在“边界“（读取外部数据）处做一次类型转换与校验，把宽松、可能出错的输入变成可信的强类型；此后整个程序内部都能安心地享受类型保护。 这是真实工程里处理外部数据的标准姿势。

扩展练习

  1. 加一个字段：给 AppConfig 增加 timeout: number（毫秒），写对应的默认值和解析逻辑。感受“改了类型，编译器会提醒你哪些地方还没补齐“。
  2. 更严的校验：让 parseNumber 支持“最小值/最大值“参数，比如 port 必须在 1~65535 之间，越界就回退默认值。
  3. 真实环境变量：把 fromEnv 换成真正的 process.env（Node 提供，类型近似 Record<string, string | undefined>）。想想为什么它的值类型是 string | undefined 而不是 string，你的解析函数是否已经能应对。
  4. 用 unknown 做输入：把 RawInput 改成 unknown（模拟“完全不可信的 JSON“），在 loadConfig 开头先用 typeof/in 判断它是不是一个对象，再往下解析。体会 unknown 如何逼你把校验写扎实。
  5. 返回校验结果：让 loadConfig 不直接返回 AppConfig，而是返回 { ok: true; config: AppConfig } | { ok: false; errors: string[] }（联合类型，下一章会正式学），把校验中发现的问题收集起来报告给调用者。

小结与自测

本章我们正式从 JavaScript 跨入 TypeScript，核心要点：

  • 为什么用 TS：动态类型让类型错误从“写代码时“推迟到“运行时“，又晚又隐蔽。TS 用类型注解在编译期发现错误，还带来智能提示和可维护性。
  • 三个世界观：TS 是 JS 的超集；TS 要编译成 JS 才能运行；类型在运行时被擦除——运行时判断类型仍靠 typeof/in/instanceof。
  • 运行方式：开发用 tsx 直接跑，tsc 负责正式编译，tsc --noEmit 只做类型检查；tsconfig.json 是项目配置，strict 是灵魂（第 20 章深入）。
  • 基础类型：number/string/boolean/null/undefined（小写！）、数组 T[] 或 Array<T>、元组 [T1, T2]（定长定位）。
  • 类型推断：能推断就别写；函数参数、对外返回值、先声明后赋值的变量才需要显式注解。
  • 函数类型：参数与返回值注解、可选参数 ?（实为 T | undefined）、默认参数、rest ...args: T[]、void、函数类型表达式 (a: T) => R。
  • 对象类型：内联对象类型、可选属性 ?、readonly（仅编译期护栏）。
  • 特殊类型：any（关闭检查，危险）vs unknown（安全，逼你先收窄）；never（不会有值）；object（非原始值，少用）。
  • 字面量类型与 as const；类型断言 as（只影响编译期、不做运行时检查，谨慎用）。
  • 严格模式尤其 strictNullChecks：null/undefined 不能乱赋，可能为空的值必须先判断——把最常见的运行时崩溃挡在编译期。

自测题（能脱口而出就说明掌握了）：

  1. “TS 是 JS 的超集” 和 “类型在运行时被擦除” 分别是什么意思？后者对你写代码有什么实际约束？
  2. 为什么说函数参数几乎总要显式写类型，而变量和返回值常常可以不写？
  3. any 和 unknown 都能装任何值，它们最关键的区别是什么？你更应该用哪个？
  4. 可选属性 email?: string 的真实类型是什么？为什么用它之前通常要先 if 判断？
  5. const x = "hi" 和 let x = "hi"，x 被推断成的类型有什么不同？为什么？
  6. 类型断言 as 会在运行时做检查或转换吗？滥用它有什么风险？

本章我们用内联对象类型和函数类型表达式描述了简单的形状，但它们写多了又长又难复用。第 14 章《接口、类型别名、联合与交叉类型》 会教你给类型“起名字“并组合它们：用 interface 和 type 定义可复用的对象形状，用联合类型（A | B） 表达“要么这个要么那个“（本章的字面量类型和可选属性其实已经用到了它），用交叉类型（A & B） 把多个类型合并。掌握了它们，你才算真正会用 TypeScript 的类型系统去“建模“你的数据。
