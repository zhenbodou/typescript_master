JavaScript & TypeScript 从入门到精通

第 23 章 · Node.js 后端开发与 REST API

本章目标

到目前为止，我们写的程序几乎都是“自己跑给自己看“——命令行脚本、测试用例、类型练习。但真实世界里，软件常常要作为一个服务（service）长期运行，等着别人来“敲门“：浏览器、手机 App、另一台服务器……它们通过网络发来请求，你的程序处理后再把结果送回去。这类程序就是后端（backend），而它们之间最通用的对话方式，就是 HTTP。

这一章我们要把“写脚本“升级成“写服务“。你会先亲手用 Node 内置的 node:http 搓一个最原始的 HTTP 服务器——不为了好用，而是为了看清框架到底替我们做了什么；然后引入业界最流行的 Web 框架 Express，理解它的路由、req/res、中间件（middleware）这套核心模型；最后用 Express + TypeScript 搭一个结构清晰、分层合理、有校验有错误处理的任务管理 REST API。

学完本章，你将能够：

  • 复习并说清 HTTP 的骨架：请求/响应、方法（GET/POST/PUT/PATCH/DELETE）、状态码、headers、body、JSON。
  • 用 node:http 手写最小服务器，理解“请求 body 是流“、“路由就是 if/else“这些底层真相。
  • 说清 REST 风格：资源、URL 设计、无状态（stateless）、幂等性（idempotency）与常见约定。
  • 用 Express 写路由，读取路由参数（route params）、查询参数（query params）和请求体（body）。
  • 真正理解中间件这套洋葱模型，并手写日志、CORS、鉴权、错误处理中间件。
  • 用 express.Router() 把路由模块化。
  • 给外部输入做请求校验，并用统一错误处理中间件收口所有错误。
  • 用 TypeScript + @types/express 写出类型安全的 Express 应用。
  • 用 curl 和一个“supertest 风格“的冒烟脚本测试 API。

本章的核心代码是一个独立子项目 code/ch23/api/，它有自己的 package.json。运行前需要先进入该目录执行 npm install，然后用 npx tsx src/server.ts 启动。

📌 一句话记住本章：后端服务 = 一个长期监听端口、按“请求→路由→处理→响应“循环工作的程序；Express 帮你把这个循环写得优雅，而 REST 是给这个循环定的一套“好好说话“的规矩。

HTTP 基础复习：一次请求到底长什么样

在写任何服务器之前，先把 HTTP 这门“对话语言“的语法过一遍。HTTP 是请求-响应（request-response）模型：客户端发一条请求（request），服务器回一条响应（response），一来一回，然后连接就可以结束了。

一条 HTTP 请求由四部分组成：

POST /tasks HTTP/1.1            ← 请求行：方法 + 路径 + 协议版本
Host: localhost:3000           ← 请求头（headers），key: value
Content-Type: application/json
                               ← 一个空行，分隔头和体
{"title":"买牛奶"}              ← 请求体（body）

逐段拆解：

  • 请求行（request line）：POST 是方法（method），表明“我想干什么“；/tasks 是路径（path），表明“对哪个资源干“；HTTP/1.1 是协议版本。
  • 请求头（headers）：一堆 键: 值，携带元信息。比如 Content-Type: application/json 告诉服务器“我的 body 是 JSON 格式“。
  • 空行：头和体之间必须有一个空行来分隔——这是协议规定。
  • 请求体（body）：真正要提交的数据。不是每个请求都有 body（GET 通常没有，POST 通常有）。

服务器的响应结构几乎对称：

HTTP/1.1 201 Created           ← 状态行：协议 + 状态码 + 原因短语
Content-Type: application/json
Location: /tasks/abc123

{"id":"abc123","title":"买牛奶","done":false}

HTTP 方法：表达“意图“

方法不是随便选的，每个都有约定的语义：

方法意图幂等？典型用途
GET读取资源是查列表、查单个
POST新建资源否创建一条数据
PUT整体替换资源是用完整数据覆盖
PATCH局部更新资源否*（约定上不保证）只改某几个字段
DELETE删除资源是删掉一条数据

📌 **幂等（idempotent）**指的是“同一个请求发一次和发 N 次，对服务器状态的最终影响相同“。DELETE /tasks/1 删一次和删十次，结果都是“1 号任务没了“，所以幂等；而 POST /tasks 发十次会创建十条任务，所以不幂等。幂等性很重要：网络会抖动，客户端可能重发，幂等的接口重发无害。

状态码：表达“结果“

响应用**状态码（status code）**告诉客户端结果如何。按首位数字分五大类：

  • 2xx 成功：200 OK（成功，通常带 body）、201 Created（新建成功）、204 No Content（成功但没有 body，常用于 DELETE）。
  • 3xx 重定向：301/302（资源换地方了），本章用不到。
  • 4xx 客户端错误：400 Bad Request（你请求有问题，比如 body 不合法）、401 Unauthorized（没登录）、403 Forbidden（登录了但没权限）、404 Not Found（资源不存在）、409 Conflict（冲突，如重复创建）。
  • 5xx 服务器错误：500 Internal Server Error（服务器自己崩了）。

⚠️ 别把 4xx 和 5xx 搞混。 4xx 是“**你（客户端）**错了“，5xx 是“**我（服务器）**错了“。校验失败该返回 400，而不是 500——500 意味着你的代码抛了没接住的异常。这个区分对排查问题至关重要。

用 node:http 手写最小服务器：看清底层

框架很好用，但如果一上来就用框架，你会把很多“魔法“当成理所当然。我们先用 Node 内置的 node:http 模块（不需要安装任何依赖）搓一个最小服务器，亲眼看看一个请求进来时，最原始的 API 长什么样。

// 文件：code/ch23/raw-http.ts
import http from "node:http";

// createServer 接收一个「请求处理函数」，每来一个请求就调用它一次。
// req 是 IncomingMessage（可读流），res 是 ServerResponse（可写流）。
const server = http.createServer((req, res) => {
  const { method, url } = req;

  // 最原始的「路由」：就是一堆 if/switch，根据 method + 路径分发。
  if (method === "GET" && url === "/hello") {
    res.statusCode = 200;
    res.setHeader("Content-Type", "application/json; charset=utf-8");
    res.end(JSON.stringify({ message: "你好，世界" }));
    return;
  }
  // …（POST /echo 与 404 见完整文件）
});

server.listen(3000, () => {
  console.log("原始 HTTP 服务器已启动：http://localhost:3000");
});

逐段拆解：

  • http.createServer(handler) 创建一个服务器对象。handler 是每个请求都会被调用一次的回调，签名是 (req, res) => void。
  • req（IncomingMessage）代表进来的请求。它是一个可读流（readable stream）——注意这点，body 不是现成的字符串，得自己收。req.method 是方法字符串，req.url 是路径加查询串。
  • res（ServerResponse）代表要发出去的响应。它是一个可写流（writable stream）。你用 res.statusCode 设状态码、res.setHeader() 设响应头、res.end(body) 把 body 写出去并结束响应。
  • “路由“在这一层就是裸的 if/else：手动比较 method 和 url。想象一下你有几十个端点，这里会变成一坨恐怖的 if 森林——这正是我们后面需要框架的原因之一。
  • server.listen(3000, cb) 让服务器**监听（listen）**3000 端口。这一步之后，进程不会退出，而是“挂“在那里等请求——这就是“服务“和“脚本“的本质区别：脚本跑完就结束，服务要一直活着。

那 POST 的 body 怎么读？这里藏着新手最容易踩的坑：body 是分块流式到达的。

// 文件：code/ch23/raw-http.ts（节选：POST /echo 的处理）
if (method === "POST" && url === "/echo") {
  const chunks: Buffer[] = [];
  req.on("data", (chunk: Buffer) => {
    chunks.push(chunk); // 每来一块数据触发一次 data
  });
  req.on("end", () => {
    const raw = Buffer.concat(chunks).toString("utf-8"); // 收齐后拼起来
    try {
      const data = raw ? JSON.parse(raw) : {}; // 自己 parse JSON
      res.statusCode = 200;
      res.setHeader("Content-Type", "application/json; charset=utf-8");
      res.end(JSON.stringify({ youSent: data }));
    } catch {
      res.statusCode = 400;
      res.end(JSON.stringify({ error: "请求体不是合法 JSON" }));
    }
  });
  return;
}

逐段拆解：

  • 请求体可能很大，网络会把它切成若干**块（chunk）**陆续送达。每到一块，req 触发一次 "data" 事件，我们把这些 Buffer 块攒进数组。
  • 当所有块都到齐，req 触发 "end" 事件。这时用 Buffer.concat 把碎块拼成完整 Buffer，再 toString 成字符串。
  • 我们约定 body 是 JSON，所以要手动 JSON.parse。如果客户端发来的不是合法 JSON，parse 会抛异常，我们接住并返回 400。
  • 看到没？“把 JSON body 解析成对象“这件小事，裸写居然要监听两个事件 + 手动拼接 + 手动 parse + 手动兜错。Express 的 express.json() 中间件，一行就替你做完了全部。

运行体验一下：

npx tsx ch23/raw-http.ts
# 另开一个终端：
curl http://localhost:3000/hello
curl -X POST http://localhost:3000/echo -H "Content-Type: application/json" -d '{"name":"张三"}'

💡 这个练习的价值不在“能用“，而在“祛魅“。 当你后面用 res.json(obj) 时，你会知道它内部无非是 setHeader('Content-Type', ...) + end(JSON.stringify(obj))；当你用 req.body 时，你会知道有个中间件替你收了流、parse 了 JSON。框架不是魔法，是“把这些重复劳动封装好的普通代码“。

REST 风格：给接口定一套好好说话的规矩

同样是“任务管理“，你可以设计成 GET /getAllTasks、POST /createTask、POST /deleteTaskById……也可以设计成一套统一、可预测的接口。后者就是 REST（Representational State Transfer）风格。REST 不是协议、不是标准，而是一套设计约定，核心思想是：把系统里的一切建模成“资源（resource）“，用 URL 定位资源，用 HTTP 方法表达对资源的操作。

以“任务“为例，REST 风格的接口长这样：

操作方法 + URL说明
列出所有任务GET /tasks集合资源
获取某个任务GET /tasks/:id单个资源
新建任务POST /tasks向集合里加一个
局部更新任务PATCH /tasks/:id改某几个字段
删除任务DELETE /tasks/:id删掉单个

REST 的几条关键约定，逐条理解：

  • 资源用名词，不用动词。 URL 里应该是 /tasks（名词复数），而不是 /getTasks（动词）。“想干什么“由 HTTP 方法表达，不由 URL 表达。GET /tasks 和 DELETE /tasks/1 里，URL 都指同一类资源，动作在方法上。
  • 用层级表达从属关系。 GET /tasks/1/comments 表示“1 号任务下的评论集合“，URL 的层级结构天然表达了资源之间的隶属。
  • 无状态（stateless）。 服务器不为某个客户端“记住上一次请求“。每个请求都必须自带一切必要信息（比如身份用 token 每次都带上）。这样任何一台服务器都能处理任何请求，方便水平扩展。⚠️ 这不代表“不能有数据库“——数据库里的持久数据是资源状态，不是会话状态；无状态说的是后者。
  • 善用状态码。 创建成功回 201，删除成功回 204，找不到回 404，校验失败回 400。让状态码承载“结果类型“，客户端一看状态码就知道大局。
  • 利用方法的幂等性。 GET/PUT/DELETE 设计成幂等的，POST/PATCH 不保证。这让客户端在网络不稳时能安全重试幂等请求。

💡 REST 不是教条，而是“降低惊讶“的工具。 当所有接口都遵循同一套约定，任何人（包括半年后的你）看到 DELETE /tasks/5 都能秒懂它干什么、成功该回什么码。一致性本身就是巨大的价值。

引入 Express：路由、req/res、中间件

现在正式请出 Express——Node 生态里最流行、资料最多的 Web 框架。它做的事，本质就是把上面那坨 if (method && url) 变成优雅的路由表，并提供一套强大的中间件机制。

本章的完整应用在 code/ch23/api/ 子项目里，它有独立的 package.json：

一个最小的 Express 应用

先不管分层，看 Express 最核心的三样东西：app、路由、req/res。

// 一个概念性的最小 Express 应用（讲解用）
import express from "express";

const app = express();          // 创建应用实例

app.get("/tasks", (req, res) => {
  res.json({ items: [] });      // res.json 自动设 Content-Type 并序列化
});

app.listen(3000, () => console.log("listening on 3000"));

逐段拆解：

  • express() 返回一个 app 对象，它既是“配置中心“也是那个能被 listen 的请求处理器。
  • app.get(path, handler) 注册一个路由：当 GET /tasks 进来时，调用 handler。类似还有 app.post、app.patch、app.delete。这就取代了裸 http 里那坨 if。
  • handler 的签名是 (req, res) => void。req 是增强版的请求对象（比裸 IncomingMessage 多了 req.params、req.query、req.body 等便利属性），res 是增强版响应对象。
  • res.json(obj) 一步到位：设 Content-Type: application/json、JSON.stringify(obj)、end。还记得裸 http 里我们手动做的三步吗？这里一行。

读取三种输入：路由参数、查询参数、请求体

一个真实接口的输入可能来自三个地方，Express 分别放在三个属性里：

// 讲解用：三种输入的读取方式
app.get("/tasks/:id", (req, res) => {
  const id = req.params.id;         // 路由参数：URL 路径里的 :id
  res.json({ id });
});

app.get("/tasks", (req, res) => {
  const done = req.query.done;      // 查询参数：?done=true 里的 done（永远是字符串！）
  const page = req.query.page;      // ?page=2
  res.json({ done, page });
});

app.post("/tasks", (req, res) => {
  const body = req.body;            // 请求体：需要 express.json() 中间件才有值
  res.json({ received: body });
});

逐段拆解：

  • 路由参数（route params）：路径里用 :id 声明的占位符，运行时从 req.params.id 取。GET /tasks/abc 时 req.params.id === "abc"。
  • 查询参数（query params）：URL 里 ? 后面的键值对。⚠️ 它们永远是字符串（或字符串数组）。?done=true 里 req.query.done 是字符串 "true"，不是布尔 true。想当布尔用必须自己转换——这是个高频陷阱。
  • 请求体（body）：req.body。但注意，Express 默认不解析 body，req.body 是 undefined！必须先挂上 express.json() 中间件，它才会把 JSON body 解析成对象。这就引出了下一个核心概念。

中间件：Express 的灵魂

中间件（middleware）是理解 Express 的钥匙。一个中间件就是一个函数，签名 (req, res, next) => void。Express 把所有中间件和路由排成一条流水线，请求进来后按注册顺序依次流过每一个中间件；每个中间件可以：

  1. 读或改 req/res（比如解析 body、加日志、校验身份）；
  2. 调用 next() 把控制权交给下一个中间件；
  3. 或者直接 res.send(...) 结束响应，不调 next，请求就此终结。
// 讲解用：中间件的流水线模型
const app = express();

app.use((req, res, next) => {
  console.log("① 我先执行");
  next();                    // 交棒给下一个
});

app.use((req, res, next) => {
  console.log("② 我第二个执行");
  next();
});

app.get("/", (req, res) => {
  console.log("③ 路由处理器，最后执行");
  res.send("done");
});

逐段拆解：

  • app.use(fn) 注册一个全局中间件，对所有请求生效。请求会先过 ①，再过 ②，最后到路由处理器 ③。
  • 顺序至关重要：注册顺序就是执行顺序。如果日志中间件注册在路由后面，它就永远不会对那些路由生效。
  • next() 是流水线的传送带。⚠️ 忘了调 next()（又没结束响应），请求就会“卡死“在这个中间件里，客户端一直等到超时——这是最常见的 Express bug。
  • 路由处理器 (req, res) => ... 其实也是一种“终点中间件“：它通常不调 next，而是直接 res.send/res.json 结束响应。

理解了这个模型，express.json() 就不神秘了——它只是一个“官方提供的、负责收 body 流并 parse JSON 挂到 req.body“的中间件。你 app.use(express.json())，它就被插进流水线，在它之后的所有处理器就都能读到 req.body 了。

用 TypeScript 组织 Express 应用：分层结构

小 demo 可以把所有代码堆在一个文件里，但真实项目需要分层，让每一层只操心一件事。本章的任务 API 分成四层，请求从上往下流：

路由层 routes/     ← 只管 HTTP：解析 req、调 service、发 res
业务层 services/   ← 只管业务规则：筛选、分页、"找不到就 404"
数据层 store/      ← 只管数据存哪：本章是内存数组，第 24 章换数据库
中间件 middleware/ ← 横切关注点：日志、CORS、错误处理

📌 分层的核心收益：可替换性。 因为“数据存哪“被隔离在 store/ 一层，第 24 章我们把内存数组换成数据库时，上面的 service 和 route 几乎不用动。这就是“面向接口而非实现“在架构层面的体现。

数据层：内存 store

先从最底层看起。数据层只暴露“增删改查“的原始能力，不含任何业务判断。

// 文件：code/ch23/api/src/store/taskStore.ts
import { randomUUID } from "node:crypto";
import type { Task } from "../types.js";

// 内存里的「表」。进程重启就没了——这正是我们要在第 24 章解决的问题。
const tasks: Task[] = [];

export const taskStore = {
  findAll(): Task[] {
    return [...tasks]; // 返回拷贝，避免外部直接改到内部数组
  },
  findById(id: string): Task | undefined {
    return tasks.find((t) => t.id === id);
  },
  create(data: { title: string; done: boolean }): Task {
    const task: Task = {
      id: randomUUID(),
      title: data.title,
      done: data.done,
      createdAt: new Date().toISOString(),
    };
    tasks.push(task);
    return task;
  },
  update(id: string, patch: Partial<Pick<Task, "title" | "done">>): Task | undefined {
    const task = tasks.find((t) => t.id === id);
    if (!task) return undefined;
    if (patch.title !== undefined) task.title = patch.title;
    if (patch.done !== undefined) task.done = patch.done;
    return task;
  },
  remove(id: string): boolean {
    const index = tasks.findIndex((t) => t.id === id);
    if (index === -1) return false;
    tasks.splice(index, 1);
    return true;
  },
};

逐段拆解：

  • const tasks: Task[] = [] 就是我们的“数据库“——一个进程内的数组。⚠️ 进程一重启，数据就全没了。这在开发阶段够用，但显然不能上生产，第 24 章会换成真数据库。
  • randomUUID() 是 Node 内置的、生成唯一 id 的函数，省得我们自己维护自增计数器。
  • findAll 返回 [...tasks] 而不是 tasks 本身——返回拷贝，防止调用方拿到内部数组的引用后乱改，破坏封装。
  • update 用了 Partial<Pick<Task, "title" | "done">>（第 16 章的工具类型）：Pick 挑出 title 和 done 两个字段，Partial 让它们都可选。语义就是“可以只改这两个字段中的任意几个“。
  • 注意这一层没有任何“找不到该怎么办“的业务判断——它诚实地返回 undefined 或 false，把“该报什么错“的决定权交给上层。

领域模型与错误类型

数据层依赖的类型和自定义错误：

// 文件：code/ch23/api/src/types.ts
export interface Task {
  id: string;
  title: string;
  done: boolean;
  createdAt: string; // ISO 8601 字符串
}
export interface CreateTaskInput { title: string; done?: boolean; }
export interface UpdateTaskInput { title?: string; done?: boolean; }

// 文件：code/ch23/api/src/errors.ts
// 让「业务错误」携带一个 HTTP 状态码，统一错误处理中间件据此决定返回什么码。
export class HttpError extends Error {
  status: number;
  constructor(status: number, message: string) {
    super(message);
    this.status = status;
    this.name = "HttpError";
  }
}
export class NotFoundError extends HttpError {
  constructor(message = "资源不存在") { super(404, message); this.name = "NotFoundError"; }
}
export class ValidationError extends HttpError {
  details: string[];
  constructor(details: string[]) {
    super(400, "请求校验失败");
    this.name = "ValidationError";
    this.details = details;
  }
}

逐段拆解：

  • 核心设计：让错误对象自己携带 HTTP 状态码。HttpError 继承内置 Error，多加一个 status 字段。
  • NotFoundError 固定 status = 404，ValidationError 固定 status = 400 并额外带一个 details 数组（具体哪些字段不合法）。
  • 这样一来，业务代码里只要 throw new NotFoundError(...)，无需关心 HTTP 细节；后面统一的错误中间件会 instanceof 判断类型，读出 status 和 message，转成合适的响应。抛错的地方和处理错的地方彻底解耦。

业务层：service

业务层把 store 的原始能力组合成“对上层有意义的操作“，并在这里落地业务规则（如“找不到就 404“、筛选、分页）。

// 文件：code/ch23/api/src/services/taskService.ts
import { NotFoundError } from "../errors.js";
import { taskStore } from "../store/taskStore.js";
import type { CreateTaskInput, Task, UpdateTaskInput } from "../types.js";

export interface ListQuery { done?: boolean; page: number; pageSize: number; }
export interface ListResult { items: Task[]; page: number; pageSize: number; total: number; }

export const taskService = {
  list(query: ListQuery): ListResult {
    let all = taskStore.findAll();
    if (query.done !== undefined) {
      all = all.filter((t) => t.done === query.done);       // 1) 筛选
    }
    all.sort((a, b) => (a.createdAt < b.createdAt ? 1 : -1)); // 2) 排序：新的在前
    const total = all.length;
    const start = (query.page - 1) * query.pageSize;
    const items = all.slice(start, start + query.pageSize);   // 3) 分页切片
    return { items, page: query.page, pageSize: query.pageSize, total };
  },
  getById(id: string): Task {
    const task = taskStore.findById(id);
    if (!task) throw new NotFoundError(`找不到 id 为 ${id} 的任务`);
    return task;
  },
  create(input: CreateTaskInput): Task {
    return taskStore.create({ title: input.title, done: input.done ?? false });
  },
  update(id: string, patch: UpdateTaskInput): Task {
    const updated = taskStore.update(id, patch);
    if (!updated) throw new NotFoundError(`找不到 id 为 ${id} 的任务`);
    return updated;
  },
  remove(id: string): void {
    if (!taskStore.remove(id)) throw new NotFoundError(`找不到 id 为 ${id} 的任务`);
  },
};

逐段拆解：

  • list 体现了列表接口的三板斧：筛选 → 排序 → 分页。先按 done 过滤，再按创建时间倒序，最后用 slice 切出当前页。返回体带上 total（筛选后总数），让客户端知道一共有多少条、能算出多少页。
  • 分页计算：第 page 页（从 1 开始）、每页 pageSize 条，起始下标是 (page - 1) * pageSize。这是分页最经典的公式，务必记牢。
  • getById/update/remove 在 store 返回“找不到“的信号（undefined/false）时，统一 throw new NotFoundError(...)。这就是“找不到就 404“这条业务规则的落点——它不该在 store 里（store 只管数据），也不该在 route 里重复写（每个用到 id 的路由都要判），放在 service 层集中处理最合适。
  • create 里的 input.done ?? false：用**空值合并（第 6 章）**给 done 一个默认值——客户端没传就默认“未完成“。

中间件：日志、CORS、错误处理

三个横切中间件。先看日志：

// 文件：code/ch23/api/src/middleware/logger.ts
import type { Request, Response, NextFunction } from "express";

export function requestLogger(req: Request, res: Response, next: NextFunction): void {
  const start = Date.now();
  // res 在响应发送完毕后触发 "finish"，这时才知道最终状态码和总耗时。
  res.on("finish", () => {
    const ms = Date.now() - start;
    console.log(`${req.method} ${req.originalUrl} → ${res.statusCode} (${ms}ms)`);
  });
  next(); // 一定要调用，否则请求卡死
}

逐段拆解：

  • 从 express 导入 Request/Response/NextFunction 三个类型（来自 @types/express），给参数标上类型——这就是“用 TS 写 Express“的日常。
  • 进入时记 start 时间戳。但此刻还不知道最终状态码（路由还没跑）。所以我们监听 res 的 "finish" 事件——它在响应完全发出后触发，那时 res.statusCode 才是最终值。
  • 计算耗时并打印一行 方法 路径 → 状态码 (耗时)。这种结构化日志对排查线上问题极有帮助。
  • 最后 next() 放行。这个中间件“只观察不拦截“，所以必须放行。

CORS 中间件——理解“CORS 本质就是加几个响应头“：

// 文件：code/ch23/api/src/middleware/cors.ts
import type { Request, Response, NextFunction } from "express";

export function simpleCors(req: Request, res: Response, next: NextFunction): void {
  res.setHeader("Access-Control-Allow-Origin", "*");          // 允许的来源
  res.setHeader("Access-Control-Allow-Methods", "GET,POST,PATCH,DELETE,OPTIONS");
  res.setHeader("Access-Control-Allow-Headers", "Content-Type, Authorization");
  if (req.method === "OPTIONS") {  // 浏览器的预检请求，直接回 204 放行
    res.sendStatus(204);
    return;
  }
  next();
}

逐段拆解：

  • **CORS（Cross-Origin Resource Sharing，跨源资源共享）**解决的是浏览器安全策略问题：默认情况下，https://a.com 的网页用 fetch 调 https://b.com 的 API 会被浏览器拦截。服务器需要通过 Access-Control-Allow-* 系列响应头，明确“我允许来自某某源的跨域请求“。
  • Access-Control-Allow-Origin: * 表示允许任意源。⚠️ 生产环境不要用 *，应收窄到你信任的具体域名，否则任何网站都能调你的 API。
  • 预检（preflight）：浏览器发跨域“非简单请求“（比如带 JSON body 的 POST）前，会先自动发一个 OPTIONS 请求“问路“。我们检测到 OPTIONS 就直接回 204 表示允许，不必往下走业务逻辑。
  • 💡 真实项目请用成熟的 cors 包，功能更全。这里手写只为让你看清 CORS 的本质——不过是几个响应头的约定。

最关键的统一错误处理中间件：

// 文件：code/ch23/api/src/middleware/errorHandler.ts
import type { Request, Response, NextFunction } from "express";
import { HttpError, ValidationError } from "../errors.js";

export function errorHandler(
  err: unknown,
  _req: Request,
  res: Response,
  _next: NextFunction, // 必须声明第 4 个参数，否则 Express 不把它当错误处理器
): void {
  if (err instanceof ValidationError) {
    res.status(err.status).json({ error: err.message, details: err.details });
    return;
  }
  if (err instanceof HttpError) {
    res.status(err.status).json({ error: err.message });
    return;
  }
  console.error("未处理的错误:", err);          // 未预期错误：打日志
  res.status(500).json({ error: "服务器内部错误" }); // 不泄露内部细节
}

export function notFoundHandler(_req: Request, res: Response): void {
  res.status(404).json({ error: "Not Found：请求的路径不存在" });
}

逐段拆解：

  • 错误处理中间件的“魔法特征“是它有 4 个参数 (err, req, res, next)。Express 靠“参数个数是否为 4“来识别它。⚠️ 即使你用不到 next，也必须把它写在签名里（哪怕加下划线前缀表示不用），否则 Express 会把它当成普通中间件，永远收不到错误。
  • 只有当某个中间件/路由抛出异常或调用 next(err) 时，Express 才会跳过后续普通中间件，直接进到错误处理中间件。
  • 处理逻辑分三档：ValidationError → 400 带 details；其它 HttpError（如 NotFoundError）→ 用它自带的 status 和 message；兜底 → 未预期的异常打到服务器日志、给客户端回 500 但不暴露内部堆栈（暴露堆栈是安全隐患）。
  • notFoundHandler 是“路径没匹配上“的兜底，注册在所有业务路由之后。它和错误处理器的区别：这个处理“路由没命中“，那个处理“处理过程中抛了错“。

📌 “统一错误处理“为什么重要？ 没有它，你得在每一个路由里写 try/catch，重复且容易漏。有了它，业务代码只管“正常流程 + 该抛错时抛错”，所有错误自动汇聚到一处转成规范响应。这是 Express 中间件机制最漂亮的应用之一。

请求校验

外部传进来的 body 是不可信的 unknown，必须校验后才能当作可信类型使用（呼应第 18 章“在边界收窄“的思想）。

// 文件：code/ch23/api/src/validation.ts（节选：创建校验）
import { ValidationError } from "./errors.js";
import type { CreateTaskInput } from "./types.js";

function isPlainObject(value: unknown): value is Record<string, unknown> {
  return typeof value === "object" && value !== null && !Array.isArray(value);
}

export function parseCreateTask(body: unknown): CreateTaskInput {
  const errors: string[] = [];
  if (!isPlainObject(body)) throw new ValidationError(["请求体必须是一个 JSON 对象"]);

  const { title, done } = body;
  if (typeof title !== "string" || title.trim() === "") {
    errors.push("title 必填，且必须是非空字符串");
  } else if (title.length > 200) {
    errors.push("title 长度不能超过 200 个字符");
  }
  if (done !== undefined && typeof done !== "boolean") {
    errors.push("done 如果提供，必须是布尔值");
  }
  if (errors.length > 0) throw new ValidationError(errors);

  return { title: (title as string).trim(), done: done as boolean | undefined };
}

逐段拆解：

  • isPlainObject 是一个类型守卫（type guard，第 18 章）：返回类型 value is Record<string, unknown>，校验通过后 TS 就知道 value 是对象，能安全解构。注意排除了 null 和数组（它们 typeof 也是 "object"）。
  • 校验策略是收集所有错误再一次性抛出，而不是遇到第一个错就返回——这样客户端一次就能看到所有问题，体验更好。
  • 每条规则都很朴素：title 必须是非空字符串且不超长；done 若提供必须是布尔。这就是“手写校验“，逻辑清晰、零依赖。
  • 校验全过后，返回一个类型为 CreateTaskInput 的可信对象（顺手 trim 了 title）。此后 service 层拿到的就是干净数据，无需再防。
  • parseUpdateTask（完整文件里）逻辑类似，额外规定“PATCH 至少要传 title 或 done 之一“，否则这个 PATCH 毫无意义。

💡 真实项目常用 zod 这类校验库来声明式地定义 schema，能自动推导类型、生成错误信息。手写校验是为了让你看清底层在做什么；学有余力可以把这里换成 zod 练手。

路由层：把 HTTP 和业务粘起来

路由层是最上面一层，只负责“HTTP 进出“：从 req 取参数、调 service、把结果写进 res。用 express.Router() 把 /tasks 相关路由封装成独立模块。

// 文件：code/ch23/api/src/routes/tasks.ts
import { Router } from "express";
import type { Request, Response } from "express";
import { taskService } from "../services/taskService.js";
import { parseCreateTask, parseUpdateTask } from "../validation.js";

export const tasksRouter = Router();

function parseListQuery(req: Request) {
  const { done, page, pageSize } = req.query;
  let doneFilter: boolean | undefined;
  if (done === "true") doneFilter = true;
  else if (done === "false") doneFilter = false;
  const pageNum = Math.max(1, Number.parseInt(String(page ?? "1"), 10) || 1);
  const sizeNum = Math.min(100, Math.max(1, Number.parseInt(String(pageSize ?? "10"), 10) || 10));
  return { done: doneFilter, page: pageNum, pageSize: sizeNum };
}

tasksRouter.get("/", (req: Request, res: Response) => {
  res.status(200).json(taskService.list(parseListQuery(req)));
});

tasksRouter.get("/:id", (req: Request, res: Response) => {
  res.status(200).json(taskService.getById(req.params.id));
});

tasksRouter.post("/", (req: Request, res: Response) => {
  const created = taskService.create(parseCreateTask(req.body));
  res.setHeader("Location", `/tasks/${created.id}`); // 指向新资源
  res.status(201).json(created);
});

tasksRouter.patch("/:id", (req: Request, res: Response) => {
  const updated = taskService.update(req.params.id, parseUpdateTask(req.body));
  res.status(200).json(updated);
});

tasksRouter.delete("/:id", (req: Request, res: Response) => {
  taskService.remove(req.params.id);
  res.status(204).end(); // 无响应体
});

逐段拆解：

  • Router() 创建一个“迷你 app“：你在它上面注册的路由，之后会被整体挂到某个前缀下。这让主应用文件保持干净，路由按资源分文件。
  • parseListQuery 专门处理查询参数的“脏活“。还记得吗——查询参数永远是字符串。所以 done 要跟字符串 "true"/"false" 比较；page/pageSize 要 parseInt 转数字，并做边界兜底（页码至少 1，每页 1~100）。这些防御让接口面对乱七八糟的输入也不会崩。
  • 每个路由都极薄：取参数 → 调 service → 写响应。没有 try/catch——因为校验和 service 抛出的错误会自动冒泡到统一错误处理中间件。这正是分层 + 统一错误处理带来的简洁。
  • 状态码严格遵守 REST 约定：查询/更新成功 200，创建成功 201（并用 Location 头指向新资源），删除成功 204（res.end() 不带 body）。
  • ⚠️ 关于异步错误：本章的处理器都是同步的，同步抛的错 Express 4 能自动接住。但如果处理器是 async 的（第 24 章接数据库后就是），async 函数里 await 抛的错 Express 4 不会自动接——你得手动 try/catch 后 next(err)，或用一个 asyncHandler 包裹器。这个坑我们留到第 24 章处理真实异步时再展开。

组装：app 与 server

最后把所有部件拼起来。刻意把“创建 app“和“监听端口“分成两个文件——这样测试能只 import app 而不真的占用端口。

// 文件：code/ch23/api/src/app.ts
import express from "express";
import { requestLogger } from "./middleware/logger.js";
import { simpleCors } from "./middleware/cors.js";
import { errorHandler, notFoundHandler } from "./middleware/errorHandler.js";
import { tasksRouter } from "./routes/tasks.js";

export function createApp() {
  const app = express();

  // —— 全局中间件，注册顺序 = 执行顺序 ——
  app.use(simpleCors);       // 1) CORS 尽早，好让预检快速返回
  app.use(requestLogger);    // 2) 日志
  app.use(express.json());   // 3) 解析 JSON body → req.body

  app.get("/health", (_req, res) => res.status(200).json({ status: "ok" }));

  app.use("/tasks", tasksRouter); // 把 router 挂到 /tasks 前缀下

  // —— 收尾：404 与统一错误处理，必须放在最后 ——
  app.use(notFoundHandler);
  app.use(errorHandler);
  return app;
}

// 文件：code/ch23/api/src/server.ts
import { createApp } from "./app.js";

const app = createApp();
const PORT = Number(process.env.PORT ?? 3000);
app.listen(PORT, () => {
  console.log(`任务 API 已启动：http://localhost:${PORT}`);
});

逐段拆解：

  • 中间件注册顺序反映流水线顺序：CORS → 日志 → JSON 解析 → 路由 → 404 → 错误处理。这个顺序不是随意的：CORS 要最早（好处理预检）；express.json() 必须在路由之前（否则路由里 req.body 是空的）；notFoundHandler 和 errorHandler 必须在最后（前者兜没命中的路径，后者兜所有抛出的错）。
  • app.use("/tasks", tasksRouter)：把 router **挂载（mount）**到 /tasks 前缀。于是 router 里写的 "/" 实际对应 /tasks，"/:id" 对应 /tasks/:id。前缀在这里统一加，router 内部不用重复写。
  • createApp() 返回配好的 app 但不监听；server.ts 才 listen。这个拆分让 scripts/smoke.ts 能 import { createApp } 后自己控制端口，方便测试（这正是 supertest 的核心思路）。
  • process.env.PORT ?? 3000 让端口可通过环境变量配置——配置从环境读，是第 24 章的重要主题。

启动它：

cd code/ch23/api
npm install     # 首次运行前必须

看到 任务 API 已启动：http://localhost:3000 就说明服务跑起来了。

测试 API：curl 与冒烟脚本

服务跑起来后，怎么验证它对不对？两种方式。

方式一：curl 手动打

curl 是命令行 HTTP 客户端，最适合手动摸接口。-i 显示响应头（含状态码），-X 指定方法，-H 加请求头，-d 带 body。

# 1) 创建一个任务
curl -i -X POST http://localhost:3000/tasks \
  -H "Content-Type: application/json" \
  -d '{"title":"买牛奶"}'

预期响应 201 Created：

HTTP/1.1 201 Created
Location: /tasks/…uuid…
Content-Type: application/json; charset=utf-8

{"id":"…uuid…","title":"买牛奶","done":false,"createdAt":"2026-07-02T08:00:00.000Z"}

# 2) 校验失败：空 title → 400，带 details
curl -i -X POST http://localhost:3000/tasks \
  -H "Content-Type: application/json" -d '{"title":""}'
# → 400 {"error":"请求校验失败","details":["title 必填，且必须是非空字符串"]}

# 3) 列表（支持筛选与分页）
curl "http://localhost:3000/tasks"
curl "http://localhost:3000/tasks?done=false&page=1&pageSize=5"
# → {"items":[…],"page":1,"pageSize":5,"total":1}

# 4) 查询单个（把上面返回的 id 填进去）
curl http://localhost:3000/tasks/<id>          # → 200 + 任务对象
curl -i http://localhost:3000/tasks/not-exist  # → 404

# 5) 局部更新：标记完成
curl -X PATCH http://localhost:3000/tasks/<id> \
  -H "Content-Type: application/json" -d '{"done":true}'
# → 200 + 更新后的任务

# 6) 删除
curl -i -X DELETE http://localhost:3000/tasks/<id>  # → 204 No Content

逐段拆解：把这六步跑一遍，你就完整验证了 CRUD（增查改删）全流程，以及校验失败（400）和资源不存在（404）两条错误路径。注意观察状态码——它们精确对应了我们在 REST 小节定的约定。

方式二：冒烟脚本（supertest 思想）

手动 curl 适合探索，但每次改代码都手打一遍太累。更好的做法是用代码把整套流程跑一遍并自动断言——这就是 supertest 这类库的思路。我们不引入新依赖，直接用 Node 内置的 fetch 写一个冒烟脚本：

// 文件：code/ch23/api/scripts/smoke.ts（节选）
import { createApp } from "../src/app.js";

const app = createApp();
// 监听随机端口（0 = 系统分配），拿到端口后用 fetch 打自己
const server = await new Promise<import("node:http").Server>((resolve) => {
  const s = app.listen(0, () => resolve(s));
});
const addr = server.address();
const port = typeof addr === "object" && addr ? addr.port : 3000;
const base = `http://localhost:${port}`;

let res = await fetch(`${base}/tasks`);
assert(res.status === 200, "GET /tasks → 200");

常见坑与最佳实践

  • ⚠️ 忘了 app.use(express.json())，req.body 是 undefined。 这是新手第一大坑。Express 默认不解析 body，必须显式挂上解析中间件，且要挂在用到 req.body 的路由之前。
  • ⚠️ 查询参数永远是字符串。 ?done=true 里 req.query.done 是 "true" 不是 true；?page=2 是 "2" 不是 2。当布尔/数字用之前必须手动转换并做边界防御。
  • ⚠️ 中间件忘了 next()。 观察型中间件（日志、CORS 放行）如果忘了调 next() 又没结束响应，请求会卡到超时。写完中间件先问自己：“我放行了吗？”
  • ⚠️ 错误处理中间件必须有 4 个参数。 少写一个 next，Express 就不认它是错误处理器。而且它必须注册在所有路由之后。
  • ⚠️ Express 4 不自动接 async 错误。 同步抛错能接住，但 async 处理器里 await 抛的错默认会“漏掉“。接数据库（第 24 章）后处理器变异步，务必用 try/catch + next(err) 或 asyncHandler 包裹。（Express 5 已改进这一点。）
  • ⚠️ 别在 5xx 里泄露内部细节。 未预期错误给客户端回笼统的 500 服务器内部错误，具体堆栈只打到服务器日志。暴露堆栈会泄露代码结构，是安全隐患。
  • 📌 状态码要用对。 客户端错误（校验失败）是 4xx 不是 5xx；创建用 201、删除用 204。状态码是 API 的“第一语言“，用错会误导调用方。
  • 📌 分层不是过度设计。 哪怕内存 store 只有几行，把它单独分层的价值会在第 24 章换数据库时兑现——上层几乎不用改。
  • 💡 CORS 生产别用 *。 把 Access-Control-Allow-Origin 收窄到你信任的域名。开发用成熟的 cors 包而非手写。
  • 💡 校验放在最外层（路由入口）。 让不可信数据在进入 service 前就被拦下，service 及以下各层拿到的永远是干净数据。

本章小项目：任务管理 REST API

前面的代码其实已经把小项目讲完了——它就是 code/ch23/api/ 这个完整、可运行、分层清晰的子项目。这里把它作为一个整体回顾一遍，帮你建立“全景图“。

目标：用 Express + TypeScript 实现任务（Task）管理 REST API，含 GET /tasks（?done= 筛选 + 分页）、GET /tasks/:id、POST /tasks（带校验）、PATCH /tasks/:id、DELETE /tasks/:id，数据存内存。

分层与文件：

src/
  server.ts               入口：createApp() 后 listen
  app.ts                  组装：中间件流水线 + 路由挂载 + 错误收口
  types.ts                Task 及输入类型
  errors.ts               HttpError / NotFoundError / ValidationError
  validation.ts           parseCreateTask / parseUpdateTask（手写校验）
  routes/tasks.ts         路由层（express.Router，只做 HTTP 进出）
  services/taskService.ts 业务层（筛选/分页/找不到就 404）
  store/taskStore.ts      数据层（内存数组，第 24 章换数据库）
  middleware/
    logger.ts             请求日志
    cors.ts               极简 CORS
    errorHandler.ts       统一错误处理 + 404 兜底
scripts/smoke.ts          冒烟测试（supertest 思想）

一个请求的完整旅程（以 POST /tasks 为例，把全章串起来）：

  1. 请求进入 → simpleCors 加 CORS 头 → requestLogger 记下起点 → express.json() 把 body 解析进 req.body。
  2. 匹配到 tasksRouter 的 POST /。处理器调 parseCreateTask(req.body) 校验：不合法就 throw ValidationError，直接跳到错误中间件回 400。
  3. 校验通过 → 调 taskService.create(input) → service 调 taskStore.create(...) 往内存数组里塞一条、生成 id 和时间戳。
  4. 处理器设 Location 头、res.status(201).json(created) 返回新资源。
  5. 响应发出，res 触发 "finish"，requestLogger 打印 POST /tasks → 201 (xms)。

小结与自测

这一章我们完成了从“写脚本“到“写服务“的跨越：

  • HTTP 骨架：请求/响应、方法、状态码、headers、body、JSON，是所有后端沟通的公共语言。

  • 工程化：用 TS + @types/express 分层组织（route→service→store + 中间件），手写校验、统一错误处理、请求日志、CORS，并用 curl 和冒烟脚本验证。
