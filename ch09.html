JavaScript & TypeScript 从入门到精通

第 9 章 · 模块化与包管理

本章目标

到目前为止，我们写的代码基本都挤在一个文件里。文件小的时候没问题，可一旦程序变大——几百行、几千行——所有函数、变量堆在一起，就会变成一团乱麻：改一个地方牵动全身，两个功能不小心用了同名变量互相覆盖，别人也读不懂你的代码从哪儿开始看。

模块化（modularity）就是解决这个问题的。它让我们把代码拆成一个个独立的模块（module），每个模块只关心自己的事，通过明确的“出口“和“入口“互相协作。学完本章，你将能够：

  • 说清楚“为什么需要模块化“，理解全局污染和依赖管理的历史。
  • 熟练使用 ES Modules（ESM）：命名导出、默认导出、各种 import 写法、再导出、动态 import()。
  • 理解 CommonJS（CJS），知道它和 ESM 的区别，以及在 Node.js 里怎么选。
  • 读懂并写出规范的 package.json，理解语义化版本（semantic versioning，简称 semver）。
  • 使用 npm 安装、运行、管理第三方包。
  • 了解把自己的包发布到 npm 的完整流程。
  • 亲手做一个可发布的 npm 工具包 str-kit。

本章所有示例用 Node.js 运行。请确认你已经安装了 Node.js（终端里 node -v 能看到版本号，本书用 Node 20+）。

为什么需要模块化

一切从“全局变量“说起

早期的 JavaScript 只跑在浏览器里，加载代码的唯一方式是一个个 <script> 标签：

// a.js
var count = 0;
function increase() { count++; }

// b.js
var count = 100; // 糟糕！和 a.js 用了同一个名字

浏览器把所有 <script> 里的顶层 var 和函数都塞进同一个全局作用域（global scope）。上面 b.js 的 count 会直接覆盖 a.js 的 count，两个文件的作者可能根本不认识彼此。这种“大家共用一个命名空间、互相踩脚“的现象，叫做全局污染（global pollution）。

📌 模块化要解决的第一个核心问题：让每个文件有自己独立的作用域，变量默认不外泄，只有主动“导出“的东西才对外可见。

依赖管理的历史（了解即可）

在标准的模块系统出现之前，社区想了很多办法：

  • IIFE + 命名空间：用立即执行函数把变量包起来，只往一个全局对象上挂东西。
  • CommonJS：Node.js 采用的方案，用 require() 和 module.exports，让服务器端 JS 有了真正的模块。
  • AMD / UMD：浏览器端的异步模块方案（如 RequireJS），现在基本被淘汰。
  • ES Modules（ESM）：2015 年随 ES6 进入语言标准，是官方的、跨浏览器和 Node 通用的模块系统。

💡 今天写新代码，首选 ESM。CJS 你仍然需要认识，因为海量的老包和 Node 内部还在用它。本章两者都讲，但以 ESM 为主。

ES Modules（ESM）基础

ESM 的核心就两个关键字：export（导出，把东西对外提供）和 import（导入，把别的模块的东西拿进来）。

我们先做一个能跑的例子。目录结构：

code/ch09/esm-demo/
├── math.js   （被导入的模块）
└── main.js   （入口，导入并使用 math.js）

导出：命名导出与默认导出

// 文件：code/ch09/esm-demo/math.js

// 命名导出（named export）：可以有很多个。
export const PI = 3.14159;

export function add(a, b) {
  return a + b;
}

export function multiply(a, b) {
  return a * b;
}

// 也可以先声明、最后统一导出（效果相同）。
function subtract(a, b) {
  return a - b;
}
export { subtract };

// 默认导出（default export）：每个模块最多一个。
export default {
  name: "mini-calculator",
  version: "1.0.0",
};

// 模块顶层的代码只会在「第一次被导入」时执行一次。
console.log("[math.js] 模块被加载了（只会看到一次这行）");

逐段拆解：

  • export const PI = ... 和 export function add(...)：直接在声明前面加 export，这就是命名导出。它的特点是导出时有名字，别人导入时要用同样的名字。一个模块可以有任意多个命名导出。
  • export { subtract }：如果你想先写完函数、最后统一列出要导出哪些，可以用这种花括号写法。效果和写在声明前一样。
  • export default { ... }：默认导出。一个模块最多只能有一个 default。它常用来表达“这个模块最主要的那个东西“。注意 default 导出的东西没有名字，导入方可以随便给它起名。
  • 最后那行 console.log：模块顶层的普通语句在第一次被加载时会执行。这一点我们待会儿在 main.js 里验证。

导入：各种 import 写法

// 文件：code/ch09/esm-demo/main.js

// 1) 按名字导入命名导出（大括号里的名字必须和导出的一致）。
import { add, multiply, PI } from "./math.js";

// 2) 默认导出用任意名字接收，不加大括号。
import calculator from "./math.js";

// 3) 导入时重命名，避免和本地变量冲突。
import { subtract as minus } from "./math.js";

// 4) 用命名空间对象一次性拿到所有命名导出。
import * as mathNS from "./math.js";

console.log("add(2, 3) =", add(2, 3));
console.log("multiply(4, 5) =", multiply(4, 5));
console.log("PI =", PI);
console.log("minus(10, 4) =", minus(10, 4));
console.log("默认导出：", calculator.name, calculator.version);
console.log("命名空间 mathNS.PI =", mathNS.PI);
console.log("命名空间里的默认导出 mathNS.default.name =", mathNS.default.name);

// 5) 再次导入同一个模块，不会重新执行 math.js 的顶层代码（单例缓存）。
const again = await import("./math.js"); // 动态 import()，返回一个 Promise
console.log("动态导入拿到 add：", again.add(1, 1));
console.log("（注意：上面没有再次打印 [math.js] 模块被加载了）");

逐段拆解：

  • ①命名导入：import { add, multiply, PI } from "./math.js"。花括号里的名字必须和 math.js 里导出的名字一模一样。⚠️ 注意 from 后面的路径是相对路径，而且 ESM 里必须写完整的扩展名 .js（这和 CJS 不同，CJS 可以省略）。
  • ②默认导入：import calculator from "./math.js"，不加花括号。这里 calculator 是我们自己起的名字——它接收的是 math.js 的 export default。你想叫它 calc、abc 都行。
  • ③重命名导入：import { subtract as minus }。用 as 把导入进来的名字改掉，常用于避免和本地变量重名。
  • ④命名空间导入：import * as mathNS from "./math.js"。把该模块所有命名导出打包成一个对象 mathNS，通过 mathNS.PI、mathNS.add 访问。默认导出会被放在 mathNS.default 上。
  • ⑤动态 import()：前面四种是静态导入，必须写在文件顶层。而 import("./math.js") 是函数调用形式，返回一个 Promise，可以写在任何地方（比如 if 里、按需加载）。这行还顺带用到了顶层 await（稍后讲）。

运行它：

cd code/ch09/esm-demo
node main.js

输出：

[math.js] 模块被加载了（只会看到一次这行）
add(2, 3) = 5
multiply(4, 5) = 20
PI = 3.14159
minus(10, 4) = 6
默认导出： mini-calculator 1.0.0
命名空间 mathNS.PI = 3.14159
命名空间里的默认导出 mathNS.default.name = mini-calculator
动态导入拿到 add： 2
（注意：上面没有再次打印 [math.js] 模块被加载了）

📌 请特别注意两点：

  1. [math.js] 模块被加载了 只打印了一次，尽管我们多次导入它（静态导入 + 动态导入）。这就是 ESM 的单例缓存（module caching / singleton）：一个模块无论被导入多少次、被多少个文件依赖，它的代码只执行一次，结果被缓存，后续都拿缓存。
  2. 它是第一行打印的。因为 ESM 的静态导入会在当前模块的其他代码执行之前，先把依赖全部加载好。

模块的两个隐藏特性

📌 ESM 自动是严格模式（strict mode）。你不用写 "use strict"，模块里就已经是严格模式了。所以像“给未声明的变量赋值“这种在普通脚本里会悄悄创建全局变量的操作，在模块里会直接报错——这正好帮你避免全局污染。

📌 每个模块有独立作用域。模块顶层用 const/let/function 声明的东西默认是私有的，外界看不到，除非你 export。这从根本上杜绝了本章开头那种“两个文件抢同一个全局 count“的问题。

再导出（export from）

有时你想把好几个小模块“打包“成一个统一入口，让使用者只需从一个地方导入。这时用再导出（re-export）：

// 假设有一个 index.js 想把 strings.js 的东西原样转发出去
export * from "./src/strings.js";             // 转发所有命名导出
export { capitalize } from "./src/strings.js"; // 只转发指定的
export { slugify as toSlug } from "./src/strings.js"; // 转发并改名

这就是“入口文件（entry point）“或“桶文件（barrel file）“的常见做法，稍后的 str-kit 项目会用到。

动态 import() 与顶层 await

动态 import() 长得像函数调用，返回 Promise，用于按需、在运行时加载模块：

// 只有真的需要时才加载某个较重的模块
async function loadWhenNeeded() {
  const { add } = await import("./math.js");
  return add(1, 2);
}

它的价值在于：静态 import 必须写在顶层、总是会被加载；而动态 import() 可以放在条件分支里，实现“用到才加载“（在浏览器里这能显著减小首屏体积）。

顶层 await（top-level await）：在 ESM 模块里，你可以直接在模块最外层写 await，不必包在 async 函数里。前面 main.js 里的 await import("./math.js") 就是这么用的。

// 在 ESM 模块顶层直接 await（普通脚本里不行）
const data = await fetch("https://example.com/data.json").then(r => r.json());
console.log(data);

⚠️ 顶层 await 只在 ESM 里可用，且会让“依赖这个模块的其他模块“也等它完成。别拿它做很慢的操作，否则会拖慢整个加载。

CommonJS（CJS）

CommonJS 是 Node.js 诞生时自带的模块系统，你会在无数老项目和 npm 包里见到它。它的两个关键字是 require（导入）和 module.exports（导出）。

// CJS 导出（注意：这是 CommonJS 写法，不是 ESM）
function add(a, b) { return a + b; }
const PI = 3.14159;

module.exports = { add, PI };
// 或者一个个挂：
// exports.add = add;

// CJS 导入
const { add, PI } = require("./math"); // 注意：可以省略 .js 扩展名
console.log(add(1, 2), PI);

CJS 和 ESM 的关键区别

这几点是面试和实战里最常问的，我们一条条讲清楚：

对比点CommonJS (CJS)ES Modules (ESM)
关键字require / module.exportsimport / export
加载时机运行时同步加载（执行到 require 那一刻才去读文件）静态分析（导入关系在代码运行前就确定）
导入的是导出对象的值拷贝（copy）到原变量的引用绑定（live binding）
路径扩展名可省略（./math）必须写全（./math.js）
严格模式默认非严格默认严格
顶层 await不支持支持

其中两点最容易踩坑，重点解释：

📌 静态 vs 动态。ESM 的 import 必须写在顶层、路径必须是字符串常量，因为工具需要在运行前就分析出“谁依赖谁“（这样才能做打包、Tree Shaking 等优化）。CJS 的 require 是普通函数，可以写在任何地方、路径可以是变量，是运行时才解析的。

📌 值拷贝 vs 引用绑定。看这个对比：

// counter.cjs（CommonJS）
let count = 0;
setInterval(() => count++, 1000);
module.exports = { count };
// 导入方拿到的是导出那一刻 count 的「拷贝」，之后 count 变了它也不知道。

// counter.mjs（ESM）
export let count = 0;
setInterval(() => count++, 1000);
// 导入方拿到的是「实时绑定」，count 变化能被看到（但导入方不能给它赋值）。

在 CJS 里，module.exports = { count } 导出的是那一瞬间 count 的值（0），之后内部再改 count，外面拿到的还是 0。在 ESM 里，导入的命名绑定是“活的“，内部改了，外面读到的也是新值。这个差异偶尔会导致费解的 bug。

Node.js 里怎么选：.mjs / .cjs / "type"

Node.js 同时支持两套系统，它靠文件扩展名和 package.json 的 "type" 字段来判断某个 .js 文件到底是 ESM 还是 CJS：

  • 扩展名 .mjs → 永远当作 ESM。
  • 扩展名 .cjs → 永远当作 CJS。
  • 扩展名 .js → 看最近的 package.json 的 "type"：
    • "type": "module" → 当作 ESM。
    • "type": "commonjs" 或没写 "type" → 当作 CJS（默认）。

💡 本书的示例目录 code/ 里有一个 "type": "module" 的 package.json，所以我们所有 .js 文件都被当作 ESM，这也是为什么前面 math.js/main.js 用 import 能直接跑。如果你在一个没有 "type": "module" 的目录里想用 ESM，把文件名改成 .mjs 即可。

ESM 与 CJS 互操作的注意点

现实中你经常需要在 ESM 里用一个 CJS 老包，或反过来。规则：

  • ✅ ESM 里 import 一个 CJS 模块：可以。module.exports 会被当作那个模块的默认导出。所以通常写 import pkg from "some-cjs-pkg"，然后 pkg.foo()。有些包 Node 能“猜“出命名导出，但不保证。
  • ⚠️ CJS 里 require 一个 ESM 模块：传统上不行（会报错），因为 require 是同步的、ESM 加载可能是异步的。老代码里要用 ESM 得改成动态 import()：const mod = await import("esm-only-pkg")。（较新的 Node 版本对这点有所放宽，但不是所有版本都支持，写库时不要依赖它。）

⚠️ 最常见的报错之一：“require() of ES Module … not supported”——这就是在 CJS 里 require 了一个纯 ESM 包。解决办法通常是把你的代码也改成 ESM，或用动态 import()。

package.json 详解

每一个 Node 项目（以及每一个 npm 包）的根目录都有一个 package.json。它是项目的“身份证 + 说明书 + 配置表“。用 npm init -y 可以生成一个默认的。下面是一个较完整的例子，逐字段讲：

{
  "name": "str-kit",
  "version": "1.2.3",
  "description": "几个常用的字符串小工具。",
  "type": "module",
  "main": "index.js",
  "exports": {
    ".": "./index.js"
  },
  "scripts": {
    "test": "vitest run",
    "build": "tsc"
  },
  "dependencies": {
    "lodash": "^4.17.21"
  },
  "devDependencies": {
    "vitest": "^2.1.8"
  },
  "peerDependencies": {
    "react": ">=18"
  }
}

name 与 version

  • name：包名。发布到 npm 时它必须全网唯一。只能小写，可含连字符。也可以带作用域（scope），如 @yourname/str-kit。
  • version：版本号，遵循语义化版本（semver）。这是重点，单独讲。

语义化版本 semver（重点，别讲错）

版本号是三段 主版本.次版本.修订号，即 MAJOR.MINOR.PATCH，例如 1.2.3：

  • MAJOR（主版本，1）：做了不兼容的改动（breaking change），升级它意味着“老代码可能跑不了了“。
  • MINOR（次版本，2）：向后兼容地新增了功能，老代码照样能跑。
  • PATCH（修订号，3）：向后兼容的问题修复（bug fix），没加新功能。

在 dependencies 里写依赖时，版本号前面常有 ^ 或 ~，它们表示“允许自动升级到哪个范围“：

📌 ^（脱字符，caret）：允许升级到不改变最左边非零数字的最新版本。对最常见的 1.x.x，通俗说就是“MAJOR 不变“。

  • ^1.2.3 → 允许 >=1.2.3 且 <2.0.0（可升到 1.9.9，但不会到 2.0.0）。
  • ^0.2.3 → 因为最左非零是 2（在 MINOR 位），允许 >=0.2.3 且 <0.3.0。（⚠️ 0.x 版本被认为不稳定，^ 对它更保守，只升 PATCH。）
  • ^0.0.3 → 允许 >=0.0.3 且 <0.0.4，几乎锁死。

📌 ~（波浪号，tilde）：一般允许只升 PATCH（MINOR 锁定）。

  • ~1.2.3 → 允许 >=1.2.3 且 <1.3.0（只升修订号）。
  • ~1.2 → 允许 >=1.2.0 且 <1.3.0。

对比记忆：^ 允许升 MINOR 和 PATCH，~ 通常只允许升 PATCH。所以 ^ 更宽松，~ 更保守。此外还有 1.2.3（精确锁定）、*（任意版本，危险）、>=18（范围）等写法。

💡 npm 默认给你装的依赖用 ^，这是社区约定：相信别人会遵守 semver，MINOR/PATCH 升级不会弄坏你的代码。

type、main、module、exports

  • type：前面讲过，"module" 让本包的 .js 按 ESM 解析，"commonjs"（或不写）按 CJS。
  • main：包的入口文件。别人 import "str-kit" 时，Node 默认找的就是 main 指向的文件。这是最老、最通用的字段。
  • module：一个非官方约定字段，用来单独指明“ESM 版本的入口“，主要给打包工具看。Node 本身不认它，能不用就不用。
  • exports：现代、官方推荐的入口声明方式，比 main 强大得多。它能精确控制“外部能导入哪些路径“（相当于给包设了一道门），还能针对 ESM/CJS 提供不同文件：
{
  "exports": {
    ".": "./index.js",
    "./utils": "./src/utils.js"
  }
}

上面表示：import "str-kit" 走 index.js，import "str-kit/utils" 走 src/utils.js。⚠️ 一旦写了 exports，没在里面列出的文件就无法从外部导入（哪怕它在包目录里），这是它比 main 更“严格“的地方。

scripts

scripts 里定义可以用 npm run <名字> 运行的命令：

"scripts": {
  "dev": "node server.js",
  "test": "vitest run",
  "build": "tsc"
}
  • npm run dev 就等于执行 node server.js。
  • test、start 等少数几个“特殊名字“可以省略 run（npm test、npm start）。其余的都得写 npm run xxx。
  • 💡 script 里可以直接调用装在 node_modules/.bin 里的工具（如 vitest、tsc），不用写完整路径——npm 会自动把它加进 PATH。

dependencies / devDependencies / peerDependencies

这三种依赖的区别很重要：

  • dependencies（运行时依赖）：你的代码运行时真正需要的包。别人安装你的包时，这些会被一起装上。例：你的库内部用了 lodash。
  • devDependencies（开发依赖）：只在开发/构建/测试阶段需要，运行时用不到。别人安装你的包时不会装它们。例：测试框架 vitest、编译器 typescript。安装时加 -D 就进这里。
  • peerDependencies（同伴依赖）：表示“我需要宿主项目里存在某个包，但由宿主自己安装，我不重复装“。最典型的是插件类库：一个 React 组件库会把 react 写成 peerDependency，意思是“用我的人自己已经装了 React，我用你那份就行，别装两份导致冲突“。

⚠️ 一个常见误区：把只在测试里用的东西写进 dependencies。这会让所有使用者被迫下载它们，拖慢安装、增大体积。判断标准很简单：运行你发布的代码时用不到的，就放 devDependencies。

npm 基础

**npm（Node Package Manager）**是随 Node.js 一起安装的包管理器。它做两件事：管理项目依赖、运行脚本。

npm init：创建 package.json

npm init          # 交互式问答，一步步生成
npm init -y       # 全部用默认值，直接生成（-y = yes）

npm install：安装依赖

npm install                 # 按 package.json 装齐所有依赖
npm install lodash          # 装 lodash，并写入 dependencies
npm install -D vitest       # 装 vitest 到 devDependencies（-D = --save-dev）
npm install -g http-server  # 全局安装（-g），装到系统里，供命令行随处调用
npm uninstall lodash        # 卸载

（npm i 是 npm install 的简写。）

node_modules 与 package-lock.json

装完之后，你会看到两样东西：

  • node_modules/ 目录：所有依赖（以及依赖的依赖）的实际代码都放这里。它可能非常大、文件极多。📌 永远不要把它提交到 Git，在 .gitignore 里忽略它。别人克隆你的项目后 npm install 就能重新生成。
  • package-lock.json 文件：记录了这次安装每个包被解析成的确切版本和依赖树。它保证团队里每个人、以及服务器上，装出来的依赖完全一致。📌 这个文件要提交到 Git。 package.json 写的是“我想要的范围“（如 ^1.2.3），lock 文件记的是“实际锁定的版本“（如 1.9.0）。

npm run：运行脚本

npm run build       # 运行 scripts.build
npm test            # 特殊名字，等于 npm run test
npm start           # 特殊名字，等于 npm run start

npx：临时执行包里的命令

npx 能执行一个命令行工具，而不需要先全局安装它：

npx create-vite my-app   # 临时下载并运行 create-vite，用完不留在全局
npx cowsay "hello"

它会优先用项目 node_modules 里已有的版本，没有就临时下载。特别适合那种“一次性脚手架工具“，避免污染全局环境。

pnpm / yarn（了解即可）

除了 npm，社区还有两个流行的包管理器，命令大同小异：

  • yarn：早期以速度快著称，命令如 yarn add lodash。
  • pnpm：用硬链接共享依赖，省磁盘、装得快，现在很受欢迎，命令如 pnpm add lodash。

💡 三者用的都是同一个 npm 仓库（registry），装的是同一批包，区别主要在安装策略和速度。初学阶段用自带的 npm 就够了。

从 npm 安装并使用第三方包（完整流程）

以给项目加一个 UUID 生成库为例，完整走一遍：

# 1. 进入你的项目目录，确保有 package.json（没有就 npm init -y）
cd my-project

# 2. 安装第三方包（会写进 dependencies，生成/更新 lock 文件）
npm install uuid

# 3. 在代码里导入使用（见下），然后运行
node index.js

// 文件：my-project/index.js（假设本项目是 "type": "module"）
import { v4 as uuidv4 } from "uuid";

console.log("新生成的 ID：", uuidv4());

背后发生了什么：npm install uuid 从 npm 仓库下载 uuid 及它的依赖到 node_modules/，在 package.json 的 dependencies 里记一行 "uuid": "^…"，并把确切版本写进 package-lock.json。之后 import ... from "uuid" 时，Node 发现这不是相对路径（不以 ./ ../ / 开头），就去 node_modules/ 里按名字找这个包，读它 package.json 的 exports/main 找到入口。

📌 裸模块名 vs 相对路径：import x from "uuid"（裸名，bare specifier）→ 去 node_modules 找包；import x from "./math.js"（相对路径）→ 找你自己的文件。

发布到 npm 的流程简介

⚠️ 本节只讲流程，不要求你真的发布（发布会占用全网唯一的包名，请勿用练习包污染 npm 仓库）。了解流程即可。

假设你的包已经准备好（有规范的 package.json、代码、README），发布大致是：

# 1. 注册并登录 npm 账号（需要先在 npmjs.com 注册）
npm login

# 2. 检查将要发布的内容（不会真的发布，只是预览打包了哪些文件）
npm publish --dry-run

# 3. 正式发布
npm publish
# 带作用域的包若想公开发布，要加 --access public：
# npm publish --access public

发布前要控制“哪些文件会被打进包里“，有两种方式：

  • files 字段（推荐，白名单）：在 package.json 里列出要发布的文件/目录，只有列出的会被发布（外加 package.json、README、LICENSE 这些总会被带上）。
"files": ["index.js", "src/", "README.md"]
  • .npmignore（黑名单）：像 .gitignore 一样列出不发布的东西。如果没有 .npmignore，npm 会退而使用 .gitignore。

💡 用 files 白名单通常更安全——你明确知道发布了什么，不会不小心把测试文件、密钥、node_modules 打进去。

发布新版本前，记得按 semver 升级版本号（不能重复发布同一个版本）。npm 提供了便捷命令：

npm version patch   # 1.2.3 -> 1.2.4（修 bug）
npm version minor   # 1.2.3 -> 1.3.0（加功能）
npm version major   # 1.2.3 -> 2.0.0（不兼容改动）

常见坑与最佳实践

  • ⚠️ ESM 里忘写扩展名。import x from "./math" 在 Node 的 ESM 下会报 ERR_MODULE_NOT_FOUND，必须写 "./math.js"。CJS 才能省略。
  • ⚠️ 搞混 export default 和命名导出。默认导出用 import x from "..."（无花括号）；命名导出用 import { x } from "..."（有花括号）。名字对不上就是 undefined。
  • ⚠️ "require() of ES Module not supported"。在 CJS 里 require 了纯 ESM 包。把代码改成 ESM，或用动态 import()。
  • ⚠️ 把 node_modules 提交进 Git，却忘了提交 package-lock.json。恰好反了：node_modules 要忽略，lock 文件要提交。
  • ⚠️ 把开发工具写进 dependencies。测试框架、编译器等只在开发时用的，应放 devDependencies（npm i -D）。
  • ⚠️ 误解 semver。再强调：^ 锁 MAJOR（可升 MINOR/PATCH），~ 通常只升 PATCH。0.x 版本下 ^ 会更保守。
  • 💡 一个模块一件事。每个模块聚焦单一职责，导出清晰。用“桶文件“（index.js 再导出）给使用者一个统一入口。
  • 💡 优先命名导出。命名导出对自动补全、重构、Tree Shaking 更友好；默认导出留给“这个模块最核心的那一个东西“。
  • 💡 能用 npx 就别全局装。脚手架类工具用 npx，保持全局环境干净。

本章小项目：可发布的 npm 工具包 str-kit

我们来做一个“看起来就能发布到 npm“的字符串工具包，把本章所有知识串起来：规范的 package.json、ESM 导出、桶文件再导出、使用示例。

目录结构

code/ch09/str-kit/
├── package.json      （包的身份证与配置）
├── README.md         （说明文档）
├── index.js          （入口/桶文件，再导出 src 的实现）
├── src/
│   └── strings.js    （三个工具函数的实现）
└── usage.js          （像使用者一样导入并演示）

第一步：写工具函数

// 文件：code/ch09/str-kit/src/strings.js

/**
 * 把字符串首字母大写，其余保持不变。
 * capitalize("hello") -> "Hello"
 */
export function capitalize(str) {
  if (typeof str !== "string") throw new TypeError("capitalize 需要一个字符串");
  if (str.length === 0) return str;
  return str[0].toUpperCase() + str.slice(1);
}

/**
 * 把一段文字转成 URL 友好的 slug。
 * slugify("Hello World! 你好") -> "hello-world-你好"
 */
export function slugify(str) {
  if (typeof str !== "string") throw new TypeError("slugify 需要一个字符串");
  return str
    .trim()
    .toLowerCase()
    .replace(/\s+/g, "-")               // 连续空白 -> 一个连字符
    .replace(/[^\p{L}\p{N}-]/gu, "")    // 去掉字母/数字/连字符以外的字符
    .replace(/-+/g, "-")                // 合并多余连字符
    .replace(/^-|-$/g, "");             // 去掉首尾连字符
}

/**
 * 把过长的字符串截断，并在末尾加上省略号。
 * truncate("这是一段很长的文字", 6) -> "这是一段很…"
 */
export function truncate(str, maxLength = 20, ellipsis = "…") {
  if (typeof str !== "string") throw new TypeError("truncate 需要一个字符串");
  if (str.length <= maxLength) return str;
  const keep = Math.max(0, maxLength - ellipsis.length);
  return str.slice(0, keep) + ellipsis;
}

逐段拆解：

  • 三个函数都用命名导出，因为它们地位平等，命名导出对使用者的自动补全也更友好。
  • 每个函数开头都做类型校验：不是字符串就抛 TypeError。健壮的库应该对错误输入给出清晰报错，而不是默默返回怪结果。
  • slugify 里用了正则。\p{L} 匹配任意语言的字母（Letter），\p{N} 匹配数字（Number），后面的 u 标志开启 Unicode 模式——这样中文、日文等都能保留，而标点、感叹号会被去掉。（正则在第 6 章讲过，这里是它的实战。）
  • truncate 的 maxLength 和 ellipsis 都有默认值（第 5 章的默认参数）。keep 用 Math.max(0, ...) 保证不会出现负的切片长度。

第二步：写入口（桶文件 + 再导出）

// 文件：code/ch09/str-kit/index.js

// 再导出（re-export）：把 strings.js 的所有命名导出原样转发出去。
export * from "./src/strings.js";

// 也提供一个「聚合默认导出」，方便一次性拿到所有函数。
import { capitalize, slugify, truncate } from "./src/strings.js";
export default { capitalize, slugify, truncate };

逐段拆解：

  • export * from "./src/strings.js"：这就是再导出。它把 strings.js 的 capitalize/slugify/truncate 原封不动转发出去。好处是：实现文件可以随便重构、拆分，只要入口 index.js 的对外接口不变，使用者就完全无感。这就是“桶文件“模式。
  • 下面再额外提供一个默认导出（一个聚合对象），照顾“想一次性拿全部“的用法。这样使用者既能 import { capitalize }，也能 import strKit。

第三步：写 package.json

{
  "name": "str-kit",
  "version": "1.0.0",
  "description": "几个常用的字符串小工具：capitalize / slugify / truncate。",
  "type": "module",
  "main": "index.js",
  "exports": {
    ".": "./index.js"
  },
  "files": ["index.js", "src/", "README.md"],
  "scripts": {
    "demo": "node usage.js"
  },
  "keywords": ["string", "slugify", "capitalize", "truncate", "utils"],
  "author": "你的名字 <you@example.com>",
  "license": "MIT"
}

逐字段回顾（都在前面详解过，这里看它们如何组合）：

  • "type": "module"：本包用 ESM。⚠️ 因为我们用 export/import，这个字段必须有（否则 Node 会按 CJS 解析 .js，export 语法会报错）。
  • "main": "index.js" 和 "exports": { ".": "./index.js" }：两个都指向入口。main 是老字段做兼容，exports 是现代写法、更严格。二者一致最稳妥。
  • "files"：发布白名单——只有 index.js、src/、README.md 会被打进包（usage.js 是演示用的，故意不发布）。
  • "scripts": { "demo": "node usage.js" }：npm run demo 就能跑演示。

第四步：写使用示例

// 文件：code/ch09/str-kit/usage.js

// 因为 usage.js 就在包内部，这里用相对路径 "./index.js" 指向入口。
import strKit, { capitalize, slugify, truncate } from "./index.js";

console.log("capitalize('hello world') =>", capitalize("hello world"));
console.log("slugify('Hello World! 你好') =>", slugify("Hello World! 你好"));
console.log("truncate('这是一段很长很长的文字', 6) =>", truncate("这是一段很长很长的文字", 6));

console.log("默认导出对象包含：", Object.keys(strKit));

// 如果这个包被别人 npm install 安装后使用，代码会长这样：
//   import { capitalize } from "str-kit";
//   console.log(capitalize("hi"));

📌 注意这里的关键区别：因为 usage.js 在包内部，它用相对路径 "./index.js" 导入。而别人安装了你的包之后，写的是裸模块名 import { capitalize } from "str-kit"——Node 会去他项目的 node_modules/str-kit/ 里，按 exports/main 找到 index.js。这正是本章反复强调的“相对路径 vs 裸名“的分野。

运行

node usage.js       # 或者：npm run demo

如果要真的发布，还需要做什么

这个包已经具备发布条件了。真要发布的话（记住：练习别真发）：

  1. 确认 name（str-kit）在 npm 上没被占用，被占了就换名或用作用域名 @你的用户名/str-kit。
  2. 写好 README.md（我们已经写了）和 LICENSE。
  3. npm login 登录账号。
  4. npm publish --dry-run 预览会打包哪些文件，确认 files 白名单正确。
  5. npm publish 发布（作用域包公开发布要加 --access public）。
  6. 之后每次改动，用 npm version patch|minor|major 按 semver 升级版本，再 npm publish。

扩展练习

  1. 加一个 camelCase 函数：把 "hello world-foo" 转成 "helloWorldFoo"，加到 src/strings.js 并从 index.js 导出。
  2. 给 truncate 加一个选项：支持“按单词边界截断“，避免把一个英文单词砍成两半。
  3. 拆分子入口：把 slugify 单独放到 src/slug.js，并在 package.json 的 exports 里加 "./slug": "./src/slug.js"，让别人能 import { slugify } from "str-kit/slug"。
  4. 写测试：为三个函数写单元测试，覆盖空字符串、非字符串输入等边界。
  5. 对比 CJS：把 strings.js 复制一份改成 CJS（module.exports），文件名用 .cjs，写一个 require 它的脚本，亲手体会两套系统的差异。

小结与自测

本章我们从“为什么需要模块“讲到“如何发布一个包“，核心要点：

  • 模块化解决全局污染和依赖管理问题，让每个文件有独立作用域。
  • ESM 是官方标准：export（命名/默认）、import（命名/默认/命名空间/重命名）、export from 再导出、动态 import()、顶层 await。模块自动严格模式，且只执行一次（单例缓存）。
  • CJS（require/module.exports）是 Node 老系统：运行时同步、值拷贝、可省扩展名。ESM 是静态、引用绑定、必须写扩展名。
  • Node 靠扩展名（.mjs/.cjs）和 package.json 的 "type" 判断用哪套。
  • package.json：name、version（semver：^ 锁 MAJOR、~ 通常只升 PATCH）、type、main/exports、scripts、三类 dependencies。
  • npm：init / install（-D/-g）/ run / npx；node_modules 要忽略、package-lock.json 要提交。
  • 发布：npm login → npm publish，用 files 白名单控制打包内容。
