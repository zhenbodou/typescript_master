# 第 26 章 · 综合大项目：全栈任务管理器

恭喜你走到了最后一章。前面 25 章里，你从「变量是什么」一路学到 TypeScript 类型体操、异步、模块、测试、Express、SQLite、部署——每一块都单独练过。但真实项目从来不是「一块一块」的，而是**所有知识拧成一股绳**：一个改动可能同时牵动类型、数据库、HTTP 和错误处理。

本章节奏会比前面快一些——你已经有基础了，重点不在「每行怎么写」，而在**如何把这些块组合成一个能跑、能测、能扩展的系统**。

## 项目总览与架构

### 两个程序，一套类型

先建立整体画面。我们有两个可执行入口：`src/server.ts`（后端）和 `src/cli.ts`（CLI 客户端）。它们跑在**不同的进程**里，通过 HTTP 对话：

```text
┌─────────────────────────┐        HTTP (JSON)        ┌──────────────────────────────┐
│   CLI 客户端 (cli.ts)    │  ───────────────────────▶ │   REST API 后端 (server.ts)  │
│                         │   POST /tasks             │                              │
│  argv 解析 → ApiClient  │   GET  /tasks?done=       │  Express → service → repo    │
│  彩色输出 / 退出码      │ ◀───────────────────────  │            ↓                 │
└─────────────────────────┘        200 / 4xx          │      node:sqlite (tasks.db)  │
                                                       └──────────────────────────────┘
        ▲                                                          ▲
        └──────────── 共享同一套 domain 类型（Task 等）────────────┘
```

关键点：CLI 和后端**共享 `src/domain/task.ts` 里的类型**。这在同一个 TS 项目里是「白拿」的类型安全——CLI 里把 `task.done` 写错成 `task.finished`，编译期就报错。真实世界里前后端常是两个仓库，那时会把类型抽成一个共享包，思路完全一样。

### 后端的分层架构

后端内部分成四层，**依赖方向永远由外向内**（外层依赖内层，内层对外层一无所知）：

```text
   HTTP 层 (http/)          ← 只懂「HTTP 怎么进出」：解析请求、设状态码、装配 Express
      │  调用
      ▼
   业务层 (service/)        ← 只懂「业务规则」：找不到就 404、筛选、分页
      │  调用
      ▼
   仓储层 (repository/)     ← 只懂「数据怎么存取」：SQL、参数化查询、行↔对象映射
      │  操作
      ▼
   领域层 (domain/)         ← 最核心：类型、错误、校验，不依赖任何框架，谁都能用
```

📌 为什么要这么切？一句话：**让每一层都能被单独替换和单独测试**。

- 想把 SQLite 换成 Postgres？只改 `repository/`，`service` 和 `http` 一行不动。
- 想给 service 写测试？塞一个内存数据库进去，根本不用启动 Express。
- `domain/` 不 import 任何框架，所以后端和 CLI 都能安全复用它。

这不是「为了架构而架构」。你在第 23 章已经见过 `store → service → router` 的雏形，本章把它做得更彻底、更可测——这正是从「能跑的 demo」到「能维护的项目」的那一步。

### 数据流：一次 `add "买菜"` 发生了什么

把上面几张图连起来，跟着一条真实请求走一遍：

```text
1. 你敲：  npx tsx src/cli.ts add "买菜"
2. cli.ts 解析 argv，得到命令 "add"、参数 "买菜"
3. ApiClient.create({ title: "买菜" }) → fetch POST http://localhost:3000/tasks
4. Express 收到请求 → express.json() 解析 body → tasksRouter 的 POST "/" 处理器
5. parseCreateTask(req.body) 校验（第 18 章思想）：title 非空？done 类型对？
6. service.create(input) → repo.create(input)
7. repo 生成 id/createdAt，执行参数化 INSERT，写进 tasks.db（第 24 章）
8. 逐层把新建的 Task 返回 → router 设 201 + Location 头 → JSON 响应
9. ApiClient 解析 JSON → cli.ts 彩色打印「已添加 ○ e09e34c4 买菜」
```

九步里，每一步都恰好对应前面某一章。接下来我们逐层实现，你会不断有「啊，这个我学过」的感觉。

## 第一层：领域模型（domain）

领域层是整个应用的「共同语言」，也是唯一**不依赖任何框架**的一层。它只有三样东西：模型、错误、校验。

### 模型：`Task`

```typescript
// 文件：code/ch26/app/src/domain/task.ts
export interface Task {
  id: string;
  title: string;
  done: boolean;
  createdAt: string; // ISO 8601 字符串，例如 "2026-07-02T08:00:00.000Z"
}

/** 创建任务时，客户端允许提交的字段（id / createdAt 由服务端生成）。 */
export interface CreateTaskInput {
  title: string;
  done?: boolean;
}

/** 更新任务时，客户端可以只提交部分字段（PATCH 语义）。 */
export interface UpdateTaskInput {
  title?: string;
  done?: boolean;
}
```

这几个类型你在第 23/24 章见过，一字未改——**这就是重点**。分层做对了，换存储、加界面都不该动模型。文件里还有 `ListQuery` 和 `ListResult`（分页查询的入参与出参），完整代码见仓库。

💡 注意 `CreateTaskInput` 里**没有** `id` 和 `createdAt`：这两个字段由**服务端**生成，客户端无权指定。用类型把这条规则钉死，比写一句注释可靠得多。

### 错误：带状态码的异常

```typescript
// 文件：code/ch26/app/src/domain/errors.ts
export class HttpError extends Error {
  status: number;
  constructor(status: number, message: string) {
    super(message);
    this.status = status;
    this.name = "HttpError";
  }
}

/** 404：资源不存在。 */
export class NotFoundError extends HttpError {
  constructor(message = "资源不存在") {
    super(404, message);
    this.name = "NotFoundError";
  }
}

/** 400：请求校验失败，可携带具体字段错误列表。 */
export class ValidationError extends HttpError {
  details: string[];
  constructor(details: string[]) {
    super(400, "请求校验失败");
    this.name = "ValidationError";
    this.details = details;
  }
}
```

思路（来自第 23 章）：**让业务错误自己携带 HTTP 状态码**。service 层只管 `throw new NotFoundError(...)`，至于「404 该返回什么」，交给最外层的错误处理中间件统一决定。业务代码不碰 `res`，HTTP 细节不渗进业务——职责就此分清。

### 校验：把 `unknown` 升级成可信类型

这是第 18 章「类型守卫 + 显式检查」思想在 HTTP 边界的落地。**外部传进来的任何东西都是 `unknown`**，校验通过后才「升级」成可信类型：

```typescript
// 文件：code/ch26/app/src/domain/validation.ts
import { ValidationError } from "./errors.js";
import type { CreateTaskInput } from "./task.js";

/** 判断一个值是不是「普通对象」（排除 null、数组、函数等）。 */
function isPlainObject(value: unknown): value is Record<string, unknown> {
  return typeof value === "object" && value !== null && !Array.isArray(value);
}

/** 校验「创建任务」的请求体。失败抛 ValidationError（会被错误中间件接住）。 */
export function parseCreateTask(body: unknown): CreateTaskInput {
  const errors: string[] = [];
  if (!isPlainObject(body)) {
    throw new ValidationError(["请求体必须是一个 JSON 对象"]);
  }

  const { title, done } = body;

  if (typeof title !== "string" || title.trim() === "") {
    errors.push("title 必填，且必须是非空字符串");
  } else if (title.length > 200) {
    errors.push("title 长度不能超过 200 个字符");
  }
  if (done !== undefined && typeof done !== "boolean") {
    errors.push("done 如果提供，必须是布尔值");
  }
  if (errors.length > 0) throw new ValidationError(errors);

  // 走到这里，title 一定是 string，done 一定是 boolean | undefined。
  return { title: (title as string).trim(), done: done as boolean | undefined };
}
```

注意函数签名 `parseCreateTask(body: unknown): CreateTaskInput`：入口是 `unknown`（我不信任你），出口是强类型（校验过了，可以信）。这个「校验 = 类型收窄」的模式，是给任何外部数据（HTTP body、命令行输入、读进来的 JSON 文件）把关的通用套路。`parseUpdateTask`（PATCH：允许只传部分字段，但不允许一个都不传）同理，完整代码见仓库。

⚠️ 别用「类型断言」偷懒，比如直接 `const input = req.body as CreateTaskInput`。`as` 只骗过编译器，运行时该崩还是崩。校验必须是**真的在运行时检查**，这也是为什么我们手写 `if`——生产项目里可以用 Zod 之类的库把这段自动化，但原理就是这些 `if`。

## 第二层：仓储（repository）—— SQLite 持久化

仓储层只回答一个问题：**数据存在哪、怎么存取**。这里用第 24 章学的 `node:sqlite`。

```typescript
// 文件：code/ch26/app/src/repository/taskRepository.ts
import { createRequire } from "node:module";
import { randomUUID } from "node:crypto";
import type { Task, CreateTaskInput, UpdateTaskInput } from "../domain/task.js";

// node:sqlite 是较新的 Node 内置模块。用 createRequire 在运行时加载它，
// 而不是写静态的 import——这样测试打包器（Vitest 底层的 Vite）不会试图
// 去「解析/打包」这个它还不认识的内置模块。运行时行为完全一致。
const require = createRequire(import.meta.url);
const { DatabaseSync } = require("node:sqlite") as typeof import("node:sqlite");

export class TaskRepository {
  private db: InstanceType<typeof DatabaseSync>;

  /** @param dbFile 数据库文件路径；传 ":memory:" 则用纯内存库（测试用）。 */
  constructor(dbFile: string) {
    this.db = new DatabaseSync(dbFile);
    this.db.exec("PRAGMA foreign_keys = ON;");
    this.migrate(); // 建表（幂等）
  }

  private migrate(): void {
    this.db.exec(`
      CREATE TABLE IF NOT EXISTS tasks (
        id         TEXT    PRIMARY KEY,
        title      TEXT    NOT NULL,
        done       INTEGER NOT NULL DEFAULT 0,   -- SQLite 无布尔，用 0/1
        created_at TEXT    NOT NULL              -- ISO 8601 字符串
      );
    `);
  }
```

💡 这里有个**实战小坑**值得单独说：为什么不直接 `import { DatabaseSync } from "node:sqlite"`？因为 Vitest 底层用的 Vite 版本，其「内置模块名单」还没收录这个较新的模块，静态 import 会让它试图去打包、然后报「找不到 sqlite」。用 `createRequire` 在运行时加载，就绕开了打包器的静态分析——这类「测试工具和新特性打架」的问题，在真实项目里很常见，解决思路就是**别让打包器看到它**。运行时行为一模一样。

核心的 CRUD 方法（`findAll` / `findById` / `create` / `update` / `remove`）沿用第 24 章，两个要点复习一下：

```typescript
  /** 按 id 查找，找不到返回 undefined。 */
  findById(id: string): Task | undefined {
    // ⚠️ 参数化查询（parameterized query）：id 作为参数传给 .get(id)，
    // 永远不会被解释成 SQL —— 这是防 SQL 注入的正解。
    const row = this.db
      .prepare("SELECT id, title, done, created_at FROM tasks WHERE id = ?")
      .get(id) as Record<string, unknown> | undefined;
    return row ? this.rowToTask(row) : undefined;
  }

  /** 把数据库的一行（snake_case、done 为 0/1）映射成领域模型 Task。 */
  private rowToTask(row: Record<string, unknown>): Task {
    return {
      id: row.id as string,
      title: row.title as string,
      done: (row.done as number) === 1, // 0/1 → boolean
      createdAt: row.created_at as string,
    };
  }
```

两个反复出现的主题：**参数化查询**（用 `?` 占位，绝不拼字符串——防 SQL 注入）和 **行↔对象映射**（数据库是 `snake_case` + `0/1`，领域模型是 `camelCase` + `boolean`，`rowToTask` 是它们之间的翻译官）。完整实现（含 `create` / `update` / `remove` / `close`）见仓库。

📌 注意这一层**没有任何「找不到就报错」的逻辑**——`findById` 找不到只是安静地返回 `undefined`。「找不到算不算错误」是**业务决定**，属于上一层。仓储只忠实反映数据库的事实。

## 第三层：业务逻辑（service）—— 依赖注入登场

service 把仓储的原始能力，组合成对上层有意义的操作，并在这里安放业务规则。相比第 23 章，这里有一个关键升级：**依赖注入**。

```typescript
// 文件：code/ch26/app/src/service/taskService.ts
import { NotFoundError } from "../domain/errors.js";
import type { TaskRepository } from "../repository/taskRepository.js";
import type {
  CreateTaskInput, ListQuery, ListResult, Task, UpdateTaskInput,
} from "../domain/task.js";

export class TaskService {
  // ↓↓↓ 依赖注入：service 不自己 new 仓储，而是通过构造函数接收一个。
  constructor(private readonly repo: TaskRepository) {}

  /** 列表：done 筛选 + 分页。 */
  list(query: ListQuery): ListResult {
    let all = this.repo.findAll();
    if (query.done !== undefined) {
      all = all.filter((t) => t.done === query.done);
    }
    const total = all.length;
    const start = (query.page - 1) * query.pageSize;
    const items = all.slice(start, start + query.pageSize);
    return { items, page: query.page, pageSize: query.pageSize, total };
  }

  /** 按 id 取单个任务，找不到抛 404。 ←── 业务规则在这里。 */
  getById(id: string): Task {
    const task = this.repo.findById(id);
    if (!task) throw new NotFoundError(`找不到 id 为 ${id} 的任务`);
    return task;
  }

  create(input: CreateTaskInput): Task {
    return this.repo.create({ title: input.title, done: input.done ?? false });
  }

  update(id: string, patch: UpdateTaskInput): Task {
    const updated = this.repo.update(id, patch);
    if (!updated) throw new NotFoundError(`找不到 id 为 ${id} 的任务`);
    return updated;
  }

  remove(id: string): void {
    const ok = this.repo.remove(id);
    if (!ok) throw new NotFoundError(`找不到 id 为 ${id} 的任务`);
  }
}
```

盯着构造函数看：`constructor(private readonly repo: TaskRepository) {}`。service **不知道**这个 `repo` 是连着磁盘文件还是内存库，它只知道「有个东西满足 `TaskRepository` 的接口」。谁来决定用哪个 repo？由**装配它的人**（`http/app.ts`，或测试）决定。

📌 这就是**依赖注入（dependency injection）**，也是本章相对第 23 章最重要的架构升级。好处马上就能兑现：

- **可测试**：测试时给它 `new TaskRepository(":memory:")`，一个内存库，跑得飞快、不落盘、互不干扰。
- **可替换**：换存储实现，service 零改动。
- **职责清晰**：`getById` 里那句 `if (!task) throw new NotFoundError(...)`——「找不到算错误」这条业务规则，明明白白待在业务层，不在仓储、也不在 HTTP。

## 类型化配置（config）

在装配 HTTP 层之前，先看配置。端口、数据库路径这些「因环境而异」的东西，不该写死在代码里。这是第 13 章「把宽松输入解析成强类型」的思想，数据源换成了环境变量 `process.env`（值全是 `string | undefined`）：

```typescript
// 文件：code/ch26/app/src/config/config.ts
export interface AppConfig {
  port: number;
  dbFile: string;
  nodeEnv: "development" | "production" | "test";
}

/** 把字符串解析成端口号；非法或缺失时回退默认值，越界则「快速失败」。 */
function parsePort(value: string | undefined, fallback: number): number {
  if (value === undefined || value.trim() === "") return fallback;
  const n = Number(value);
  if (!Number.isInteger(n) || n < 1 || n > 65535) {
    throw new Error(`配置错误：PORT 必须是 1~65535 的整数，收到的是 "${value}"`);
  }
  return n;
}

export function loadConfig(
  env: Record<string, string | undefined> = process.env,
): AppConfig {
  return {
    port: parsePort(env.PORT, 3000),
    dbFile: env.DB_FILE ? path.resolve(env.DB_FILE) : defaultDbFile,
    nodeEnv: parseNodeEnv(env.NODE_ENV),
  };
}
```

两个原则：**敏感/环境相关的信息来自环境变量**（不写死、不进 git，配一份 `.env.example` 做模板）；**非法配置要快速失败（fail fast）**——端口写成 `"abc"`，启动时就抛错，而不是等运行到一半才莫名其妙地崩。`loadConfig` 默认读 `process.env`，但也允许传入自定义对象，这让它**可测试**（同样是「不硬编码依赖」的思想）。

## 第四层：HTTP —— 用 Express 把各层拼起来

### 路由：只管 HTTP 进出

路由层负责「HTTP 怎么进出」：解析参数、调 service、设状态码。它用一个**工厂函数**接收 service（延续依赖注入）：

```typescript
// 文件：code/ch26/app/src/http/tasksRouter.ts
import { Router } from "express";
import type { Request, Response, NextFunction } from "express";
import type { TaskService } from "../service/taskService.js";
import { parseCreateTask, parseUpdateTask } from "../domain/validation.js";

/** 把可能抛错的同步处理器包一层，抛出的错误统一转交 next()。 */
function wrap(handler: (req: Request, res: Response) => void) {
  return (req: Request, res: Response, next: NextFunction) => {
    try {
      handler(req, res);
    } catch (err) {
      next(err); // 交给统一错误处理中间件
    }
  };
}

export function createTasksRouter(service: TaskService): Router {
  const router = Router();

  // POST /tasks —— 创建，返回 201 + Location 头。
  router.post("/", wrap((req, res) => {
    const created = service.create(parseCreateTask(req.body));
    res.setHeader("Location", `/tasks/${created.id}`);
    res.status(201).json(created);
  }));

  // PATCH /tasks/:id —— 局部更新。
  router.patch("/:id", wrap((req, res) => {
    const updated = service.update(req.params.id!, parseUpdateTask(req.body));
    res.status(200).json(updated);
  }));

  // DELETE /tasks/:id —— 成功返回 204（无响应体）。
  router.delete("/:id", wrap((req, res) => {
    service.remove(req.params.id!);
    res.status(204).end();
  }));

  // GET /tasks、GET /tasks/:id 同理，见仓库。
  return router;
}
```

看那个 `wrap` 小工具：它把「处理器抛的错」统一 `next(err)` 出去。这样每个路由都能**直接 `throw`**（比如校验失败抛 `ValidationError`、找不到抛 `NotFoundError`），不必层层写 `try/catch`。错误最终都汇到一个地方处理——下面就是那个地方。

💡 `req.params.id!` 后面的 `!`：因为路由是 `/:id`，命中时 `id` 一定存在，但 TS 在严格模式（`noUncheckedIndexedAccess`）下认为它可能 `undefined`。这里我们比编译器更了解运行时，用 `!` 明确告诉它「放心，一定有」。这种「你确实知道、而类型系统不知道」的场合，正是非空断言的正当用途。

### 统一错误处理：一个出口

```typescript
// 文件：code/ch26/app/src/http/errorHandler.ts
import type { Request, Response, NextFunction } from "express";
import { HttpError, ValidationError } from "../domain/errors.js";

export function errorHandler(
  err: unknown,
  _req: Request,
  res: Response,
  _next: NextFunction, // ← 必须声明第 4 个参数，Express 靠「4 个参数」识别错误中间件
): void {
  if (err instanceof ValidationError) {
    res.status(err.status).json({ error: err.message, details: err.details });
    return;
  }
  if (err instanceof HttpError) {
    res.status(err.status).json({ error: err.message });
    return;
  }
  // 兜底：未预期错误 → 500，打日志但不泄露内部细节。
  console.error("未处理的错误:", err);
  res.status(500).json({ error: "服务器内部错误" });
}
```

所有错误在这里**归一**成统一的 JSON 形状 `{ error, details? }`。这带来两个好处：客户端只需处理一种错误格式；业务代码里可以放心 `throw`，不用操心怎么变成 HTTP 响应。`err instanceof ValidationError` 这类判断，是第 18 章 `instanceof` 收窄的直接应用——先判具体、再判宽泛、最后兜底。

### 装配现场（composition root）

现在把四层拼起来。`app.ts` 是全后端**唯一**知道「用哪个 repo、哪个 service」的地方，这个位置有个专门的名字叫 **composition root**：

```typescript
// 文件：code/ch26/app/src/http/app.ts
import express from "express";
import { TaskRepository } from "../repository/taskRepository.js";
import { TaskService } from "../service/taskService.js";
import { createTasksRouter } from "./tasksRouter.js";
import { errorHandler, notFoundHandler } from "./errorHandler.js";

export function createApp(deps: { dbFile: string }) {
  // —— 依赖注入：自底向上组装 repository → service → router ——
  const repo = new TaskRepository(deps.dbFile);
  const service = new TaskService(repo);

  const app = express();
  app.use(express.json()); // 解析 JSON 请求体到 req.body

  // 健康检查端点，方便部署探活（第 25 章）。
  app.get("/health", (_req, res) => {
    res.status(200).json({ status: "ok", time: new Date().toISOString() });
  });

  app.use("/tasks", createTasksRouter(service));

  // 收尾：404 与统一错误处理，必须放最后。
  app.use(notFoundHandler);
  app.use(errorHandler);

  return { app, repo }; // 返回 repo，好让入口在退出时 close 它
}
```

三行装配 `repo → service → router` 就是依赖注入的「组装线」。把 `createApp`（造应用）和 `listen`（开端口）分开，是刻意的：这样测试能只 import `createApp`、拿到 `app`，而不必真的占用一个端口。

### 入口 + 优雅关闭

```typescript
// 文件：code/ch26/app/src/server.ts
import { loadConfig } from "./config/config.js";
import { createApp } from "./http/app.js";

const config = loadConfig(); // 非法配置在这里「快速失败」
const { app, repo } = createApp({ dbFile: config.dbFile });

const server = app.listen(config.port, () => {
  console.log(`✅ 任务 API 已启动：http://localhost:${config.port}`);
  console.log(`   环境：${config.nodeEnv}  数据库：${config.dbFile}`);
});

/** 优雅关闭：先停收新请求，再关数据库连接，最后退出。 */
function shutdown(signal: string) {
  console.log(`\n收到 ${signal}，正在关闭…`);
  server.close(() => {
    repo.close();
    console.log("已关闭数据库连接，再见 👋");
    process.exit(0);
  });
}
process.on("SIGINT", () => shutdown("SIGINT"));   // Ctrl+C
process.on("SIGTERM", () => shutdown("SIGTERM")); // kill / 容器停止
```

💡 **优雅关闭（graceful shutdown）**是 demo 和生产的一条分界线。直接 `process.exit()` 会粗暴掐断正在处理的请求、可能留下未刷盘的数据。正确做法是：监听终止信号 → `server.close()` 停止接收新连接、等现有请求处理完 → 关掉数据库连接 → 干净退出。部署到容器里时（第 25 章），编排系统正是靠发 `SIGTERM` 来通知你的程序「该收摊了」。

## CLI 客户端

后端完工，来做第二个程序：命令行客户端。一个好 CLI 也分层——底层是「怎么跟后端说话」，上层是「怎么跟用户说话」。

### API 客户端：封装 fetch

```typescript
// 文件：code/ch26/app/src/client/apiClient.ts
import type { CreateTaskInput, ListResult, Task, UpdateTaskInput } from "../domain/task.js";

/** 后端返回非 2xx 时抛出，携带状态码与后端给的消息。 */
export class ApiError extends Error {
  constructor(public status: number, message: string) {
    super(message);
    this.name = "ApiError";
  }
}

export class ApiClient {
  constructor(private readonly baseUrl: string) {}

  /** 统一请求封装：拼 URL、发请求、按状态码分流、解析 JSON。 */
  private async request<T>(path: string, init?: RequestInit): Promise<T> {
    let res: Response;
    try {
      res = await fetch(`${this.baseUrl}${path}`, {
        ...init,
        headers: { "Content-Type": "application/json", ...init?.headers },
      });
    } catch {
      // fetch 只在「网络层」失败时 reject（比如后端根本没启动）。
      throw new ApiError(0, `无法连接到服务 ${this.baseUrl}（后端启动了吗？）`);
    }

    if (res.status === 204) return undefined as T; // 删除成功，无响应体

    const text = await res.text();
    const data = text ? JSON.parse(text) : undefined;
    if (!res.ok) {
      const msg = data?.error ?? `HTTP ${res.status}`;
      const detail = data?.details ? `：${data.details.join("；")}` : "";
      throw new ApiError(res.status, `${msg}${detail}`);
    }
    return data as T;
  }

  list(done?: boolean): Promise<ListResult> {
    const query = done === undefined ? "?pageSize=100" : `?done=${done}&pageSize=100`;
    return this.request<ListResult>(`/tasks${query}`);
  }
  create(input: CreateTaskInput): Promise<Task> {
    return this.request<Task>("/tasks", { method: "POST", body: JSON.stringify(input) });
  }
  update(id: string, patch: UpdateTaskInput): Promise<Task> {
    return this.request<Task>(`/tasks/${id}`, { method: "PATCH", body: JSON.stringify(patch) });
  }
  remove(id: string): Promise<void> {
    return this.request<void>(`/tasks/${id}`, { method: "DELETE" });
  }
}
```

`ApiClient` 把「和后端对话」的所有细节（拼 URL、JSON 序列化、`Content-Type`、错误分流、204 特判）关进一个盒子。上层调 `client.create({ title })` 就完事了。注意两处**边界处理**：`fetch` 只在网络层失败时才 reject（后端没开机），HTTP 4xx/5xx 反而是「成功收到响应」，得自己看 `res.ok` 判断——这是第 20/22 章 `fetch` 语义的直接应用。返回类型 `Promise<Task>` 让调用方也是全类型的，尽管数据实际来自网络。

### 彩色输出

真实 CLI 会给输出上色。ANSI 转义码零依赖就能做，还要记得**不是终端时自动关掉颜色**（比如输出被管道到文件）：

```typescript
// 文件：code/ch26/app/src/client/colors.ts
const enabled = process.stdout.isTTY === true && process.env.NO_COLOR === undefined;

function wrap(open: number, close: number) {
  return (s: string) => (enabled ? `\x1b[${open}m${s}\x1b[${close}m` : s);
}

export const color = {
  green: wrap(32, 39),  red: wrap(31, 39),  yellow: wrap(33, 39),
  gray: wrap(90, 39),   cyan: wrap(36, 39), bold: wrap(1, 22),
};
```

💡 `process.stdout.isTTY` 判断输出是不是「真的连着终端」。管道给别的程序时它是 `undefined`，此时输出纯文本，免得 `\x1b[32m` 这种乱码污染下游。顺带尊重了社区约定的 `NO_COLOR` 环境变量——这些小细节是「玩具脚本」和「像样工具」的区别。

### CLI 主程序：解析 argv 并分发

```typescript
// 文件：code/ch26/app/src/cli.ts
import { ApiClient, ApiError } from "./client/apiClient.js";
import { color } from "./client/colors.js";

const API_URL = process.env.API_URL ?? "http://localhost:3000";
const client = new ApiClient(API_URL);

/** 用法错误（区别于 ApiError）：提示用户命令写法不对。 */
class UsageError extends Error {}

async function cmdAdd(args: string[]): Promise<void> {
  const title = args.join(" ").trim();
  if (!title) throw new UsageError('add 需要一个标题，例如：add "买菜"');
  const task = await client.create({ title });
  console.log(`${color.green("已添加")}  ${formatTask(task)}`);
}

async function main(): Promise<void> {
  // process.argv 前两项是 [node, 脚本路径]，真正参数从第 3 项起。
  const [command, ...rest] = process.argv.slice(2);
  switch (command) {
    case "add":  return cmdAdd(rest);
    case "list": return cmdList(rest);
    case "done": return cmdDone(rest);
    case "rm":   return cmdRm(rest);
    case undefined: case "help": case "--help": case "-h":
      printUsage(); return;
    default:
      throw new UsageError(`未知命令：${command}`);
  }
}

// 顶层错误处理：把不同错误翻译成友好提示，并设置退出码。
main().catch((err: unknown) => {
  if (err instanceof UsageError) {
    console.error(color.red(`用法错误：${err.message}`));
  } else if (err instanceof ApiError) {
    console.error(color.red(`请求失败（${err.status}）：${err.message}`));
  } else {
    console.error(color.red("发生未知错误："), err);
  }
  process.exitCode = 1; // ← 非 0 退出码，让脚本/CI 能感知失败
});
```

三个 CLI 惯用法值得记住：

1. **`process.argv.slice(2)`**：前两项永远是 `[node路径, 脚本路径]`，你的参数从第 3 项开始。用解构 `[command, ...rest]` 一步拆出「命令 + 其余参数」。
2. **顶层 `.catch` 统一处理错误**：跟后端的错误中间件是同一个思想——错误集中在一个出口翻译。这里用 `instanceof` 区分「用法错误」和「请求失败」，给不同的提示。
3. **`process.exitCode = 1`**：命令失败必须用非 0 退出码，这样它被别的脚本或 CI 调用时，对方能通过退出码知道「这一步挂了」（第 25 章）。

还有一个**贴心设计**：`done`/`rm` 支持用 **id 前缀**定位任务，你不用复制整段 UUID，敲前几位就行。实现很简单——先 `list` 拉全部，找 id 以该前缀开头的那个，命中 0 个或多个都报错。完整的 `cmdList` / `cmdDone` / `cmdRm` / `resolveId` / `formatTask` 见仓库。

## 测试

分层 + 依赖注入的红利在这里兑现。给 service 塞一个**内存库**（`":memory:"`），就能全速测试完整的「service → repository → SQL」链路，不启动 Express、不落盘、每个用例互相隔离：

```typescript
// 文件：code/ch26/app/test/taskService.test.ts
import { describe, it, expect, beforeEach } from "vitest";
import { TaskRepository } from "../src/repository/taskRepository.js";
import { TaskService } from "../src/service/taskService.js";
import { NotFoundError } from "../src/domain/errors.js";

function makeService(): TaskService {
  // 关键：内存库。每次都是全新、隔离、不落盘的数据库。
  return new TaskService(new TaskRepository(":memory:"));
}

describe("TaskService", () => {
  let service: TaskService;
  beforeEach(() => { service = makeService(); });

  it("create 后能在 list 中查到", () => {
    const created = service.create({ title: "买菜" });
    expect(created.done).toBe(false);
    const { items, total } = service.list({ page: 1, pageSize: 20 });
    expect(total).toBe(1);
    expect(items[0]?.title).toBe("买菜");
  });

  it("getById 找不到时抛 NotFoundError", () => {
    expect(() => service.getById("不存在的-id")).toThrow(NotFoundError);
  });

  it("list 支持 done 筛选", () => {
    const a = service.create({ title: "A" });
    service.create({ title: "B" });
    service.update(a.id, { done: true });
    expect(service.list({ done: true, page: 1, pageSize: 20 }).total).toBe(1);
    expect(service.list({ done: false, page: 1, pageSize: 20 }).total).toBe(1);
  });
});
```

📌 `new TaskRepository(":memory:")` 就是依赖注入设计的**直接回报**：因为 service 从不自己 new 仓储，测试才能轻松「换」一个内存实现进去。测试断言业务规则——create 默认未完成、找不到抛 404、筛选生效、删两次第二次报错、分页正确——而不去操心 HTTP 或磁盘。另有 `validation.test.ts` 单独测校验函数（空标题被拒、`trim` 生效等）。两个测试文件共 12 个用例。

⚠️ 一个真实的工具坑：Vitest 底层用 Vite，而这个 Vite 版本的「内置模块名单」还不认识 `node:sqlite`——这正是前面 repository 里用 `createRequire` 加载它的原因。绕开打包器的静态分析后，测试就干净通过了。这类「新语言特性 vs 老工具链」的摩擦，是真实项目的日常，解决它本身也是一种能力。

## 下一步：你可以如何扩展

这个项目是一个扎实的地基，往上加东西时，你会立刻体会到分层的好处——每个新功能大多只动一两层。挑几个练手：

- **加字段**：给任务加 `dueDate`（截止日期）或 `priority`（优先级）。顺着改一遍 `domain`（类型 + 校验）→ `repository`（列 + 迁移）→ 也许 `service`（按优先级排序），你会摸清「一个字段穿过所有层」的完整路径。
- **换存储**：把 `TaskRepository` 换成 Postgres 或 Prisma 实现。因为 service 只依赖它的方法签名，理论上 service/http **一行不改**——亲手验证一下这个承诺。
- **加认证**：写一个中间件校验 `Authorization` 头，给任务加 `userId`，实现「每人只看自己的任务」。
- **换前端**：CLI 只是众多客户端之一。后端不变，另写一个浏览器网页（第 15~20 章的 DOM/fetch），或一个手机端——它们都连同一个 API。这就是「前后端分离」的实感。
- **补测试**：给 HTTP 层加集成测试。用 `supertest` 或直接 `fetch` 打 `createApp()` 起的临时服务，覆盖 201/400/404/204 各条路径。
- **上生产**：按第 25 章把后端塞进 Docker、配上环境变量、挂上健康检查探针，部署到云上。优雅关闭这时就派上用场了。

---

## 结语

如果你从第 1 章一路跟到这里，请停下来认真感受一下这段距离：开头我们还在辨析 `let` 和 `const`，现在你已经能独立设计并实现一个分层清晰、类型完备、带持久化和测试、还有配套 CLI 的全栈应用了。

更重要的是，你收获的不是「记住了多少 API」，而是几种能迁移到任何项目的思维方式：

- **分层**——让每部分只关心一件事，系统才不会随规模腐化。
- **类型驱动**——用类型把规则钉在编译期，把「运行时才发现的错」提前到「写代码时就发现」。
- **边界校验**——所有外部输入都是 `unknown`，验过才可信。
- **依赖注入**——不硬编码依赖，换实现和写测试才有回旋余地。
- **统一错误处理**——错误集中在一个出口翻译，业务代码才能保持干净。

这些原则，换成 Python、Go、Rust 一样成立；换成前端框架、后端服务、CLI 工具一样管用。语言和框架会更新换代，但「如何把复杂度组织好」的功夫会跟你很久。

技术永远学不完，但你现在已经有了继续自学的全部底子：会读文档、会查类型、会写测试验证猜想、会把大问题拆成分层的小问题。接下来，去做一个**你自己真正想要的东西**吧——那会是比任何教程都好的老师。
