# 第 22 章 · 测试与 TDD

## 本章目标

前 21 章里，我们写了大量代码：闭包、Promise、类、泛型、装饰器、工具链……但有一件事我们一直在偷偷做，做得还很原始——**验证代码到底对不对**。

回想一下，你每写完一个函数，是不是都要在文件末尾加一句 `console.log(...)`，跑一下 `node`，肉眼看看输出对不对？看完再把这句 `console.log` 删掉？这套"打印—肉眼比对—删除"的流程有三个致命问题：

1. **不可重复**：你删掉 `console.log` 之后，下次改动代码，还得重新写一遍来验证。
2. **不可回归**：你改 A 函数时，怎么保证没顺手弄坏 B 函数？你不可能每次都手动把所有函数都 `console.log` 一遍。
3. **不是文档**：`console.log` 删掉就没了，别人（和三个月后的你）根本不知道这个函数"应该"有什么行为。

**自动化测试（automated testing）**就是把"这段代码应该产生什么结果"用代码固化下来，让机器帮你一遍遍、飞快地验证。它是专业开发和玩票的分水岭。

学完本章你会：

- 想清楚**为什么要测试**：回归保护、重构信心、活文档，以及**测试金字塔（test pyramid）**。
- 上手 **Vitest**：`describe`/`it`/`test`/`expect`、常用**匹配器（matcher）**、`beforeEach`/`afterEach`、`npx vitest run` 与 watch 模式。
- 测试**异步代码**：`async/await`、`.resolves`/`.rejects`。
- 掌握 **Mock / Stub / Spy**：`vi.fn()`、`vi.spyOn`、mock 模块、**假定时器（fake timers）**——用来测 `debounce` 这种带副作用、依赖外部的函数。
- 理解**测试覆盖率（coverage）**，并且知道为什么"别唯覆盖率论"。
- 完整走一遍 **TDD（测试驱动开发，Test-Driven Development）**：红 → 绿 → 重构。
- 认识**好测试的特征**：AAA 模式、独立、可读、快。
- 见识一下**快照测试（snapshot testing）**。

本书的 `code` 目录已经装好了 Vitest，所有示例都能直接 `npx vitest run` 跑绿。我们从"测试到底能带来什么"讲起。

## 22.1 为什么要测试：回归、重构信心、活文档

先说三个最能打动人的理由。

**理由一：回归保护（regression protection）。**
"回归（regression）"指的是"本来好好的功能，被后来的改动弄坏了"。软件越大，这种事越防不胜防：你改了 A 模块，结果依赖 A 的 B 模块悄悄崩了，而你根本没意识到。有了自动化测试，你每次改完跑一遍全部测试，只要有谁被弄坏，测试立刻变红告诉你。**测试是你代码的安全网。**

**理由二：重构信心（confidence to refactor）。**
"重构（refactoring）"指在不改变外部行为的前提下改进代码内部结构。没有测试时，你根本不敢动祖传代码——万一改坏了呢？有了测试，你可以大胆地把一坨乱麻重写得清清爽爽，只要测试还全绿，就说明**行为没变**。测试把"我猜没问题"变成了"我知道没问题"。

**理由三：活文档（living documentation）。**
一份好的测试，读起来就是"这个函数在各种情况下应该怎么表现"的说明书。而且它是**活的**——如果代码行为变了但测试没更新，测试会失败，逼你保持文档与代码同步。注释会过期，测试不会。

📌 一句话总结：**测试的本质，是把"正确性"从你脑子里的临时想法，变成可以反复运行、永不遗忘的代码。**

## 22.2 测试金字塔：单元、集成、端到端

测试不止一种。业界用一个"金字塔"来描述它们的比例：

```
        /\
       /e2e\        ← 端到端测试：少而慢，覆盖整条用户路径
      /------\
     / 集成   \      ← 集成测试：中等数量，测多个模块协作
    /----------\
   /   单元      \   ← 单元测试：又多又快，测单个函数/类
  /--------------\
```

- **单元测试（unit test）**：测最小的单元——一个函数、一个类的方法。**又快又多**，是金字塔的地基。本章重点。
- **集成测试（integration test）**：测多个模块**协作**是否正确，比如"服务层调数据库能不能存对"。数量中等，速度中等。
- **端到端测试（end-to-end test，e2e）**：模拟**真实用户**从头到尾操作，比如用浏览器自动点开页面、填表单、提交。最真实，但**又慢又脆**（一点点 UI 变动就可能挂），所以要少。

📌 为什么是"金字塔"而不是"倒金字塔"？因为单元测试便宜、快、稳定，能精确定位问题；e2e 贵、慢、容易误报。**多写便宜稳定的，少写昂贵脆弱的**，整体投入产出比最高。

💡 新手常犯的错是想直接写一堆 e2e"一步到位"，结果测试跑十分钟、动不动就无端变红，最后没人愿意维护。先把单元测试写扎实，收益立竿见影。

## 22.3 Vitest 快速上手

本书选用 **Vitest**：它是专为现代 TypeScript / ESM 项目设计的测试框架，开箱即用地支持 TS，速度极快，API 和老牌的 Jest 几乎一致（会 Vitest 就基本会 Jest）。

`code` 目录的 `package.json` 里已经装好了 `vitest`。如果你是全新克隆的仓库，先在 `code` 目录里装依赖：

```bash
npm install
```

写第一个测试。测试文件的命名约定是 `*.test.ts`（或 `*.spec.ts`），Vitest 会自动找到它们：

```typescript
// 文件：code/ch22/first.test.ts（演示用，可自行创建）
import { describe, it, expect } from "vitest";

describe("加法", () => {
  it("1 + 1 应该等于 2", () => {
    expect(1 + 1).toBe(2);
  });
});
```

**逐段拆解：**

- `import { describe, it, expect } from "vitest";`：从 vitest 引入三个核心 API。（Vitest 也支持配置成"全局可用"免 import，但**显式 import 更清晰**，本书一律显式引入。）
- `describe("加法", () => { ... })`：**测试套件（test suite）**，把相关的测试分组，第一个参数是分组名。
- `it("1 + 1 应该等于 2", () => { ... })`：一个**测试用例（test case）**。`it` 读起来像句子——"it（它）应该 1+1 等于 2"。`it` 和 `test` 是同义词，用哪个都行，本书用 `it`。
- `expect(1 + 1).toBe(2)`：**断言（assertion）**。`expect(实际值)` 包住你要检查的值，`.toBe(期望值)` 是**匹配器**，声明它俩应该相等。断言不成立，测试就失败。

运行它：

```bash
# 跑一次全部测试，跑完就退出（CI / 提交前用这个）
npx vitest run

# 只跑名字含 ch22 的文件
npx vitest run ch22
```

你会看到绿色的 `✓`。

💡 **watch 模式**：开发时用 `npx vitest`（不加 `run`），它会常驻，一保存文件就自动重跑相关测试，红绿反馈实时到眼前。写测试驱动开发时几乎离不开它。跑一次就退出用 `npx vitest run`。

## 22.4 常用匹配器一览

匹配器决定"怎么比"。挑最常用的记熟：

```typescript
// 文件：code/ch22/matchers.test.ts（演示用）
import { describe, it, expect } from "vitest";

describe("常用匹配器", () => {
  it("toBe：比较基本值（用 Object.is，约等于 ===）", () => {
    expect(2 + 2).toBe(4);
    expect("ab").toBe("ab");
  });

  it("toEqual：递归比较对象/数组的内容（结构相等）", () => {
    expect({ a: 1, b: [2, 3] }).toEqual({ a: 1, b: [2, 3] });
    // ⚠️ 这里不能用 toBe：两个对象引用不同，toBe 会失败
  });

  it("toContain：数组/字符串包含某元素/子串", () => {
    expect([1, 2, 3]).toContain(2);
    expect("hello world").toContain("world");
  });

  it("toThrow：断言函数会抛错", () => {
    const boom = () => {
      throw new RangeError("超范围");
    };
    expect(boom).toThrow(); // 抛了就行
    expect(boom).toThrow(RangeError); // 抛的是指定类型
    expect(boom).toThrow("超范围"); // 错误信息含此子串
  });

  it("toBeCloseTo：比较浮点数，避开精度陷阱", () => {
    // 0.1 + 0.2 === 0.30000000000000004，用 toBe 会失败！
    expect(0.1 + 0.2).toBeCloseTo(0.3);
  });

  it("其他好用的", () => {
    expect(null).toBeNull();
    expect(undefined).toBeUndefined();
    expect(0).toBeFalsy();
    expect("x").toBeTruthy();
    expect([1, 2, 3]).toHaveLength(3);
    expect(3).toBeGreaterThan(2);
    expect(1).not.toBe(2); // .not 取反
  });
});
```

**逐段拆解，重点讲两个坑：**

- ⚠️ **`toBe` vs `toEqual`**：`toBe` 用 `Object.is` 比，对对象/数组来说等于比"是不是同一个引用"。`{a:1}` 和另一个 `{a:1}` 是**两个不同的对象**，`toBe` 会失败。比较对象/数组内容要用 `toEqual`（递归比每个字段）。基本值（数字、字符串、布尔）用 `toBe`。
- ⚠️ **浮点数别用 `toBe`**：`0.1 + 0.2` 因为二进制浮点精度，结果是 `0.30000000000000004`。要用 `toBeCloseTo`，它允许一个极小的误差。这个坑第 2 章讲过，测试里同样要小心。
- `.not` 可以接在任何匹配器前取反。

📌 匹配器很多，记不住就查文档。日常 90% 用得着的就是 `toBe / toEqual / toThrow / toContain / toBeCloseTo` 这几个。

## 22.5 `beforeEach` / `afterEach`：准备与清理

很多测试需要"每个用例跑之前都先准备一份新数据，跑完再清理"。手动在每个 `it` 里复制粘贴太蠢，Vitest 提供了**生命周期钩子（lifecycle hooks）**：

```typescript
// 文件：code/ch22/hooks.test.ts（演示用）
import { describe, it, expect, beforeEach, afterEach } from "vitest";

describe("购物车", () => {
  let cart: string[];

  beforeEach(() => {
    // 每个 it 运行"之前"都会执行：给一份全新的空购物车
    cart = [];
  });

  afterEach(() => {
    // 每个 it 运行"之后"都会执行：这里做清理（本例其实不需要）
  });

  it("加一件商品后长度为 1", () => {
    cart.push("apple");
    expect(cart).toHaveLength(1);
  });

  it("新用例拿到的是全新的空车（互不干扰）", () => {
    expect(cart).toHaveLength(0); // 不受上一个 it 影响
  });
});
```

**逐段拆解：**

- `beforeEach` 在**每一个** `it` 之前重新执行，所以第二个用例里 `cart` 又是空的，**两个用例互不干扰**。
- 这正体现了好测试的一条铁律：**测试之间必须独立**。用例 A 不该依赖用例 B 是否跑过、跑没跑成。`beforeEach` 帮你保证每个用例从干净状态起步。
- 还有 `beforeAll` / `afterAll`（整个 `describe` 只跑一次），用于开销大的一次性准备（如启动数据库连接）。默认优先用 `beforeEach`，更安全。

💡 后面测 `debounce` 时，我们会用 `beforeEach(() => vi.useFakeTimers())` 和 `afterEach(() => vi.useRealTimers())`，把"接管时钟"和"还原时钟"分别放进钩子，保证每个用例都从真实时钟状态出发。

## 22.6 测试异步代码

第 8 章讲了 `async/await`。异步函数的测试有个天然陷阱：**如果不 `await`，测试可能在断言之前就"结束"了**，导致假绿。Vitest 支持异步测试——只要把回调写成 `async`。

```typescript
// 文件：code/ch22/async.test.ts（演示用）
import { describe, it, expect } from "vitest";

function delay<T>(value: T, ms: number): Promise<T> {
  return new Promise((resolve) => setTimeout(() => resolve(value), ms));
}

async function loadName(): Promise<string> {
  return delay("Ada", 10);
}

async function boom(): Promise<never> {
  throw new Error("网络挂了");
}

describe("异步测试", () => {
  it("用 async/await 直接断言结果", async () => {
    const name = await loadName();
    expect(name).toBe("Ada");
  });

  it("用 .resolves 断言 resolve 的值", async () => {
    await expect(loadName()).resolves.toBe("Ada");
  });

  it("用 .rejects 断言会 reject", async () => {
    await expect(boom()).rejects.toThrow("网络挂了");
  });
});
```

**逐段拆解：**

- 第一种写法最直观：`const name = await loadName();` 等它完成，再 `expect`。**别忘了 `await`**，否则 `name` 会是个 Promise，断言就错了。
- `.resolves` / `.rejects`：接在 `expect(promise)` 后面，专门断言 Promise 的结果。`.resolves.toBe(...)` 断言"这个 Promise 成功且值是……"；`.rejects.toThrow(...)` 断言"这个 Promise 会失败、抛出的错含此信息"。
- ⚠️ **`.resolves`/`.rejects` 前面务必加 `await`**（或 `return`）！`await expect(promise).rejects.toThrow(...)`。漏了 `await`，测试不会等断言完成就退出，一个本该失败的用例可能假装通过。这是异步测试头号坑。

📌 记住两条：异步测试回调用 `async`；断言 Promise 用 `.resolves`/`.rejects` 且**前面加 `await`**。

## 22.7 Mock、Stub、Spy：隔离被测代码

有些函数不"纯"：它依赖网络、依赖时间、依赖随机数、依赖数据库。测这种函数时，我们不想真的发网络、真的等 5 秒，那样又慢又不稳定。解决办法是用**替身**把这些外部依赖换掉。三个术语常被混用，先分清：

- **Spy（间谍）**：包住一个函数，**记录**它被调用了几次、参数是什么，但（通常）保留原行为。用来验证"我的代码有没有正确地调用它"。
- **Stub（桩）**：提供一个**预设的假返回值**，替代真实实现（比如让假 `fetch` 直接返回一份写死的数据）。
- **Mock（模拟）**：功能最全的替身，既能预设返回值（像 stub），又能记录调用（像 spy）。Vitest 里 `vi.fn()` 造出来的就是这种全能替身。

日常口语里大家统称"mock"，不必太纠结。看 Vitest 怎么用：

```typescript
// 文件：code/ch22/mock-demo.test.ts（演示用）
import { describe, it, expect, vi } from "vitest";

describe("vi.fn 基础", () => {
  it("记录调用次数与参数", () => {
    const fn = vi.fn(); // 造一个空的 mock 函数
    fn("a");
    fn("b", 2);
    expect(fn).toHaveBeenCalledTimes(2);
    expect(fn).toHaveBeenCalledWith("a");
    expect(fn).toHaveBeenLastCalledWith("b", 2);
  });

  it("预设返回值（当 stub 用）", () => {
    const getPrice = vi.fn().mockReturnValue(9.9);
    expect(getPrice()).toBe(9.9);

    const fetchData = vi.fn().mockResolvedValue({ ok: true });
    // mockResolvedValue：让它返回一个 resolve 成该值的 Promise
    return expect(fetchData()).resolves.toEqual({ ok: true });
  });
});

describe("vi.spyOn：监视对象上已有的方法", () => {
  it("监视 console.log 而不真的打印", () => {
    const spy = vi.spyOn(console, "log").mockImplementation(() => {});
    console.log("hi"); // 被替换成空实现，不会真打印
    expect(spy).toHaveBeenCalledWith("hi");
    spy.mockRestore(); // 用完还原，别污染别的测试
  });
});
```

**逐段拆解：**

- `vi.fn()` 造一个全新的"间谍函数"。调它之后，`toHaveBeenCalledTimes`、`toHaveBeenCalledWith`、`toHaveBeenLastCalledWith` 这些专用匹配器就能检查它的调用记录。
- `.mockReturnValue(x)` 让 mock 同步返回 `x`；`.mockResolvedValue(x)` 让它返回一个成功 resolve 为 `x` 的 Promise（测异步依赖时超常用）；对应还有 `.mockRejectedValue(err)`。
- `vi.spyOn(对象, "方法名")`：不是凭空造函数，而是**接管一个已存在的方法**（比如 `console.log`、全局 `fetch`）。加 `.mockImplementation(() => {})` 可替换其实现。⚠️ 用完记得 `mockRestore()`（或在 `afterEach` 里 `vi.restoreAllMocks()`）**还原**，否则会污染后续测试——这也是"测试要独立"的要求。

💡 **mock 整个模块**：`vi.mock("./path")` 能把一个模块整体替换掉。但过度 mock 会让测试脱离现实（你测的是"假模块"而不是真代码）。**能用依赖注入（把依赖当参数传进去）就别 mock 模块**——后面小项目里的 `fetchUser` 就是把 `fetch` 当参数传，天然好测。

## 22.8 假定时器：测试 `debounce` 这类"和时间赛跑"的代码

`debounce`（防抖）会用 `setTimeout` 延迟执行。如果测试真的 `setTimeout(200ms)` 再等，那测试就慢了 200 毫秒；测一堆定时器，几秒钟就没了。Vitest 的**假定时器（fake timers）**能让你**手动拨动时钟**，瞬间"快进"，既快又可控。

先看被测的 `debounce`（完整实现，也是小项目的一部分）：

```typescript
// 文件：code/ch22/debounce.ts
export type Debounced<Args extends unknown[]> = ((...args: Args) => void) & {
  cancel: () => void;
};

export function debounce<Args extends unknown[]>(
  fn: (...args: Args) => void,
  wait: number,
): Debounced<Args> {
  let timer: ReturnType<typeof setTimeout> | null = null;

  const debounced = (...args: Args): void => {
    if (timer !== null) clearTimeout(timer);
    timer = setTimeout(() => {
      timer = null;
      fn(...args);
    }, wait);
  };

  debounced.cancel = (): void => {
    if (timer !== null) {
      clearTimeout(timer);
      timer = null;
    }
  };

  return debounced;
}
```

**逐段拆解：**

- 每次调用 `debounced`，先 `clearTimeout` 掉上一个还没到点的定时器（这就是"计时重置"），再重新 `setTimeout`。于是**只有最后一次调用之后的 `wait` 毫秒过去了，`fn` 才真正执行**。
- `debounced.cancel()` 允许你主动取消尚未触发的那次调用。
- 泛型 `Args extends unknown[]` 让 `debounced` 的参数类型和原函数一致，类型安全。

现在用假定时器测它：

```typescript
// 文件：code/ch22/debounce.test.ts（节选：debounce 部分）
import { describe, it, expect, vi, beforeEach, afterEach } from "vitest";
import { debounce } from "./debounce";

describe("debounce（用 fake timers 测定时器逻辑）", () => {
  beforeEach(() => {
    vi.useFakeTimers(); // 接管 setTimeout 等，改由我们手动拨时钟
  });
  afterEach(() => {
    vi.useRealTimers(); // 还原真实定时器
  });

  it("wait 之前不触发，wait 之后才触发一次", () => {
    const spy = vi.fn();
    const debounced = debounce(spy, 200);

    debounced("a");
    expect(spy).not.toHaveBeenCalled(); // 还没到点

    vi.advanceTimersByTime(199);
    expect(spy).not.toHaveBeenCalled(); // 差 1ms

    vi.advanceTimersByTime(1);
    expect(spy).toHaveBeenCalledTimes(1); // 到点！
    expect(spy).toHaveBeenCalledWith("a");
  });

  it("连续调用只触发最后一次（计时重置）", () => {
    const spy = vi.fn();
    const debounced = debounce(spy, 200);

    debounced("first");
    vi.advanceTimersByTime(100);
    debounced("second"); // 重置计时
    vi.advanceTimersByTime(100);
    expect(spy).not.toHaveBeenCalled(); // 距 second 才 100ms

    vi.advanceTimersByTime(100); // 距 second 满 200ms
    expect(spy).toHaveBeenCalledTimes(1);
    expect(spy).toHaveBeenCalledWith("second"); // 只保留最后一次参数
  });

  it("cancel 后不再触发", () => {
    const spy = vi.fn();
    const debounced = debounce(spy, 200);
    debounced("x");
    debounced.cancel();
    vi.advanceTimersByTime(500);
    expect(spy).not.toHaveBeenCalled();
  });
});
```

**逐段拆解：**

- `vi.useFakeTimers()` 一开，`setTimeout` 就不再真的等，而是把回调"挂起"，等你手动推进。放在 `beforeEach` 里，保证每个用例都从假时钟开始。
- `vi.advanceTimersByTime(ms)`：**把假时钟往前拨 ms 毫秒**，所有到点的定时器回调立刻执行。第一个用例里，拨到 199ms 还没触发、拨到 200ms 才触发，精确验证了"等满 wait 才执行"。
- 第二个用例演示"计时重置"：中途再调一次 `debounced("second")`，前一个定时器被清掉，计时从 second 重新开始，所以只有 second 生效。用 `toHaveBeenCalledWith("second")` 验证参数是最后那次的。
- `afterEach(() => vi.useRealTimers())` 还原真实定时器，不污染别的测试。

📌 这一节浓缩了本章多个核心武器：**spy（`vi.fn()`）验证调用**、**fake timers 快进时间**、**钩子做准备与清理**。测"和时间相关"的逻辑，套路就是它。

## 22.9 Mock 异步依赖：测试一个 `fetch` 包装

再看一个"依赖外部网络"的例子——包装 `fetch` 拉取用户。测试时我们**绝不真的联网**，而是把 `fetch` 换成假的。

```typescript
// 文件：code/ch22/user-service.ts
export interface User {
  id: number;
  name: string;
}

export async function fetchUser(
  id: number,
  fetchFn: typeof fetch = fetch,
): Promise<User> {
  const res = await fetchFn(`https://api.example.com/users/${id}`);
  if (!res.ok) {
    throw new Error(`请求用户 ${id} 失败：HTTP ${res.status}`);
  }
  return (await res.json()) as User;
}
```

**逐段拆解：**

- 注意第二个参数 `fetchFn: typeof fetch = fetch`：这是**依赖注入（dependency injection）**。默认用全局 `fetch`，但测试时可以传入一个假的。**把外部依赖做成可替换的参数，是让代码"可测"的关键技巧。**
- 成功（`res.ok`）就解析 JSON；失败（如 404）就抛出带状态码的错误。这两条分支都要测。

测试：

```typescript
// 文件：code/ch22/debounce.test.ts（节选：fetchUser 部分）
import { describe, it, expect, vi, afterEach } from "vitest";
import { fetchUser } from "./user-service";

describe("fetchUser（用 mock 测异步 + 依赖外部）", () => {
  afterEach(() => {
    vi.restoreAllMocks();
  });

  it("成功时解析并返回用户", async () => {
    const fakeFetch = vi.fn().mockResolvedValue({
      ok: true,
      status: 200,
      json: async () => ({ id: 1, name: "Ada" }),
    } as Response);

    const user = await fetchUser(1, fakeFetch as unknown as typeof fetch);

    expect(user).toEqual({ id: 1, name: "Ada" });
    expect(fakeFetch).toHaveBeenCalledWith("https://api.example.com/users/1");
    expect(fakeFetch).toHaveBeenCalledTimes(1);
  });

  it("HTTP 失败时抛出带状态码的错误（用 rejects 断言）", async () => {
    const fakeFetch = vi.fn().mockResolvedValue({
      ok: false,
      status: 404,
      json: async () => ({}),
    } as Response);

    await expect(
      fetchUser(99, fakeFetch as unknown as typeof fetch),
    ).rejects.toThrow("HTTP 404");
  });

  it("也可以用 spyOn 监视全局 fetch", async () => {
    const spy = vi.spyOn(globalThis, "fetch").mockResolvedValue({
      ok: true,
      status: 200,
      json: async () => ({ id: 7, name: "Lin" }),
    } as Response);

    const user = await fetchUser(7); // 不传第二参，走全局 fetch
    expect(user).toEqual({ id: 7, name: "Lin" });
    expect(spy).toHaveBeenCalledOnce();
  });
});
```

**逐段拆解：**

- 第一个用例造了个 `fakeFetch`，用 `mockResolvedValue` 让它返回一个假的 `Response`（只需提供被测代码真正用到的字段：`ok`、`status`、`json`）。然后不仅断言**返回值**对（`toEqual`），还断言**行为**对——URL 拼对了、只调了一次。**既验证结果，也验证交互**，是 mock 测试的价值所在。
- 第二个用例把 `ok` 设成 `false`、`status` 设 `404`，断言 `fetchUser` 会 `rejects.toThrow("HTTP 404")`——覆盖了错误分支。
- 第三个用例展示另一条路：用 `vi.spyOn(globalThis, "fetch")` 直接接管全局 `fetch`，这样连参数都不用传。`afterEach` 里的 `vi.restoreAllMocks()` 把它还原，防止污染其他测试。
- ⚠️ 那个 `as Response` 是因为我们只造了 `Response` 的一小部分字段，TS 会抱怨类型不全，用断言告诉它"我知道我在干嘛"。测试里这种局部断言可以接受。

## 22.10 覆盖率：有用，但别唯它论

**测试覆盖率（code coverage）**衡量"你的测试跑到了多少代码"。常见指标：

- **行覆盖率（line）**：多少比例的代码行被执行到。
- **分支覆盖率（branch）**：`if/else`、`? :` 的每个分支是否都走到过。
- **函数覆盖率（function）**：多少函数被调用过。

Vitest 生成覆盖率报告（需要额外装一次覆盖率插件）：

```bash
# 覆盖率是可选插件，第一次用先装：
npm install -D @vitest/coverage-v8
# 然后带 --coverage 跑：
npx vitest run ch22 --coverage
```

你会得到一张表，标出每个文件的各项覆盖率百分比，以及**哪些行没被测到**——这非常有用，能提醒你"这个错误分支我忘了测"。

⚠️ **但千万别"唯覆盖率论"。** 覆盖率高 ≠ 测得好：

- 覆盖率只说"这行**跑过了**"，不保证"这行的**结果被断言检查过**"。你可以写一堆调用但一个 `expect` 都没有，覆盖率照样 100%，却什么都没验证。
- 盲目追求 100% 会诱使你为不值得测的边角写一堆脆弱测试，性价比极低。
- 覆盖率是**发现盲区的工具**，不是**目标**。看到某个关键分支没覆盖，去补；但别为了把数字从 96% 刷到 100% 而写垃圾测试。

📌 正确姿势：把覆盖率当"体检报告"，重点看**有没有重要逻辑完全没测到**，而不是盯着总分。

## 22.11 TDD：红 → 绿 → 重构

前面我们都是"先写实现，再补测试"。**TDD（测试驱动开发，Test-Driven Development）** 反过来——**先写测试，再写实现**。它有一个著名的三步循环：

1. **红（Red）**：先写一个会失败的测试（因为功能还没实现），跑，看它变红。这一步逼你**先想清楚"我要的行为到底是什么"**。
2. **绿（Green）**：写**刚好够用**的实现，让测试变绿。不多写，不优化，先让它过。
3. **重构（Refactor）**：测试保护下，放心地把实现改漂亮。改完再跑，仍绿即安全。

然后回到第 1 步，加下一个测试。小步快跑。

我们用 `capitalize`（首字母大写）完整走一遍。

**第 1 步 · 红。** 先写测试，此时 `string-utils.ts` 里还没有 `capitalize`：

```typescript
// 假设先只写这个测试（此刻实现还不存在）
import { capitalize } from "./string-utils";

it("把首字母大写、其余小写", () => {
  expect(capitalize("hello")).toBe("Hello");
  expect(capitalize("wORLD")).toBe("World");
});
```

跑 `npx vitest run`——**红**：报错找不到 `capitalize`。这是好事，说明测试确实在检查一个尚不存在的行为。

**第 2 步 · 绿。** 写刚好够用的实现：

```typescript
export function capitalize(input: string): string {
  return input[0].toUpperCase() + input.slice(1).toLowerCase();
}
```

再跑——**绿**。

**第 3 步 · 想边界，再红再绿。** 现在问自己：空串会怎样？`input[0]` 是 `undefined`，`.toUpperCase()` 直接抛错。补一个测试逼出这个 bug：

```typescript
it("空串返回空串（边界）", () => {
  expect(capitalize("")).toBe("");
});
```

跑——**红**（当前实现对空串会崩）。于是补一行守卫：

```typescript
export function capitalize(input: string): string {
  if (input.length === 0) return ""; // 处理边界
  return input[0].toUpperCase() + input.slice(1).toLowerCase();
}
```

再跑——**绿**。

📌 TDD 的精髓不是"测试"本身，而是**用测试驱动你把需求想清楚、把边界考虑全**。"空串怎么办""负数怎么办"——这些问题在 TDD 里会被自然地、一个个逼出来，而不是等上线后由用户帮你发现。下面的小项目，我们就整章用 TDD 的思路推进。

## 常见坑与最佳实践

⚠️ **坑 1：异步测试漏 `await`。** `expect(p).rejects.toThrow()` 前面忘了 `await`，测试可能在断言完成前就退出，假绿。异步断言一律 `await`。

⚠️ **坑 2：`toBe` 比对象。** 两个内容相同的对象引用不同，`toBe` 失败。比内容用 `toEqual`。

⚠️ **坑 3：mock/spy 不还原。** `vi.spyOn` 或 `vi.useFakeTimers` 用完不还原，会污染后续测试，导致"单独跑绿、一起跑红"的诡异现象。用 `afterEach(() => { vi.restoreAllMocks(); vi.useRealTimers(); })` 兜底。

⚠️ **坑 4：测试互相依赖。** 用例 B 依赖用例 A 留下的状态，一旦调整顺序或单独跑就崩。每个用例都要能独立运行，用 `beforeEach` 重置状态。

⚠️ **坑 5：过度 mock。** 把什么都 mock 掉，测的其实是"假实现"，真代码坏了都测不出来。**能用依赖注入就别 mock 模块**，尽量测真实逻辑。

💡 **实践 1：AAA 模式。** 每个测试分三段——**Arrange（准备数据）→ Act（执行被测代码）→ Assert（断言结果）**。结构清晰，一眼看懂。

💡 **实践 2：一个测试只测一件事。** 用例名就是它验证的那句话（"空串返回空串"）。测试失败时，名字直接告诉你哪坏了。

💡 **实践 3：好测试要快。** 单元测试应在毫秒级完成。慢测试（真联网、真等定时器）没人愿意频繁跑，也就失去了安全网的意义——这正是 mock 和 fake timers 的用武之地。

💡 **实践 4：测行为，别测实现细节。** 断言"输入 X 得到输出 Y"，而不是"内部调了哪个私有方法"。这样重构内部实现时测试不会无谓地红。

### 第二步：写实现，把灯变绿

对着测试写实现，每个函数都刚好满足测试列出的行为：

```typescript
// 文件：code/ch22/string-utils.ts
export function capitalize(input: string): string {
  if (input.length === 0) return "";
  return input[0].toUpperCase() + input.slice(1).toLowerCase();
}

export function slugify(input: string): string {
  return input
    .trim()
    .toLowerCase()
    .replace(/[^a-z0-9]+/g, "-") // 连续的非字母数字压成一个 "-"
    .replace(/^-+|-+$/g, ""); // 去掉首尾多余的 "-"
}

export function truncate(
  input: string,
  maxLength: number,
  ellipsis = "…",
): string {
  if (maxLength < 0) {
    throw new RangeError(`maxLength 不能为负数，收到 ${maxLength}`);
  }
  if (input.length <= maxLength) return input;
  if (maxLength <= ellipsis.length) return ellipsis.slice(0, maxLength);
  return input.slice(0, maxLength - ellipsis.length) + ellipsis;
}

export function wordCount(input: string): number {
  const trimmed = input.trim();
  if (trimmed.length === 0) return 0;
  return trimmed.split(/\s+/).length;
}

export function isPalindrome(input: string): boolean {
  const cleaned = input.toLowerCase().replace(/[^a-z0-9]/g, "");
  let left = 0;
  let right = cleaned.length - 1;
  while (left < right) {
    if (cleaned[left] !== cleaned[right]) return false;
    left++;
    right--;
  }
  return true;
}
```

**逐段拆解（重点看边界是怎么被测试逼出来的）：**

- `capitalize`：第一行 `if (input.length === 0) return "";` 就是被"空串"那个测试逼出来的——没有它，`input[0]` 会是 `undefined` 而崩。
- `slugify`：用正则 `/[^a-z0-9]+/g` 把**连续**的非字母数字整段替换成一个 `-`（`+` 是关键，否则 `a---b` 里每个 `-` 都换一次就压不掉了），再用 `/^-+|-+$/g` 削掉首尾的 `-`。"纯符号返回空串"和"连续分隔符压成一个"两个测试，正好验证了这两条正则。
- `truncate`：三道关卡——负数抛错（异常）、不超长原样返回（边界）、`maxLength` 比省略号还短时只返回被裁的省略号（极端边界）。`truncate("hello world", 8)` 取前 7 个字符 + `…` = 8 个字符，正好不超 `maxLength`。
- `wordCount`：先 `trim`，空了就返回 0（否则 `"".split(/\s+/)` 会返回 `[""]`，长度 1，算错）。这个坑也是被"纯空白算 0"的测试逼出来的。
- `isPalindrome`：先清洗成只剩小写字母数字，再双指针两端夹逼。空串清洗后还是空串，循环不进入，直接返回 `true`。

### 快照测试小示例

`debounce.test.ts` 末尾还有一个**快照测试**演示：

```typescript
// 文件：code/ch22/debounce.test.ts（节选：快照示例）
it("对象结构与快照一致", () => {
  const profile = { id: 1, name: "Ada", roles: ["admin", "author"] };
  expect(profile).toMatchInlineSnapshot(`
    {
      "id": 1,
      "name": "Ada",

        "author",
      ],
```

**逐段拆解：**

## 小结与自测

这一章我们把"验证代码正确性"从 `console.log` 升级成了专业的自动化测试：

- **为什么测**：回归保护、重构信心、活文档。**测试金字塔**：单元多、集成中、e2e 少。
- **Vitest**：`describe`/`it`/`expect`，匹配器 `toBe`/`toEqual`/`toThrow`/`toContain`/`toBeCloseTo`，`beforeEach`/`afterEach`，`npx vitest run` 与 watch。
- **异步测试**：`async` 回调、`.resolves`/`.rejects`（前面必须 `await`）。
- **Mock/Stub/Spy**：`vi.fn()`、`vi.spyOn`、`mockResolvedValue`；**fake timers**（`vi.useFakeTimers` + `advanceTimersByTime`）测 `debounce`；**依赖注入**让 `fetch` 好测。
- **覆盖率**：发现盲区的工具，别唯它论。
- **TDD**：红 → 绿 → 重构，用测试驱动你想清需求与边界。
- **好测试**：AAA、独立、可读、快。

**自测题**（答不上来就回翻对应小节）：

1. `toBe` 和 `toEqual` 什么时候用哪个？为什么比较对象不能用 `toBe`？
2. 测一个会 reject 的异步函数，正确写法是什么？漏了 `await` 会怎样？
3. `vi.fn()`、`vi.spyOn`、`vi.mock` 分别在什么场景用？
4. 为什么测 `debounce` 要用 fake timers 而不是真的 `setTimeout` 等待？
5. TDD 的三步循环是什么？"红"这一步的价值在哪？
6. 为什么说"覆盖率 100% 不等于测得好"？

答完这六题，你就具备给任何代码补测试的能力了。
