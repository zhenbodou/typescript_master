## 25.1 性能：先测量，再优化

关于性能，最重要的一句话不是任何技巧，而是一条纪律：

> 📌 **先测量，再优化（Measure first, optimize later）。** 未经测量的优化，十有八九是在优化一个根本不慢的地方，还顺手把代码改得更难懂。

### Big-O：一点点直觉

**Big-O** 描述的是"当数据量 n 变大时，耗时/内存怎么增长"，它不关心常数，只关心趋势。你只需要建立几个直觉：

| 记法 | 名字 | 典型例子 | n 翻倍时 |
| --- | --- | --- | --- |
| `O(1)` | 常数 | `Map.get`、`Set.has`、数组按下标访问 | 不变 |
| `O(log n)` | 对数 | 二分查找 | 几乎不变 |
| `O(n)` | 线性 | 遍历一遍数组、`Array.includes` | 翻倍 |
| `O(n log n)` | 线性对数 | `Array.sort` | 略超两倍 |
| `O(n²)` | 平方 | 双重循环、循环里套 `includes` | 变四倍 |

⚠️ 新手最常见的性能灾难，就是**在循环里做了一次 `O(n)` 的查找**，于是整体变成 `O(n²)`。数据量一大就卡死。

### 用 `performance.now()` 做基准测试

我们用一个统一的计时工具 `timeit`，把同一段逻辑跑很多次、量出总耗时：

```typescript
// 文件：code/ch25/bench.ts
import { performance } from "node:perf_hooks";

// 工具：把一段函数跑 rounds 次，返回耗时（毫秒）。
function timeit(label: string, fn: () => void, rounds = 1): number {
  // warm-up：先空跑一次，避免第一次运行的即时编译（JIT）预热影响读数。
  fn();
  const start = performance.now();
  for (let i = 0; i < rounds; i++) fn();
  const elapsed = performance.now() - start;
  console.log(`${label.padEnd(28)} ${elapsed.toFixed(3)} ms  (${rounds} 次)`);
  return elapsed;
}
```

逐段拆解：

- **`performance.now()`**：来自 `node:perf_hooks`，返回一个高精度的毫秒时间戳（带小数），比 `Date.now()` 精确得多，专门用于测耗时。两次相减就是这段代码花的时间。
- **warm-up（预热）**：V8 引擎会对热点代码做即时编译（JIT，Just-In-Time compilation）优化。第一次运行往往偏慢，先空跑一次把它"热起来"，读数更公平。
- **`rounds` 循环**：单次运行可能只要零点几毫秒，噪声太大。跑几千次再看总时间，结论才稳。

### 演示 1：把循环里的重复工作提出去

```typescript
// 文件：code/ch25/bench.ts（节选）
const data = Array.from({ length: 50_000 }, (_, i) => i);

function sumBad(): number {
  let total = 0;
  for (let i = 0; i < data.length; i++) {
    // ⚠️ 每一轮都重新构造同一个 config——重复计算，纯属浪费。
    const config = buildConfig();
    total += data[i] * config.factor;
  }
  return total;
}

function sumGood(): number {
  let total = 0;
  const len = data.length;      // 长度提到循环外。
  const config = buildConfig(); // 只构造一次，提到循环外。
  for (let i = 0; i < len; i++) {
    total += data[i] * config.factor;
  }
  return total;
}
```

`buildConfig()` 每次返回的都是同一个结果，但 `sumBad` 在 5 万次循环里**每一轮都重新算一遍**。`sumGood` 把不随循环变化的东西（`config`、`len`）提到循环外，只算一次。实测差距是数量级的：

```text
=== 演示 1：循环里的重复工作 ===
sumBad                       15076.503 ms  (2000 次)
sumGood                      95.147 ms  (2000 次)
```

> 💡 这类"循环不变量外提（loop-invariant hoisting）"是最朴素也最有效的优化：不需要任何高级技巧，只是别做无用功。

### 演示 2：用数据结构把 O(n²) 降到 O(n)

```typescript
// 文件：code/ch25/bench.ts（节选）
function countHitsQuadratic(): number {
  let hits = 0;
  for (const n of needles) {
    if (haystack.includes(n)) hits++; // includes 是 O(n)，套在循环里就是 O(n²)。
  }
  return hits;
}

function countHitsLinear(): number {
  const set = new Set(haystack); // 一次性建索引，O(n)。
  let hits = 0;
  for (const n of needles) {
    if (set.has(n)) hits++; // has 是 O(1)。
  }
  return hits;
}
```

`Array.includes` 要从头扫到尾，是 `O(n)`；套在另一个循环里就成了 `O(n²)`。而 `Set.has` 是 `O(1)`——只要先花 `O(n)` 把数组灌进 `Set` 建好"索引"，后面每次查找都是常数时间。实测：

```text
=== 演示 2：O(n^2) vs O(n) ===
countHitsQuadratic           81.533 ms  (20 次)
countHitsLinear              5.320 ms  (20 次)
```

> 📌 这是第 7 章 `Map`/`Set` 的实战价值：**当你需要频繁判断"存不存在"或"按 key 取值"，用 `Set`/`Map` 而不是数组。**

### 演示 3：memoize（记忆化）缓存纯函数结果

对于**纯函数**（同样的输入永远给同样的输出，见第 4 章），可以把算过的结果缓存起来，下次直接取：

```typescript
// 文件：code/ch25/bench.ts（节选）
function memoize<A, R>(fn: (arg: A) => R): (arg: A) => R {
  const cache = new Map<A, R>();
  return (arg: A): R => {
    if (cache.has(arg)) return cache.get(arg)!;
    const result = fn(arg);
    cache.set(arg, result);
    return result;
  };
}
```

这正是第 4 章闭包 + 第 16 章泛型的组合：闭包保存 `cache`，泛型 `<A, R>` 让它对任意"单参数纯函数"都类型安全。实测第二次调用几乎瞬间返回：

```text
=== 演示 3：memoize ===
slowFib(32) 第一次              20.478 ms  (1 次)
fastFib(32) 第一次              0.001 ms  (1 次)
fastFib(32) 第二次(命中缓存)        0.000 ms  (1 次)
```

⚠️ memoize 不是银弹：缓存会占内存、只对纯函数安全、参数是对象时用 `Map` 按引用比较可能永远命不中。用对场景才有效。

### 演示 4：`console.time` —— 最省事的临时计时

想快速量一段代码，不必写 `timeit`，用内置的一对即可：

```typescript
// 文件：code/ch25/bench.ts（节选）
console.time("build-big-string");
let s = "";
for (let i = 0; i < 100_000; i++) s += "x";
console.timeEnd("build-big-string"); // 打印 build-big-string: X ms
```

`console.time(标签)` 和 `console.timeEnd(同一标签)` 之间的耗时会被打印出来。适合临时插桩。

## 25.2 Node 性能：别阻塞事件循环

回顾第 8 章：Node 是**单线程 + 事件循环**。这意味着——

> ⚠️ **一段同步的、耗时的代码会把整个事件循环卡住**，在它跑完之前，服务器无法处理任何其它请求。

典型元凶：在请求处理里做了一个巨大的同步循环、同步读一个大文件（`fs.readFileSync`）、或一次 `JSON.parse` 一个几百 MB 的字符串。应对：

- **I/O 用异步版本**：`fs/promises` 而不是 `fs.readFileSync`（第 11 章）。异步 I/O 交给底层去等，不占用事件循环。
- **大数据用流（stream）**：边读边处理，不要一次性载入内存（第 11 章的日志分析 CLI 就是范例）。
- **CPU 密集任务用 `worker_threads`**：图像处理、加密、大规模计算这类"纯算"的活，放进 worker 线程，别占用主线程。（本书不展开，知道有这个出口即可。）

> 📌 一句话记忆：**Node 擅长 I/O 密集（等数据库、等网络），不擅长 CPU 密集（狂算）。** 后者要么拆成异步分片，要么丢给 worker。

## 25.3 调试：让 bug 无处可藏

`console.log` 能解决大部分问题，但当调用栈很深、状态很复杂时，**断点调试**效率高得多。

### VS Code 断点调试 TypeScript

VS Code 通过 `.vscode/launch.json` 配置调试任务。本章在 `code/ch25/launch.json` 放了一份示例（用时把它拷到项目的 `.vscode/launch.json`）：

```jsonc
// 文件：code/ch25/launch.json（节选）
{
  "version": "0.2.0",
  "configurations": [
    {
      // 配置一：用 tsx 直接跑 TS 源码，可在 .ts 文件上打断点（推荐日常调试）。
      "name": "Debug: graceful-shutdown (tsx)",
      "type": "node",
      "request": "launch",
      "runtimeExecutable": "npx",
      "runtimeArgs": ["tsx", "${workspaceFolder}/code/ch25/graceful-shutdown.ts"],
      "console": "integratedTerminal",
      "skipFiles": ["<node_internals>/**"]
    }
  ]
}
```

关键字段：

- **`request: "launch"`**：启动一个新进程来调试（另一种是 `"attach"`，附加到已运行的进程）。
- **`runtimeExecutable` + `runtimeArgs`**：用 `npx tsx` 直接跑 `.ts`，于是你能**直接在 TypeScript 源码上打断点**，无需先编译。这是日常最省事的方式。
- **`skipFiles: ["<node_internals>/**"]`**：调试时跳过 Node 内部代码，别让单步执行掉进引擎源码里。

打好断点后按 F5，程序会停在断点处，你可以查看变量、单步执行（F10 逐行 / F11 步入函数）、看**调用栈（call stack）**面板了解"是谁调用到这里的"。条件断点（右键断点设条件）在循环里只想停某一轮时特别有用。

### `--inspect` + Chrome DevTools

不想用 VS Code 也行。给 Node 加 `--inspect`：

```bash
node --inspect dist/app.js
# 或调试 TS：
node --inspect --import tsx ch25/graceful-shutdown.ts
```

然后在 Chrome 打开 `chrome://inspect`，就能用熟悉的 DevTools 面板下断点、看内存快照。

### source map：为什么调试 TS 却停在 TS 上

你运行的其实是编译后的 JavaScript，为什么断点能停在 `.ts` 源码？靠的是 **source map（源码映射）**——一份记录"编译后某行对应源码某行"的映射文件（`.js.map`）。只要编译时开启 `"sourceMap": true`，调试器就能把运行位置反查回 TS 源码。上面 `launch.json` 的"配置二"就演示了调试已编译产物时用 `sourceMaps: true` + `outFiles` 定位到原始 TS。

### 读懂 stack trace

程序抛错时打印的**调用栈（stack trace）**是定位 bug 的地图，从上往下读：最上面是**出错的那一行**，往下是**一层层调用它的函数**。学会看第一行的文件名:行号，直接跳过去，比盲猜快得多（第 6 章讲过 Error 的 `stack` 属性）。

## 25.4 内存：常见泄漏来源

JavaScript 有垃圾回收（GC），但**只要还有引用指向一个对象，它就不会被回收**。"内存泄漏（memory leak）"往往是你无意间留了引用：

- ⚠️ **未清理的定时器 / 监听器**：`setInterval`、`addEventListener`、`emitter.on` 注册了却从不 `clear`/`off`，回调闭包引用的东西就一直活着。用完记得清。
- ⚠️ **不断增长的全局缓存**：一个 `Map` 当缓存却从不淘汰旧数据，越积越大（考虑用有上限的 LRU 缓存或 `WeakMap`）。
- ⚠️ **闭包意外捕获大对象**：闭包引用了一个大数组，而这个闭包被长期持有（比如塞进了全局数组），大数组就跟着一直不释放。

> 💡 排查手段：`node --inspect` 打开 DevTools 的 Memory 面板，抓两次 **heap snapshot（堆快照）**做对比，看哪类对象在持续增长——那通常就是泄漏点。

## 25.5 打包产物：从 TS 源码到可分发的 dist

开发时用 `tsx` 直接跑 `.ts` 很爽，但**分发/部署时应该交付编译后的 JavaScript**（更快启动、不依赖 tsx）。回顾第 20、21 章，产物由 tsc 或 tsup 生成。一个"库/服务"典型的构建配置会开启：

- `"outDir": "dist"`：产物输出目录。
- `"declaration": true`：额外生成 `.d.ts` 类型声明（第 20 章），别人用你的库时能拿到类型。
- `"sourceMap": true`：生成 source map，线上报错能映射回 TS 源码行号。

`package.json` 里用 `main` / `exports` / `bin` / `files` 字段（第 9 章）声明入口与要发布的文件。构建后 `node dist/xxx.js` 就能直接运行，不再需要 TypeScript。

## 25.6 发布部署：让服务在真实世界活下去

一个能"上线"的服务，比"本地能跑"多了几样东西。我们用 `code/ch25/graceful-shutdown.ts` 这个最小 HTTP 服务把它们都演示一遍。

### 配置来自环境变量

线上不能把端口、密钥写死在代码里，要从**环境变量（environment variables）**读（第 24 章）：

```typescript
// 文件：code/ch25/graceful-shutdown.ts（节选）
interface Config {
  port: number;
  nodeEnv: string;
  shutdownTimeoutMs: number;
}

function loadConfig(): Config {
  return {
    port: Number(process.env.PORT ?? 3000),
    nodeEnv: process.env.NODE_ENV ?? "development",
    shutdownTimeoutMs: Number(process.env.SHUTDOWN_TIMEOUT_MS ?? 10_000),
  };
}
```

用 `??`（空值合并，第 1 章）给每个配置项一个合理默认值，并用 `Number(...)` 把字符串环境变量转成数字。`NODE_ENV=production` 是业界约定，很多库据此关闭调试信息、开启优化。

### 结构化日志

线上日志会被采集系统（如 Loki、CloudWatch）收走检索，输出**一行 JSON**远比人类可读的花括号好解析：

```typescript
// 文件：code/ch25/graceful-shutdown.ts（节选）
function log(level: "info" | "warn" | "error", msg: string, extra: Record<string, unknown> = {}): void {
  const line = JSON.stringify({ ts: new Date().toISOString(), level, msg, ...extra });
  // 错误走 stderr，其余走 stdout——这是进程管理器/容器的惯例。
  if (level === "error") console.error(line);
  else console.log(line);
}
```

每条日志都带时间戳 `ts`、级别 `level`、消息 `msg`，外加任意结构化字段（`...extra`）。⚠️ 注意 **error 走 `stderr`、其余走 `stdout`**——这是容器和进程管理器区分正常输出与错误的惯例。

### 计时中间件 + 健康检查端点

```typescript
// 文件：code/ch25/graceful-shutdown.ts（节选）
function handleRequest(req: IncomingMessage, res: ServerResponse): void {
  const start = performance.now();
  const { method = "GET", url = "/" } = req;

  // --- 计时中间件：请求结束时记录耗时 ---
  res.on("finish", () => {
    const ms = performance.now() - start;
    log("info", "request", { method, url, status: res.statusCode, ms: Number(ms.toFixed(2)) });
  });

  const send = (status: number, body: unknown): void => {
    res.writeHead(status, { "content-type": "application/json" });
    res.end(JSON.stringify(body));
  };

  // --- 健康检查端点 ---
  if (url === "/health") {
    if (shuttingDown) return send(503, { status: "shutting_down" });
    return send(200, { status: "ok", uptime: process.uptime() });
  }
  // ...业务端点略
}
```

- **计时中间件**：请求一进来记 `start`，监听响应的 `finish` 事件，在请求真正结束时算出耗时并记日志。这样你能观测每个端点的响应时间（呼应 25.1 的"先测量"）。
- **健康检查端点 `/health`**：容器编排（如 Kubernetes）和负载均衡会定期请求它，判断这个实例"活着且能服务吗"。注意：一旦开始关闭，它返回 `503`，好让负载均衡**把流量从这个即将退出的实例上摘走**。

### 优雅关闭（graceful shutdown）

这是"上线"和"玩具"最大的区别之一。当运维要停服务时，操作系统会发一个**信号（signal）**：容器/pm2 发 `SIGTERM`，你按 Ctrl+C 发 `SIGINT`。如果进程收到信号就立刻硬退出，**正在处理中的请求会被粗暴掐断**，用户看到报错。优雅关闭的做法是：停止接收新请求，等手头的请求处理完，再退出。

```typescript
// 文件：code/ch25/graceful-shutdown.ts（节选）
const shutdown = (signal: string): void => {
  if (shuttingDown) return; // 防止重复触发。
  shuttingDown = true;
  log("warn", "shutdown_start", { signal });

  // server.close 会等所有活跃连接结束，然后回调。
  server.close(() => {
    log("info", "shutdown_complete");
    process.exit(0);
  });

  // 兜底：超过 shutdownTimeoutMs 还没关完，就强制退出，避免进程"卡死"。
  const timer = setTimeout(() => {
    log("error", "shutdown_forced_timeout");
    process.exit(1);
  }, config.shutdownTimeoutMs);
  timer.unref(); // 让这个定时器不要单独把进程"钉住"。
};

process.on("SIGTERM", () => shutdown("SIGTERM")); // 容器/pm2 停止时发送。
process.on("SIGINT", () => shutdown("SIGINT")); // Ctrl+C。
```

逐段拆解：

- **`server.close(cb)`**：停止接收新连接，等所有**活跃连接**处理完毕后调用 `cb`，我们在里面 `process.exit(0)` 正常退出。
- **超时兜底**：万一有请求卡住导致迟迟关不掉，`setTimeout` 到点就强制退出（exit code 1），避免进程僵死。`timer.unref()` 让这个定时器不要单独把事件循环撑住（否则明明可以关了却因为定时器还在而不退出）。
- **`process.on("SIGTERM" / "SIGINT")`**：注册信号处理器。`shuttingDown` 标志防止收到多个信号时重复触发。

> 📌 进程管理：生产上常用 **pm2** 或直接由容器编排托管进程，负责崩溃自动重启、多实例、日志收集。它们停止进程的方式正是发 `SIGTERM`——所以你的优雅关闭必须响应它。

### Docker：把服务打包成镜像

**Docker** 把"代码 + 运行时 + 依赖"一起打包成**镜像（image）**，做到"这台能跑，那台也能跑"。我们用**多阶段构建（multi-stage build）**：一个阶段负责编译，另一个只带上产物，最终镜像更小更安全。

```dockerfile
# 文件：code/ch25/Dockerfile
# ---------- 阶段 1：构建 ----------
FROM node:22-slim AS builder
WORKDIR /app

# 先只拷贝清单文件，利用 Docker 层缓存：依赖没变时不重复 npm ci。
COPY package.json package-lock.json ./
RUN npm ci

# 再拷贝源码并编译。假设 npm run build 会把 TS 编译到 dist/。
COPY . .
RUN npm run build

# ---------- 阶段 2：运行 ----------
FROM node:22-slim AS runner
WORKDIR /app
ENV NODE_ENV=production

# 只安装生产依赖（跳过 devDependencies，如 typescript / vitest）。
COPY package.json package-lock.json ./
RUN npm ci --omit=dev

# 从 builder 阶段拷贝编译好的产物。
COPY --from=builder /app/dist ./dist

# 不用 root 运行：node 官方镜像自带非特权用户 node。
USER node
EXPOSE 3000
CMD ["node", "dist/graceful-shutdown.js"]
```

几个要点：

- **先拷清单再拷源码**：`package.json` 没变时，Docker 会复用 `npm ci` 那一层的缓存，不必每次改代码都重装依赖——构建快很多。这是"利用层缓存"的经典技巧。
- **两个阶段**：`builder` 装了全部依赖（含 TypeScript）来编译；`runner` 只 `npm ci --omit=dev` 装生产依赖、再从 builder 拷 `dist/`。最终镜像里没有 TypeScript、没有源码，更小、攻击面更少。
- **`USER node`**：不用 root 跑应用，降低被攻破后的风险。
- **`CMD` 用 `node dist/...`**：跑编译产物，PID 1 收到编排发来的 `SIGTERM` 正好被我们的优雅关闭接住。

配套的 `.dockerignore`（形同 `.gitignore`）避免把 `node_modules`、`.env`、`.git` 拷进构建上下文：

```text
# 文件：code/ch25/.dockerignore（节选）
node_modules
dist
.git
.env
.env.*
```

⚠️ **务必忽略 `.env`**——绝不能把密钥打进镜像。

常用命令（本机需装 Docker）：

```bash
docker build -t task-service .      # 构建镜像
docker run -p 3000:3000 task-service # 运行，映射端口
```

### CI：每次提交自动检查

**CI（持续集成，Continuous Integration）**让每次 `push`/PR 自动跑类型检查、测试、构建，坏代码进不了主干。下面是一条最小的 **GitHub Actions** 流水线：

```yaml
# 文件：code/ch25/.github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: "22"
          cache: "npm"
      - name: Install deps
        run: npm ci
      - name: Typecheck
        run: npm run typecheck
      - name: Test
        run: npm test
      - name: Build
        run: npm run build
```

逐段：

- **`on`**：触发时机——推送到 `main` 或任何 PR。
- **`runs-on`**：在 GitHub 提供的全新 Ubuntu 虚拟机上跑。
- **`steps`**：检出代码 → 装 Node（带 npm 缓存加速）→ `npm ci` 精确安装 → 类型检查 → 测试 → 构建。**任一步失败，整个流水线失败**，PR 上显示红叉，评审就知道这次改动有问题。

> 💡 把这份文件放到项目根的 `.github/workflows/` 下，推到 GitHub 即自动生效。CI 绿了再合并，是团队协作的底线纪律。

## 常见坑与最佳实践

- ⚠️ **凭感觉优化**：改了半天"感觉更快"，其实没量过。✅ 永远先 `performance.now()` / 基准测试定位真正的瓶颈。
- ⚠️ **循环里套 `O(n)` 查找**导致 `O(n²)`。✅ 需要频繁"查存在/取值"就上 `Set`/`Map`。
- ⚠️ **在请求处理里跑同步重活**卡死事件循环。✅ I/O 用异步、大数据用流、纯算用 worker_threads。
- ⚠️ **忘了清理定时器/监听器**造成内存泄漏。✅ 有 `on`/`setInterval` 就要有对应的 `off`/`clearInterval`。
- ⚠️ **收到 SIGTERM 直接硬退出**掐断在途请求。✅ 实现优雅关闭：`server.close` + 超时兜底。
- ⚠️ **把密钥写死在代码或打进镜像**。✅ 用环境变量，`.env` 加进 `.gitignore` 和 `.dockerignore`。
- ⚠️ **部署编译前的 TS 或带着 devDependencies**，镜像臃肿。✅ 交付 `dist/`，生产只装 `--omit=dev`，用多阶段构建。
- ✅ **上线三件套别省**：健康检查端点、结构化日志、CI 流水线。

## 小结与自测

**本章重点回顾：**

- 性能第一纪律是**先测量再优化**；用 `performance.now()` / `console.time` 量，用 Big-O 直觉找 `O(n²)` 陷阱，用 `Set`/`Map`、循环不变量外提、memoize 消除浪费。
- Node 单线程，**别用同步重活阻塞事件循环**；I/O 异步、大数据用流、CPU 密集用 worker。
- 调试用 VS Code 断点 / `--inspect`，靠 **source map** 直接调试 TS，会读调用栈。
- 内存泄漏多来自**未清理的定时器/监听器、无上限缓存、闭包意外持有大对象**。
- 部署产物是编译后的 `dist/`（带 `.d.ts` 与 source map），配置走环境变量。
- 可上线服务要有**健康检查、结构化日志、优雅关闭**；用**多阶段 Dockerfile** 打小镜像，用 **CI** 守住质量底线。

**自测题：**
