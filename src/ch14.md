# 第 14 章 · 接口、类型别名、联合与交叉类型

## 本章目标

第 13 章我们迈进了 TypeScript 的大门：给变量、函数、对象加上**类型注解（type annotation）**，理解了类型推断、`any` 与 `unknown`、字面量类型与类型断言。那些工具让我们能描述"一个值是什么类型"。

但真实项目里的类型远比"一个 `number`"复杂：一个用户对象有 `id`、`name`、`email`；一条通知可能是邮件、短信或推送，各自字段还不一样；一个 API 请求要么成功返回数据、要么失败返回错误码。**如何把这些复杂的、有结构的、"多选一"的领域概念用类型精确表达出来？** 这正是本章要解决的。

学完本章，你会：

- 用 **接口（interface）** 描述对象的"形状（shape）"：可选属性、只读属性、方法签名、索引签名，以及函数类型接口、可调用/可构造接口。
- 掌握接口的 **继承（extends，可多继承）** 与 **声明合并（declaration merging）**。
- 用 **类型别名（type alias）** 给任意类型起名字，并搞清楚 `type` 与 `interface` **到底该用哪个**。
- 彻底吃透 **联合类型（union type）** `A | B`——TypeScript 建模"多选一"的核心武器，配合字面量类型做出"字符串枚举"的效果。
- 理解 **交叉类型（intersection type）** `A & B`——把多个类型合并成一个。
- 攻克本章的重头戏 **可辨识联合（discriminated union）**：带公共标签字段的联合，配合 `switch` 做**完全穷尽处理（exhaustiveness）**，用 `never` 做编译期兜底检查。这是 TS 里最优雅、最实用的建模模式，没有之一。
- 入门 **类型收窄（narrowing）**、`keyof` 与 **索引访问类型（indexed access type）** `T["key"]`。

我们从最基础的"描述一个对象长什么样"开始。

## 14.1 接口：描述对象的形状

第 13 章我们用**内联对象类型**注解过对象，像 `{ x: number; y: number }`。当这个形状要被反复使用时，每次都手写一遍既啰嗦又难维护。**接口（interface）** 就是给对象形状起个名字，一处定义、处处复用。

```typescript
// 文件：code/ch14/01-interface.ts
interface User {
  readonly id: number; // 只读，创建后不可改
  name: string;
  age: number;
  email?: string; // 可选属性，可有可无
  greet(): string; // 方法签名
  updateAge(newAge: number): void; // 带参数的方法签名
}

const alice: User = {
  id: 1,
  name: "Alice",
  age: 30,
  greet() {
    return `你好，我叫 ${this.name}`;
  },
  updateAge(newAge) {
    this.age = newAge;
  },
};

console.log(alice.greet()); // 你好，我叫 Alice
alice.updateAge(31);
console.log("新年龄：", alice.age); // 31

console.log("email 是否存在：", alice.email ?? "（未填写）");

// alice.id = 2; // ⚠️ 取消注释会报错：Cannot assign to 'id' because it is a read-only property.
```

**逐段拆解：**

- `interface User { ... }` 定义了一个名叫 `User` 的接口。注意：接口**只是编译期的类型描述**，编译成 JavaScript 后会被完全抹掉，运行时不存在 `User` 这个东西。
- `readonly id: number`：`readonly` 修饰符表示这个属性**只能在创建时赋值，之后不能改**。这在建模"实体 ID"这类不该变的字段时特别有用。
- `name: string` 和 `age: number` 是普通的**必填属性**。
- `email?: string`：属性名后加 `?` 表示**可选属性（optional property）**。有它没它都符合 `User` 类型。⚠️ 注意可选属性的实际类型是 `string | undefined`，所以取用前最好用 `??` 或先判断。
- `greet(): string` 是**方法签名（method signature）**：一个不接参数、返回 `string` 的方法。`updateAge(newAge: number): void` 则接收一个 `number` 参数、无返回值（`void`）。
- 定义 `alice` 时，TS 会逐一核对：必填属性都在吗？类型对吗？方法签名匹配吗？少一个必填属性、或类型对不上，都会立刻报错。注意 `updateAge(newAge)` 里我们**没写参数类型**——TS 根据接口"上下文推断"出 `newAge` 是 `number`，这叫**上下文类型（contextual typing）**。

📌 接口的本质是一份"契约"：它规定了对象**必须长什么样**。只要一个对象满足这份契约（该有的属性都有、类型都对），它就能被当作这个类型使用——这就是 TypeScript 的**结构化类型系统（structural typing）**，也叫"鸭子类型"：走起来像鸭子、叫起来像鸭子，它就是鸭子。

### 索引签名：描述"键名不固定"的对象

有时对象的键名不是提前定死的，比如一个"字符串到数字"的映射表。这时用**索引签名（index signature）**：

```typescript
// 文件：code/ch14/01-interface.ts（续）
interface StringToNumber {
  [key: string]: number;
}

const scores: StringToNumber = {};
scores["math"] = 95;
scores["english"] = 88;
console.log("数学成绩：", scores["math"]); // 95
console.log("所有分数：", scores); // { math: 95, english: 88 }
```

- `[key: string]: number` 读作"任意 `string` 类型的键，对应的值都是 `number`"。`key` 只是个占位名字，叫 `k`、`prop` 都行。
- 于是 `scores["math"] = 95` 合法，而 `scores["math"] = "满分"` 会报错（值不是 `number`）。

⚠️ 索引签名很灵活但也"很松"：它允许任意键，编译器无法帮你 catch 拼写错误。能用固定属性描述的对象，优先用固定属性；只有键名确实动态时才用索引签名。

## 14.2 函数类型接口、可调用与可构造接口

接口不只能描述"带属性的对象"，还能描述**函数**本身，以及"既能调用又带属性"的特殊对象。

```typescript
// 文件：code/ch14/02-callable-construct.ts
// 1) 用接口描述"函数类型"
interface BinaryOp {
  (a: number, b: number): number;
}

const add: BinaryOp = (a, b) => a + b;
const mul: BinaryOp = (a, b) => a * b;
console.log("add(2, 3) =", add(2, 3)); // 5
console.log("mul(2, 3) =", mul(2, 3)); // 6
```

**逐段拆解：**

- `interface BinaryOp { (a: number, b: number): number }` 里那行 `(a, b): number` 没有方法名，它是一个**调用签名（call signature）**，表示"这个接口本身是一个函数"：接收两个 `number`，返回 `number`。
- 于是 `add`、`mul` 都被约束成"符合 `BinaryOp` 形状的函数"。函数体里的 `a`、`b` 类型由接口推断而来，不必重写。

💡 描述纯函数类型，用**类型别名** `type BinaryOp = (a: number, b: number) => number` 通常更简洁（下一节讲）。函数类型接口的真正价值，在于函数**同时还带属性**的场景——这就是**可调用接口（callable interface）**：

```typescript
// 文件：code/ch14/02-callable-construct.ts（续）
// 2) 可调用接口：既能当函数调用，本身又能带属性
interface Counter {
  (): number; // 可调用签名：直接 counter() 返回当前值
  count: number; // 附加属性
  reset(): void; // 附加方法
}

function makeCounter(): Counter {
  const fn = (() => {
    fn.count += 1;
    return fn.count;
  }) as Counter;
  fn.count = 0;
  fn.reset = () => {
    fn.count = 0;
  };
  return fn;
}

const c = makeCounter();
console.log(c(), c(), c()); // 1 2 3
c.reset();
console.log("重置后：", c()); // 1
```

- `Counter` 接口既有调用签名 `(): number`，又有属性 `count` 和方法 `reset`。这精确描述了"一个函数，同时挂着状态字段"这种在 JS 里很常见的模式。
- `makeCounter` 内部先把箭头函数用 `as Counter` 断言（因为此刻还没挂上 `count`/`reset`，需要我们告诉编译器"稍后会补齐"），再补上属性和方法。

再看**可构造接口（constructable interface）**，它用 `new` 签名描述"能被 `new` 的东西"，也就是构造器类型：

```typescript
// 文件：code/ch14/02-callable-construct.ts（续）
// 3) 可构造接口：用 new 签名描述"能被 new 的东西"
interface PointCtor {
  new (x: number, y: number): { x: number; y: number };
}

function createInstance(Ctor: PointCtor, x: number, y: number) {
  return new Ctor(x, y);
}

class Point {
  constructor(public x: number, public y: number) {}
}

const p = createInstance(Point, 3, 4);
console.log("构造出的点：", p); // Point { x: 3, y: 4 }
```

- `new (x, y): {...}` 是**构造签名（construct signature）**，表示"用 `new` 调用它、传两个 `number`、得到一个 `{x, y}` 对象"。
- `createInstance` 接收一个**类本身**（不是实例）作为参数，在函数内 `new` 它。这在写"工厂""依赖注入"这类框架代码时很常见——把类当值传来传去。

📌 记住三种签名的区别：`(a): R` 是普通调用签名，`new (a): R` 是构造签名，`m(a): R` 是方法签名。绝大多数日常代码只用方法签名，另外两种在写工具/框架时才登场，知道有它们、看得懂即可。

## 14.3 接口继承与声明合并

### extends：接口继承（可多继承）

接口可以用 `extends` **继承**另一个接口，白拿它的全部成员，再补充自己的：

```typescript
// 文件：code/ch14/03-extends-merge.ts
interface Person {
  name: string;
  age: number;
}

interface Admin extends Person {
  role: "admin";
  permissions: string[];
}

const boss: Admin = {
  name: "Boss",
  age: 40,
  role: "admin",
  permissions: ["read", "write", "delete"],
};
console.log(`${boss.name} 拥有权限：${boss.permissions.join(", ")}`);
```

- `Admin extends Person` 意味着 `Admin` 拥有 `Person` 的 `name`、`age`，再加上自己的 `role`、`permissions`。所以定义 `boss` 时四个属性都得给全。
- 注意 `role: "admin"` 用的是**字面量类型**：`role` 不是任意 `string`，而是**必须恰好等于字符串 `"admin"`**。这为后面讲可辨识联合埋下伏笔。

接口还能**多继承**——一次 `extends` 多个接口，用逗号分隔：

```typescript
// 文件：code/ch14/03-extends-merge.ts（续）
interface HasId {
  id: number;
}
interface Timestamped {
  createdAt: Date;
  updatedAt: Date;
}
interface Article extends HasId, Timestamped {
  title: string;
}

const post: Article = {
  id: 100,
  title: "TypeScript 入门",
  createdAt: new Date("2026-01-01"),
  updatedAt: new Date("2026-07-01"),
};
console.log(`文章 #${post.id}《${post.title}》创建于 ${post.createdAt.getFullYear()}`);
```

- `Article extends HasId, Timestamped` 同时继承了两个"能力接口"，`Article` 因此拥有 `id`、`createdAt`、`updatedAt`、`title` 全部四个字段。
- 💡 这是一种很好的建模习惯：把公共字段（如 `id`、时间戳）抽成小接口，各实体按需组合。本章小项目就用到了它。

### 声明合并：同名接口自动合并

TypeScript 有个独特机制：**同名的 `interface` 会自动合并成一个**，这叫**声明合并（declaration merging）**。

```typescript
// 文件：code/ch14/03-extends-merge.ts（续）
interface Box {
  width: number;
}
interface Box {
  height: number;
}
// 现在 Box 同时拥有 width 和 height
const b: Box = { width: 10, height: 20 };
console.log("盒子面积：", b.width * b.height); // 200
```

- 两次声明 `interface Box`，TS 不会报"重复定义"，而是把它们**合并**成 `{ width: number; height: number }`。
- 这个特性最大的用途是**扩展第三方库或全局类型**：比如给 `window` 添加自定义属性、给某个库的接口补字段，你无需修改库源码，只要在自己的文件里再声明一次同名接口即可。

⚠️ 声明合并是把双刃剑：它很强大，但也意味着"接口的定义可能分散在多处"，容易在不经意间被别处扩展。**这也是 `interface` 与 `type` 的关键区别之一——`type` 不能声明合并**（重复定义同名 `type` 会直接报错）。

## 14.4 类型别名：给任意类型起名字

**类型别名（type alias）** 用 `type` 关键字，给**任意类型**起一个名字——注意是"任意类型"，不只是对象。这是它比接口更宽广的地方。

```typescript
// 文件：code/ch14/04-type-alias.ts
type ID = number | string; // 联合类型别名
type Point = { x: number; y: number }; // 对象类型别名
type Pair<T> = [T, T]; // 元组类型别名（带泛型，第 19 章细讲）

const userId: ID = "u_123";
const origin: Point = { x: 0, y: 0 };
const coords: Pair<number> = [3, 4];
console.log(userId, origin, coords);
```

**逐段拆解：**

- `type ID = number | string`：给一个**联合类型**起名叫 `ID`。这是接口做不到的——`interface` 只能描述对象/函数形状，没法给"`number` 或 `string`"这种非对象类型命名。
- `type Point = { ... }`：给对象形状起名，这和 `interface Point { ... }` 效果几乎一样。
- `type Pair<T> = [T, T]`：给一个**元组类型**起名，还带了泛型参数（第 19 章详讲）。

`type` 也能用 `&`（交叉类型，下节讲）实现类似继承的扩展：

```typescript
// 文件：code/ch14/04-type-alias.ts（续）
type Animal = { name: string };
type Dog = Animal & { breed: string }; // 交叉，等价于接口的 extends
const d: Dog = { name: "旺财", breed: "柴犬" };
console.log(`${d.name} 是一只 ${d.breed}`);
```

- `Dog = Animal & { breed: string }` 让 `Dog` 同时拥有 `Animal` 的 `name` 和额外的 `breed`，效果类似 `interface Dog extends Animal`。

### interface vs type：到底用哪个？

这是初学者最纠结的问题。先看对比表：

| 能力 | `interface` | `type` |
| --- | --- | --- |
| 描述对象形状 | ✅ | ✅ |
| 描述函数类型 | ✅（调用签名） | ✅（`(a) => b`，更简洁） |
| 给联合类型命名 | ❌ | ✅ `A \| B` |
| 给基础/元组/字面量命名 | ❌ | ✅ |
| 继承/扩展 | ✅ `extends` | ✅ 用 `&` |
| 声明合并（同名自动合并） | ✅ | ❌（重复报错） |
| 被 `class` 用 `implements` | ✅ | ✅（若是对象类型） |

📌 **实用选择建议：**

- **描述对象/类的形状**——优先用 `interface`。它更贴近"面向对象"的直觉，`extends` 语义清晰，报错信息更友好，还支持声明合并（利于被扩展）。
- **需要联合、交叉、元组、给基础类型/函数类型起名、或做类型运算（如 `keyof`、映射类型）**——必须用 `type`，因为 `interface` 做不到。

一句话记忆：**"形状用 interface，其它一律 type"**。当然这不是铁律，团队保持一致最重要。

```typescript
// 文件：code/ch14/04-type-alias.ts（续）
type Direction = "up" | "down" | "left" | "right"; // 只能是这四个字符串之一
type Handler = (e: string) => void; // 函数类型
type ReadonlyPoint = Readonly<Point>; // 基于已有类型变换

const move: Direction = "up";
const onClick: Handler = (e) => console.log("事件：", e);
const rp: ReadonlyPoint = { x: 1, y: 2 };
console.log(move); // up
onClick("click"); // 事件： click
console.log(rp); // { x: 1, y: 2 }
// rp.x = 5; // ⚠️ 报错：Readonly 使属性只读
```

- 这三行展示了 `type` 独有的场景：字面量联合、函数类型、以及用内置工具类型 `Readonly<T>` 基于已有类型**变换**出新类型（工具类型第 20 章详讲）。这些都是 `interface` 无法表达的。

## 14.5 联合类型：建模"多选一"

**联合类型（union type）** 用竖线 `|` 连接多个类型，表示"这个值是其中之一"。它是 TypeScript 建模能力的核心。

```typescript
// 文件：code/ch14/05-union.ts
function formatId(id: number | string): string {
  return `ID: ${id}`;
}
console.log(formatId(123)); // ID: 123
console.log(formatId("abc")); // ID: abc
```

- `id: number | string` 表示 `id` 可以是 `number` **或** `string`。
- ⚠️ 关键约束：当一个值是联合类型时，你**只能访问所有成员共有的成员**。比如 `id` 上不能直接调 `.toFixed()`（`string` 没有）也不能调 `.toUpperCase()`（`number` 没有）——除非先**收窄**（见 14.7）。上面代码只用了模板字符串（对两者都合法），所以没问题。

### 字面量类型 + 联合 = "字符串枚举"

联合类型配合**字面量类型**威力巨大，能做出比枚举更轻量、更灵活的"多选一取值"：

```typescript
// 文件：code/ch14/05-union.ts（续）
type LogLevel = "debug" | "info" | "warn" | "error";

function log(level: LogLevel, message: string): void {
  console.log(`[${level.toUpperCase()}] ${message}`);
}
log("info", "服务已启动"); // [INFO] 服务已启动
log("error", "连接失败"); // [ERROR] 连接失败
// log("verbose", "..."); // ⚠️ 报错：'verbose' 不在 LogLevel 允许的取值内
```

**逐段拆解：**

- `type LogLevel = "debug" | "info" | "warn" | "error"` 定义了一个只能取这四个字符串之一的类型。传入 `"verbose"` 会立刻报错。
- 相比第 15 章要讲的 `enum`，这种"字面量联合"更轻量：它编译后**不产生任何运行时代码**（就是普通字符串），还能享受编辑器的自动补全。现代 TS 代码里，用字面量联合替代字符串枚举是主流做法。

数字字面量也能这么玩：

```typescript
// 文件：code/ch14/05-union.ts（续）
type DiceRoll = 1 | 2 | 3 | 4 | 5 | 6;
function roll(): DiceRoll {
  return (Math.floor(Math.random() * 6) + 1) as DiceRoll;
}
console.log("掷出：", roll());
```

- `DiceRoll` 限定只能是 1~6。这里用了 `as DiceRoll` 断言，因为 `Math.floor(...) + 1` 的类型是宽泛的 `number`，编译器无法自动确认它落在 1~6 内，需要我们担保。

## 14.6 交叉类型：合并多个类型

**交叉类型（intersection type）** 用 `&` 连接，表示"**同时满足**所有成员"。如果说联合 `A | B` 是"或"，交叉 `A & B` 就是"与"。

```typescript
// 文件：code/ch14/06-intersection.ts
type WithId = { id: number };
type WithName = { name: string };
type WithTimestamps = { createdAt: Date; updatedAt: Date };

// 交叉：同时拥有全部三组属性
type Entity = WithId & WithName & WithTimestamps;

const user: Entity = {
  id: 1,
  name: "Alice",
  createdAt: new Date("2026-01-01"),
  updatedAt: new Date("2026-07-01"),
};
console.log(`#${user.id} ${user.name}`);
```

- `Entity = WithId & WithName & WithTimestamps` 把三个小类型"合并"成一个，`Entity` 必须**同时具备**全部字段。这在效果上很像接口的多继承，是"组合小类型"的常用手段。

一个典型用途是给类型"混入（mixin）"额外能力：

```typescript
// 文件：code/ch14/06-intersection.ts（续）
type Serializable = { toJSON(): string };
type Loggable = { log(): void };
type Model = WithId & Serializable & Loggable;

const model: Model = {
  id: 42,
  toJSON() {
    return JSON.stringify({ id: this.id });
  },
  log() {
    console.log("Model:", this.toJSON());
  },
};
model.log(); // Model: {"id":42}
```

- `Model` 既是"有 id 的东西"，又是"可序列化的东西"，还是"可打日志的东西"。交叉类型让我们把这些正交的能力自由拼装。

⚠️ **交叉的陷阱**：如果两边有**同名属性但类型不兼容**，交叉的结果会坍缩成 `never`（一个"不可能有值"的类型），导致这个类型无法被赋值：

```typescript
// 文件：code/ch14/06-intersection.ts（续）
type A = { kind: "a" };
type B = { kind: "b" };
type Impossible = A & B; // kind 同时要是 "a" 又要是 "b" → never
// const imp: Impossible = { kind: "a" }; // 报错：无论填什么都不满足

type KindType = Impossible["kind"]; // never
const _check: KindType extends never ? true : false = true;
console.log("Impossible.kind is never:", _check); // true
```

- `kind` 被要求"既是 `"a"` 又是 `"b"`"，这不可能同时成立，于是 `kind` 的类型变成 `never`，整个 `Impossible` 就没法有合法的值了。
- 📌 记住：交叉 `&` 适合**合并不冲突的对象类型**；如果你想表达的是"要么 A 要么 B"，那需要的是**联合 `|`**，别用错了。

## 14.7 可辨识联合：TS 建模的利器

现在到了本章最重要的部分。前面我们说过，联合类型的值"只能访问共有成员"。那如果各成员字段差异很大，怎么安全地区分处理？答案就是**可辨识联合（discriminated union）**，也叫"标签联合（tagged union）"。

它的配方只有三步：
1. 每个成员是一个对象类型；
2. 每个成员都有一个**同名的、值为字面量的公共字段**（俗称"标签"或"辨识符"，如 `kind`、`type`、`status`）；
3. 用这些成员组成联合。

之后就能靠检查这个标签字段，让编译器**自动收窄**到对应的成员类型。

```typescript
// 文件：code/ch14/07-discriminated-union.ts
interface Circle {
  kind: "circle";
  radius: number;
}
interface Rectangle {
  kind: "rectangle";
  width: number;
  height: number;
}
interface Triangle {
  kind: "triangle";
  base: number;
  height: number;
}

type Shape = Circle | Rectangle | Triangle;
```

- 三个接口的**公共字段都叫 `kind`**，且每个的 `kind` 都是一个**唯一的字面量**（`"circle"`/`"rectangle"`/`"triangle"`）。这个字段就是"辨识符"。
- `Shape` 是它们的联合：一个 `Shape` 要么是圆、要么是矩形、要么是三角形。

现在用 `switch` 分支处理，见证收窄的魔力：

```typescript
// 文件：code/ch14/07-discriminated-union.ts（续）
function area(shape: Shape): number {
  switch (shape.kind) {
    case "circle":
      // 这里 shape 被收窄为 Circle，可以安全访问 radius
      return Math.PI * shape.radius ** 2;
    case "rectangle":
      return shape.width * shape.height;
    case "triangle":
      return (shape.base * shape.height) / 2;
    default: {
      // 完全穷尽检查（exhaustiveness）：若上面漏了某个成员，
      // shape 在这里就不是 never，赋值会报错，编译期就能发现遗漏
      const _exhaustive: never = shape;
      return _exhaustive;
    }
  }
}
```

**逐段拆解（这段务必吃透）：**

- `switch (shape.kind)` 对辨识符做分支。在 `case "circle"` 这个分支里，编译器**知道** `shape.kind === "circle"`，于是自动把 `shape` 的类型收窄为 `Circle`。因此 `shape.radius` 合法——而如果你在这个分支里写 `shape.width`，会立刻报错，因为 `Circle` 没有 `width`。这就是可辨识联合的核心价值：**编译器帮你保证"只在正确的分支里访问正确的字段"**。
- 每个 `case` 分支里，`shape` 都被精确收窄到对应类型，享受完整的属性提示与检查。
- 重点看 `default` 分支的 `const _exhaustive: never = shape`：这是**完全穷尽检查（exhaustiveness check）**的经典写法。它的原理是——如果前面的 `case` 已经覆盖了 `Shape` 的**全部**成员，那么代码走到 `default` 时，`shape` 的类型会被收窄为 `never`（因为所有可能都被排除了）。而 `never` 可以赋值给 `never`，编译通过。
- ✨ **但如果你以后给 `Shape` 加了第四种图形（比如 `Pentagon`）却忘了加对应的 `case`**，那么 `default` 分支里 `shape` 的类型就还剩 `Pentagon`（不是 `never`），`const _exhaustive: never = shape` 就会**编译报错**："类型 `Pentagon` 不能赋值给 `never`"。于是你在**编译期**就被提醒"有个分支忘了处理"，而不是等到运行时出 bug。

📌 这套"字面量标签 + switch 收窄 + never 兜底"的组合拳，是 TypeScript 建模状态机、事件、API 响应、领域对象的**最佳实践**。请务必把它变成肌肉记忆。

### 用 `in` 与 `typeof` 收窄

可辨识联合是最优雅的收窄方式，但不是唯一的。当成员之间**没有统一标签字段**时，还能用 `in` 操作符（检查属性是否存在）或 `typeof` 收窄：

```typescript
// 文件：code/ch14/07-discriminated-union.ts（续）
type Fish = { swim: () => void };
type Bird = { fly: () => void };
function move(animal: Fish | Bird) {
  if ("swim" in animal) {
    animal.swim(); // 收窄为 Fish
  } else {
    animal.fly(); // 收窄为 Bird
  }
}
move({ swim: () => console.log("游泳") }); // 游泳
move({ fly: () => console.log("飞行") }); // 飞行
```

- `"swim" in animal` 为真时，编译器推断 `animal` 是拥有 `swim` 的那个成员（`Fish`），否则是 `Bird`。
- 而 14.5 里用 `typeof x === "number"` 收窄基础类型，是同一个思路。这些收窄手段第 18 章会系统展开，这里先建立直觉：**通过运行时可检查的条件，把宽泛的联合类型逐步缩小到具体类型。**

（补充：14.5 里 `describe` 函数用 `typeof` 分别收窄了 `number`、`string`、`boolean`，可对照回看。）

## 14.8 keyof 与索引访问类型入门

最后两个小工具，为后续泛型章节铺路。

**`keyof T`** 取出一个对象类型**所有键组成的联合**；**索引访问类型 `T["key"]`** 取出某个属性的类型。

```typescript
// 文件：code/ch14/08-keyof-indexed.ts
interface Product {
  id: number;
  title: string;
  price: number;
  inStock: boolean;
}

// keyof Product 得到所有键组成的联合："id" | "title" | "price" | "inStock"
type ProductKey = keyof Product;

// 索引访问类型：取出某个属性的类型
type PriceType = Product["price"]; // number
type IdOrTitle = Product["id" | "title"]; // number | string

const key: ProductKey = "title";
const price: PriceType = 99.9;
console.log(key, price); // title 99.9
```

**逐段拆解：**

- `keyof Product` 的结果是字面量联合 `"id" | "title" | "price" | "inStock"`。它让"键名"本身变成一种类型。
- `Product["price"]` 用属性名"索引"进类型，取出该属性的类型 `number`。
- `Product["id" | "title"]` 甚至能用**联合当索引**，取出 `number | string`。

这两个特性组合起来，能写出**类型安全的属性读取函数**：

```typescript
// 文件：code/ch14/08-keyof-indexed.ts（续）
function getProp<T, K extends keyof T>(obj: T, k: K): T[K] {
  return obj[k];
}

const laptop: Product = { id: 1, title: "笔记本", price: 6999, inStock: true };
const t = getProp(laptop, "title"); // t 的类型是 string
const p = getProp(laptop, "price"); // p 的类型是 number
console.log(`${t}：￥${p}`); // 笔记本：￥6999
// getProp(laptop, "color"); // ⚠️ 报错："color" 不是 Product 的键
```

- `K extends keyof T` 约束了第二个参数 `k` 只能是 `T` 真实存在的键——传 `"color"` 会报错。
- 返回类型 `T[K]` 精确地是"那个键对应的类型"：取 `"title"` 返回 `string`，取 `"price"` 返回 `number`，绝不含糊。

💡 泛型（`<T, K>`）第 19 章才正式讲，这里你只需感受到：`keyof` 和索引访问让类型能"随取值联动"，这是 TypeScript 类型系统真正强大的地方。看懂即可，暂不必自己写。

## 常见坑与最佳实践

- ⚠️ **可选属性 `?` 的类型含 `undefined`**。`email?: string` 的实际类型是 `string | undefined`，直接当 `string` 用（如 `email.length`）会在 `strict` 模式报错，务必先判断或用 `??` 兜底。
- ⚠️ **联合类型只能访问共有成员**。想访问某个成员独有的属性，必须先收窄（`typeof`/`in`/字面量标签），别硬来。
- ⚠️ **该用联合的地方误用了交叉**。"要么 A 要么 B"是联合 `|`；"同时是 A 和 B"才是交叉 `&`。对两个标签不同的对象类型做交叉，会得到 `never`。
- ⚠️ **可辨识联合一定要写 `default` 的 `never` 兜底**。这是"加了新成员却忘了处理"的编译期保险，几乎零成本，收益巨大。别省。
- 📌 **辨识符字段要用字面量类型**，不能是宽泛的 `string`。写成 `kind: string` 就丧失了收窄能力，编译器无从区分。
- 📌 **形状优先 `interface`，其它优先 `type`**。需要联合/交叉/元组/映射时只能用 `type`；描述对象/类形状时 `interface` 更合适、报错更友好、可声明合并。
- 💡 **公共字段抽成小接口再组合**（`extends` 多个或 `&`），比在每个大接口里重复写 `id`、时间戳更易维护。
- 💡 **优先用字面量联合替代字符串 `enum`**：更轻量、无运行时代码、补全一样好用（枚举的取舍第 15 章讨论）。

## 本章小项目：领域模型建模

我们来给一个**小型电商领域**建模，把本章所有知识点串起来：用 `interface` + 交叉定义实体，用**可辨识联合**建模支付方式、通知、API 响应，并用 `switch` + `never` 做完全穷尽处理。

先看实体建模——公共字段抽成小接口，靠继承组合：

```typescript
// 文件：code/ch14/domain-model.ts
interface HasId {
  readonly id: string;
}
interface Timestamped {
  createdAt: Date;
  updatedAt: Date;
}

interface User extends HasId, Timestamped {
  name: string;
  email: string;
  vipLevel?: 1 | 2 | 3; // 可选 + 字面量联合
}

interface Product extends HasId, Timestamped {
  title: string;
  price: number; // 单位：分，避免浮点误差
  stock: number;
}
```

- `HasId` 和 `Timestamped` 是可复用的"能力接口"，`User` 与 `Product` 都 `extends` 它们，避免重复写 `id`、时间戳。
- `vipLevel?: 1 | 2 | 3` 一行里同时用到了**可选属性**和**数字字面量联合**：要么不填，要么只能是 1/2/3。

接着是核心——用可辨识联合建模**支付方式**，每种方式字段各不相同，靠 `method` 辨识：

```typescript
// 文件：code/ch14/domain-model.ts（续）
interface AlipayPayment {
  method: "alipay";
  account: string;
}
interface WechatPayment {
  method: "wechat";
  openId: string;
}
interface CardPayment {
  method: "card";
  cardNumber: string;
  bank: string;
}
interface CashOnDelivery {
  method: "cod"; // 货到付款，无需额外字段
}

type Payment = AlipayPayment | WechatPayment | CardPayment | CashOnDelivery;

function describePayment(p: Payment): string {
  switch (p.method) {
    case "alipay":
      return `支付宝支付（账号 ${maskTail(p.account)}）`;
    case "wechat":
      return `微信支付（openId ${maskTail(p.openId)}）`;
    case "card":
      return `${p.bank}银行卡支付（尾号 ${p.cardNumber.slice(-4)}）`;
    case "cod":
      return "货到付款";
    default:
      return assertNever(p); // 漏掉任何一种 method，这里就会编译报错
  }
}
```

**逐段拆解：**

- 四个接口各有独特字段（支付宝有 `account`、微信有 `openId`、银行卡有 `cardNumber`/`bank`、货到付款啥都不加），但都带 `method` 标签。
- `describePayment` 在每个 `case` 里，`p` 都被收窄到对应类型，所以能安全访问各自的独有字段：`case "card"` 里 `p.bank`、`p.cardNumber` 都合法，换到别的分支就不合法。
- `default` 用 `assertNever(p)` 兜底，这是把 14.7 的 `never` 检查**抽成了一个可复用函数**（见下文）。

再看**通知**，同样的模式，辨识符换成 `channel`：

```typescript
// 文件：code/ch14/domain-model.ts（续）
interface EmailNotification {
  channel: "email";
  to: string;
  subject: string;
  body: string;
}
interface SmsNotification {
  channel: "sms";
  phone: string;
  text: string;
}
interface PushNotification {
  channel: "push";
  deviceToken: string;
  title: string;
  badge: number;
}

type Notification = EmailNotification | SmsNotification | PushNotification;

function renderNotification(n: Notification): string {
  switch (n.channel) {
    case "email":
      return `邮件 → ${n.to}｜主题：${n.subject}`;
    case "sms":
      return `短信 → ${n.phone}｜${truncate(n.text, 10)}`;
    case "push":
      return `推送 → 设备 ${maskTail(n.deviceToken)}｜${n.title}（角标 ${n.badge}）`;
    default:
      return assertNever(n);
  }
}
```

- 完全是支付方式的翻版，说明这套模式**可以套用到任何"多态数据"上**。邮件/短信/推送的展示逻辑各不相同，但都被统一在一个函数里安全处理。

第三个例子——用**泛型 + 可辨识联合**建模 API 响应"要么成功、要么失败"：

```typescript
// 文件：code/ch14/domain-model.ts（续）
type ApiResult<T> =
  | { status: "ok"; data: T }
  | { status: "error"; code: number; message: string };

function handleResult<T>(res: ApiResult<T>): string {
  switch (res.status) {
    case "ok":
      return `成功：${JSON.stringify(res.data)}`;
    case "error":
      return `失败[${res.code}]：${res.message}`;
    default:
      return assertNever(res);
  }
}
```

- `ApiResult<T>` 用 `status` 辨识：成功分支带泛型数据 `data: T`，失败分支带 `code`/`message`。这正是真实项目里包装接口返回的标准做法——调用方 `switch` 一下就能安全区分成功和失败，绝不会在失败时误读 `data`。

最后是共用的工具函数，尤其是 `assertNever`：

```typescript
// 文件：code/ch14/domain-model.ts（续）
// exhaustiveness 检查：参数类型必须是 never，否则说明联合没被穷尽
function assertNever(x: never): never {
  throw new Error(`未处理的分支：${JSON.stringify(x)}`);
}

function maskTail(s: string): string {
  if (s.length <= 3) return "***";
  return s.slice(0, 3) + "***";
}

function truncate(s: string, max: number): string {
  return s.length > max ? s.slice(0, max) + "…" : s;
}
```

- `assertNever(x: never)` 是完全穷尽检查的"官方"写法：参数类型是 `never`，只有当调用点的实参确实被收窄成了 `never`（即所有成员都已处理）才编译通过。一旦联合新增了成员而某个 `switch` 没跟上，对应的 `assertNever(...)` 调用就会**编译报错**，精确指出遗漏。它的返回类型也是 `never`，所以能放在 `return` 后面而不破坏函数的返回类型。运行时若真被执行（理论上不该发生），则抛错，双保险。

运行演示：

```typescript
// 文件：code/ch14/domain-model.ts（续，演示部分节选）
const payments: Payment[] = [
  { method: "alipay", account: "alice@example.com" },
  { method: "wechat", openId: "ox_abc123def" },
  { method: "card", cardNumber: "6222021234567890", bank: "招商" },
  { method: "cod" },
];
console.log("=== 支付方式 ===");
for (const p of payments) console.log("-", describePayment(p));
```

用 `npx tsx ch14/domain-model.ts` 运行，输出：


📌 回头看整个设计，本章知识点全被串起来了：**`interface` + `extends` 定义实体、抽公共能力接口复用；字面量类型当辨识符；联合类型建模"多选一"；`switch` 收窄让每个分支安全访问独有字段；`assertNever` + `never` 做完全穷尽检查**。这套组合是 TypeScript 在真实业务里最高频、最可靠的建模范式。

## 小结与自测

本章我们掌握了 TypeScript 描述"有结构的数据"和"多选一的数据"的全套武器：

- **接口（interface）** 描述对象形状：可选属性 `?`、只读属性 `readonly`、方法签名、索引签名；还能描述函数类型（调用签名）、可调用接口、可构造接口（`new` 签名）。
- 接口用 **`extends`** 继承（可多继承）；**同名接口自动声明合并**——这是 `interface` 独有、`type` 没有的能力。
- **类型别名 `type`** 能给**任意类型**命名（联合、交叉、元组、函数、字面量）。选择口诀：**形状用 `interface`，其它用 `type`**。
- **联合类型 `A | B`** 表示"多选一"，配合**字面量类型**做出轻量的"字符串枚举"效果；联合值只能访问共有成员，需收窄后才能访问独有成员。
- **交叉类型 `A & B`** 表示"同时满足"，用来合并/混入类型；同名冲突字段会坍缩成 `never`。
- **可辨识联合**（字面量标签 + `switch` 收窄）是 TS 建模的利器；**`assertNever` + `never`** 的完全穷尽检查能在编译期揪出漏处理的分支。
- **收窄**入门：`typeof`、`in`、字面量标签。**`keyof T`** 取键的联合，**`T["key"]`** 取属性类型，二者组合能写出类型安全的取值函数。
