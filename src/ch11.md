# 第 11 章 · Node.js 运行时基础

## 本章目标

前面十章，我们的代码要么在浏览器控制台里跑，要么用 `node` 命令直接执行一个小脚本。但我们一直没认真讲过：**Node.js 到底是什么？它能干什么？** 这一章就来补上这块。

学完前面的模块化（第 9 章）和异步（第 8 章），你已经具备了理解 Node 的全部前置知识。Node.js（简称 Node）让 JavaScript 走出浏览器，能直接读写文件、访问网络、操作系统、写命令行工具、跑后端服务器。它是你从"写页面脚本"迈向"写真正的程序"的第一步。

本章你将学会：

- 认识 **Buffer**。
- 亲手做一个**日志分析命令行工具**。

本章所有示例用 Node.js 运行（本书用 Node 20+，示例在 Node 24 上验证过）。请确认终端里 `node -v` 能看到版本号。

## Node.js 是什么（复习一下）

**Node.js** 是一个 **JavaScript 运行时（runtime）**。所谓"运行时"，就是"能把 JavaScript 代码跑起来的环境"。浏览器是一种运行时，Node 是另一种。

它们的心脏是同一个东西——Google 的 **V8 引擎**（就是 Chrome 浏览器里那个 JS 引擎）。V8 负责把 JavaScript 翻译成机器能执行的指令。但光有引擎还不够：引擎只懂"语言本身"（变量、函数、循环这些），它并不知道怎么读文件、怎么开网络端口。

📌 **关键区分**：

- **语言本身**（ECMAScript 标准）：`let`、`for`、`Array`、`Promise`……这些无论在浏览器还是 Node 里都一样。
- **运行时提供的能力（API）**：浏览器给你 `document`、`fetch`、`localStorage`；Node 给你 `fs`（文件系统）、`http`（网络服务器）、`process`（进程）。这些**不是**语言的一部分，是各自运行时"额外附赠"的工具箱。

Node 在 V8 之外，用 C++ 补上了操作系统级别的能力（文件、网络、进程），再用一个叫 **libuv** 的库提供了我们在第 8 章讲过的**事件循环（event loop）**。这就是为什么 Node 天生擅长处理大量并发的 I/O（输入输出）——它不会傻等一个文件读完，而是先去干别的，读完了再回来。

## Node 与浏览器的全局对象差异

在浏览器里，最顶层的全局对象叫 `window`。你在第 10 章见过它：`window.document`、`window.location`……

在 Node 里，**没有 `window`，也没有 `document`**——因为 Node 根本没有网页、没有 DOM。Node 的全局对象叫 `global`。

```javascript
// 在 Node 里跑这段
console.log(typeof global);      // 'object'
console.log(typeof globalThis);  // 'object'
console.log(global === globalThis); // true

console.log(typeof window);      // 'undefined' —— Node 里没有 window
console.log(typeof document);    // 'undefined' —— Node 里也没有 document
```

**逐段拆解：**

- `global` 是 Node 的全局对象，相当于浏览器里的 `window`。
- `globalThis` 是 ES2020 引入的**统一名字**：无论在浏览器还是 Node，`globalThis` 都指向当前环境的全局对象。💡 **写跨环境代码时，永远用 `globalThis`**，别用 `window` 或 `global`，这样代码搬到哪儿都能跑。
- `window` 和 `document` 在 Node 里是 `undefined`。如果你把一段依赖 DOM 的浏览器代码直接丢进 Node 跑，就会报"`document is not defined`"——现在你知道原因了。

⚠️ **常见误区**：初学者从网上抄一段操作页面的代码放进 Node 里跑，结果报 `window is not defined`，然后一脸懵。记住：**DOM 是浏览器的东西，Node 里没有。** 反过来，`fs`、`process` 这些是 Node 的东西，浏览器里也没有。

Node 里还有几个"两边都有"的东西，比如 `console`、`setTimeout`；而 `process`（下面会讲）是 Node 独有的。

## 导入内置模块：`node:` 前缀

Node 自带一大堆现成的模块，不用安装就能用，叫**内置模块（built-in module）**或**核心模块（core module）**：`fs`（文件）、`path`（路径）、`os`（系统）、`http`（网络）、`events`（事件）……

导入它们时，**推荐加 `node:` 前缀**：

```javascript
import fs from "node:fs";
import path from "node:path";
import os from "node:os";
```

你可能在老代码里见过不带前缀的写法 `import fs from "fs"`，它现在仍然能用。但 `node:` 前缀有两个好处：

📌 **为什么推荐 `node:` 前缀**：

1. **一眼就能看出这是 Node 内置模块**，而不是你 `npm install` 装的第三方包。
2. **避免歧义**：万一 npm 上有个包也叫 `fs`，`node:fs` 明确告诉 Node"我要的是内置那个"，杜绝被冒名顶替（这其实是一种安全防护）。

本章一律使用 `node:` 前缀。

## fs：读写文件

**fs（file system，文件系统）** 是最常用的 Node 模块，负责所有和文件、目录打交道的事：读、写、追加、删除、列目录……

fs 的一个特点是：**同一个功能它提供了三种风格的 API**。搞懂这三种风格的区别，是本章最重要的知识点之一。

### 三种风格：同步、回调异步、Promise

我们先看一个完整例子，同一件事（读写文件）用三种方式各做一遍：

```javascript
// 文件：code/ch11/01-fs.js
import { readFileSync, writeFileSync, existsSync } from "node:fs";
import fs from "node:fs";                 // 回调风格也在这个默认导出上
import fsp from "node:fs/promises";       // Promise 版
import os from "node:os";
import path from "node:path";

// 在系统临时目录里造一个本例专用的文件路径
const tmpFile = path.join(os.tmpdir(), "ch11-fs-demo.txt");

// ---------- 1) 同步（sync）：会阻塞，直到读写完成 ----------
writeFileSync(tmpFile, "第一行\n", "utf8");
const text1 = readFileSync(tmpFile, "utf8");
console.log("[sync] 读到：", JSON.stringify(text1));
console.log("[sync] 文件存在吗？", existsSync(tmpFile));

// ---------- 2) 回调异步（callback）：不阻塞，结果通过回调返回 ----------
fs.appendFile(tmpFile, "第二行\n", "utf8", (err) => {
  if (err) {
    console.error("追加失败：", err);
    return;
  }
  fs.readFile(tmpFile, "utf8", (err2, data) => {
    if (err2) {
      console.error("读取失败：", err2);
      return;
    }
    console.log("[callback] 追加后内容：", JSON.stringify(data));
    runPromiseDemo();
  });
});

async function runPromiseDemo() {
  // ---------- 3) Promise 版（推荐）：配合 async/await 最清爽 ----------
  await fsp.appendFile(tmpFile, "第三行\n", "utf8");
  const text3 = await fsp.readFile(tmpFile, "utf8");
  console.log("[promises] 最终内容：\n" + text3);

  await fsp.rm(tmpFile); // 清理临时文件
  console.log("[promises] 已清理临时文件");
}
```

运行 `node 01-fs.js`，输出：

```
[sync] 读到： "第一行\n"
[sync] 文件存在吗？ true
[callback] 追加后内容： "第一行\n第二行\n"
[promises] 最终内容：
第一行
第二行
第三行

[promises] 已清理临时文件
```

**逐段拆解：**

**导入部分。** 注意三种风格从不同地方来：

- 同步和回调风格都在 `node:fs` 上。同步函数名字都带 `Sync` 后缀（`readFileSync`、`writeFileSync`）；不带 `Sync` 的（`fs.readFile`、`fs.appendFile`）是回调风格。
- Promise 版单独在 `node:fs/promises` 里，我们把它命名为 `fsp`。

**临时文件路径。** 我们用 `os.tmpdir()` 拿到系统临时目录，再用 `path.join` 拼出一个文件名。💡 **为什么写到临时目录？** 这样示例跑完清理掉，不会在项目里留下垃圾文件。`path` 和 `os` 下面会细讲。

**① 同步风格。** `writeFileSync`、`readFileSync` 会**阻塞（block）**——代码执行到这一行就停下来死等，直到磁盘操作彻底完成才往下走。所以 `text1` 拿到的一定是刚写进去的内容，非常直观。`existsSync` 判断文件是否存在，返回 `true`/`false`。

**② 回调异步风格。** `fs.appendFile(...)` **不阻塞**：它把"追加文件"这件事交给底层，然后立刻返回，代码继续跑。等操作真正完成，Node 才回来调用你给的**回调函数（callback）**。回调的第一个参数**永远是错误对象** `err`（没出错时是 `null`），这是 Node 回调的铁律，叫 "error-first callback"。你看到我们嵌套了两层回调（追加完再读），这就是臭名昭著的**回调地狱（callback hell）**的雏形——层层缩进，越写越右。

**③ Promise 风格。** `fsp` 里的每个函数都返回 Promise，可以直接 `await`。对比一下就知道它有多清爽：没有嵌套、没有 `err` 参数（出错会抛异常，用 `try/catch` 接），读到的结果直接是返回值。

📌 **该用哪一种？**

| 风格 | 特点 | 何时用 |
|------|------|--------|
| **Promise（`fs/promises`）** | 不阻塞，`async/await` 写法清爽 | ✅ **默认首选**，绝大多数情况 |
| 回调异步 | 不阻塞，但容易回调地狱 | 对接老 API 时不得已才用 |
| 同步（`Sync`） | 会阻塞整个进程 | ⚠️ 仅限脚本启动时读配置等一次性场景 |

⚠️ **同步 API 的陷阱**：Node 处理你的 JS 代码是**单线程**的。一个同步的 `readFileSync` 读大文件时，**整个程序都卡住**——如果这是个服务器，所有用户的请求都得排队干等。所以在服务器等长期运行的程序里，**几乎永远不要用 `Sync` 版本**。同步版只适合命令行工具启动时读一下配置文件这种"就这么一次、卡一下无所谓"的场合。

### 常用 fs 操作速查

Promise 版（`fsp`）的常用方法：

- `fsp.readFile(path, "utf8")` —— 读文件（带编码返回字符串，不带编码返回 Buffer）。
- `fsp.writeFile(path, data)` —— 写文件（**会覆盖**已有内容）。
- `fsp.appendFile(path, data)` —— 追加到文件末尾。
- `fsp.rm(path)` —— 删除文件；加 `{ recursive: true }` 可删目录。
- `fsp.mkdir(path)` —— 创建目录；加 `{ recursive: true }` 可创建多级。
- `fsp.readdir(path)` —— 读目录，返回文件名数组。
- `fsp.access(path)` —— 检查能否访问（存在/有权限），不能则抛错。

### 目录操作示例

```javascript
// 文件：code/ch11/01b-fs-dir.js
import fsp from "node:fs/promises";
import os from "node:os";
import path from "node:path";

async function main() {
  // mkdtemp 会创建一个带随机后缀的临时目录，避免冲突
  const dir = await fsp.mkdtemp(path.join(os.tmpdir(), "ch11-dir-"));
  console.log("创建目录：", dir);

  // 往目录里写三个文件
  await fsp.writeFile(path.join(dir, "a.txt"), "AAA");
  await fsp.writeFile(path.join(dir, "b.txt"), "BBB");
  await fsp.writeFile(path.join(dir, "c.log"), "CCC");

  // 读目录：默认返回文件名字符串数组
  const names = await fsp.readdir(dir);
  console.log("目录内容：", names); // [ 'a.txt', 'b.txt', 'c.log' ]

  // withFileTypes 让我们能区分文件和子目录
  const entries = await fsp.readdir(dir, { withFileTypes: true });
  for (const e of entries) {
    console.log(`  ${e.name} —— 是文件吗？ ${e.isFile()}`);
  }

  // 递归删除整个目录（recursive: true 相当于 rm -rf）
  await fsp.rm(dir, { recursive: true, force: true });
  console.log("已删除目录");
}

main();
```

**逐段拆解：**

- `fsp.mkdtemp(前缀)` 创建一个**独一无二**的临时目录，Node 会在你给的前缀后面自动加随机字符。这比自己写死一个目录名安全——不会和别的程序或别的运行撞车。
- 我们往里写了三个文件，然后用 `readdir` 读回来。默认它只给你**文件名字符串**。
- 加上 `{ withFileTypes: true }`，返回的就是 **Dirent 对象**，每个对象能用 `.isFile()`、`.isDirectory()` 判断类型。处理真实目录时这个很有用（你常需要区分文件和子文件夹）。
- 最后 `fsp.rm(dir, { recursive: true, force: true })` 递归删掉整个目录。`recursive` 表示连里面的东西一起删，`force` 表示"就算不存在也别报错"。

⚠️ **`writeFile` 会覆盖！** 很多人以为 `writeFile` 是"往里加内容"，其实它会把文件**整个重写**。想追加请用 `appendFile`。

## path：跨平台处理路径

**path** 模块专门处理文件路径字符串。你可能会想："路径不就是字符串拼一拼吗，用 `+` 不行？" 不行——因为不同操作系统的路径分隔符不一样：Linux/macOS 用 `/`，Windows 用 `\`。手动拼接容易在换系统时出 bug。`path` 帮你屏蔽这些差异。

### 常用方法

```javascript
// 文件：code/ch11/02-path.js
import path from "node:path";
import { fileURLToPath } from "node:url";

// ESM 里没有 __dirname / __filename，需要自己算（下一节详解）：
const __filename = fileURLToPath(import.meta.url);
const __dirname = path.dirname(__filename);

// join：把片段拼成路径，自动处理分隔符和多余的斜杠
console.log("join：", path.join(__dirname, "data", "sample.log"));

// resolve：从右往左拼，直到拼出一个绝对路径（以 cwd 为基准）
console.log("resolve：", path.resolve("data", "sample.log"));

// 拆解路径
const p = "/var/log/app/error.log";
console.log("basename：", path.basename(p));         // error.log
console.log("basename(去后缀)：", path.basename(p, ".log")); // error
console.log("dirname：", path.dirname(p));           // /var/log/app
console.log("extname：", path.extname(p));           // .log
console.log("parse：", path.parse(p));

console.log("path.sep：", JSON.stringify(path.sep)); // 当前系统的分隔符
```

**逐段拆解：**

- **`path.join(...片段)`**：把若干片段用当前系统的分隔符拼起来，还会自动清理多余的斜杠。这是**日常拼路径最常用**的方法。
- **`path.resolve(...片段)`**：也拼路径，但保证结果是**绝对路径**。它从右往左处理，遇到绝对路径就停；如果拼到头还是相对的，就以**当前工作目录（cwd）**为基准补成绝对路径。💡 **`join` vs `resolve`**：想要"相对于某个已知目录"就用 `join`；想要"变成绝对路径"就用 `resolve`。
- **`basename(p)`**：取路径最后一段（文件名）。传第二参数 `".log"` 可以顺便去掉后缀。
- **`dirname(p)`**：取所在目录（去掉最后一段）。
- **`extname(p)`**：取扩展名（含点，如 `.log`）。
- **`parse(p)`**：一次性把路径拆成 `{ root, dir, base, ext, name }` 五部分。
- **`path.sep`**：当前系统的路径分隔符（macOS/Linux 是 `"/"`，Windows 是 `"\\"`）。

### ESM 下没有 `__dirname`（重点！）

这是从 CommonJS 转到 ESM 的人**最容易踩的坑**。

在旧的 CommonJS 模块里，Node 自动给你两个变量：`__filename`（当前文件的绝对路径）和 `__dirname`（当前文件所在目录）。很多老教程、老代码都直接用它们。

📌 **但在 ES Modules（也就是本书用的 `import`/`export`）里，`__filename` 和 `__dirname` 根本不存在。** 你写 `console.log(__dirname)` 会直接报错 `__dirname is not defined`。

ESM 提供的替代品是 `import.meta.url`——它是当前模块的 **URL**（形如 `file:///Users/.../02-path.js`）。要把这个 URL 转成普通文件路径，用 `node:url` 里的 `fileURLToPath`：

```javascript
import path from "node:path";
import { fileURLToPath } from "node:url";

const __filename = fileURLToPath(import.meta.url); // 当前文件绝对路径
const __dirname = path.dirname(__filename);        // 当前文件所在目录
```

这两行几乎是每个需要读取"和脚本放在一起的文件"的 ESM 脚本的**标准开头**，记熟它。本章的日志分析工具就要用它来找默认的 `sample.log`。

⚠️ **别把 `cwd` 和 `__dirname` 搞混**：`process.cwd()`（当前工作目录）是**你运行 `node` 命令时所在的目录**，可能和脚本文件本身的位置完全不同。而 `__dirname`（用上面方法算出的）永远指向**脚本文件所在的目录**。想读"跟脚本放一起的资源文件"，用 `__dirname`；想解析"用户在命令行里给的相对路径"，用 `cwd`。

## process：与进程打交道

**process** 是一个全局对象（不用 import 就能用），代表**当前这个 Node 进程**。它是你的程序和外部世界沟通的重要通道。

```javascript
// 文件：code/ch11/02b-process-os.js
import os from "node:os";

// argv：命令行参数数组
console.log("argv：", process.argv);
const userArgs = process.argv.slice(2);
console.log("用户参数：", userArgs);

// env：环境变量（一个对象）
console.log("自定义变量 MY_NAME：", process.env.MY_NAME ?? "(未设置)");

// cwd：当前工作目录
console.log("cwd：", process.cwd());

// 平台与版本
console.log("platform：", process.platform); // 'darwin' | 'win32' | 'linux'
console.log("Node 版本：", process.version);

// stdout：process.stdout.write 不会自动换行
process.stdout.write("这是标准输出，没有换行");
process.stdout.write("——接着上一句\n");

// exit：主动结束进程，0 表示成功，非 0 表示出错
process.exit(0);
```

运行 `MY_NAME=Claude node 02b-process-os.js foo bar`，关键输出：

```
argv： [
  '/usr/.../node',
  '/Users/.../02b-process-os.js',
  'foo',
  'bar'
]
用户参数： [ 'foo', 'bar' ]
自定义变量 MY_NAME： Claude
platform： darwin
...
```

**逐段拆解：**

- **`process.argv`**：命令行参数**数组**。📌 **注意前两个是固定的**：`argv[0]` 是 node 可执行文件的路径，`argv[1]` 是你的脚本路径，**从 `argv[2]` 开始才是用户真正传的参数**。所以取用户参数几乎总是写 `process.argv.slice(2)`。上面命令传了 `foo bar`，`slice(2)` 就拿到 `['foo', 'bar']`。
- **`process.env`**：**环境变量（environment variable）**对象。命令 `MY_NAME=Claude node ...` 在运行前设置了 `MY_NAME`，脚本里就能通过 `process.env.MY_NAME` 读到。💡 环境变量是配置程序的标准方式——数据库密码、API 密钥这类**不该写死在代码里**的东西，通常都通过环境变量传入。用 `??`（第 8 章的空值合并）给个"未设置"的默认值是好习惯。
- **`process.cwd()`**：当前工作目录（见上一节的对比说明）。
- **`process.platform`**：操作系统标识，`'darwin'` 是 macOS，`'win32'` 是 Windows，`'linux'` 是 Linux。写跨平台代码时用它判断。
- **`process.stdout` / `process.stderr` / `process.stdin`**：标准输出、标准错误、标准输入这三个**流（Stream）**。`console.log` 其实底层就是往 `process.stdout` 写。直接用 `process.stdout.write()` 的区别是**它不会自动加换行**，适合需要精细控制输出的场景。`process.stderr` 用来输出错误信息（`console.error` 走的就是它）。
- **`process.exit(code)`**：立刻结束进程。`0` 表示成功，任何**非 0**表示失败。命令行工具出错时，规范做法是打印错误后 `process.exit(1)`，这样调用它的脚本能通过退出码知道"出问题了"。

## os：了解运行环境

**os（operating system）** 模块提供关于操作系统的信息。用得没 fs/path 那么频繁，但偶尔很有用。

```javascript
// 文件：code/ch11/02b-process-os.js（节选）
import os from "node:os";

console.log("CPU 核心数：", os.cpus().length);
console.log("总内存(GB)：", (os.totalmem() / 1024 ** 3).toFixed(1));
console.log("空闲内存(GB)：", (os.freemem() / 1024 ** 3).toFixed(1));
console.log("os.platform()：", os.platform());
console.log("临时目录：", os.tmpdir());
```

**逐段拆解：**

- **`os.cpus()`**：返回一个数组，每个元素描述一个 CPU 逻辑核心。`.length` 就是核心数——需要"开几个并行任务合适"时常参考它。
- **`os.totalmem()` / `os.freemem()`**：总内存 / 空闲内存，单位是**字节**。`1024 ** 3` 就是 1 GB（`**` 是第 1 章的幂运算），所以除以它换算成 GB。
- **`os.platform()`**：和 `process.platform` 一样返回平台标识。
- **`os.tmpdir()`**：系统临时目录路径。前面我们写临时文件就靠它。

## 事件：EventEmitter

Node 的很多东西是**事件驱动（event-driven）**的：不主动去"问结果好了没"，而是"注册一个监听器，等事情发生时被通知"。这套机制的核心就是 `node:events` 里的 **EventEmitter（事件发射器）**。

其实你在第 10 章的浏览器 `addEventListener` 里已经见过类似思想。`EventEmitter` 是 Node 版的通用事件系统。

```javascript
// 文件：code/ch11/03-events.js
import { EventEmitter } from "node:events";

// 1) 直接使用一个 EventEmitter 实例
const bus = new EventEmitter();

// on：注册监听器，同一个事件可以注册多个
bus.on("greet", (name) => {
  console.log(`你好，${name}！`);
});
bus.on("greet", (name) => {
  console.log(`（第二个监听器也收到了：${name}）`);
});

// once：只触发一次，之后自动移除
bus.once("boot", () => {
  console.log("系统启动，这句只会打印一次");
});

// emit：触发事件，后面的参数会传给监听器
bus.emit("greet", "小明");
bus.emit("boot");
bus.emit("boot"); // 第二次不会有反应
```

**逐段拆解：**

- **`emitter.on(事件名, 回调)`**：注册一个监听器。同一个事件可以 `on` 很多次，触发时按注册顺序**依次全部**调用——所以 `greet` 触发一次，两个监听器都打印。
- **`emitter.once(事件名, 回调)`**：只监听**一次**，触发后自动移除。所以 `boot` 我们 `emit` 了两次，只打印了一次。
- **`emitter.emit(事件名, ...参数)`**：**触发（发射）**事件。后面跟的参数会原样传给每个监听器。`emit("greet", "小明")` 里的 `"小明"` 就成了监听器的 `name`。

再看更实际的用法——**自定义一个继承 EventEmitter 的类**，让你自己的对象也能发事件：

```javascript
// 文件：code/ch11/03-events.js（续）
class Downloader extends EventEmitter {
  start(url) {
    this.emit("start", url);
    for (let percent = 25; percent <= 100; percent += 25) {
      this.emit("progress", percent); // 模拟分块进度
    }
    this.emit("done", url);
  }
}

const d = new Downloader();
d.on("start", (url) => console.log(`开始下载：${url}`));
d.on("progress", (p) => console.log(`进度：${p}%`));
d.on("done", (url) => console.log(`下载完成：${url}`));
d.start("https://example.com/file.zip");
```

**逐段拆解：**

- `Downloader extends EventEmitter`（第 5 章的继承）让 `Downloader` 实例天生具备 `on`/`emit`/`once` 能力。
- `start()` 方法里，我们在不同阶段 `this.emit(...)` 发出 `start`、`progress`、`done` 三种事件。
- 外部代码只需 `on` 自己关心的事件。这就实现了**解耦**：`Downloader` 不需要知道谁在听、听了要干嘛，它只管在合适的时机"喊一嗓子"。这正是 Node 生态大量 API（比如流、HTTP 服务器）的设计范式。

📌 输出会依次是：开始下载 → 进度 25/50/75/100% → 下载完成。

## 流（Stream）：处理大文件的正确姿势

### 为什么需要流

假设你要处理一个 **5 GB** 的日志文件。如果用 `fsp.readFile` 一次性读进来，会发生什么？Node 会试图把整整 5 GB 塞进内存的一个字符串里——你的机器很可能直接内存爆掉、程序崩溃。

**流（Stream）** 就是为解决这个问题而生的。流不是一次性搬运整个文件，而是把数据切成一小块一小块（**chunk**），**像水管里的水一样一段段流过**。你处理完一块，它再给你下一块，内存里任何时刻只有一小块数据。

📌 **一句话记住流**：**用固定的、很小的内存，处理任意大的数据。**

流分几种，最常打交道的两种：

- **可读流（Readable Stream）**：数据从这里"流出来"，比如从文件读、从网络请求读。
- **可写流（Writable Stream）**：数据"流进去"，比如写文件、发网络响应。

### 用 pipeline 复制文件

把一个可读流"接"到一个可写流上，就能实现边读边写。这个操作叫 **pipe（管道）**，形象得很——就是拿根管子把两头接上让水流过去。

```javascript
// 文件：code/ch11/04-stream.js
import fsp from "node:fs/promises";
import { createReadStream, createWriteStream } from "node:fs";
import { createInterface } from "node:readline";
import os from "node:os";
import path from "node:path";
import { pipeline } from "node:stream/promises";

const bigFile = path.join(os.tmpdir(), "ch11-big.txt");
const copyFile = path.join(os.tmpdir(), "ch11-big-copy.txt");

async function main() {
  // 先造一个"大"文件：1000 行
  let content = "";
  for (let i = 1; i <= 1000; i++) content += `第 ${i} 行\n`;
  await fsp.writeFile(bigFile, content, "utf8");

  // 1) 用 pipeline 把可读流接到可写流，实现"边读边写"的复制
  const readable = createReadStream(bigFile);
  const writable = createWriteStream(copyFile);
  await pipeline(readable, writable); // 自动处理错误和收尾
  console.log("已用流复制文件");
```

**逐段拆解：**

- `createReadStream(路径)` 创建一个**可读流**；`createWriteStream(路径)` 创建一个**可写流**。注意它们来自 `node:fs`（不是 `fs/promises`）。
- `pipeline(可读流, 可写流)` 把两者接起来，让数据从读流一块块流进写流。💡 我们用的是 `node:stream/promises` 里的 `pipeline`（返回 Promise，能 `await`），它比老式的 `readable.pipe(writable)` **更安全**——会自动处理出错和关闭流的收尾工作，避免资源泄漏。整个过程内存里从不会同时装下 1000 行的全部内容。

### 用 readline 按行读取

处理日志、CSV 这类"一行一条记录"的文件时，最舒服的是**按行读**。`node:readline` 模块把一个可读流包装成"逐行产出"的接口：

```javascript
// 文件：code/ch11/04-stream.js（续）
  // 2) 用 readline 按行读取（大文件按行处理的标准做法）
  const rl = createInterface({
    input: createReadStream(copyFile, "utf8"),
    crlfDelay: Infinity, // 正确处理 Windows 的 \r\n 换行
  });

  let lineCount = 0;
  for await (const line of rl) {
    lineCount++;
    if (lineCount <= 3) console.log(`readline 拿到：${line}`);
  }
  console.log(`一共读到 ${lineCount} 行`); // 1000
```

**逐段拆解：**

- `createInterface({ input: 可读流 })` 创建一个 readline 接口。`input` 我们给它一个文件的可读流。
- `crlfDelay: Infinity` 是个很实用的选项：它让 readline 正确地把 Windows 的 `\r\n` 当成一个换行处理，不会多出空行。⚠️ 处理可能来自 Windows 的文本文件时，加上它能省掉一类恼人的 bug。
- **`for await (const line of rl)`** 是关键：这是第 7 章的**异步迭代**。它一行一行地把内容交给你，处理完这一行才读下一行——所以哪怕文件有几个 GB，内存占用也稳定在很小的水平。这正是我们下面日志分析工具的核心技术。

### Buffer 简介

前面读文件时我们都带了 `"utf8"` 编码，拿到的是**字符串**。如果**不带编码**，fs 给你的是一个 **Buffer**：

```javascript
// 文件：code/ch11/04-stream.js（续）
  // 3) 不带编码读，拿到的是二进制 Buffer
  const buf = await fsp.readFile(copyFile);
  console.log("Buffer 长度(字节)：", buf.length);       // 11893
  console.log("前 10 个字节：", buf.subarray(0, 10));
  // <Buffer e7 ac ac 20 31 20 e8 a1 8c 0a>

  await fsp.rm(bigFile);
  await fsp.rm(copyFile);
}
main();
```

**Buffer** 是 Node 用来表示**二进制原始数据**的类型——本质是一段固定长度的字节序列。文本文件读成字符串很方便，但图片、音频、压缩包这些**二进制数据**没法用字符串正确表示，就得用 Buffer。

- `buf.length` 是**字节数**（注意：一个中文字符在 UTF-8 里占 3 个字节，所以字节数和字符数不相等）。
- 打印出来的 `<Buffer e7 ac ac ...>` 是每个字节的**十六进制**值。头三个字节 `e7 ac ac` 正是"第"这个汉字的 UTF-8 编码。

📌 **初学阶段你只需知道**：读写文本用字符串（带 `"utf8"`），处理二进制才碰 Buffer。它是流在底层传递数据块时用的载体。

## 常见坑与最佳实践

⚠️ **坑 1：在 ESM 里用 `__dirname`。** ES Modules 里 `__dirname`/`__filename` 不存在。用 `fileURLToPath(import.meta.url)` + `path.dirname` 自己算。这是从 CJS 转过来的头号坑。

⚠️ **坑 2：滥用同步 API。** 在服务器或任何需要响应的程序里用 `readFileSync` 会阻塞整个进程，卡住所有并发任务。默认用 `node:fs/promises` + `await`，同步版只留给"启动时读配置"这类一次性场合。

⚠️ **坑 3：`writeFile` 覆盖了原有内容。** `writeFile` 是整体重写，不是追加。要追加请用 `appendFile`。

⚠️ **坑 4：`process.argv` 前两项当成了用户参数。** 记住 `argv[0]`、`argv[1]` 是 node 和脚本路径，用 `process.argv.slice(2)` 取真正的参数。

⚠️ **坑 5：一次性读大文件。** 几百 MB 以上的文件别用 `readFile`，用流或 readline 逐块/逐行处理，否则内存会爆。

⚠️ **坑 6：混淆 `cwd` 和脚本目录。** `process.cwd()` 是运行命令时所在的目录；脚本自己的目录要用 `__dirname`（算出来的那个）。读"和脚本放一起的资源"用后者。

💡 **最佳实践 1：内置模块一律加 `node:` 前缀**，一眼区分内置和第三方，还更安全。

💡 **最佳实践 2：路径拼接永远用 `path.join` / `path.resolve`**，不要用字符串 `+` 或手写斜杠，保证跨平台。

💡 **最佳实践 3：CLI 工具出错时打印到 `stderr` 并 `process.exit(1)`**，让调用方能通过退出码判断成败。

## 本章小项目：日志分析 CLI

现在把本章学的东西串起来，做一个真正能用的命令行工具。它读取一个日志文件，**用 readline 流式逐行处理**（所以哪怕日志几个 GB 也不怕），统计各日志级别的数量、按小时统计请求数、提取所有 ERROR 行，最后打印一份报告。

先看示例日志文件（每行格式为 `日期 时间 级别 消息`）：

```
2026-07-01 08:01:12 INFO  GET /api/users 200 12ms
2026-07-01 09:05:51 ERROR failed to connect to database: timeout
2026-07-01 09:12:44 WARN  high memory usage 87%
...
```

完整代码（示例日志放在 `code/ch11/sample.log`，共 20 行，含 INFO/WARN/ERROR 三种级别）：

```javascript
// 文件：code/ch11/log-analyzer.js
// 用法：node log-analyzer.js [日志文件路径]
// 不传参数时，默认分析同目录下的 sample.log。
import { createReadStream } from "node:fs";
import { access } from "node:fs/promises";
import { createInterface } from "node:readline";
import path from "node:path";
import { fileURLToPath } from "node:url";

// ESM 下没有 __dirname，需要自己算出脚本所在目录
const __dirname = path.dirname(fileURLToPath(import.meta.url));

// 1) 解析命令行参数：process.argv 从下标 2 开始才是用户传的参数
const inputArg = process.argv[2];
const logPath = inputArg
  ? path.resolve(process.cwd(), inputArg) // 相对用户当前目录解析
  : path.join(__dirname, "sample.log");   // 默认值

// 解析一行日志：2026-07-01 09:05:51 ERROR failed to connect ...
function parseLine(line) {
  const trimmed = line.trim();
  if (!trimmed) return null;               // 跳过空行

  const parts = trimmed.split(/\s+/);
  if (parts.length < 3) return null;

  const time = parts[1];                   // HH:MM:SS
  const level = parts[2].toUpperCase();    // INFO / WARN / ERROR
  const hour = time.slice(0, 2);           // 取前两位当小时
  const message = parts.slice(3).join(" ");
  return { level, hour, message, raw: trimmed };
}

async function analyze(filePath) {
  // 先确认文件存在，给出友好错误而不是崩溃
  try {
    await access(filePath);
  } catch {
    console.error(`❌ 找不到日志文件：${filePath}`);
    process.exit(1);                        // 非 0 退出码表示失败
  }

  const stats = {
    total: 0,
    byLevel: { INFO: 0, WARN: 0, ERROR: 0, OTHER: 0 },
    byHour: {},                             // { "09": 5, ... }
    errors: [],                             // 所有 ERROR 行原文
  };

  const rl = createInterface({
    input: createReadStream(filePath, "utf8"),
    crlfDelay: Infinity,
  });

  // 逐行处理，一行处理完才拿下一行 —— 这就是流式处理
  for await (const line of rl) {
    const parsed = parseLine(line);
    if (!parsed) continue;

    stats.total++;

    if (parsed.level in stats.byLevel) {
      stats.byLevel[parsed.level]++;
    } else {
      stats.byLevel.OTHER++;
    }

    stats.byHour[parsed.hour] = (stats.byHour[parsed.hour] || 0) + 1;

    if (parsed.level === "ERROR") {
      stats.errors.push(parsed.raw);
    }
  }

  return stats;
}

function printReport(filePath, stats) {
  console.log("========================================");
  console.log(" 日志分析报告");
  console.log("========================================");
  console.log(`文件：${filePath}`);
  console.log(`总行数：${stats.total}\n`);

  console.log("按级别统计：");
  for (const [level, count] of Object.entries(stats.byLevel)) {
    if (count === 0) continue;
    console.log(`  ${level.padEnd(6)} ${count}`);
  }
  console.log("");

  console.log("按小时统计（请求数）：");
  for (const h of Object.keys(stats.byHour).sort()) {
    const count = stats.byHour[h];
    const bar = "█".repeat(count);          // 简易柱状图
    console.log(`  ${h}:00  ${String(count).padStart(3)} ${bar}`);
  }
  console.log("");

  console.log(`ERROR 明细（共 ${stats.errors.length} 条）：`);
  if (stats.errors.length === 0) {
    console.log("  （无）");
  } else {
    for (const e of stats.errors) console.log(`  - ${e}`);
  }
  console.log("========================================");
}

const stats = await analyze(logPath);
printReport(logPath, stats);
```

运行 `node log-analyzer.js`（不传参数，用默认的 `sample.log`），输出：

```
========================================
 日志分析报告
========================================
文件：/Users/.../code/ch11/sample.log
总行数：20

按级别统计：
  INFO   13
  WARN   3
  ERROR  4

按小时统计（请求数）：
  08:00    4 ████
  09:00    6 ██████
  10:00    5 █████
  11:00    5 █████

ERROR 明细（共 4 条）：
  - 2026-07-01 09:05:51 ERROR failed to connect to database: timeout
  - 2026-07-01 09:06:00 ERROR retry 1 to database failed
  - 2026-07-01 10:04:27 ERROR unhandled exception in /api/checkout: null pointer
  - 2026-07-01 11:23:14 ERROR payment gateway returned 502
========================================
```

也可以显式指定文件：`node log-analyzer.js sample.log`，或换成任意别的日志路径。

**逐段拆解：**

**开头三段（导入 + 参数解析）** 集中体现了本章的核心技能：

- 用 `fileURLToPath(import.meta.url)` + `path.dirname` 算出**脚本目录**（ESM 里没有 `__dirname`）。
- 用 `process.argv[2]` 读用户参数。**带默认值**的逻辑很重要：传了就用 `path.resolve(process.cwd(), inputArg)`——注意这里用 **`cwd`** 解析，因为用户是站在自己当前目录敲的命令，给的相对路径应相对于他所在的位置；没传就 `path.join(__dirname, "sample.log")` 用**脚本目录**里的示例文件。这正是前面"cwd vs 脚本目录"那个坑的实战应用。

**`parseLine`** 把一行文本拆成结构化对象。`split(/\s+/)` 按空白切分（`\s+` 是正则，下一章会细讲），取出时间、级别、消息。`time.slice(0, 2)` 取小时。返回 `null` 表示"这行跳过"（空行或格式不对），调用处用 `continue` 忽略。

**`analyze`** 是核心：

- 先用 `access` 检查文件是否存在。存不到就打印错误到 **`stderr`**（`console.error`）并 `process.exit(1)`——这是 CLI 工具处理"文件找不到"的规范做法，而不是让程序抛一堆栈崩溃。
- `stats` 对象收集所有统计结果。
- 核心的 `for await (const line of rl)` **逐行处理**：读一行、解析、累加计数。因为是流式的，这个工具处理 20 行和处理两千万行，内存占用几乎一样。
- 三类统计同时进行：按级别（`byLevel`）、按小时（`byHour`，用"存在就 +1、不存在就设 1"的经典写法）、收集 ERROR 原文（`errors` 数组）。

**`printReport`** 负责漂亮地打印。`padEnd`/`padStart`（第 3 章的字符串方法）对齐列；`"█".repeat(count)` 用重复字符画一个简易**柱状图**，让按小时的分布一眼可见。

### 扩展练习

1. **Top N 错误消息**：统计每种 ERROR 消息出现的次数（用 `parseLine` 返回的 `message`），打印出现最多的前 3 条。（提示：用一个对象当计数器，最后 `Object.entries` + `sort`。）
2. **支持第二个参数过滤级别**：`node log-analyzer.js sample.log ERROR` 只统计/打印 ERROR 行。用 `process.argv[3]` 读第二个参数。
3. **输出到文件**：加一个选项，把报告用 `fsp.writeFile` 写到 `report.txt`，而不是打印到屏幕。
4. **响应时间统计**：日志里的 `12ms`、`30ms` 是响应耗时。提取这些数字，算出平均响应时间和最慢的一条请求。（下一章的正则会让提取数字更轻松。）
5. **健壮性**：故意传一个不存在的路径，确认工具打印友好错误并以退出码 1 结束（在终端里 `echo $?` 可以看上一条命令的退出码）。

## 小结与自测

本章我们走进了浏览器之外的 JavaScript 世界：

- **Node.js** 是基于 V8 的 JS 运行时，在语言之外提供了文件、网络、进程等操作系统能力。
- Node 的全局对象是 **`global` / `globalThis`**，**没有** `window`、`document`；跨环境代码用 `globalThis`。
- 内置模块导入加 **`node:` 前缀**。
- **fs** 有同步、回调、Promise 三种风格，**默认用 `node:fs/promises` + `await`**，同步版只用于一次性场景。
- **path** 跨平台拼路径；ESM 下用 `fileURLToPath(import.meta.url)` 代替不存在的 `__dirname`。
- **process** 提供 `argv`（`slice(2)` 取用户参数）、`env`、`cwd`、`stdout/stderr`、`exit`。
- **os** 给出 CPU、内存、平台、临时目录等信息。
- **EventEmitter** 用 `on`/`once`/`emit` 实现事件驱动。
- **流** 用固定小内存处理任意大数据；`pipeline` 复制、`readline` + `for await` 逐行处理大文件；二进制数据用 **Buffer** 表示。
