# 第 16 章 · 泛型（Generics）

## 本章目标

前面几章我们学了接口、类型别名、联合类型、类。你已经能给函数的参数和返回值标上类型，让 TypeScript 帮你挡住一大批低级错误。但很快你会撞上一堵墙：**有些函数、有些容器，天生就要能装「任意类型」，可我们又不想丢掉类型信息。**

举个最直白的例子：写一个"取数组第一个元素"的函数。数组里可能是数字、字符串、用户对象……你希望：传进去数字数组，返回值就是数字；传进去字符串数组，返回值就是字符串。用目前学过的工具，你要么给每种类型各写一份（啰嗦到爆），要么用 `any`（等于把类型系统关掉）。

**泛型（generics）**就是为解决这类问题而生的。它让类型也能像函数参数一样被"传进来"，在真正调用时才确定。学完本章，你将能够：

- 说清楚"为什么需要泛型"——不用它会付出什么代价。
- 写出泛型函数，理解**类型推断（type inference）**和**显式指定**两种用法。
- 使用多个类型参数，处理数组与元组。
- 掌握**泛型约束（constraints）** `extends`，特别是用 `keyof` 实现**类型安全的属性访问**。
- 给类型参数设置**默认值**。
- 写出泛型接口、泛型类型别名、泛型类。
- 用泛型表达常见模式：包装容器、`Result` / `Option` 类型。
- 认识 TypeScript 内置的**工具类型（utility types）**：`Partial` / `Required` / `Readonly` / `Record` / `Pick` / `Omit` / `ReturnType`。
- 避开泛型的常见坑：过度泛型化、以及"泛型 vs 联合类型"怎么选。
- 亲手实现一个**类型安全的仓储 `Repository<T>`**。

本章代码用 `npx tsx` 运行，例如 `npx tsx ch16/01-why-generics.ts`。

📌 **一句话记住泛型**：泛型就是**"类型的参数"**。普通函数的参数是"值的占位符"（`function f(x)` 里的 `x`），泛型是"类型的占位符"（`function f<T>()` 里的 `T`），在调用时才被填上具体类型。

## 为什么需要泛型：不用它会怎样

我们把"取数组第一个元素"这件小事，用三种"不带泛型"的思路各写一遍，看看各自的痛处。

```typescript
// 文件：code/ch16/01-why-generics.ts

// 做法一：为每种类型各写一份，啰嗦且无法穷尽。
function firstNumber(arr: number[]): number | undefined {
  return arr[0];
}
function firstString(arr: string[]): string | undefined {
  return arr[0];
}

// 做法二：用 any，虽然「通用」了，但丢掉了全部类型信息。
function firstAny(arr: any[]): any {
  return arr[0];
}

// 做法三：泛型。类型参数 T 像一个「占位符」，在调用时才被填上。
function first<T>(arr: T[]): T | undefined {
  return arr[0];
}
```

逐段拆解：

- **做法一（各写一份）**：`firstNumber`、`firstString`……逻辑一模一样（都是 `return arr[0]`），只有类型不同。这叫**重复代码（duplication）**。而且你永远写不完——布尔数组、日期数组、你自定义的 `User` 数组，每来一种就得再抄一份。
- **做法二（用 `any`）**：`firstAny` 确实"通用"了，一个函数吃所有类型。但代价是**类型信息全丢了**。`any` 的意思是"我放弃对这个值做任何类型检查"。返回 `any` 意味着后面你对结果做什么操作，TS 都不管了——保护网被你亲手剪断。
- **做法三（泛型）**：`first<T>` 里的 `<T>` 声明了一个**类型参数（type parameter）**，名字叫 `T`（type 的首字母，习惯用大写单字母，但你完全可以起有意义的名字）。参数 `arr` 的类型是 `T[]`（元素类型为 `T` 的数组），返回值是 `T | undefined`。`T` 具体是什么，现在还不知道——**要等调用时才确定**。

现在看看三者用起来的差别：

```typescript
// 文件：code/ch16/01-why-generics.ts（续）
const a = firstNumber([1, 2, 3]); // number | undefined
const b = firstString(["x", "y"]); // string | undefined

const bad = firstAny([1, 2, 3]); // 类型是 any，下面这行不会报编译错误（危险！）
try {
  bad.toUpperCase(); // 编译通过，但 bad 其实是 number，运行时才炸
} catch (e) {
  console.log("any 埋下的雷在运行时爆炸:", (e as Error).message);
}

const n = first([1, 2, 3]); // T 被推断为 number，n: number | undefined
const s = first(["hello", "world"]); // T 被推断为 string，s: string | undefined
```

逐段拆解：

- `firstAny([1, 2, 3])` 返回 `any`。于是 `bad.toUpperCase()`（字符串方法）**在编译期一路绿灯**，因为 `any` 关掉了检查。可 `bad` 实际是数字 `1`，运行到这行才崩——这正是 `any` 最阴险的地方：**把错误从编译期推迟到了运行期**，还是在用户面前爆炸。
- 而 `first([1, 2, 3])`，TS 看到实参是 `number[]`，就把 `T` **推断**为 `number`，于是 `n` 的类型精确地是 `number | undefined`。如果你对 `n` 调用字符串方法，TS 会当场拦下来。**一份代码，类型却随输入精确变化**——这就是泛型的魔力。

⚠️ **`any` 不是"通用"，而是"投降"。** 每当你想写 `any`，先停一下问自己：是不是该用泛型？泛型能做到"通用"的同时**保留**类型，`any` 是"通用"的同时**丢掉**类型。

运行看看：

```bash
npx tsx ch16/01-why-generics.ts
```

你会看到 `any` 那行在运行时抛出 `bad.toUpperCase is not a function`，而泛型版本一切安好。

## 泛型函数：推断 vs 显式指定

泛型函数最经典的例子是 `identity`（恒等函数）——原样返回传入的值。它虽然没什么实际用处，却是理解泛型语法的最佳教具。

```typescript
// 文件：code/ch16/02-generic-func.ts

function identity<T>(x: T): T {
  return x;
}

// 1) 类型推断（inference）：不写 <...>，TS 从实参推断 T。
const r1 = identity(42); // T = number，r1: number
const r2 = identity("hi"); // T = string，r2: string
const r3 = identity(true); // T = boolean，r3: boolean

// 2) 显式指定（explicit）：手动写 <...>，覆盖推断结果。
const r4 = identity<string>("abc"); // 明确 T = string
// const r5 = identity<number>("abc"); // 报错：字符串不能赋给 number
```

逐段拆解：

- `function identity<T>(x: T): T` 读作："这个函数接受一个类型参数 `T`；它有一个值参数 `x`，类型是 `T`；它返回一个 `T`。" 三个 `T` 必须是**同一个**类型——传进来是什么，返回就是什么。
- **推断**：`identity(42)`，你没写 `<...>`，TS 看到 `42` 是 `number`，自动令 `T = number`。这是**最常用**的方式——大多数时候你不需要手写类型参数，让 TS 推断即可。
- **显式指定**：`identity<string>("abc")`，你用尖括号手动把 `T` 钉死为 `string`。当推断不出来、或推断结果不合你意时才需要这么写。
- 被注释掉的 `identity<number>("abc")` 会报错：你说了 `T = number`，却传了字符串，矛盾。

💡 **优先靠推断，需要时才显式指定。** 显式写 `<...>` 会让代码变啰嗦；只在 TS 推断不出、或你想收窄/放宽类型时才动手。

### 多个类型参数、元组

一个泛型函数可以有**多个**类型参数，用逗号分隔。它们之间还能互相引用。

```typescript
// 文件：code/ch16/02-generic-func.ts（续）

// 多个类型参数：pair 把两个值组成一个元组。
function pair<A, B>(a: A, b: B): [A, B] {
  return [a, b];
}
const p = pair("age", 30); // p: [string, number]

// swap：交换元组的两个元素，返回类型也随之交换。
function swap<A, B>(t: [A, B]): [B, A] {
  return [t[1], t[0]];
}
const sw = swap(p); // sw: [number, string]

// 类型参数之间可以互相引用：把数组每一项做映射。
function mapArray<T, U>(arr: T[], fn: (item: T) => U): U[] {
  const result: U[] = [];
  for (const item of arr) {
    result.push(fn(item));
  }
  return result;
}
const lengths = mapArray(["a", "bb", "ccc"], (s) => s.length); // number[]
```

逐段拆解：

- `pair<A, B>`：两个类型参数 `A`、`B`，分别对应两个值参数。返回**元组** `[A, B]`。调用 `pair("age", 30)` 时，`A` 推断为 `string`、`B` 推断为 `number`，所以 `p` 的类型是精确的 `[string, number]`。
- `swap<A, B>(t: [A, B]): [B, A]`：注意返回类型把顺序反过来了。传入 `[string, number]`，返回类型就是 `[number, string]`——**类型跟着逻辑一起翻转**，这是普通类型标注做不到的。
- `mapArray<T, U>`：这是 `Array.prototype.map` 的简化版。它体现了泛型的精髓——**类型的"流动"**：输入是 `T[]`，映射函数把每个 `T` 变成 `U`（`fn: (item: T) => U`），所以输出是 `U[]`。调用 `mapArray(["a","bb","ccc"], s => s.length)` 时，`T = string`（数组元素），`fn` 返回 `number` 所以 `U = number`，最终 `lengths` 的类型是 `number[]`。全程没写一个类型标注，TS 全推出来了。

## 泛型约束：给类型参数设"门槛"

泛型很自由——`T` 可以是任何类型。但有时"太自由"反而办不成事。看这个想求"更长的那个"的函数：

```typescript
// 没有约束时，T 可能是任何类型，不能假设它有 .length。
// function longest<T>(a: T, b: T): T {
//   return a.length >= b.length ? a : b; // 报错：T 上不存在 length
// }
```

因为 `T` 可能是 `number`（没有 `length`），TS 不敢让你访问 `a.length`。我们需要告诉 TS："`T` 不是随便什么类型，它**至少**得有个 `length`。" 这就是**泛型约束**，用 `extends` 表达。

```typescript
// 文件：code/ch16/03-constraints.ts

// 约束 T 必须有 length: number 这个属性。
interface HasLength {
  length: number;
}
function longest<T extends HasLength>(a: T, b: T): T {
  return a.length >= b.length ? a : b;
}

const longerStr = longest("hello", "hi"); // string 有 length ✓
const longerArr = longest([1, 2, 3], [9]); // 数组有 length ✓
// longest(10, 20); // 报错：number 没有 length，不满足约束
```

逐段拆解：

- `T extends HasLength` 读作："`T` 必须是 `HasLength` 的**子类型**——也就是说，`T` 至少要具备 `length: number` 这个属性。" 这里的 `extends` **不是继承**，而是"满足……的形状"的意思。
- 有了约束，函数体里访问 `a.length` 就合法了，因为 TS 保证任何 `T` 都有 `length`。
- `longest("hello", "hi")`：字符串有 `length`，✓。`longest([1,2,3], [9])`：数组也有 `length`，✓。而 `longest(10, 20)` 会报错——`number` 没有 `length`，不满足门槛。
- 注意：约束限制了"能传什么"，但**没有丢掉具体类型**。`longerStr` 的类型仍然是精确的 `string`，而不是 `HasLength`。

### `keyof` 约束：类型安全的属性访问

泛型约束最强大、最常用的场景，是配合 **`keyof`** 做"按 key 取属性值"。`keyof T` 会得到 `T` 所有键名组成的联合类型。

```typescript
// 文件：code/ch16/03-constraints.ts（续）

// K 被约束为「T 的所有键之一」，返回类型精确到 T[K]。
function getProp<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const user = { id: 1, name: "Alice", active: true };
const uid = getProp(user, "id"); // number
const uname = getProp(user, "name"); // string
// getProp(user, "email"); // 报错："email" 不是 user 的键

// 反过来：按 key 安全地写入属性值。
function setProp<T, K extends keyof T>(obj: T, key: K, value: T[K]): void {
  obj[key] = value;
}
setProp(user, "name", "Bob"); // ✓ value 必须是 string
// setProp(user, "id", "oops"); // 报错：id 需要 number
```

逐段拆解：

- 对于 `user`，它的类型是 `{ id: number; name: string; active: boolean }`，那么 `keyof typeof user` 就是联合类型 `"id" | "name" | "active"`。
- `getProp<T, K extends keyof T>`：两个类型参数。`T` 是对象类型；`K` 被约束为 `keyof T`，也就是"`T` 的某个键名"。返回类型是 **`T[K]`**——这叫**索引访问类型（indexed access type）**，意思是"`T` 里 `K` 这个键对应的值的类型"。
- 于是 `getProp(user, "id")`，`K` 被推断为字面量 `"id"`，返回类型精确到 `T["id"]` 即 `number`；`getProp(user, "name")` 返回 `string`。**返回类型随你传的 key 精确变化**，这是普通标注绝对做不到的。
- `getProp(user, "email")` 会报错——`"email"` 不在 `keyof user` 里，约束不满足。**拼错属性名当场被抓**。
- `setProp` 是反方向：写入时 `value` 的类型被约束为 `T[K]`，所以给 `"name"` 只能传 `string`，给 `"id"` 传字符串会报错。读写两端都类型安全。

📌 **`K extends keyof T` + 返回 `T[K]` 是 TS 最经典的泛型套路之一**，任何"按字段名操作对象"的工具函数都靠它。我们的本章小项目 `Repository.where` 就用到它。

## 泛型默认值

和函数参数可以有默认值一样，类型参数也可以有**默认值**，写作 `<T = 默认类型>`。当调用方没指定、也推断不出来时，就用默认值。

```typescript
// 文件：code/ch16/04-default-and-shapes.ts

// 不指定时 T 默认为 string。
interface Box<T = string> {
  value: T;
}
const b1: Box = { value: "默认就是 string" }; // T 用默认值 string
const b2: Box<number> = { value: 123 }; // 显式指定 number
```

逐段拆解：

- `interface Box<T = string>`：类型参数 `T` 的默认值是 `string`。
- `const b1: Box`——注意 `Box` 后面没写 `<...>`，此时 `T` 取默认值 `string`，所以 `value` 必须是字符串。
- `const b2: Box<number>`——显式指定 `T = number`，`value` 就必须是数字。

💡 默认值在设计"大多数情况用某个类型，偶尔需要换"的 API 时很有用（比如很多库的配置对象），能让常见用法省去写尖括号。

## 泛型接口、类型别名、类

泛型不只用于函数，**接口、类型别名、类**都能带类型参数。

```typescript
// 文件：code/ch16/04-default-and-shapes.ts（续）

// 泛型接口：描述「可以装任意类型」的容器。
interface Container<T> {
  items: T[];
  add(item: T): void;
  get(index: number): T | undefined;
}

// 泛型类型别名：Result 表示「成功或失败」二选一。
type Result<T, E = Error> =
  | { ok: true; value: T }
  | { ok: false; error: E };

function parseIntSafe(s: string): Result<number> {
  const n = Number(s);
  if (Number.isNaN(n)) {
    return { ok: false, error: new Error(`不是合法数字: ${s}`) };
  }
  return { ok: true, value: n };
}
```

逐段拆解：

- `interface Container<T>`：一个泛型接口。`Container<string>` 就是"装字符串的容器"，`Container<User>` 就是"装用户的容器"，一份定义覆盖所有元素类型。
- `type Result<T, E = Error>`：这是一个**极其实用**的模式。它是可辨识联合（第 14 章学过）：要么 `{ ok: true, value: T }`（成功，带上结果值），要么 `{ ok: false, error: E }`（失败，带上错误）。`E` 有默认值 `Error`。很多语言（Rust、Go 风格）用这种类型代替"抛异常"，让"可能失败"这件事**写进类型里**，调用方不得不处理。
- `parseIntSafe` 返回 `Result<number>`：成功时 `value` 是数字，失败时 `error` 是 `Error`。

再来看泛型类，以及 `Result` 怎么用——靠 `ok` 字段**收窄类型**：

```typescript
// 文件：code/ch16/04-default-and-shapes.ts（续）

// 泛型类：Stack<T> 是一个类型安全的栈。
class Stack<T> {
  private data: T[] = [];
  push(item: T): void {
    this.data.push(item);
  }
  pop(): T | undefined {
    return this.data.pop();
  }
  get size(): number {
    return this.data.length;
  }
}

const numStack = new Stack<number>();
numStack.push(1);
numStack.push(2);
const top = numStack.pop(); // number | undefined

const r1 = parseIntSafe("42");
const r2 = parseIntSafe("abc");
// 用「可辨识联合」的 ok 字段收窄类型：
if (r1.ok) console.log("解析成功:", r1.value); // 这个分支里 r1.value 是 number
if (!r2.ok) console.log("解析失败:", r2.error.message); // 这个分支里 r2.error 是 Error
```

逐段拆解：

- `class Stack<T>`：泛型类。类型参数写在**类名后面**，然后 `T` 就能在整个类体里用——字段 `data: T[]`、方法 `push(item: T)`、`pop(): T | undefined`。
- `new Stack<number>()`：实例化时指定 `T = number`，于是这个栈只接受数字，`pop()` 返回 `number | undefined`。想要字符串栈就 `new Stack<string>()`，一份类定义通吃。
- `if (r1.ok)` 之后，TS 知道 `r1` 一定是 `{ ok: true, value: number }` 那个分支，所以能安全访问 `r1.value`；`if (!r2.ok)` 分支里能安全访问 `r2.error`。**类型系统逼你先判断成功/失败，再取对应字段**，从根上杜绝了"忘了处理错误"。

⚠️ **泛型类的类型参数只影响类型，不影响运行时。** 例如 `Stack<T>` 里没法在运行时知道 `T` 到底是什么（TS 类型在编译后会被擦除，这叫**类型擦除 / type erasure**）。想在运行时判断类型，得靠值本身（`typeof`、字段标记等），不能靠 `T`。

## 内置工具类型入门

TypeScript 内置了一批**工具类型（utility types）**——它们本质上就是官方帮你写好的泛型类型别名，专门用来"基于已有类型造新类型"。这里先学**怎么用**，它们的**实现原理**（映射类型、条件类型）我们留到第 17 章。

```typescript
// 文件：code/ch16/05-utility-types.ts

interface User {
  id: number;
  name: string;
  email: string;
  age: number;
}

// Partial<T>：把 T 的所有属性变成可选，常用于「更新时只传一部分字段」。
function updateUser(user: User, patch: Partial<User>): User {
  return { ...user, ...patch };
}
const u0: User = { id: 1, name: "Alice", email: "a@x.com", age: 20 };
const u1 = updateUser(u0, { age: 21 }); // 只更新 age，合法

// Required<T>：把所有属性变成必填（Partial 的反面）。
interface Config {
  host?: string;
  port?: number;
}
const fullConfig: Required<Config> = { host: "localhost", port: 8080 };

// Readonly<T>：把所有属性变成只读。
const frozen: Readonly<User> = u0;
// frozen.age = 99; // 报错：只读属性不能赋值
```

逐段拆解：

- **`Partial<T>`**：把 `T` 每个属性都变成可选（`?`）。`Partial<User>` 就是"User 的任意子集"，特别适合"局部更新"——`updateUser` 允许只传 `{ age: 21 }`，其余字段沿用旧值。
- **`Required<T>`**：反过来，把所有可选属性变成必填。`Config` 里 `host`、`port` 本来可选，`Required<Config>` 要求两个都得给。
- **`Readonly<T>`**：把所有属性变成只读，赋值就报错。适合表达"这份数据不许改"。

```typescript
// 文件：code/ch16/05-utility-types.ts（续）

// Record<K, V>：构造「键为 K、值为 V」的对象类型。
const scores: Record<string, number> = { math: 90, english: 85 };
type Role = "admin" | "guest";
const perms: Record<Role, boolean> = { admin: true, guest: false };

// Pick<T, K>：从 T 中「挑出」若干属性。
type UserPreview = Pick<User, "id" | "name">;
const preview: UserPreview = { id: 1, name: "Alice" };

// Omit<T, K>：从 T 中「去掉」若干属性（Pick 的反面）。
type UserWithoutEmail = Omit<User, "email">;
const noEmail: UserWithoutEmail = { id: 1, name: "Alice", age: 20 };

// ReturnType<F>：取出函数的返回值类型。
function makeUser() {
  return { id: 1, name: "Alice" };
}
type MadeUser = ReturnType<typeof makeUser>; // { id: number; name: string }
const made: MadeUser = { id: 2, name: "Bob" };
```

逐段拆解：

- **`Record<K, V>`**：造一个"所有键类型为 `K`、所有值类型为 `V`"的对象类型。`Record<string, number>` 是"任意字符串键、值为数字"的字典；`Record<Role, boolean>` 更严格——键**必须恰好是** `"admin"` 和 `"guest"`，少一个都报错。
- **`Pick<T, K>`**：从 `T` 里**挑出** `K` 指定的那几个属性。`Pick<User, "id" | "name">` 只保留 `id` 和 `name`。常用于造"精简版对象"（如列表预览）。
- **`Omit<T, K>`**：`Pick` 的反面，从 `T` 里**去掉** `K` 指定的属性。`Omit<User, "email">` 就是"去掉 email 的 User"。
- **`ReturnType<F>`**：抽取一个函数类型的**返回值类型**。`ReturnType<typeof makeUser>` 得到 `{ id: number; name: string }`。注意 `typeof makeUser` 先拿到函数的类型，再交给 `ReturnType`。这在"不想手写、想直接复用某函数返回结构"时非常方便。

📌 **这些工具类型全是泛型。** 记不住没关系，用到时查即可。常见的还有 `NonNullable<T>`（去掉 `null`/`undefined`）、`Parameters<F>`（取参数元组类型）、`Awaited<T>`（取 Promise 解开后的类型）等。它们全靠第 17 章要讲的映射类型和条件类型实现——学完那章，你能自己写出 `Partial`、`Pick`。

## 常见坑与最佳实践

### 坑一：过度泛型化

看到泛型好用，新手容易"见谁都套个 `<T>`"。但**泛型只有在"类型需要流动、需要关联"时才有意义**。

```typescript
// ❌ 过度泛型：T 只用了一次，等于没用，反而更难读。
function logIt<T>(value: T): void {
  console.log(value);
}

// ✅ 直接用 unknown 即可，函数根本不关心 value 是什么。
function logItBetter(value: unknown): void {
  console.log(value);
}
```

判断法则很简单：**如果一个类型参数在签名里只出现一次，它多半是多余的。** 泛型的价值在于把多个位置的类型"绑在一起"（如 `identity` 把参数和返回值绑成同一个 `T`，`getProp` 把 key 和返回值绑起来）。只出现一次，就没有"绑定"可言，用 `unknown` 或具体类型更清晰。

### 坑二：约束加得不够，或加得太狠

- 约束**太松**（不加）：函数体里没法安全地用 `T` 的属性（回忆 `longest` 不加约束就访问不了 `.length`）。
- 约束**太狠**（如直接把参数标成 `HasLength` 而不是 `T extends HasLength`）：会**丢掉具体类型**。

```typescript
// ❌ 参数直接标成接口：返回类型也退化成接口，丢了 string / 数组的具体类型。
function badLongest(a: HasLength, b: HasLength): HasLength {
  return a.length >= b.length ? a : b;
}
const x = badLongest("hello", "hi"); // x: HasLength，不再是 string！

// ✅ 用 T extends HasLength：既能约束，又保留具体类型（见前面 longest）。
```

用 `T extends HasLength` 时结果会是 `string`；直接标 `HasLength` 时结果退化成 `HasLength`，你就没法再当字符串用了。**能用泛型约束保留具体类型时，就别直接标接口。**

### 坑三：泛型 vs 联合类型，怎么选？

这是最容易混淆的决策。两者都能"处理多种类型"，但语义完全不同：

```typescript
// 联合类型：这个函数能接受 string 或 number，但两者是「或」的关系。
function formatUnion(x: string | number): string {
  return String(x);
}

// 泛型：调用时确定「一种」具体类型，并保持它。
function wrap<T>(x: T): { value: T } {
  return { value: x };
}
```

选择标准：

- **用联合类型**：当一个参数本身就"可能是这几种之一"，而且你要在函数里**分别处理**它们（`typeof` 收窄）。它表达的是"**或**"。
- **用泛型**：当类型需要在**参数与返回值之间流动/关联**，或者需要**保留调用方传入的精确类型**。它表达的是"**同一个、由调用方决定**"。

一句话：**问"这几种类型之间有没有关联"。** 有关联（输入决定输出）用泛型；没关联（就是几选一）用联合。

💡 **最佳实践清单**：类型参数优先靠推断；名字有意义时别吝啬（`Repository<TEntity>` 比 `<T>` 更清晰，虽然本书多数场景 `T` 已足够）；约束尽量精确；一个类型参数只出现一次时删掉它。

## 本章小项目：类型安全的仓储 `Repository<T>`

我们来做一个能"装任意实体、且全程类型安全"的内存仓储（Repository）。它的每个方法——增删改查、按字段筛选、分页、排序——返回类型都会**随实体类型自动变化**。这正是泛型 + 约束 + `keyof` 的综合演武场。

设计要点：

- 所有实体必须有 `id: number`，用约束 `T extends Entity` 表达。
- `getById` 返回 `T | undefined`，`findAll` 返回 `T[]`——类型随 `T` 变。
- `where<K extends keyof T>(key, value)`：只能按 `T` 真实存在的字段筛选，`value` 类型必须匹配该字段。
- 扩展：`sortBy`（泛型排序）、`paginate`（泛型分页）。

先看核心的仓储类：

```typescript
// 文件：code/ch16/repository.ts
// 运行：npx tsx ch16/repository.ts

// 所有能被仓储管理的实体，都必须有一个 number 类型的 id。
interface Entity {
  id: number;
}

interface PageOptions {
  page: number; // 从 1 开始
  pageSize: number;
}

// 分页结果：随 T 变化的泛型返回类型。
interface Page<T> {
  data: T[];
  page: number;
  pageSize: number;
  total: number;
  totalPages: number;
}

class Repository<T extends Entity> {
  private store = new Map<number, T>();

  add(entity: T): T {
    if (this.store.has(entity.id)) {
      throw new Error(`id=${entity.id} 已存在`);
    }
    this.store.set(entity.id, entity);
    return entity;
  }

  getById(id: number): T | undefined {
    return this.store.get(id);
  }

  // 局部更新：patch 是 T 的「部分字段」，用 Partial 表达；
  // 再用 Omit 去掉 id，避免调用方偷偷改主键。
  update(id: number, patch: Partial<Omit<T, "id">>): T | undefined {
    const existing = this.store.get(id);
    if (!existing) return undefined;
    const updated = { ...existing, ...patch } as T;
    this.store.set(id, updated);
    return updated;
  }

  remove(id: number): boolean {
    return this.store.delete(id);
  }

  findAll(): T[] {
    return [...this.store.values()];
  }

  findBy(predicate: (entity: T) => boolean): T[] {
    return this.findAll().filter(predicate);
  }

  // 按「某个字段等于某个值」筛选。类型安全的关键：
  // K 被约束为 keyof T，value 精确到 T[K]。
  where<K extends keyof T>(key: K, value: T[K]): T[] {
    return this.findAll().filter((entity) => entity[key] === value);
  }

  // 扩展：按某字段排序。key 必须是 T 的键。
  sortBy<K extends keyof T>(key: K, order: "asc" | "desc" = "asc"): T[] {
    const sign = order === "asc" ? 1 : -1;
    return this.findAll().sort((a, b) => {
      if (a[key] < b[key]) return -1 * sign;
      if (a[key] > b[key]) return 1 * sign;
      return 0;
    });
  }

  // 扩展：分页，返回 Page<T>。
  paginate(options: PageOptions): Page<T> {
    const { page, pageSize } = options;
    const all = this.findAll();
    const total = all.length;
    const start = (page - 1) * pageSize;
    return {
      data: all.slice(start, start + pageSize),
      page,
      pageSize,
      total,
      totalPages: Math.ceil(total / pageSize),
    };
  }

  get count(): number {
    return this.store.size;
  }
}
```

逐段拆解——重点看泛型如何保证类型安全：

- `interface Entity { id: number }` + `class Repository<T extends Entity>`：约束保证**每种实体都有 `id`**，所以 `add` 里的 `entity.id`、以及用 `Map<number, T>` 以 id 作键，全都合法。任何不带 `id` 的类型都无法作为 `Repository` 的实体，编译期就被挡住。
- `getById(id): T | undefined`：找不到返回 `undefined`，所以返回类型是 `T | undefined`。调用方**被迫**处理"可能没有"的情况——这是类型安全，不是麻烦。
- `update(id, patch: Partial<Omit<T, "id">>)`：这里把两个工具类型叠起来用。`Omit<T, "id">` 先去掉 `id`（禁止改主键），再用 `Partial<...>` 变成"部分可选"，于是你能只更新几个字段，且**碰不到 id**。`{ ...existing, ...patch }` 处用了 `as T` 断言——因为 TS 无法静态证明"旧对象 + 部分补丁"必然还是完整的 `T`，这是合理的类型断言。
- `findBy(predicate: (entity: T) => boolean)`：谓词函数接收的 `entity` 类型是 `T`，所以你在谓词里能安全访问实体的所有字段。
- **`where<K extends keyof T>(key: K, value: T[K])`**：本项目的类型安全之魂。`key` 只能是 `T` 真实存在的字段名；`value` 的类型被钉死为 `T[K]`（那个字段的值类型）。传错字段名或传错值类型，全在编译期报错。
- `sortBy<K extends keyof T>`：同样的 `keyof` 约束——只能按真实字段排序。`a[key] < b[key]` 的比较对字符串和数字字段都成立。
- `paginate(options): Page<T>`：返回泛型 `Page<T>`，其中 `data: T[]` 随实体类型变化。

现在定义两种截然不同的实体，各建一个仓储：

```typescript
// 文件：code/ch16/repository.ts（续）

interface User extends Entity {
  id: number;
  name: string;
  age: number;
  role: "admin" | "guest";
}

interface Product extends Entity {
  id: number;
  title: string;
  price: number;
  inStock: boolean;
}

const users = new Repository<User>();
users.add({ id: 1, name: "Alice", age: 30, role: "admin" });
users.add({ id: 2, name: "Bob", age: 25, role: "guest" });
users.add({ id: 3, name: "Carol", age: 35, role: "guest" });

const products = new Repository<Product>();
products.add({ id: 101, title: "键盘", price: 199, inStock: true });
products.add({ id: 102, title: "鼠标", price: 89, inStock: false });
products.add({ id: 103, title: "显示器", price: 1299, inStock: true });
```

逐段拆解：

- `User` 和 `Product` 都 `extends Entity`（都带 `id`），所以都能塞进 `Repository`。但它们的其余字段天差地别——`User` 有 `role`，`Product` 有 `price`。
- `new Repository<User>()` 把 `T` 钉成 `User`：这个仓储的 `add` 只接受 `User`，`getById` 返回 `User | undefined`。给它塞 `Product` 会报错。**一份 `Repository` 类，服务两套完全不同的数据，且各自类型精确。**

最后是使用与演示：

```typescript
// 文件：code/ch16/repository.ts（续）

const alice = users.getById(1); // User | undefined
console.log("getById(1):", alice);

// where：key 只能是 User 的键，value 类型必须匹配。
const guests = users.where("role", "guest"); // ✓ "guest" 是合法的 role
console.log("where role=guest:", guests.map((u) => u.name));
// users.where("role", "boss"); // 报错：不是合法的 role 字面量
// users.where("email", "x"); // 报错：User 没有 email 字段

const adults = users.findBy((u) => u.age >= 30);
console.log("findBy age>=30:", adults.map((u) => u.name));

users.update(2, { age: 26 }); // patch 是 Partial，且不能改 id
console.log("update(2) 后:", users.getById(2));

// where 在另一种实体上同样类型安全
const available = products.where("inStock", true);
console.log("在售商品:", available.map((p) => p.title));

console.log(
  "按 price 降序:",
  products.sortBy("price", "desc").map((p) => `${p.title}=${p.price}`)
);
console.log("用户分页(第1页,每页2条):", users.paginate({ page: 1, pageSize: 2 }));

users.remove(3);
console.log("删除后用户数:", users.count);
```

逐段拆解：

- `users.where("role", "guest")`：`"role"` 是 `User` 的合法键，`"guest"` 是合法的 `role` 值——通过。而注释里的 `where("role", "boss")` 会报错（`"boss"` 不是 `"admin" | "guest"`），`where("email", ...)` 也报错（`User` 没有 `email`）。**筛选条件的正确性由类型系统保证**，你不可能按不存在的字段查询。
- `products.where("inStock", true)`：同一个 `where` 方法，在 `Product` 仓储上照样类型安全——这里 `"inStock"` 要求 `value` 是 `boolean`。这就是泛型的复用威力。
- `users.update(2, { age: 26 })`：只更新 `age`，合法；若写 `{ id: 999 }` 会报错，因为 `Omit<T, "id">` 禁止改主键。

```bash
npx tsx ch16/repository.ts
```

### 扩展练习（加分项）

1. **多字段排序**：让 `sortBy` 接受一个 `Array<{ key: keyof T; order: "asc" | "desc" }>`，实现"先按 A 升序、再按 B 降序"。想想类型怎么写才安全。
2. **`whereMany`**：接受一个 `Partial<T>` 作为查询条件，返回"所有字段都匹配"的实体。提示：遍历 `Object.keys(query)`，用 `keyof` 收窄。
3. **给 `paginate` 加排序**：让分页选项里能带一个 `sortKey: keyof T`，返回排序后的那一页。
4. **`Option` 类型**：仿照 `Result`，定义 `type Option<T> = { some: true; value: T } | { some: false }`，把 `getById` 的返回从 `T | undefined` 改成 `Option<T>`，体会两种"可能没有"表达方式的取舍。

## 小结与自测

本章我们把 TypeScript 的核心——泛型——从头讲透：

- **为什么需要泛型**：为每种类型各写一份太啰嗦，用 `any` 又丢掉类型（把错误推迟到运行时）。泛型让一份代码同时"通用"且"保留类型"，让类型**流动**起来。
- **泛型函数**：`function identity<T>(x: T): T`。优先靠**类型推断**，需要时用 `<...>` **显式指定**。可有多个类型参数（`pair<A, B>`），配合数组、元组使用（`swap`、`mapArray`）。
- **泛型约束** `extends`：给 `T` 设"至少要满足什么形状"的门槛（`T extends HasLength`）。特别是 **`K extends keyof T` + 返回 `T[K]`**，实现类型安全的属性读写。
- **泛型默认值** `<T = string>`：调用方不指定时的兜底。
- **泛型接口 / 类型别名 / 类**：`Container<T>`、`Result<T, E>`、`Stack<T>`。`Result` 是把"可能失败"写进类型的实用模式。
- **内置工具类型**：`Partial` / `Required` / `Readonly` / `Record` / `Pick` / `Omit` / `ReturnType`——都是官方写好的泛型，先会用，原理下章讲。
- **常见坑**：别过度泛型化（只出现一次的类型参数多半多余）；约束别太松也别太狠（用 `T extends X` 而非直接标 `X` 以保留具体类型）；泛型 vs 联合看"类型间有没有关联"。
- **小项目**：`Repository<T extends Entity>` 把上述全部串起来，做出返回类型随实体变化、按字段筛选/排序/分页全程类型安全的仓储。

**自测题**（先自己想，再回代码验证）：

1. `function identity<T>(x: T): T` 里三个 `T` 是同一个类型吗？如果我写 `identity<string>(123)` 会怎样？
2. 为什么 `longest` 不加 `T extends HasLength` 约束时，函数体里访问 `a.length` 会报错？
3. `getProp<T, K extends keyof T>(obj: T, key: K): T[K]`——`getProp(user, "name")` 的返回类型是什么？为什么不是 `string | number | boolean`？
4. `Partial<User>` 和 `Required<Config>` 分别把属性变成什么？`Omit<User, "email">` 得到什么类型？
5. 什么时候该用泛型，什么时候该用联合类型 `A | B`？用一句话概括判断标准。
6. `Repository.where("email", "x")` 为什么会编译报错？这体现了泛型的什么价值？

答得上来，说明你已经掌握了泛型的核心。

---

泛型是"参数化类型"的能力，但它本身还比较"直"——传进什么类型，就用什么类型。真正让 TypeScript 类型系统变得像一门"小型编程语言"的，是**在类型层面做计算**：根据条件选择类型、遍历并改造一个类型的每个属性、从字符串模板里提取类型、甚至从复杂类型里"反推"出某个部分。

**第 17 章 · 高级类型**将带你进入这个更强大的世界：**条件类型（conditional types）** `T extends U ? X : Y`、**映射类型（mapped types）**（`Partial`、`Pick` 到底是怎么写出来的）、**模板字面量类型（template literal types）**，以及神奇的 **`infer`** 关键字。学完那章，你不仅会**用**工具类型，还能**亲手造**出自己的工具类型。
