# 第 18 章 · 类型收窄、类型守卫与断言

## 本章目标

到目前为止，你已经会用**联合类型（union type）** `A | B` 来表达"这个值可能是好几种类型之一"，也见过 `unknown` 这个"安全的任意类型"。但很快你会发现一个尴尬的处境：**一个 `number | string` 类型的变量，你既不能对它调用 `.toFixed()`（那是 `number` 的方法），也不能调用 `.toUpperCase()`（那是 `string` 的方法）。** 因为在那一刻，TypeScript 并不知道它究竟是哪一种，只好把两边都不允许，免得你出错。

要安全地使用它，你必须先做一件事——**类型收窄（narrowing）**：通过某种检查，把"宽"的联合类型缩小到"窄"的具体类型，让 TypeScript 确信"在这段代码里，它一定是 `number`"。这一章就是关于收窄的方方面面。

学完本章，你将能够：

- 说清楚**为什么需要收窄**：联合类型、`unknown` 必须先收窄才能安全使用。
- 熟练运用各种收窄手段：`typeof`、`instanceof`、`in`、真值收窄、相等收窄、**可辨识联合（discriminated union）** 的 `switch`。
- 理解 TypeScript 的**控制流分析（control flow analysis, CFA）** 是怎么"跟着代码走"推算类型的。
- 编写**用户自定义类型守卫（user-defined type guard）**，即返回值为 `x is T` 的**类型谓词（type predicate）**函数：`isString`、`isUser`……
- 编写**断言函数（assertion function）**：`asserts condition` 与 `asserts x is T`。
- 用好各种**类型断言（type assertion）**：`as`、`as const`、非空断言 `!`、双重断言 `as unknown as T`，并牢记它们的危险。
- 分清 `unknown` 与 `any`，掌握 `never` 的用途（穷尽检查、不可能分支）。
- 让可选链 `?.`、空值合并 `??` 与收窄默契配合。
- 认清**运行时校验与类型之间的鸿沟**：TS 类型在运行时不存在，外部数据（JSON / 网络）必须做运行时校验。你将**亲手写一个迷你版 zod**，让运行时校验与编译期类型联动。

本章代码用 `npx tsx` 运行，例如 `npx tsx ch18/01-why-narrowing.ts`。

📌 **一句话记住收窄**：收窄就是**"用运行时能做的检查（`typeof`、`in`、`instanceof`……），换取编译期更精确的类型"**。你在代码里做了一次判断，TypeScript 就顺着这次判断，把变量的类型"变窄"。

## 为什么需要收窄：联合类型的两难

先把开头那个"两难"的处境用代码坐实。

```typescript
// 文件：code/ch18/01-why-narrowing.ts

// 一个联合类型的参数：id 可能是数字，也可能是字符串。
function formatId(id: number | string): string {
  // 此时 id 的类型是 number | string，
  // 直接调用 .toFixed()（只有 number 有）会报错，
  // 直接调用 .toUpperCase()（只有 string 有）也会报错。
  // id.toFixed(2);    // ❌ 报错：string 上没有 toFixed
  // id.toUpperCase(); // ❌ 报错：number 上没有 toUpperCase

  // 必须先「收窄」到某一个具体类型，才能安全使用它专属的方法。
  if (typeof id === "number") {
    // 这个分支里，TS 知道 id 一定是 number
    return `#${id.toFixed(0)}`;
  } else {
    // 排除了 number，剩下只能是 string
    return `#${id.toUpperCase()}`;
  }
}

console.log(formatId(42)); // #42
console.log(formatId("abc")); // #ABC
```

逐段拆解：

- 参数 `id` 类型是 `number | string`。在函数体最开始，TS 对 `id` 只允许你做"两种类型都支持"的操作。`.toFixed` 只有 `number` 有，`.toUpperCase` 只有 `string` 有，所以两个都被拦下——这不是 TS 在刁难你，而是在**保护**你：万一实参传的是字符串，你却调 `.toFixed()`，运行时就会崩。
- `if (typeof id === "number")`：这是一次**运行时检查**。TS 看懂了这个检查的含义，于是在 `if` 的**内部**，把 `id` 的类型收窄为 `number`。此时 `id.toFixed(0)` 完全合法。
- `else` 分支：既然进了 `else`，说明 `typeof id === "number"` 为假，那么在 `number | string` 里排除掉 `number`，**剩下只能是 `string`**。TS 自动把 `id` 收窄为 `string`，`id.toUpperCase()` 合法。

📌 **收窄的本质**：`if` 判断把一个类型"劈"成两半，每一半里变量的类型都比外面更精确。TS 会**分别**记住每个分支里的类型。

运行：

```bash
npx tsx ch18/01-why-narrowing.ts
```

## 收窄手段（一）：typeof、instanceof、in

TS 认识好几种"运行时检查"的写法，看到它们就会自动收窄。先看最常用的三种。

```typescript
// 文件：code/ch18/02-typeof-instanceof-in.ts

// ---------- 1) typeof 收窄：适合原始类型 ----------
function double(x: number | string): number | string {
  if (typeof x === "number") {
    return x * 2; // x: number
  }
  return x.repeat(2); // x: string
}

// typeof 能识别的字符串："string" | "number" | "bigint" | "boolean"
// | "symbol" | "undefined" | "object" | "function"
function describe(x: unknown): string {
  if (typeof x === "string") return `字符串:${x}`;
  if (typeof x === "number") return `数字:${x}`;
  if (typeof x === "boolean") return `布尔:${x}`;
  if (typeof x === "function") return `函数`;
  return `其他:${String(x)}`;
}
```

逐段拆解：

- `typeof x === "number"`：`typeof` 是 JavaScript 运行时就有的运算符，返回一个字符串。TS 专门认得这个模式，能据此收窄。`double` 里进入 `if` 后 `x: number`，剩下的自然是 `string`。
- `describe` 展示 `typeof` 能返回的**全部 8 种字符串**。⚠️ 注意 `typeof null === "object"`（JavaScript 的历史遗留 bug），所以想判断"是不是对象且非 null"，光靠 `typeof x === "object"` 是不够的，还得额外 `x !== null`——这一点在后面校验库里会反复用到。

```typescript
// 文件：code/ch18/02-typeof-instanceof-in.ts（续）

// ---------- 2) instanceof 收窄：适合类 / 构造函数 ----------
class Dog {
  bark(): string {
    return "汪";
  }
}
class Cat {
  meow(): string {
    return "喵";
  }
}

function speak(animal: Dog | Cat): string {
  if (animal instanceof Dog) {
    return animal.bark(); // animal: Dog
  }
  return animal.meow(); // animal: Cat
}

// instanceof 对内置类同样有效
function toISO(x: Date | string): string {
  if (x instanceof Date) {
    return x.toISOString(); // x: Date
  }
  return x; // x: string
}
```

逐段拆解：

- `animal instanceof Dog`：`instanceof` 检查"某个对象是不是由某个构造函数（类）创建的"。适合区分**类的实例**。进入 `if` 后 `animal` 收窄为 `Dog`。
- `x instanceof Date`：对内置类（`Date`、`Array`、`Map`、`RegExp` 等）同样有效。⚠️ `typeof` 只能区分原始类型，遇到对象一律返回 `"object"`，区分不了具体是哪个类——那正是 `instanceof` 的用武之地。

```typescript
// 文件：code/ch18/02-typeof-instanceof-in.ts（续）

// ---------- 3) in 收窄：靠「属性是否存在」区分对象 ----------
type Fish = { swim: () => string };
type Bird = { fly: () => string };

function move(animal: Fish | Bird): string {
  if ("swim" in animal) {
    return animal.swim(); // animal: Fish
  }
  return animal.fly(); // animal: Bird
}
```

逐段拆解：

- `"swim" in animal`：`in` 运算符检查"对象上有没有某个属性"。当两个类型的**区别在于"有没有某个字段"**时，`in` 是最自然的收窄方式。`Fish` 有 `swim`，`Bird` 没有，所以 `"swim" in animal` 为真时收窄为 `Fish`。
- 这类"用普通对象字面量类型（而非 `class`）"的场景，用不了 `instanceof`（`Fish` 不是一个构造函数），`in` 就成了首选。

运行三种收窄：

```bash
npx tsx ch18/02-typeof-instanceof-in.ts
```

💡 **怎么选**：区分原始类型用 `typeof`；区分类的实例用 `instanceof`；区分"形状不同的普通对象"用 `in`（或后面讲的可辨识联合，更佳）。

## 收窄手段（二）：真值收窄、相等收窄，与控制流分析

除了显式的 `typeof`/`instanceof`/`in`，TS 还认得更"隐式"的收窄。

```typescript
// 文件：code/ch18/03-truthy-equality-cfa.ts

// ---------- 1) 真值收窄（truthiness narrowing） ----------
// 一个值可能是 string，也可能是 null / undefined。
function greet(name: string | null | undefined): string {
  // if (name) 会同时排除 null、undefined 以及空字符串 ""
  if (name) {
    return `你好, ${name.toUpperCase()}`; // name: string
  }
  return "你好, 陌生人";
}

// ⚠️ 真值收窄的陷阱：0、""、NaN 也是「假值」，会被一起排除
function lengthOf(list: string[] | null): number {
  if (list) {
    return list.length; // list: string[]
  }
  return 0;
}
```

逐段拆解：

- `if (name)`：把 `name` 放进条件里做"真假判断"，就是**真值收窄**。JavaScript 的"假值（falsy）"有：`false`、`0`、`""`、`null`、`undefined`、`NaN`。进入 `if` 说明 `name` 是真值，于是 `null` 和 `undefined` 都被排除，剩下 `string`。
- ⚠️ **陷阱**：真值收窄会把**空字符串 `""` 一起当假值排除**！如果 `name` 传入 `""`，会走进 `else`。若你只想排除 `null`/`undefined` 而保留 `""`，就不能用真值收窄，得用下面的 `!= null`。
- `lengthOf` 里 `list` 是 `string[] | null`，`if (list)` 排除 `null`（对象永远是真值），得到 `string[]`。这里没有 `""` 的问题，用真值收窄很安全。

```typescript
// 文件：code/ch18/03-truthy-equality-cfa.ts（续）

// ---------- 2) 相等收窄（equality narrowing） ----------
function compare(a: string | number, b: string | boolean): void {
  if (a === b) {
    // 只有当两者相等时才可能进入——此时公共类型只能是 string
    console.log(a.toUpperCase(), b.toUpperCase()); // a、b 都是 string
  }
}

// 用 != null 一次排除 null 和 undefined
function trim(s: string | null | undefined): string {
  if (s != null) {
    return s.trim(); // s: string
  }
  return "";
}
```

逐段拆解：

- `if (a === b)`：`a` 是 `string | number`，`b` 是 `string | boolean`。两者要相等，只可能在它们**类型的交集**上相等——`string | number` 和 `string | boolean` 的交集是 `string`。所以 `if` 内部 `a` 和 `b` 都被收窄为 `string`。这就是**相等收窄**。
- `if (s != null)`：这里用的是**宽松不等 `!=`**（不是 `!==`）。JavaScript 里 `null != undefined` 为假——也就是说 `!= null` 会**同时**把 `null` 和 `undefined` 都算作"等于 null"。所以这一个判断就排除了 `null` **和** `undefined`，只剩 `string`。这是排除空值的经典惯用法。

```typescript
// 文件：code/ch18/03-truthy-equality-cfa.ts（续）

// ---------- 3) 控制流分析（control flow analysis, CFA） ----------
function cfa(input: string | number): string {
  let out: string;
  if (typeof input === "string") {
    out = input; // 这里 input: string
  } else {
    out = input.toFixed(2); // 这里 input: number
  }
  // 两个分支都给 out 赋了 string，合流之后 out: string
  return out;
}

// 提前 return 也会影响后续类型：这叫「收窄的持续生效」
function afterReturn(x: string | null): string {
  if (x === null) {
    return "空";
  }
  // 上面 null 分支已经 return，走到这里 x 只可能是 string
  return x.toLowerCase(); // x: string
}
```

逐段拆解：

- 这两个例子展示 TS 的**控制流分析**：编译器像"跟着程序执行一样"，逐行推算每个位置上每个变量的类型。它不是"整个函数里 `input` 是一个固定类型"，而是"在这一行、这个分支里，`input` 是什么类型"。
- `afterReturn` 里，`if (x === null) return` 提前返回了。TS 明白：能走到 `if` **之后**的代码，一定意味着 `x !== null`。于是后面 `x` 被收窄为 `string`——**尽管这里没有把后续代码包在 `else` 里**。这种"提前 return 造成后续收窄"的模式叫**早返回（early return）**，是收窄配合 CFA 最常用、最清爽的写法。

运行：

```bash
npx tsx ch18/03-truthy-equality-cfa.ts
```

📌 **CFA 心智模型**：不要问"这个变量是什么类型"，要问"在**这一行**，这个变量是什么类型"。TS 的答案会随着你写的每一个 `if`、`return`、赋值而变化。

## 收窄手段（三）：可辨识联合与 never 穷尽检查

这是收窄里最优雅、也最值得掌握的一招。你在前面章节见过联合类型，这里我们深入它的"可辨识"变体。

```typescript
// 文件：code/ch18/04-discriminated-union.ts

// 可辨识联合（discriminated union）：每个成员都带一个「公共字面量标签」字段。
// 这里的公共字段是 kind，取值互不相同。
type Circle = { kind: "circle"; radius: number };
type Square = { kind: "square"; side: number };
type Rectangle = { kind: "rectangle"; width: number; height: number };

type Shape = Circle | Square | Rectangle;

// never 的经典用途：穷尽检查（exhaustiveness check）
function assertNever(x: never): never {
  throw new Error(`未处理的分支: ${JSON.stringify(x)}`);
}

function area(shape: Shape): number {
  switch (shape.kind) {
    case "circle":
      // 这个分支里 shape 被收窄成 Circle
      return Math.PI * shape.radius ** 2;
    case "square":
      return shape.side ** 2;
    case "rectangle":
      return shape.width * shape.height;
    default:
      // 如果上面覆盖了所有成员，走到这里 shape 的类型就是 never。
      return assertNever(shape);
  }
}
```

逐段拆解：

- **可辨识联合**的关键：每个成员都有一个**公共字段**（这里叫 `kind`），且它的类型是**互不相同的字面量类型**（`"circle"`、`"square"`、`"rectangle"`）。这个字段叫**判别式（discriminant）**。
- `switch (shape.kind)`：TS 看到你在对判别式做 `switch`，就在每个 `case` 里把 `shape` 收窄为对应成员。`case "circle"` 里 `shape` 是 `Circle`，能安全访问 `.radius`；而 `.side`、`.width` 在这个分支里不可用。这比一堆 `in` 判断清爽得多。
- `default` 分支里的 `assertNever(shape)` 是精髓所在。**如果前面三个 `case` 已经覆盖了 `Shape` 的所有成员**，那么走到 `default` 时，`shape` 的类型经过层层排除会变成 `never`（一个成员都不剩）。`assertNever` 的参数类型正是 `never`，所以这里能编译通过。
- 现在设想：将来有人给 `Shape` 加了个 `Triangle` 成员，却忘了在 `switch` 里加 `case`。此时 `default` 里的 `shape` 就还剩 `Triangle`，**不再是 `never`**，`assertNever(shape)` 会**编译报错**——编译器逼着你补上遗漏的分支。这就是 **`never` 穷尽检查（exhaustiveness check）**：用类型系统保证"所有情况都处理到了"。

运行：

```bash
npx tsx ch18/04-discriminated-union.ts
```

💡 **强烈推荐**：任何"一个值有有限几种形态"的建模，都优先用可辨识联合 + `switch` + `assertNever`。它既让每个分支自动收窄，又在新增成员时给你编译期提醒，是 TS 里少数"越用越安心"的模式。

## 用户自定义类型守卫：`x is T` 类型谓词

前面的收窄都靠内置运算符。但有时判断逻辑比较复杂（比如"这是不是一个合法的 User 对象"），你想把它抽成一个函数复用。问题来了：**普通的返回 `boolean` 的函数，TS 不会用它来收窄。**

```typescript
// 文件：code/ch18/05-type-predicate.ts

// 普通的布尔函数：TS 只知道它返回 boolean，不会用它来收窄类型。
function isStringPlain(x: unknown): boolean {
  return typeof x === "string";
}

// 类型守卫：返回值类型写成 `x is string`。
// 当它返回 true 时，TS 就把实参收窄为 string。
function isString(x: unknown): x is string {
  return typeof x === "string";
}

function demo(value: unknown): void {
  if (isStringPlain(value)) {
    // ⚠️ value 仍然是 unknown，下面这行会报错（已注释）
    // value.toUpperCase();
    console.log("isStringPlain 判定为字符串，但类型没有收窄");
  }

  if (isString(value)) {
    // ✅ 这里 value 被收窄为 string
    console.log("收窄为 string:", value.toUpperCase());
  }
}
```

逐段拆解：

- `isStringPlain` 返回 `boolean`。你在 `if (isStringPlain(value))` 里明明知道 `value` 是字符串了，但 TS **不知道**——它只看到"一个返回 `boolean` 的函数"，`value` 在 `if` 里仍是 `unknown`。
- `isString` 的返回类型写成 **`x is string`**——这就是**类型谓词（type predicate）**。它的读法是："如果这个函数返回 `true`，那么参数 `x` 就是 `string`。" TS 于是把 `isString` 当作一个**收窄工具**：`if (isString(value))` 内部，`value` 被收窄为 `string`。
- ⚠️ **责任在你**：写 `x is string` 是你对编译器的**承诺**。TS **不会验证**你的函数体是否真的正确判断了！如果你写 `function isString(x: unknown): x is string { return true; }`，TS 会照单全收，然后运行时爆炸。类型守卫的函数体必须**如实地**检查它承诺的类型。

再看给自定义对象写守卫，以及一个超实用的应用：用类型守卫过滤数组。

```typescript
// 文件：code/ch18/05-type-predicate.ts（续）

interface User {
  id: number;
  name: string;
}

function isUser(x: unknown): x is User {
  // 注意：函数体里要真的把「所有必要条件」都检查到，
  // 否则守卫在「撒谎」——运行时和类型会脱节。
  return (
    typeof x === "object" &&
    x !== null &&
    typeof (x as Record<string, unknown>).id === "number" &&
    typeof (x as Record<string, unknown>).name === "string"
  );
}

// filter + 类型守卫：把 (T | null)[] 过滤成 T[]
function isNotNull<T>(x: T | null): x is T {
  return x !== null;
}

const raw: (User | null)[] = [
  { id: 1, name: "Alice" },
  null,
  { id: 2, name: "Bob" },
];

// 用了 isNotNull 后，users 精确地是 User[]。
const users: User[] = raw.filter(isNotNull);
```

逐段拆解：

- `isUser` 逐条检查：是对象、非 `null`、`id` 是数字、`name` 是字符串。四个条件全满足才返回 `true`。注意这里用了 `x as Record<string, unknown>`——因为在 `typeof x === "object" && x !== null` 之后，`x` 只是 `object`，TS 还不允许直接访问任意属性，需要用断言"打开"它来读字段（这是收窄链条中常见的小妥协）。
- `isNotNull` 是个**泛型类型守卫**：`x is T` 里的 `T` 会随调用推断。它的妙用在最后一行。
- `raw.filter(isNotNull)`：`Array.filter` 有一个特殊重载——如果传给它的回调是**类型守卫**，那么返回数组的类型会被收窄。这里 `raw` 是 `(User | null)[]`，用 `isNotNull` 过滤后，`users` 精确地是 **`User[]`**（`null` 被从类型里剔除了）。若换成普通 `raw.filter(x => x !== null)`，结果类型仍是 `(User | null)[]`，你还得再断言一次。**这是类型守卫在日常代码里最高频的应用之一。**

运行：

```bash
npx tsx ch18/05-type-predicate.ts
```

## 断言函数：`asserts condition` 与 `asserts x is T`

类型守卫是"返回布尔、交给你 `if` 分流"。但有时你想要的语义是"**不满足就直接抛错**，满足了就默认成立、往下走"。这就是**断言函数（assertion function）**。

```typescript
// 文件：code/ch18/06-assertion-functions.ts

// 形式一：asserts condition —— 断言「某个布尔条件成立」。
function assert(condition: unknown, message: string): asserts condition {
  if (!condition) {
    throw new Error(message);
  }
}

function useAssert(x: string | null): string {
  assert(x !== null, "x 不应为 null");
  // 上面这行之后，TS 知道 x !== null 恒成立 → x: string
  return x.toUpperCase();
}
```

逐段拆解：

- `assert` 的返回类型写成 **`asserts condition`**（`condition` 是它的参数名）。含义是："**如果这个函数正常返回了（没抛错），那么 `condition` 一定为真。**" 它自己返回 `void`，不返回布尔。
- `assert(x !== null, ...)`：调用之后，TS 知道"既然没抛错，说明 `x !== null` 成立"，于是把 `x` 从 `string | null` 收窄为 `string`。**不需要 `if`**——断言函数是"直接改变后续代码里的类型"。
- 这与 Node.js 内置的 `assert` 模块、许多测试框架的断言用法完全一致，只是这里额外带上了类型收窄的能力。

```typescript
// 文件：code/ch18/06-assertion-functions.ts（续）

// 形式二：asserts x is T —— 断言「参数就是某个类型」。
function assertIsString(x: unknown): asserts x is string {
  if (typeof x !== "string") {
    throw new Error(`期望 string，实际是 ${typeof x}`);
  }
}

function useAssertIsString(input: unknown): number {
  assertIsString(input);
  // 断言之后 input: string
  return input.length;
}
```

逐段拆解：

- `asserts x is string`：这是断言函数的第二种形式。含义是："**如果正常返回，那么 `x` 就是 `string`。**" 相比 `asserts condition`（断言一个布尔表达式），这个直接断言"参数的类型"。
- `useAssertIsString` 里，`assertIsString(input)` 之后 `input` 从 `unknown` 收窄为 `string`，`input.length` 合法。

📌 **类型守卫 vs 断言函数，怎么选**：
- **类型守卫 `x is T`**：返回布尔，你要自己写 `if` 分流两个情况——"是 T 就这样，不是 T 就那样"。适合**两种情况都要处理**。
- **断言函数 `asserts x is T`**：不返回布尔，不满足**直接抛错**，之后代码默认成立。适合**"这里必须是 T，否则就是 bug / 脏数据，直接停下"**的场景。

运行：

```bash
npx tsx ch18/06-assertion-functions.ts
```

## 类型断言：as / as const / 非空断言 / 双重断言

前面所有收窄都是"TS 根据你的检查，**自己推断**出更窄的类型"。而**类型断言（type assertion）**是另一回事：你**强行告诉**编译器"它就是这个类型，别管你怎么想的"。断言不做任何运行时检查，因此**危险**。

```typescript
// 文件：code/ch18/07-assertions-as.ts

// 1) as：告诉编译器「相信我，它就是这个类型」。不做任何运行时检查！
const el = { tagName: "INPUT" } as { tagName: string; value: string };
// 编译期 el.value 合法，但运行时 el 里根本没有 value → 得到 undefined
console.log("as 断言（危险）:", el.value); // undefined

// 2) 双重断言 as unknown as T：强行绕过限制（几乎总是坏味道）。
const forced = "abc" as unknown as number;
console.log("双重断言（更危险）:", forced, typeof forced); // 运行时还是 string！
```

逐段拆解：

- `as T`：让 TS **无条件相信**这个值是 `T`。上面 `el` 明明没有 `value` 字段，但断言后 TS 允许你访问 `el.value`——运行时得到 `undefined`。**`as` 只改变编译期的看法，不改变运行时的现实。**
- ⚠️ `as` 有个安全底线：它只允许在"**类型有交集**"时转换。`"abc" as number` 会直接报错（`string` 与 `number` 毫无重叠）。
- **双重断言 `as unknown as T`**：先断言成 `unknown`（任何类型都能变成 `unknown`），再从 `unknown` 断言成目标类型（`unknown` 能变成任何类型），**绕过**了上面的安全底线。`"abc" as unknown as number` 得到一个类型上是 `number`、运行时却是字符串的"骗局值"。**这几乎总是代码坏味道**，只在极少数与第三方类型定义打架时才勉强用，且必须写注释说明为什么。

```typescript
// 文件：code/ch18/07-assertions-as.ts（续）

// 3) as const：把字面量「锁死」为最窄的只读类型。
const config = {
  method: "GET",
  retries: 3,
} as const;
// config.method 的类型是 "GET"（不是 string），且整个对象只读

const roles = ["admin", "guest"] as const;
// roles 的类型是 readonly ["admin", "guest"]，可用来派生联合类型
type Role = (typeof roles)[number]; // "admin" | "guest"
```

逐段拆解：

- `as const`：和 `as T` 是两回事，它是个**安全、常用**的断言。它把值"锁死"为**最窄的只读类型**：普通 `const config = { method: "GET" }` 里 `method` 会被推断为 `string`，加上 `as const` 后 `method` 的类型精确地是字面量 `"GET"`，且整个对象变成只读（`config.retries = 5` 会报错）。
- `roles ... as const`：数组加 `as const` 变成只读元组 `readonly ["admin", "guest"]`。配合 `(typeof roles)[number]`（取元组所有元素类型的联合）就能**从一个值派生出联合类型** `Role = "admin" | "guest"`。这是"单一数据源（single source of truth）"的常用技巧——值和类型不会各写一份、互相脱节。

```typescript
// 文件：code/ch18/07-assertions-as.ts（续）

// 4) 非空断言 !：断言「这个值一定不是 null / undefined」。
function findUser(id: number): { id: number } | undefined {
  return id > 0 ? { id } : undefined;
}
const u = findUser(1)!; // 断言结果非空 → u: { id: number }

// ⚠️ 若断言错了，运行时就会「炸得莫名其妙」：
const maybe = findUser(-1)!; // 类型上是 { id: number }，运行时其实是 undefined
try {
  console.log(maybe.id); // TypeError: Cannot read properties of undefined
} catch (e) {
  console.log("非空断言的雷:", (e as Error).message);
}
```

逐段拆解：

- **非空断言 `!`**：写在表达式后面，断言"这个值一定不是 `null`/`undefined`"，把类型里的 `null | undefined` 去掉。`findUser(1)!` 让 `u` 从 `{ id: number } | undefined` 变成 `{ id: number }`。
- ⚠️ **它的危险和 `as` 一样**：不做运行时检查。`findUser(-1)` 明明返回 `undefined`，你却用 `!` 断言它非空，TS 就不再提醒你判空，于是 `maybe.id` 在运行时抛出经典的 `Cannot read properties of undefined`——而且报错位置离你写 `!` 的地方可能很远，很难查。**能用收窄（`if`、`??`）就别用 `!`。**

## unknown 与 never：安全的顶层与不可能的底层

这两个类型是收窄体系的"两极"。搞清楚它们，收窄的图景才完整。

```typescript
// 文件：code/ch18/08-unknown-never.ts

// any：彻底关闭检查。下面全部「编译通过」，但运行时会炸。
function withAny(x: any): void {
  x.foo.bar.baz; // 编译不拦
  x(); // 编译不拦
}

// unknown：安全的顶层类型。可以「接收任何值」，但用之前必须先收窄。
function withUnknown(x: unknown): void {
  // x.foo;            // ❌ 报错：对象类型未知
  // x();              // ❌ 报错：不确定能不能调用
  if (typeof x === "object" && x !== null && "foo" in x) {
    console.log("收窄后可安全访问:", (x as { foo: unknown }).foo);
  }
}
```

逐段拆解：

- `any` 和 `unknown` 都能"接收任何值"，区别在**之后**：`any` **彻底关闭检查**——`x.foo.bar.baz`、`x()` 全部编译通过，把错误推迟到运行时爆炸。`any` 是"我放弃类型检查"。
- `unknown` 是"**安全的 `any`**"：它同样能接收任何值，但**用之前必须先收窄**。直接 `x.foo` 会报错，你得先 `typeof`/`in` 检查证明它的形状，TS 才让你用。`unknown` 把"必须先验证"这件事**强制**了下来。

```typescript
// 文件：code/ch18/08-unknown-never.ts（续）

// 处理外部数据（如 JSON.parse）时，unknown 是最佳落点。
function parseConfig(json: string): { port: number } {
  const data: unknown = JSON.parse(json); // 不要用 any！
  if (
    typeof data === "object" &&
    data !== null &&
    "port" in data &&
    typeof (data as Record<string, unknown>).port === "number"
  ) {
    return { port: (data as { port: number }).port };
  }
  throw new Error("配置格式不合法");
}

// never：「不可能存在的值」的类型。
// 用途 1：永不返回的函数（总是抛错或死循环）。
function fail(msg: string): never {
  throw new Error(msg);
}
```

逐段拆解：

- `JSON.parse` 的返回类型其实是 `any`——这是个历史遗留的"类型黑洞"。⚠️ **最佳实践**：拿到 `JSON.parse` 的结果，立刻用 `const data: unknown = ...` 把它"降级"为 `unknown`，逼自己在使用前校验。这正是本章小项目要系统化解决的问题。
- `never` 是**"不可能有值"的类型**，它是所有类型的"底"（`unknown` 是"顶"）。三大用途：(1) 永不返回的函数（`fail` 总是抛错，返回 `never`）；(2) **穷尽检查**（前面 `assertNever` 的参数）；(3) 表示"空联合"——某个联合被收窄到一个成员都不剩时，就是 `never`（例如 `string & number` 没有任何公共值，即为 `never`）。

运行：

```bash
npx tsx ch18/08-unknown-never.ts
```

📌 **口诀**：外部数据先落到 `unknown`，收窄后再用；`any` 是投降，`unknown` 是"先证明再使用"；`never` 是"这里不该有值"。

## 可选链 ?. 与空值合并 ?? 如何配合收窄

处理"可能缺失"的数据时，`?.` 和 `??` 是收窄的好搭档。

```typescript
// 文件：code/ch18/09-optional-chaining-nullish.ts

interface Address {
  city: string;
  zip?: string; // 可选
}
interface Profile {
  name: string;
  address?: Address; // 可选
}

// 可选链 ?.：链条中任意一环是 null / undefined，整体短路为 undefined。
function getCity(p: Profile | null | undefined): string {
  const city = p?.address?.city;
  // city 的类型是 string | undefined
  // 用 ?? 提供「仅在 null/undefined 时」生效的默认值
  return city ?? "未知城市";
}

// ⚠️ ?? 与 || 的区别
function withDefaults(count: number | undefined): void {
  console.log("|| :", count || 10); // count=0 时 → 10（可能不是你想要的）
  console.log("?? :", count ?? 10); // count=0 时 → 0（更精确）
}
```

逐段拆解：

- `p?.address?.city`：**可选链**。链条中任意一环是 `null`/`undefined`，整个表达式**短路**返回 `undefined`，不会报错。所以 `city` 的类型是 `string | undefined`。可选链本身就是一种"收窄的自动化"——你不必写一串 `if (p && p.address)`。
- `city ?? "未知城市"`：**空值合并**。仅当左边是 `null`/`undefined` 时才用右边的默认值。
- ⚠️ **`??` vs `||` 的关键区别**：`||` 对**所有假值**（`0`、`""`、`false`、`NaN`）都会走默认值；`??` **只对 `null`/`undefined`** 走默认值。当 `count` 是合法的 `0` 时，`count || 10` 错误地变成 `10`，而 `count ?? 10` 正确地保留 `0`。处理数字、布尔时，几乎总该用 `??`。

```typescript
// 文件：code/ch18/09-optional-chaining-nullish.ts（续）

// 可选链配合收窄：收窄之后 ?. 就不再需要
function describe(p: Profile | null): string {
  if (p == null) return "无资料";
  // 这里 p: Profile，但 address 仍是可选
  if (p.address) {
    // 收窄后 p.address: Address，直接访问不用 ?.
    return `${p.name} 住在 ${p.address.city}`;
  }
  return `${p.name}（地址未填）`;
}
```

逐段拆解：

- 这个例子展示"**先收窄，就不必反复 `?.`**"。`if (p == null) return` 之后 `p` 已是 `Profile`；`if (p.address)` 之后 `p.address` 已是 `Address`。此时直接写 `p.address.city` 即可，不需要 `p.address?.city`。可选链适合"一次性访问"，而当你要多次访问同一条链时，先用 `if` 收窄一次更清爽、也更高效。

运行：

```bash
npx tsx ch18/09-optional-chaining-nullish.ts
```

## 常见坑与最佳实践

把本章容易踩的坑集中列一遍：

- ⚠️ **类型守卫会"撒谎"**：`x is T` 是你对编译器的承诺，TS **不验证**函数体。守卫内部必须如实检查它承诺的所有条件，否则运行时和类型就脱节了——这种 bug 极难查，因为编译期一片绿。
- ⚠️ **真值收窄误伤 `0` / `""` / `false`**：想只排除 `null`/`undefined` 时，用 `x != null`（宽松不等）或 `x !== undefined && x !== null`，别用 `if (x)`。
- ⚠️ **`as` 和 `!` 不做运行时检查**：它们只改变编译期的看法。断言错了，运行时照样炸，且报错位置往往离断言处很远。**能用收窄就别用断言**。把 `as`、`!` 当成"我为这行的安全负全责"的签字。
- ⚠️ **双重断言 `as unknown as T` 是最后手段**：它绕过了 `as` 的安全底线，几乎总是坏味道。用它必写注释解释原因。
- 💡 **优先可辨识联合 + `assertNever`**：给"有限几种形态"的数据建模时，加一个字面量判别字段，用 `switch` 收窄，`default` 里放 `assertNever`。新增成员忘了处理时，编译器会当场提醒你。
- 💡 **外部数据一律先落 `unknown`**：`JSON.parse`、`fetch().json()`、`localStorage.getItem` 之后，别信它们标注的类型（往往是 `any`），先当 `unknown`，校验后再用。
- 💡 **`filter` + 类型守卫**：`arr.filter(isNotNull)` 能把 `(T | null)[]` 收窄成 `T[]`，比事后断言优雅得多。
- 📌 **收窄会"失效"**：如果在收窄后，你在两次检查之间调用了一个函数、或给变量重新赋值，TS 可能"忘记"之前的收窄（因为它无法确定那期间值没变）。遇到这种情况，把值存进一个 `const` 局部变量再用，收窄会更稳。

## 本章小项目：运行时校验 + 类型联动

现在把本章所有知识拧成一股绳，解决一个**真实且棘手**的问题：

> TypeScript 的类型在**运行时不存在**——编译成 JS 后，所有类型注解都被抹掉了。所以当你从**外部**（HTTP 接口、`JSON.parse`、用户输入）拿到数据时，你标注的 `User` 类型**保证不了任何事**。数据可能字段缺失、类型不对、多出脏字段。你必须在运行时**真的去检查**。

我们要手写一个**迷你校验库**，让"运行时校验"和"编译期类型"**联动**：定义一组校验器（每个都是类型守卫思路的产物），组合起来校验一个 `unknown` 输入；校验通过后，TS 能把类型**自动收窄**成强类型的 `User`——而且 `User` 的静态类型是从校验规则**反推**出来的，不用手写两遍。这正是 `zod` 这类库的核心思想。

### 第一步：校验器的类型契约

```typescript
// 文件：code/ch18/validate.ts（节选）

export type Ok<T> = { ok: true; value: T };
export type Err = { ok: false; errors: string[] };
export type Result<T> = Ok<T> | Err;

// Validator<T>：把 unknown 校验为 T 的函数。
export interface Validator<T> {
  (input: unknown, path: string): Result<T>;
  // 幽灵字段：只用来在类型层面记住 T，运行时不存在。
  readonly _type?: T;
}

// 从 Validator 中「反推」出它校验的类型：Infer<typeof userSchema> === User
export type Infer<V> = V extends Validator<infer T> ? T : never;
```

逐段拆解：

- `Result<T>` 是个**可辨识联合**（判别字段 `ok`）：成功带 `value: T`，失败带 `errors: string[]`。整个库都用它传递结果——你马上会看到 `if (r.ok)` 的收窄贯穿始终。
- `Validator<T>` 是一个**可调用的接口（callable interface）**：本体是函数 `(input, path) => Result<T>`。`path` 参数用来记录"当前校验到数据的哪个位置"（如 `user.tags[1]`），失败时好给出清晰路径。
- `readonly _type?: T` 是一个**幽灵字段（phantom field）**：运行时永远是 `undefined`，存在的唯一目的是让 `T` 出现在类型结构里，方便下面的 `Infer` 把它"抠"出来。
- `Infer<V>` 用**条件类型 + `infer`**（第 17 章学过）从一个 `Validator<T>` 反推出 `T`。有了它，我们能做到"**写一遍校验规则，静态类型自动派生**"，不必手写 `interface User` 再和校验规则对齐——两者永不脱节。

### 第二步：基础校验器（原始类型，本质是类型守卫）

```typescript
// 文件：code/ch18/validate.ts（节选）

const ok = <T>(value: T): Ok<T> => ({ ok: true, value });
const err = (path: string, expected: string, got: unknown): Err => ({
  ok: false,
  errors: [`${path || "<root>"}: 期望 ${expected}，实际得到 ${typeInfo(got)}`],
});

function typeInfo(x: unknown): string {
  if (x === null) return "null";
  if (Array.isArray(x)) return "array";
  return typeof x;
}

export const isString: Validator<string> = (input, path) =>
  typeof input === "string" ? ok(input) : err(path, "string", input);

export const isNumber: Validator<number> = (input, path) =>
  typeof input === "number" && !Number.isNaN(input)
    ? ok(input)
    : err(path, "number", input);

export const isBoolean: Validator<boolean> = (input, path) =>
  typeof input === "boolean" ? ok(input) : err(path, "boolean", input);
```

逐段拆解：

- `ok` / `err` 是两个小工厂，构造成功 / 失败结果。`err` 把"路径 + 期望类型 + 实际类型"拼成一条清晰的错误信息。`typeInfo` 修正了 `typeof null === "object"` 和"数组也是 object"这两个 JS 陷阱，让错误信息更准确。
- `isString` 标注为 `Validator<string>`：它接收 `unknown`，内部用 `typeof` 收窄——**这就是类型守卫的思路**，只不过我们不返回布尔，而是返回带值的 `Result<string>`。校验通过时 `ok(input)` 里的 `input` 已被收窄为 `string`，所以 `ok<string>` 成立。
- `isNumber` 额外排除了 `NaN`（`NaN` 的 `typeof` 也是 `"number"`，但通常不是你想要的合法数字）。这体现了"运行时校验能比类型系统更严"——类型系统区分不了 `NaN` 和普通数字，运行时校验可以。

### 第三步：组合校验器（数组、可选、字面量联合）

```typescript
// 文件：code/ch18/validate.ts（节选）

// isArrayOf(item)：校验「元素都满足 item 的数组」。
export function isArrayOf<T>(item: Validator<T>): Validator<T[]> {
  return (input, path) => {
    if (!Array.isArray(input)) return err(path, "array", input);
    const out: T[] = [];
    const errors: string[] = [];
    input.forEach((el, i) => {
      const r = item(el, `${path}[${i}]`);
      if (r.ok) out.push(r.value);
      else errors.push(...r.errors);
    });
    return errors.length ? { ok: false, errors } : ok(out);
  };
}

// optional(v)：允许该值缺失（undefined），否则按 v 校验。
export function optional<T>(v: Validator<T>): Validator<T | undefined> {
  return (input, path) =>
    input === undefined ? ok(undefined) : v(input, path);
}

// literalUnion(...values)：校验值是给定字面量之一。
export function literalUnion<const T extends readonly (string | number | boolean)[]>(
  ...values: T
): Validator<T[number]> {
  return (input, path) =>
    values.includes(input as T[number])
      ? ok(input as T[number])
      : err(path, values.map((v) => JSON.stringify(v)).join(" | "), input);
}
```

逐段拆解：

- `isArrayOf(item)` 是一个**高阶校验器**：接收一个校验单个元素的 `Validator<T>`，返回一个校验 `T[]` 的 `Validator<T[]>`。它遍历数组，对每个元素调用 `item`，并把**索引拼进路径**（`${path}[${i}]`）——所以错误会精确到 `tags[1]`。注意它**收集所有错误**再一起返回，而不是遇到第一个就停，用户体验更好。
- `optional(v)`：把一个 `Validator<T>` 包装成 `Validator<T | undefined>`。输入是 `undefined` 就直接放行，否则交给内层校验。返回类型里的 `| undefined` 正好对应对象里的"可选字段"。
- `literalUnion(...values)`：校验"值是给定字面量之一"。`<const T ...>` 里的 **`const` 类型参数**（TS 5.0+）让传入的 `"admin", "guest"` 被推断为字面量元组而非 `string[]`，于是 `T[number]` 得到精确的联合 `"admin" | "guest"`——这就是校验规则和类型联动的又一处体现。

### 第四步：对象校验器 `isObjectOf(schema)`——联动的核心

```typescript
// 文件：code/ch18/validate.ts（节选）

// ShapeOf：把 { name: Validator<string>, age: Validator<number> }
// 映射成      { name: string; age: number }
type ShapeOf<S extends Record<string, Validator<unknown>>> = {
  [K in keyof S]: Infer<S[K]>;
};

export function isObjectOf<S extends Record<string, Validator<any>>>(
  schema: S
): Validator<ShapeOf<S>> {
  return (input, path) => {
    if (typeof input !== "object" || input === null || Array.isArray(input)) {
      return err(path, "object", input);
    }
    const record = input as Record<string, unknown>;
    const out: Record<string, unknown> = {};
    const errors: string[] = [];
    for (const key of Object.keys(schema)) {
      const childPath = path ? `${path}.${key}` : key;
      const r = schema[key](record[key], childPath);
      if (r.ok) out[key] = r.value;
      else errors.push(...r.errors);
    }
    return errors.length
      ? { ok: false, errors }
      : ok(out as ShapeOf<S>);
  };
}
```

逐段拆解：

- `ShapeOf<S>` 是个**映射类型（mapped type）**（第 17 章）：它遍历 schema 的每个键 `K`，用 `Infer<S[K]>` 把那个字段的校验器 `Validator<X>` 反推成 `X`。于是 `{ id: isNumber, name: isString }` 这样一个**校验器对象**，在类型层面被"翻译"成 `{ id: number; name: string }`——这就是**运行时规则 → 编译期类型**的自动联动。
- 运行时逻辑：先确认 `input` 是"非 null、非数组的对象"（三重检查，绕开 `typeof null` 陷阱），再逐字段校验，把**字段名拼进路径**（`${path}.${key}`），收集所有错误。
- 返回类型 `Validator<ShapeOf<S>>`：这就是全部魔法的落点——**校验通过后，调用方拿到的值类型是从 schema 推导出的强类型对象**。

### 第五步：便捷入口 `parse` 与断言版 `assertValid`

```typescript
// 文件：code/ch18/validate.ts（节选）

// parse：校验成功返回强类型值，失败抛出带完整错误路径的异常。
export function parse<T>(validator: Validator<T>, input: unknown): T {
  const r = validator(input, "");
  if (r.ok) return r.value;
  throw new Error("校验失败:\n" + r.errors.map((e) => "  - " + e).join("\n"));
}

// assertValid：断言函数版本 —— 校验失败即抛，之后 input 被收窄为 T。
export function assertValid<T>(
  validator: Validator<T>,
  input: unknown
): asserts input is T {
  const r = validator(input, "");
  if (!r.ok) {
    throw new Error("校验失败:\n" + r.errors.map((e) => "  - " + e).join("\n"));
  }
}
```

逐段拆解：

- `parse` 返回强类型 `T`（成功）或抛错（失败）。注意 `if (r.ok) return r.value` 这行——`r` 是可辨识联合 `Result<T>`，`r.ok` 为真时 TS 把 `r` 收窄为 `Ok<T>`，才允许访问 `r.value`。**本章开头讲的收窄，在这个库的实现里无处不在。**
- `assertValid` 是**断言函数**版本：返回类型 `asserts input is T`。校验失败就抛；正常返回后，调用处的 `input` 变量会被**收窄为 `T`**。适合"我就地校验一个已有变量，之后直接用它"的场景。

### 第六步：跑起来看联动效果

```typescript
// 文件：code/ch18/validate.demo.ts（节选）

const userSchema = isObjectOf({
  id: isNumber,
  name: isString,
  active: isBoolean,
  role: literalUnion("admin", "guest"),
  tags: isArrayOf(isString),
  nickname: optional(isString),
});

// 关键：用 Infer 从 schema「反推」出静态类型，无需手写 interface！
type User = Infer<typeof userSchema>;
// User === {
//   id: number; name: string; active: boolean;
//   role: "admin" | "guest"; tags: string[]; nickname: string | undefined;
// }

// 模拟一段来自网络、类型未知的数据
const rawGood: unknown = JSON.parse(
  '{"id":1,"name":"Alice","active":true,"role":"admin","tags":["x","y"]}'
);

const user: User = parse(userSchema, rawGood);
console.log("  name:", user.name.toUpperCase()); // string 方法安全
console.log("  role:", user.role); // "admin" | "guest"
console.log("  tags:", user.tags.join(", ")); // string[]
```

逐段拆解：

- 我们只写了 **一遍** `userSchema`（运行时规则），`User` 类型由 `Infer<typeof userSchema>` **自动派生**。改 schema，`User` 跟着变，永不脱节——这是相比"手写 `interface User` + 手写校验函数"的巨大进步。
- `rawGood` 特意标注为 `unknown`（模拟 `JSON.parse` 的真实处境）。经过 `parse(userSchema, rawGood)`，`user` 就成了完全强类型的 `User`：`.name.toUpperCase()`、`.role`（联合）、`.tags.join()`（数组）全部类型安全。**运行时的校验，换来了编译期的强类型**——这就是"类型联动"。

再看非法输入的错误路径：

```typescript
// 文件：code/ch18/validate.demo.ts（节选）

const rawBad: unknown = {
  id: "not-a-number", // 类型错
  name: "Bob",
  active: "yes", // 应为 boolean
  role: "superuser", // 不在字面量联合内
  tags: ["ok", 42, "fine"], // tags[1] 类型错
};

try {
  parse(userSchema, rawBad);
} catch (e) {
  console.log((e as Error).message);
}
```

运行：

```bash
npx tsx ch18/validate.demo.ts
```

预期输出（节选）：

```
✅ 校验通过:
  name: ALICE
  role: admin
  tags: x, y
  nickname: (无)

❌ 校验失败，错误路径清晰:
校验失败:
  - id: 期望 number，实际得到 string
  - active: 期望 boolean，实际得到 string
  - role: 期望 "admin" | "guest"，实际得到 string
  - tags[1]: 期望 string，实际得到 number
```

逐段拆解：

- 每条错误都带**精确路径**：`id`、`active`、`role`、`tags[1]`。`tags[1]` 尤其能看出 `isArrayOf` 把索引拼进路径的价值——它没有笼统地说"tags 有问题"，而是指到出错的那个元素。
- 库**收集了全部错误**再一次性抛出，而不是只报第一个。对调用方（尤其是表单校验、API 网关）来说，一次看清所有问题远比"改一个报一个"高效。

### 扩展练习（加分项）

1. **`nullable(v)`**：仿照 `optional`，写一个允许 `null` 的包装器，返回 `Validator<T | null>`。想清楚它和 `optional` 的语义差别。
2. **`min` / `max` 约束**：给 `isNumber` 加一个变体 `numberRange(min, max)`，校验数字在区间内，错误信息里带上区间。体会"运行时校验能表达类型系统表达不了的约束"。
3. **`union(...validators)`**：写一个"满足其中任一校验器即可"的组合器，返回 `Validator<A | B | ...>`。提示：依次尝试，全失败才报错。
4. **严格模式**：给 `isObjectOf` 加一个选项，当输入对象**含有 schema 里没有的多余字段**时报错（防止脏数据混入）。
5. **收集 vs 快速失败**：给 `parse` 加一个 `{ abortEarly: true }` 选项，遇到第一个错误就停，对比两种模式的取舍。
6. **和 `assertNever` 结合**：给校验结果加一种 `warning` 状态，用可辨识联合 + `switch` + `assertNever` 处理三种状态，体会穷尽检查如何防止你漏掉分支。

## 小结与自测

本章我们把"如何安全地使用宽类型"这件事讲透了：

- **为什么要收窄**：联合类型、`unknown` 在收窄前只允许"两边都支持"的操作。你在运行时做一次检查，TS 就把类型变窄，让你能用专属方法。
- **收窄手段**：`typeof`（原始类型）、`instanceof`（类实例）、`in`（对象属性）、真值收窄（注意误伤 `0`/`""`）、相等收窄（`===` 取交集、`!= null` 一并排除空值）、**可辨识联合 `switch`**（最优雅）。
- **控制流分析**：TS 逐行推算类型，`return`、赋值、`if` 都会改变某个位置的类型。用**早返回**能写出清爽的收窄。
- **类型守卫 `x is T`**：把复杂判断抽成可复用的收窄函数。经典应用是 `arr.filter(isNotNull)` 收窄数组。⚠️ 守卫会"撒谎"，函数体必须如实检查。
- **断言函数 `asserts`**：`asserts condition` 和 `asserts x is T`——"不满足就抛，满足就默认成立"，不需要 `if`。
- **类型断言**：`as`（无条件相信，危险）、`as const`（锁死为只读字面量，安全常用）、非空断言 `!`（危险）、双重断言 `as unknown as T`（最后手段）。断言都**不做运行时检查**。
- **`unknown` vs `any`**：`unknown` 是"先证明再使用"，`any` 是"放弃检查"。外部数据一律先落 `unknown`。**`never`**：不返回、穷尽检查、空联合。
- **`?.` 与 `??`**：可选链自动短路，空值合并只对 `null`/`undefined` 兜底（区别于 `||`）。
- **小项目**：手写迷你校验库，用类型守卫思路 + 条件类型 `Infer` + 映射类型 `ShapeOf`，让**运行时校验**与**编译期类型**联动——一份 schema，既在运行时把关脏数据（带精确错误路径），又在编译期派生出强类型。

收窄、守卫、断言，让你能在**值的层面**安全地驾驭类型。但 TypeScript 还有一个更"元"的能力——在**声明的层面**给类、方法、属性附加行为和元数据，而不改动它们的实现。想象一下：给一个方法加上 `@log` 就自动记录调用、给一个类加上 `@Entity` 就自动注册到框架、给一个属性加上 `@Column` 就映射到数据库字段……这些"贴在声明上的小标签"就是**装饰器（decorator）**。
