## 4.1 作用域：变量的"可见范围"

**作用域（scope）** 指的是：一个变量在代码里的哪些地方"可见"（可以被访问）。可以把它类比成"权限范围"——你在自己家里能拿冰箱里的饮料，但不能随便进邻居家的冰箱。

JavaScript 里有三种作用域：

- **全局作用域（global scope）**：写在所有函数外面的变量，哪里都能访问。
- **函数作用域（function scope）**：写在函数内部的变量，只有函数内部能访问。
- **块级作用域（block scope）**：用 `let` / `const` 写在 `{ }` 里的变量，只有这个块内能访问。

```javascript
// 文件：code/ch04/01-scope.js
const appName = "mini-lodash"; // 全局

function greet() {
  const outer = "外层"; // 函数作用域

  function inner() {
    const innerOnly = "最里层"; // inner 的函数作用域
    console.log(appName); // ✅ 找到全局的
    console.log(outer); // ✅ 找到 greet 里的
    console.log(innerOnly); // ✅ 自己的
  }

  inner();
  // console.log(innerOnly); // ⚠️ 报错：innerOnly 只在 inner 内可见
}

greet();
```

**逐段拆解：**

- `appName` 在最外层声明，所以从任何地方都能读到它。
- `outer` 在 `greet` 内声明，只有 `greet`（以及它内部的 `inner`）能读到。
- `innerOnly` 在 `inner` 内声明，出了 `inner` 就"不存在"了。

📌 关键规律：**里层能看到外层，外层看不到里层。** 就像站在楼里，你能透过窗户看到外面的大街，但街上的人看不到你屋里。

### 块级作用域与 var 的历史遗留

```javascript
// 文件：code/ch04/01-scope.js（续）
{
  const blockVar = "只活在这个块里";
  console.log(blockVar); // ✅
}
// console.log(blockVar); // ⚠️ ReferenceError

if (true) {
  var y = 20; // var 没有块级作用域！
}
console.log("var y 泄漏到了外面：y =", y); // 20，居然能访问到
```

⚠️ **陷阱**：`let` / `const` 遵守块级作用域，出了 `{ }` 就访问不到。但老关键字 `var` **没有块级作用域**，它会"泄漏"到最近的函数（或全局）作用域。上面 `y` 明明是在 `if` 块里声明的，块外却还能访问。这正是 `var` 制造 bug 的根源之一。

💡 **技巧**：现代代码请**只用 `let` 和 `const`，永远不要用 `var`**。本书除了讲解历史坑，其余一律用 `let` / `const`。

## 4.2 作用域链与词法作用域

上一节 `inner` 里访问 `appName`，JS 是怎么找到它的？答案是**作用域链（scope chain）**。

当代码用到一个变量时，引擎会：

1. 先在**当前作用域**找；
2. 找不到就去**外层作用域**找；
3. 再找不到就继续往外，直到全局作用域；
4. 全局也没有？那就报 `ReferenceError`。

这一层层向外查找的链条，就叫作用域链。你可以想象成一串"套娃"：最里面的娃找不到东西，就问外面一层的娃要，一直问到最外面。

那么"外层"到底指哪一层？这里有个重要概念：**词法作用域（lexical scope）**，也叫**静态作用域**。

> 📌 **词法作用域的意思是：一个函数的外层作用域，是由它在代码里"写在哪儿"决定的，而不是由它"在哪儿被调用"决定的。**

```javascript
const value = "全局";

function outer() {
  const value = "outer 的";
  function inner() {
    console.log(value); // 打印什么？
  }
  return inner;
}

const fn = outer();
fn(); // 输出："outer 的"
```

**逐段拆解**：`inner` 是**写在** `outer` 内部的，所以它的外层永远是 `outer`——哪怕我们把 `inner` 拿到外面（`const fn = outer()`）再调用，它找 `value` 时依然先看 `outer` 的作用域。函数的"出生地"决定了它的作用域链，这个决定在你写代码的那一刻就定死了。

记住这句话，因为它是理解下一节**闭包**的钥匙。

## 4.3 变量提升复习与暂时性死区（TDZ）

第 2 章我们提过"提升（hoisting）"。这里快速复习并补上一个新概念。

```javascript
// 文件：code/ch04/02-hoisting-tdz.js
console.log(add(2, 3)); // ✅ 5，调用写在定义前也行
function add(a, b) {
  return a + b;
}

console.log("var 提升演示 v =", v); // undefined（不是报错）
var v = 100;
```

**逐段拆解：**

- **函数声明**会被整体提升到作用域顶部，所以可以"先调用后定义"。
- `var` 变量的**声明**被提升，但**赋值**留在原地。于是在赋值前访问它，拿到的是 `undefined`——这种"不报错但值诡异"的行为很坑。

那 `let` / `const` 呢？它们其实也会被提升，但有一道保护机制：

```javascript
// 文件：code/ch04/02-hoisting-tdz.js（续）
try {
  console.log(letVar); // 抛出 ReferenceError
  let letVar = 1;
} catch (e) {
  console.log("访问 TDZ 中的变量报错：", e.constructor.name);
}
```

从作用域开始，到 `let` / `const` 声明语句执行为止，这段"变量已存在但不许访问"的区间，叫**暂时性死区（Temporal Dead Zone, TDZ）**。在 TDZ 里访问变量会直接**报错**。

📌 这其实是好事：`var` 会给你一个静默的 `undefined` 让 bug 潜伏，而 `let` / `const` 会当场报错，让你第一时间发现"用了还没声明的变量"。这又是一条"优先用 `let` / `const`"的理由。

## 4.4 闭包：本章的重中之重

现在到了 JavaScript 最迷人的部分。

**闭包（closure）** 的一句话定义：**一个函数，连同它出生时所在的外层作用域（那些它引用到的变量），被一起打包带走——这个组合就是闭包。**

换个类比：函数就像一个背包客，出门（被返回、被传走）时，会把出生地那些"它用得着的变量"打包进背包，走到哪儿背到哪儿。哪怕外层函数早已执行完毕，这些变量也不会被销毁，因为背包里还留着它们。

### 4.4.1 最小的闭包：计数器

```javascript
// 文件：code/ch04/03-closure.js
function makeCounter() {
  let count = 0; // 被内层函数"闭"住的变量
  return function () {
    count += 1;
    return count;
  };
}

const counter = makeCounter();
console.log(counter()); // 1
console.log(counter()); // 2
console.log(counter()); // 3
```

**背后发生了什么？**

正常想法是：`makeCounter()` 执行完，它内部的 `count` 就该被回收了。但这里返回的那个匿名函数**引用了 `count`**。根据词法作用域，这个匿名函数的外层永远是 `makeCounter` 的作用域。所以只要这个匿名函数还活着（被 `counter` 变量拿着），`count` 就不会被销毁——它被"闭包"保护起来了。

于是每次 `counter()` 都在**同一个** `count` 上加一，值能持续累积。这就是闭包"记住状态"的能力。

再看一个关键细节：

```javascript
// 文件：code/ch04/03-closure.js（续）
const counterA = makeCounter();
const counterB = makeCounter();
console.log(counterA()); // 1
console.log(counterA()); // 2
console.log(counterB()); // 1（B 有自己独立的 count）
```

📌 **每次调用 `makeCounter()` 都会创建一份全新的、互相独立的 `count`。** `counterA` 和 `counterB` 各背各的背包，互不干扰。

### 4.4.2 用闭包实现私有变量

JavaScript 早期没有"私有属性"这种语法。人们用闭包来模拟：把变量藏在函数里，只通过返回的方法访问，外界拿不到变量本身。

```javascript
// 文件：code/ch04/03-closure.js（续）
function createAccount(initial) {
  let balance = initial; // 私有，外界无法直接访问
  return {
    deposit(amount) {
      balance += amount;
      return balance;
    },
    withdraw(amount) {
      if (amount > balance) {
        console.log("余额不足");
        return balance;
      }
      balance -= amount;
      return balance;
    },
    getBalance() {
      return balance;
    },
  };
}

const acc = createAccount(100);
console.log(acc.deposit(50)); // 150
console.log(acc.withdraw(30)); // 120
console.log(acc.balance); // undefined：没暴露，无法篡改
```

**逐段拆解**：`balance` 只存在于 `createAccount` 的作用域里。返回的对象里三个方法都是闭包，它们共享同一个 `balance`。外界只能通过 `deposit` / `withdraw` / `getBalance` 间接操作它——`acc.balance` 是 `undefined`，你没法直接把余额改成一百万。这就是**数据封装**。

### 4.4.3 函数工厂

闭包还能用来"批量生产"定制化的函数：

```javascript
// 文件：code/ch04/03-closure.js（续）
function makeMultiplier(factor) {
  return (n) => n * factor; // factor 被闭包记住
}

const double = makeMultiplier(2);
const triple = makeMultiplier(3);
console.log(double(10)); // 20
console.log(triple(10)); // 30
```

`double` 和 `triple` 是同一个模板产出的两个函数，各自记住了不同的 `factor`。这种"用配置生产函数"的模式，在实际项目里非常常见。

## 4.5 循环里的闭包经典坑（var vs let）

这是面试高频题，也是理解闭包的试金石。

```javascript
// 文件：code/ch04/04-loop-closure.js
const funcsVar = [];
for (var i = 0; i < 3; i++) {
  funcsVar.push(function () {
    return i;
  });
}
console.log(funcsVar[0]()); // 3  ⚠️
console.log(funcsVar[1]()); // 3  ⚠️
console.log(funcsVar[2]()); // 3  ⚠️
```

**为什么全是 3？**

因为 `var i` 没有块级作用域，整个循环里**只有一个 `i`**。三个函数闭的都是**同一个 `i`**。循环结束时 `i` 已经变成 3，所以后面无论调用哪个函数，读到的都是这个共享 `i` 的最终值 3。

**用 `let` 就对了：**

```javascript
// 文件：code/ch04/04-loop-closure.js（续）
const funcsLet = [];
for (let j = 0; j < 3; j++) {
  funcsLet.push(() => j);
}
console.log(funcsLet[0]()); // 0
console.log(funcsLet[1]()); // 1
console.log(funcsLet[2]()); // 2
```

📌 `let` 在 `for` 循环里有个特殊待遇：**每一轮迭代都创建一个全新的 `j`**。于是三个函数各自闭住了各自那一轮的 `j`，值分别是 0、1、2。

在还没有 `let` 的年代，人们用 **IIFE**（下一节讲）复制一份变量来解决：

```javascript
// 文件：code/ch04/04-loop-closure.js（续）
const funcsIIFE = [];
for (var k = 0; k < 3; k++) {
  funcsIIFE.push(
    (function (captured) {
      return () => captured; // captured 是每轮拷贝的独立副本
    })(k)
  );
}
console.log(funcsIIFE[0](), funcsIIFE[1](), funcsIIFE[2]()); // 0 1 2
```

理解了这段代码，你就真正理解闭包了。今天当然直接用 `let`。

## 4.6 this：取决于"怎么调用"，而非"在哪定义"

`this` 是很多人学 JS 的第一道大坎。难点在于：**普通函数里 `this` 的值，不是在定义时决定的，而是在调用时决定的。** 同一个函数换个调用方式，`this` 就变了。

记住四条绑定规则，你就能一眼看出任何 `this` 指向谁。

### 规则 2：隐式绑定（最常见）

（先讲最直观的规则 2，规则 1 放最后。）

**谁在"点号"左边调用这个函数，`this` 就是谁。**

```javascript
// 文件：code/ch04/05-this.js
const person = {
  name: "小明",
  greet() {
    return `你好，我是 ${this.name}`;
  },
};
console.log(person.greet()); // 你好，我是 小明（this = person）
```

调用是 `person.greet()`，点号左边是 `person`，所以函数体里的 `this` 就是 `person`。

⚠️ **陷阱**：一旦把方法"摘"下来单独调用，隐式绑定就丢了：

```javascript
// 文件：code/ch04/05-this.js（续）
const g = person.greet;
try {
  console.log(g()); // this 不再是 person！
} catch (e) {
  console.log("摘出来调用出问题：", e.constructor.name); // TypeError
}
```

`g()` 直接调用，前面没有点号左边的对象了，`this` 变成 `undefined`（严格模式 / ES 模块下），`this.name` 就报错。**函数体没变，只是换了调用方式，`this` 就变了**——这就是理解 `this` 的核心。

### 规则 3：显式绑定 call / apply / bind

你可以用三个方法**手动指定** `this`：

```javascript
// 文件：code/ch04/05-this.js（续）
function introduce(city, job) {
  return `${this.name} 来自 ${city}，职业是 ${job}`;
}
const user = { name: "小红" };

console.log(introduce.call(user, "北京", "工程师")); // 参数逐个传
console.log(introduce.apply(user, ["上海", "设计师"])); // 参数用数组传
const boundIntro = introduce.bind(user); // 返回绑好 this 的新函数
console.log(boundIntro("广州", "老师"));
```

**逐段拆解：**

- **`call(thisArg, arg1, arg2, ...)`**：立刻调用，第一个参数指定 `this`，其余参数逐个传。
- **`apply(thisArg, [args])`**：和 `call` 一样，只是参数用一个数组传。（记忆：apply 里有 a，array 也有 a。）
- **`bind(thisArg)`**：**不立即调用**，而是返回一个"永久绑定了 `this`"的新函数。之后无论怎么调用它，`this` 都不会再变。`bind` 常用来解决"摘出来调用丢 this"的问题。

### 规则 4：new 绑定

用 `new` 调用函数时，会创建一个全新的对象，并把它作为 `this`：

```javascript
// 文件：code/ch04/05-this.js（续）
function Animal(type) {
  this.type = type; // this 指向 new 出来的新对象
}
const cat = new Animal("猫");
console.log(cat.type); // 猫
```

`new Animal("猫")` 背后：新建一个空对象 → 把它当作 `this` → 执行函数体给它加属性 → 返回这个对象。第 5 章讲面向对象时会深入。

### 规则 1：默认绑定（兜底）

如果一个普通函数**直接调用**（没有点号左边、没有 call/apply、没有 new），就走默认绑定：

- 非严格模式：`this` 指向全局对象。
- 严格模式 / ES 模块：`this` 是 `undefined`。

现代代码基本都在模块/严格模式下，所以直接调用普通函数时 `this` 多为 `undefined`——上面 `g()` 报错就是这个原因。

📌 **四条规则的优先级**（从高到低）：`new` > 显式绑定（call/apply/bind）> 隐式绑定（点号）> 默认绑定。

### 箭头函数：没有自己的 this

**箭头函数（arrow function）不遵守上面四条规则。它根本没有自己的 `this`，而是直接用"定义它时"外层作用域的 `this`。** 而且这个 `this` 一旦确定就无法改变（call/apply/bind 也改不了）。

这个特性专治"回调里 this 丢失"的病：

```javascript
// 文件：code/ch04/05-this.js（续）
const team = {
  title: "前端组",
  members: ["A", "B", "C"],
  // ❌ 普通函数作回调，this 丢了
  listWrong() {
    return this.members.map(function (m) {
      return `${this && this.title}-${m}`; // this 是 undefined
    });
  },
  // ✅ 箭头函数继承 listRight 的 this（即 team）
  listRight() {
    return this.members.map((m) => `${this.title}-${m}`);
  },
};
console.log(team.listWrong()); // ['undefined-A', 'undefined-B', 'undefined-C']
console.log(team.listRight()); // ['前端组-A', '前端组-B', '前端组-C']
```

**逐段拆解：**

- `listWrong` 里，`map` 的回调是**普通函数**，它被 `map` 内部直接调用，走默认绑定，`this` 是 `undefined`，于是 `this.title` 拿不到。
- `listRight` 里，回调是**箭头函数**，它没有自己的 `this`，就用外层 `listRight` 的 `this`——而 `listRight` 是被 `team.listRight()` 调用的，`this` 正是 `team`。于是 `this.title` 正确拿到 `"前端组"`。

💡 **技巧**：需要"继承外层 this"的回调（尤其是数组方法、定时器回调、事件回调）用**箭头函数**；需要"由调用方决定 this"的对象方法（如上面的 `greet`）用**普通函数/方法简写**。

📌 一句话总结全节：
- **普通函数**的 `this` = 调用时点号左边是谁 / call 传谁 / new 谁 / 否则默认。
- **箭头函数**的 `this` = 定义时外层的 `this`，且不可更改。

## 4.7 高阶函数、纯函数与副作用

**高阶函数（higher-order function）** 的定义很简单：**接收函数作为参数，或者返回一个函数**——满足其一即是高阶函数。

你其实早就用过：第 3 章的 `map` / `filter` / `reduce` 都是高阶函数，因为它们接收一个函数当参数。

### 函数作为参数

```javascript
// 文件：code/ch04/06-hof.js
function repeat(n, action) {
  for (let i = 0; i < n; i++) {
    action(i); // action 是传进来的函数
  }
}
repeat(3, (i) => console.log("第", i, "次"));
```

把"做什么"（`action`）从"做几次"（循环逻辑）里抽出来，让调用方决定行为——这就是高阶函数带来的**抽象能力**。

### 函数作为返回值

```javascript
// 文件：code/ch04/06-hof.js（续）
function greaterThan(n) {
  return (m) => m > n; // 返回一个新函数
}
const greaterThan10 = greaterThan(10);
console.log(greaterThan10(11)); // true
console.log(greaterThan10(9)); // false
```

这其实就是 4.4 节的"函数工厂"，本质是闭包 + 高阶函数。

### 纯函数与副作用

**纯函数（pure function）** 满足两个条件：

1. **相同的输入永远得到相同的输出**（不依赖外部会变的状态）。
2. **不产生副作用**——不修改外部变量、不改传入的对象、不打印、不发请求、不写文件。

**副作用（side effect）** 就是函数对"外部世界"造成的任何改动。

```javascript
// 文件：code/ch04/06-hof.js（续）
function pureAdd(a, b) {
  return a + b; // 纯函数：只依赖参数，不碰外部
}

let total = 0;
function impureAdd(n) {
  total += n; // 副作用：改了外部的 total
  return total;
}
console.log(pureAdd(2, 3)); // 永远 5
console.log(impureAdd(2)); // 2
console.log(impureAdd(2)); // 4，同样输入不同输出！
```

📌 **为什么推崇纯函数？** 因为它们**可预测、好测试、好复用、易并行**。你不需要担心"调用它会不会偷偷改了别处的东西"。**函数式编程（functional programming）** 的核心思想之一，就是尽量把逻辑写成纯函数，把不可避免的副作用（打印、请求、写库）收拢到程序边缘。

当然，程序不可能没有副作用（否则什么都干不了）。思路是：**让纯的部分尽量多、纯，让副作用尽量少、集中。**

### IIFE：立即执行函数

**IIFE（Immediately Invoked Function Expression，立即执行函数表达式）** 是一个"定义完立刻执行"的函数，主要用来制造一个独立作用域、避免变量污染外部。

```javascript
// 文件：code/ch04/06-hof.js（续）
const config = (function () {
  const secret = "1234"; // 外部访问不到
  return { version: "1.0", check: (s) => s === secret };
})();
console.log(config.version); // 1.0
console.log(config.check("1234")); // true
console.log(config.secret); // undefined
```

**逐段拆解**：把函数用 `()` 包起来变成表达式，末尾再加一对 `()` 立即调用它。`secret` 被藏在 IIFE 的作用域里，外部只能通过返回的对象间接使用——这又是闭包在做封装。在模块化（第 9 章）普及前，IIFE 是隔离作用域的主力，今天你仍会在很多库的源码里见到它。

## 4.8 实用模式：柯里化、偏函数、组合、防抖、节流

这些模式全都建立在**闭包 + 高阶函数**之上。理解了它们，你就理解了现代 JS 库里大量"看起来很魔法"的写法。它们同时也是下面小项目的组成部分，这里先建立直觉。

- **偏函数（partial application）**：预先固定一个函数的部分参数，得到一个"参数更少"的新函数。比如把 `greet(greeting, name)` 固定 `greeting="你好"`，得到只需传 `name` 的 `sayHi`。
- **柯里化（currying）**：把 `fn(a, b, c)` 变换成可以 `fn(a)(b)(c)` 这样"一次传一个参数"逐步调用的形式。它和偏函数是近亲，都靠闭包累积参数。
- **组合（compose / pipe）**：把多个小函数"串"成一个大函数。`pipe(f, g)(x)` 等于 `g(f(x))`（从左到右），`compose(f, g)(x)` 等于 `f(g(x))`（从右到左）。像流水线一样，数据从一个函数流到下一个。
- **防抖（debounce）**：事件停止触发一段时间后才执行一次。典型场景：搜索框输入时，等用户停下打字再发请求。
- **节流（throttle）**：无论触发多频繁，每隔固定时间最多执行一次。典型场景：滚动、拖拽时限制处理频率。

这些概念光看文字容易忘，我们直接在小项目里把它们**亲手实现一遍**，看闭包如何保存状态。

## 常见坑与最佳实践

- ⚠️ **别用 `var`。** 它没有块级作用域、提升行为反直觉、在循环里制造闭包坑。一律 `let` / `const`，默认 `const`，需要重新赋值才用 `let`。
- ⚠️ **循环中创建函数/回调时，用 `let` 而非 `var`**，否则所有回调会共享同一个变量的最终值。
- ⚠️ **闭包会让被引用的变量常驻内存**。绝大多数情况这没问题，但如果你在长生命周期对象里闭住了大数组/大对象又不释放，可能造成内存占用偏高。用完记得让引用它的函数被回收（例如置为 `null`）。
- ⚠️ **`this` 只看调用方式，不看定义位置。** 把对象方法当回调传出去前，先想好 `this` 会不会丢——需要保住就用 `bind` 或箭头函数。
- ⚠️ **不要给需要"由调用者决定 this"的对象方法用箭头函数**（比如原型上的方法、需要复用 `this` 的方法），否则 `this` 会被固定成外层的，通常不是你想要的。
- 💡 **优先写纯函数**：不改参数、不依赖外部可变状态、返回新值而非原地修改。这样的函数好测、好复用。
- 💡 **需要私有状态时，闭包是最轻量的封装手段**（第 5 章的 `class` 私有字段是另一种）。
- 💡 **调试 `this` 时，把"这个函数最终是被怎么调用的"找出来**，答案就在四条规则里。

## 本章小项目：mini-lodash 工具库

[Lodash](https://lodash.com/) 是 JavaScript 生态里最著名的工具库之一。我们不用它，而是**亲手实现一个迷你版**，把本章所有知识点（闭包、高阶函数、`this`、函数式模式）全部用上。

新建 `code/ch04/mini-lodash.js`，逐个实现工具函数。

### once：只执行一次

```javascript
// 文件：code/ch04/mini-lodash.js
export function once(fn) {
  let called = false; // 闭包保存的状态
  let result;
  return function (...args) {
    if (!called) {
      called = true;
      result = fn.apply(this, args); // 保留 this 与参数
    }
    return result;
  };
}
```

**原理**：返回的新函数闭住了 `called` 和 `result`。第一次调用时执行 `fn` 并记下结果、把 `called` 置 `true`；以后再调用就直接返回缓存的 `result`，不再执行 `fn`。用 `fn.apply(this, args)` 是为了让被包装的函数保留原本的 `this` 和参数，不改变它的行为。适合"初始化只能跑一次"的场景。

### memoize：缓存结果

```javascript
// 文件：code/ch04/mini-lodash.js（续）
export function memoize(fn) {
  const cache = new Map(); // 闭包保存的缓存
  return function (...args) {
    const key = JSON.stringify(args);
    if (cache.has(key)) return cache.get(key);
    const value = fn.apply(this, args);
    cache.set(key, value);
    return value;
  };
}
```

**原理**：闭包里放一个 `Map` 当缓存。每次调用把参数序列化成字符串当作 key，命中就直接返回缓存值，没命中才真正计算并存起来。这就是**用闭包保存状态**的典型应用——`cache` 只属于这一个被 `memoize` 包装的函数，跨多次调用持续存在。适合计算昂贵、参数重复的纯函数。

### partial：偏函数

```javascript
// 文件：code/ch04/mini-lodash.js（续）
export function partial(fn, ...preset) {
  return function (...rest) {
    return fn.apply(this, [...preset, ...rest]);
  };
}
```

**原理**：`preset` 是预先固定的参数，被闭包记住。返回的函数收到剩余参数 `rest` 后，把两部分拼起来调用原函数。这样就"锁定"了前几个参数。

### curry：柯里化

```javascript
// 文件：code/ch04/mini-lodash.js（续）
export function curry(fn) {
  return function curried(...args) {
    if (args.length >= fn.length) {
      return fn.apply(this, args); // 参数够了就执行
    }
    return function (...more) {
      return curried.apply(this, [...args, ...more]); // 继续收集
    };
  };
}
```

**原理**：`fn.length` 是原函数声明的参数个数。`curried` 每次被调用就检查已收到的参数够不够：够了就执行；不够就返回一个新函数，继续通过闭包累积参数（`[...args, ...more]`），直到集齐。于是 `curry(sum3)` 既能 `f(1)(2)(3)`，也能 `f(1,2)(3)`。这是闭包"层层记住已有参数"的漂亮应用。

### compose 与 pipe：函数组合

```javascript
// 文件：code/ch04/mini-lodash.js（续）
export function compose(...fns) {
  return function (x) {
    return fns.reduceRight((acc, fn) => fn(acc), x);
  };
}

export function pipe(...fns) {
  return function (x) {
    return fns.reduce((acc, fn) => fn(acc), x);
  };
}
```

**原理**：`pipe` 用 `reduce` 从左到右把初始值 `x` 依次喂给每个函数，上一个的输出是下一个的输入，像流水线。`compose` 用 `reduceRight` 从右到左，符合数学上 `f(g(x))` 的写法。两者都是高阶函数（接收函数、返回函数）。

### debounce：防抖

```javascript
// 文件：code/ch04/mini-lodash.js（续）
export function debounce(fn, wait) {
  let timer = null; // 闭包记住的定时器
  return function (...args) {
    if (timer) clearTimeout(timer);
    timer = setTimeout(() => {
      fn.apply(this, args);
      timer = null;
    }, wait);
  };
}
```

**原理**：`timer` 被闭包记住。每次触发都先清掉上一个定时器，再重设一个 `wait` 毫秒后执行的定时器。只要触发得比 `wait` 快，定时器就一直被"顶掉"，直到停止触发超过 `wait`，最后那次才真正执行。注意回调用**箭头函数**，这样内部的 `this` / `args` 继承外层——`this` 会丢的坑在这里就被箭头函数化解了。

### throttle：节流

```javascript
// 文件：code/ch04/mini-lodash.js（续）
export function throttle(fn, wait) {
  let last = 0; // 闭包记住上次执行时间
  return function (...args) {
    const now = Date.now();
    if (now - last >= wait) {
      last = now;
      fn.apply(this, args);
    }
  };
}
```

**原理**：闭包记住上次执行的时间戳 `last`。每次触发时比较"现在离上次是否过了 `wait`"，够了才执行并更新 `last`，否则直接忽略。于是无论触发多密集，执行频率被限制在每 `wait` 毫秒最多一次。

### 演示：把工具库跑起来

新建 `code/ch04/demo.js` 导入并演示：

```javascript
// 文件：code/ch04/demo.js
import { once, memoize, partial, curry, compose, pipe, debounce, throttle } from "./mini-lodash.js";

// once：只真正执行一次
const init = once(() => {
  console.log("（真正执行了初始化）");
  return "done";
});
console.log(init(), init(), init()); // 只打印一次初始化，都返回 done

// memoize：相同参数命中缓存
const slowSquare = memoize((n) => {
  console.log(`  计算 ${n} 的平方...`);
  return n * n;
});
console.log(slowSquare(4)); // 会计算
console.log(slowSquare(4)); // 命中缓存，不再计算

// partial + curry
const greet = (greeting, name) => `${greeting}，${name}！`;
console.log(partial(greet, "你好")("小明")); // 你好，小明！
const sum3 = (a, b, c) => a + b + c;
console.log(curry(sum3)(1)(2)(3)); // 6

// compose / pipe
const addOne = (x) => x + 1;
const timesTwo = (x) => x * 2;
console.log(compose(timesTwo, addOne)(5)); // (5+1)*2 = 12
console.log(pipe(addOne, timesTwo)(5)); // (5+1)*2 = 12

// debounce：连触发 5 次，只有最后一次生效
const onSearch = debounce((q) => console.log(`  发起搜索：${q}`), 100);
["a", "ab", "abc", "abcd", "abcde"].forEach(onSearch); // 只搜 abcde

// throttle：100ms 内多次触发只放行一次
const onScroll = throttle(() => console.log("  处理滚动"), 100);
onScroll(); // 执行
onScroll(); // 被拦截
setTimeout(onScroll, 250); // 250ms 后再执行
```

运行：

```bash
node demo.js
```

你会看到 `once` 只初始化一次、`memoize` 第二次不再计算、`curry` 分步传参也能算对、`compose`/`pipe` 结果一致、`debounce` 五次输入只搜最后一次、`throttle` 拦掉了中间的触发。

📌 回头看这七个函数，它们有一个共同点：**都返回了一个内层函数，而这个内层函数靠闭包记住了某种状态**（`called`、`cache`、`preset`、累积的参数、`timer`、`last`）。这就是本章最核心的收获——**闭包 = 函数 + 它记住的状态**。整个 Lodash、RxJS、React Hooks 的底层，都是这套机制在支撑。

## 小结与自测

这一章我们啃下了 JavaScript 最硬的骨头：

- **作用域**分全局、函数、块级；变量查找沿**作用域链**层层向外；外层由代码书写位置决定，即**词法作用域**。
- `var` 无块级作用域、提升给 `undefined`；`let` / `const` 有块级作用域、有 **TDZ** 会当场报错。请只用 `let` / `const`。
- **闭包**是"函数 + 它出生时记住的变量"。它能实现计数器、私有变量、函数工厂，是本章乃至整门语言的核心。循环里创建函数要用 `let` 避免共享变量的坑。
- **this** 由**调用方式**决定，四条规则优先级：`new` > 显式（call/apply/bind）> 隐式（点号）> 默认。**箭头函数没有自己的 this**，继承外层且不可改。
- **高阶函数**接收或返回函数；**纯函数**无副作用、可预测；**IIFE** 制造独立作用域。
- 柯里化、偏函数、组合、防抖、节流全都建立在闭包 + 高阶函数之上，我们在 **mini-lodash** 里亲手实现了它们。
