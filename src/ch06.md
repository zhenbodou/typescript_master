# 第 6 章 · 错误处理与调试

## 本章目标

前五章你已经能写出有模有样的程序了：变量、控制流、函数、对象、闭包、class……但真实世界的程序不是活在理想国里。用户会输入乱七八糟的数据、文件可能不存在、网络会断、接口返回的 JSON 可能是残缺的。**一段没有错误处理的代码，就像一辆没有安全带和气囊的车——平时看不出问题，一出事就是灾难。**

这一章我们专门学“程序出错时该怎么办”，以及“出了 bug 怎么快速找到它”。学完之后你会：

- 理解**错误（error）到底是什么**，会用 `throw` 主动抛错，看懂 Error 对象的 `message` / `name` / `stack`。
- 熟练使用 **`try` / `catch` / `finally`**，知道什么时候该捕获、什么时候该**重新抛出（rethrow）**。
- 认识 JavaScript 的**内置错误类型**：TypeError、RangeError、SyntaxError、ReferenceError……看到报错能立刻判断“哪类问题”。
- 会写**自定义错误类**（`class MyError extends Error`），并用 **`Error.cause`** 串起错误链。
- 分清**同步错误**和**异步错误**处理的区别（异步细节留到第 8 章，这里先建立意识）。
- 掌握**防御性编程（defensive programming）**：参数校验、早返回、断言。
- 学会一套实用**调试技巧**：`console` 家族、`console.assert`、读懂调用栈、`debugger` 语句、Node `--inspect` 断点。
- 建立**优雅降级（graceful degradation）**与错误恢复的思维。
- 本章小项目：亲手封装一个**健壮的 JSON 解析器** `safeParseJSON`。

我们从最根本的问题问起：错误，到底是什么？

## 6.1 错误是什么？throw 与 Error 对象

在 JavaScript 里，**“错误”本质上就是一个被“抛（throw）”出来的值**。当程序执行到某个地方发现“干不下去了”，它会 `throw` 一个值，然后**当前的正常执行流会立刻中断**，这个值像坐电梯一样一路“向上”传播，直到被某个 `catch` 接住；如果一路没人接，程序就崩溃退出。

你可以类比成公司里的“逐级上报”：基层员工遇到搞不定的事，把问题上报给主管；主管处理不了，再上报给经理……一直报到 CEO，CEO 也搞不定，公司就“崩”了。

先看最原始的抛错：

```javascript
// 文件：code/ch06/01-throw-error.js
function withdraw(balance, amount) {
  if (amount > balance) {
    // 不推荐：抛一个普通字符串，信息太少
    throw "余额不足";
  }
  return balance - amount;
}

try {
  withdraw(100, 200);
} catch (e) {
  console.log("捕获到（字符串）：", e); // "余额不足"
  console.log("它有 stack 吗？", e.stack); // undefined，字符串没有 stack
}
```

**逐段拆解：**

- `throw` 后面**可以跟任何值**——字符串、数字、对象都行。这里我们抛了个字符串 `"余额不足"`。
- `try { ... } catch (e) { ... }` 是捕获机制：`try` 块里一旦有东西被抛出，控制权立刻跳到 `catch`，被抛出的值绑定到 `e` 上。
- ⚠️ **但抛字符串是坏习惯**：字符串没有 `stack`（调用栈），你根本不知道错误“从哪儿冒出来的”。调试时这会要了你的命。

📌 **正确姿势是抛一个 `Error` 对象。** `Error` 是 JavaScript 内置的错误类型，它自带三个关键属性：

- `message`：人类可读的错误描述（你写的那句话）。
- `name`：错误的类别名，默认 `"Error"`。
- `stack`：**调用栈快照**——记录了“错误是在哪个函数、哪一行产生的，以及是被谁一层层调用到这里的”。

```javascript
// 文件：code/ch06/01-throw-error.js（续）
function withdraw2(balance, amount) {
  if (amount > balance) {
    throw new Error(`余额不足：想取 ${amount}，但只有 ${balance}`);
  }
  return balance - amount;
}

try {
  withdraw2(100, 200);
} catch (e) {
  console.log("  name    =", e.name); // "Error"
  console.log("  message =", e.message); // 我们写的那句话
  console.log("  是 Error 实例吗？", e instanceof Error); // true
  console.log(e.stack.split("\n").slice(0, 2).join("\n")); // stack 前两行
}
```

**逐段拆解：**

- `new Error(...)` 创建一个错误对象，括号里的字符串就是 `message`。
- `e.name` 是 `"Error"`，`e.message` 是我们写的内容，`e instanceof Error` 为 `true`。
- `e.stack` 是一段多行文本，第一行是 `Error: 你的 message`，接下来每行是 `at 函数名 (文件:行:列)`。**这就是排错的黄金线索**，本章后面会专门教怎么读它。

💡 **技巧**：`message` 里尽量写清“出了什么错、相关的值是多少”。`余额不足：想取 200，但只有 100` 远比干巴巴的 `余额不足` 有用。

再看关键的一点：**抛错会“中断”后续代码**。

```javascript
// 文件：code/ch06/01-throw-error.js（续）
function a() {
  b();
  console.log("a 的这行永远不会执行"); // 被跳过
}
function b() {
  throw new Error("b 出问题了");
}

try {
  a();
} catch (e) {
  console.log("在最外层 catch 到了：", e.message);
}
console.log("程序继续正常往下走。");
```

**逐段拆解：** `a()` 调用 `b()`，`b` 抛错后，`a` 里 `b()` 之后的那行**直接被跳过**——错误“穿过” `a` 一路上升，最终被最外层的 `try/catch` 接住。接住之后，程序不会崩，会继续执行 `catch` 之后的代码。这就是错误“逐级上报、被接住就恢复正常”的完整过程。

> 📌 记住三句话：**`throw` 抛出一个值 → 正常流程立刻中断 → 值向上传播直到被 `catch` 接住（否则程序崩溃）。永远抛 `Error` 对象，不要抛字符串。**

## 6.2 try / catch / finally

上一节我们已经用到了 `try/catch`。这一节把它讲透，再补上第三块 `finally`。

完整结构是三段：

```javascript
// 文件：code/ch06/02-try-catch-finally.js
function parseAge(input) {
  try {
    const n = Number(input);
    if (Number.isNaN(n)) {
      throw new Error(`"${input}" 不是有效数字`);
    }
    console.log("try：解析成功，年龄 =", n);
    return n;
  } catch (e) {
    console.log("catch：出错了 ->", e.message);
    return -1; // 出错时返回一个兜底值
  } finally {
    console.log("finally：无论成功失败都会执行（比如关闭资源）");
  }
}

parseAge("30"); // 走 try + finally
parseAge("abc"); // 走 catch + finally
```

**逐段拆解：**

- **`try` 块**：放“可能出错”的代码。一旦里面任何一处抛错，剩下的部分立刻跳过，转到 `catch`。
- **`catch` 块**：处理错误。`e` 是被抛出的错误对象。这里我们打印信息并返回兜底值 `-1`。
- **`finally` 块**：**不管是否出错，最后一定执行**。它的经典用途是“清理”——关闭文件、断开数据库连接、隐藏加载动画。因为无论成功还是失败你都得清理，写在 `finally` 里最保险。

📌 执行流：合法输入 `"30"` 走 `try` → `finally`；非法输入 `"abc"` 走 `catch` → `finally`。**`finally` 两种情况都会执行**。

### catch 的绑定可以省略

如果你**不关心错误对象本身**，只想知道“成没成”，可以省略 `catch` 后面的 `(e)`：

```javascript
// 文件：code/ch06/02-try-catch-finally.js（续）
function tryRun(fn) {
  try {
    fn();
    return true;
  } catch {
    // 注意：没有写 catch (e)
    return false;
  }
}
console.log(tryRun(() => JSON.parse("{}"))); // true
console.log(tryRun(() => JSON.parse("{坏}"))); // false
```

**逐段拆解：** `catch {}`（无参数）是 ES2019 引入的语法，叫**可选 catch 绑定（optional catch binding）**。这里我们只想知道 `fn` 会不会抛错，不需要错误详情，省掉 `(e)` 更干净。

### ⚠️ finally 的陷阱：它的 return 会“赢”

```javascript
// 文件：code/ch06/02-try-catch-finally.js（续）
function trap() {
  try {
    return "来自 try";
  } finally {
    return "来自 finally"; // ⚠️ 这个会赢
  }
}
console.log(trap()); // "来自 finally"
```

**逐段拆解：** 直觉上 `try` 里已经 `return "来自 try"` 了，应该返回它才对。但 `finally` **一定会执行**，如果 `finally` 里也有 `return`，它会**覆盖** `try` 的返回值。这是个著名的坑。

💡 **技巧**：**不要在 `finally` 里写 `return`（也别在里面 `throw`）**。`finally` 应该只做清理，不参与“返回什么值”这件事。

### 重新抛出（rethrow）：处理不了就往上传

有时候你在 `catch` 里发现“这个错我处理不了，或者不该我处理”。这时**不要吞掉它**，应该记录一下、必要时包装一层，然后重新抛出去，让更上层决定怎么办。

```javascript
// 文件：code/ch06/02-try-catch-finally.js（续）
function readConfig(raw) {
  try {
    return JSON.parse(raw);
  } catch (e) {
    console.log("readConfig：解析失败，记录日志后重新抛出");
    // 包装成更有业务含义的错误，并用 cause 保留原始错误
    throw new Error("配置文件损坏", { cause: e });
  }
}

try {
  readConfig("这不是 JSON");
} catch (e) {
  console.log("上层收到：", e.message); // 配置文件损坏
  console.log("原始错误（cause）：", e.cause.message);
}
```

**逐段拆解：**

- `catch` 里我们没有“假装没事发生”，而是记了日志，然后 `throw` 一个新错误，把底层的 `SyntaxError` 塞进 `{ cause: e }`。
- `cause` 是错误链的关键（下一节详讲）：上层拿到“配置文件损坏”这个业务级错误，同时还能顺着 `e.cause` 追到“到底是 JSON 语法哪儿错了”。

⚠️ **最该避免的反模式是“吞错误”**：

```javascript
try {
  doSomething();
} catch (e) {
  // 什么都不做 —— 错误被彻底吞掉，程序带病继续运行，问题被藏起来
}
```

这叫“空 catch”，是新手最容易犯的错。它让 bug 无声无息地消失，等你发现数据不对时早已无从查起。**要么真正处理，要么重新抛出，绝不能默默吞掉。**

## 6.3 内置错误类型

`Error` 是所有错误的“祖先”，但 JavaScript 还内置了几个更具体的子类型。**看到不同的错误名，就能快速判断是哪一类问题。**

```javascript
// 文件：code/ch06/03-builtin-errors.js
function show(label, fn) {
  try {
    fn();
    console.log(`${label}：没有抛错`);
  } catch (e) {
    console.log(`${label}：${e.name} -> ${e.message}`);
  }
}

// TypeError：对某个值做了它不支持的操作
show("TypeError-1", () => {
  const n = null;
  n.toString(); // 读 null 的属性
});
show("TypeError-2", () => {
  const notAFn = 42;
  notAFn(); // 把数字当函数调用
});

// RangeError：数值/长度超出允许范围
show("RangeError-1", () => new Array(-1)); // 数组长度不能为负
show("RangeError-2", () => {
  (function recurse() {
    return recurse(); // 无限递归 -> 栈溢出
  })();
});

// ReferenceError：使用了根本不存在的变量
show("ReferenceError", () => console.log(thisVariableDoesNotExist));

// SyntaxError：语法错误（这里用 JSON.parse 触发）
show("SyntaxError", () => JSON.parse("{ bad json }"));
```

运行后你会看到每类错误的名字与描述。逐个认识一下：

| 错误类型 | 什么时候出现 | 典型例子 |
| --- | --- | --- |
| **TypeError** | 值的类型不对，或对它做了不支持的操作 | 读 `null`/`undefined` 的属性；把非函数当函数调用 |
| **RangeError** | 数值或长度超出合法范围 | `new Array(-1)`；递归太深导致“栈溢出（stack overflow）” |
| **ReferenceError** | 引用了一个不存在的变量 | 变量名拼错；使用了未声明的变量 |
| **SyntaxError** | 代码/数据的语法不合法 | `JSON.parse` 收到非法 JSON；代码本身写错语法 |
| **URIError** | URI 编解码函数用错 | `decodeURIComponent("%")` |
| **EvalError** | 与 `eval` 相关（历史遗留，几乎不用） | —— |

⚠️ **关于 SyntaxError 的特殊性**：如果是你**自己写的代码**语法错了（比如少个括号），SyntaxError 会在“解析阶段”就抛出，整个文件根本没机会运行，`try/catch` 也抓不住。上面例子里能抓到 SyntaxError，是因为它来自 `JSON.parse` 在**运行时**解析一段字符串。

📌 这些内置类型**全都继承自 `Error`**：

```javascript
// 文件：code/ch06/03-builtin-errors.js（续）
try {
  null.x;
} catch (e) {
  console.log(e instanceof Error); // true
  console.log(e instanceof TypeError); // true
  console.log(e instanceof RangeError); // false
}
```

**逐段拆解：** 一个 `TypeError` 既是 `TypeError` 也是 `Error`（因为继承关系，回顾第 5 章的 `instanceof` 与原型链）。所以 `catch (e)` 里可以用 `e instanceof TypeError` 精确判断“是不是这一类错误”，从而做不同处理——这在实际项目里非常常用。

## 6.4 自定义错误类与 Error.cause

内置类型够用于语言层面的错误，但**业务错误需要你自己定义**。比如“用户名不能为空”“订单已过期”“余额不足”——这些不属于 TypeError 也不属于 RangeError，你应该造一个**专属的错误类型**。

做法很简单：写一个类 `extends Error`。

```javascript
// 文件：code/ch06/04-custom-error.js
export class ValidationError extends Error {
  constructor(message, field) {
    super(message); // 一定要调用 super，把 message 传给 Error
    this.name = "ValidationError"; // 覆盖默认的 "Error"
    this.field = field; // 自定义字段：出错的字段名
  }
}
```

**逐段拆解：**

- `extends Error` 让 `ValidationError` 继承 `Error` 的全部能力（`message`、`stack` 等）。
- 构造函数里**第一件事必须是 `super(message)`**——把 message 交给父类 `Error` 去处理，`stack` 才会正确生成。（回顾第 5 章：子类构造函数用 `this` 之前必须先 `super`。）
- `this.name = "ValidationError"`：⚠️ **这一步很关键**。如果不写，`name` 会继承为 `"Error"`，日志里就分不清是哪种错误。手动设成类名。
- `this.field`：这就是自定义错误的价值——你可以携带**任意结构化信息**，这里记录了“到底哪个字段校验失败”。

来抛出并捕获它：

```javascript
// 文件：code/ch06/04-custom-error.js（续）
function checkUser(user) {
  if (!user.name) throw new ValidationError("用户名不能为空", "name");
  if (typeof user.age !== "number") throw new ValidationError("年龄必须是数字", "age");
  return true;
}

try {
  checkUser({ name: "" });
} catch (e) {
  console.log(e.name); // ValidationError
  console.log(e instanceof Error); // true
  console.log(e instanceof ValidationError); // true
  console.log("出错字段:", e.field); // name
}
```

**逐段拆解：** 抛出的 `ValidationError` 既是 `Error` 又是 `ValidationError`，所以上层可以用 `if (e instanceof ValidationError)` 单独处理“校验类错误”（比如提示用户改输入），而对其它错误做别的处理。`e.field` 直接告诉你是哪个字段的锅。

### Error.cause：把错误串成链

真实系统是分层的：底层是网络/数据库，上面是业务逻辑。**底层错误往往“看不懂”，业务层需要把它翻译成“听得懂”的错误，但又不能丢掉原始信息。** ES2022 给 `Error` 加了标准的 `cause`（原因）字段来解决这个问题。

```javascript
// 文件：code/ch06/04-custom-error.js（续）
export class HttpError extends Error {
  constructor(status, message, options) {
    super(message, options); // options 可含 { cause }
    this.name = "HttpError";
    this.status = status;
  }
}

function loadUserFromApi() {
  try {
    throw new Error("ECONNREFUSED 127.0.0.1:5432"); // 假装底层连接失败
  } catch (lowLevel) {
    // 包装成业务能懂的错误，但用 cause 保留原因
    throw new HttpError(503, "服务暂时不可用", { cause: lowLevel });
  }
}

try {
  loadUserFromApi();
} catch (e) {
  console.log(e.name, e.status, "->", e.message); // HttpError 503 -> 服务暂时不可用
  console.log("底层原因:", e.cause.message); // ECONNREFUSED ...
}
```

**逐段拆解：**

- `new Error(message, { cause })` 或自定义类里把 `options` 透传给 `super(message, options)`，就能设置 `cause`。
- 上层拿到的是清晰的业务错误 `HttpError 503 服务暂时不可用`，同时通过 `e.cause` 一路追到底层真凶 `ECONNREFUSED`。
- 这形成一条**错误链（error chain）**：你可以顺着 `e.cause.cause.cause...` 一层层往下挖，定位问题根源，而每一层都保留了它那一层的语义。

💡 **技巧**：凡是“捕获底层错误 → 重新抛出业务错误”的场景，都用 `{ cause: 原错误 }` 保留现场。这是现代 JavaScript 的标准实践，别自己发明 `originalError` 之类的字段。

📌 **导出细节**：`04-custom-error.js` 用 `export class` 把两个错误类导出（本项目 `package.json` 里 `"type": "module"`，即 ESM）。文件末尾用 `if (import.meta.url === ...)` 判断“是否被直接运行”，只有直接运行时才跑演示——被 `import` 时不会执行演示代码。这是编写“既能独立运行、又能被复用”的模块的常用技巧。

## 6.5 同步 vs 异步错误处理

这里有个新手一定会踩的大坑：**`try/catch` 只能抓住“同步”错误，抓不住“异步回调”里的错误。**

先看正常情况——同步错误 `try/catch` 抓得住：

```javascript
// 文件：code/ch06/05-async-errors.js
try {
  throw new Error("同步错误");
} catch (e) {
  console.log("同步错误被 catch 到：", e.message); // ✅
}
```

再看异步回调，这里就“漏”了：

```javascript
// 文件：code/ch06/05-async-errors.js（续）
function asyncBoomCallback() {
  try {
    setTimeout(() => {
      throw new Error("定时器里的错误"); // 未来才抛出
    }, 10);
  } catch (e) {
    console.log("这行不会执行"); // ⚠️ 永远进不来
  }
}
asyncBoomCallback();
```

**逐段拆解——这是全章最重要的坑之一：**

- `setTimeout` 会“**在未来**（10 毫秒后）”才执行回调里的代码。
- 但 `try/catch` 是**同步**的：`try` 块执行到 `setTimeout(...)` 这一行时，只是“登记了一个定时任务”就立刻结束了，`try` 块随即退出。
- 10 毫秒后回调真正执行、抛出错误时，**那个 `try` 早就执行完、不在“现场”了**——所以 `catch` 根本接不到。
- 结果：这个错误变成“**未捕获异常（uncaught exception）**”。（在示例文件里我们特意用 `process.on("uncaughtException", ...)` 把它兜住并打印，好让文件能跑完；真实项目里不要依赖这个来吞错误。）

那异步错误怎么处理？答案是 **Promise**（第 8 章会系统讲，这里先建立印象）：

```javascript
// 文件：code/ch06/05-async-errors.js（续）
function mayFail(ok) {
  return new Promise((resolve, reject) => {
    setTimeout(() => {
      if (ok) resolve("成功数据");
      else reject(new Error("Promise 失败"));
    }, 10);
  });
}

// 方式一：用 .catch()
mayFail(false)
  .then((data) => console.log("then:", data))
  .catch((e) => console.log("Promise 用 .catch 抓到：", e.message));

// 方式二：在 async 函数里，就能用熟悉的 try/catch！
async function main() {
  try {
    const data = await mayFail(false); // await 把 reject 变成“抛错”
    console.log("拿到数据：", data);
  } catch (e) {
    console.log("async 里用 try/catch 抓到：", e.message); // ✅
  }
}
main();
```

**逐段拆解：**

- Promise 表示“一个未来才有结果的操作”，成功走 `resolve`，失败走 `reject`。
- 处理它的错误，要么链式 `.catch(e => ...)`，要么——**这是 `async/await` 最大的好处**——在 `async` 函数里对 `await` 表达式用 `try/catch`。`await` 会把 Promise 的 `reject` **重新变成一次“抛错”**，于是你熟悉的 `try/catch` 又能用了。

> 📌 现在只需记住一条铁律：**普通 `try/catch` 抓不住异步回调（如 `setTimeout`）里的错误；异步错误要靠 Promise 的 `.catch()`，或在 `async` 函数里对 `await` 用 `try/catch`。** 完整的异步世界，第 8 章见。

## 6.6 防御性编程：校验、早返回、断言

前面讲的是“出错后怎么办”。这一节反过来——**怎么尽量别出错、或者让错误尽早、清晰地暴露**。这套思维叫**防御性编程（defensive programming）**：假设外部输入都是不可信的，主动设防。

### 参数校验 + 早返回（guard clause）

新手常把校验写成层层嵌套的 `if`，代码缩进成“金字塔”，很难读：

```javascript
// 反例（金字塔）
function createUserBad(user) {
  if (user) {
    if (user.name) {
      if (typeof user.age === "number") {
        return `创建成功：${user.name}, ${user.age} 岁`;
      } else {
        throw new TypeError("age 必须是数字");
      }
    } else {
      throw new TypeError("缺少 name");
    }
  } else {
    throw new TypeError("user 不能为空");
  }
}
```

**早返回（guard clause，也叫“卫语句”）** 把它拍平：**先把所有“不合格”的情况尽早挡掉并返回/抛错，主流程留在最后、缩进最浅。**

```javascript
// 文件：code/ch06/06-defensive.js
function createUser(user) {
  if (!user) throw new TypeError("user 不能为空");
  if (!user.name) throw new TypeError("缺少 name");
  if (typeof user.age !== "number") throw new TypeError("age 必须是数字");

  // 走到这里，说明前置条件全满足，逻辑清爽
  return `创建成功：${user.name}, ${user.age} 岁`;
}
```

**逐段拆解：** 两个版本行为完全一样，但 `createUser` 一眼就能读懂：“不满足这几条就出局，否则执行主逻辑”。校验用**内置错误类型**（`TypeError` 表示类型不对）语义清晰。这种“把异常路径提前处理掉、主干代码保持在最左侧”的写法，是专业代码的标志。

💡 **技巧**：函数开头几行做参数校验、用早返回挡掉非法输入，是提升代码可读性最划算的习惯之一。

### 断言（assert）

**断言（assertion）** 用来表达一个“我确信此处必然成立”的前提，如果不成立就立刻抛错。Node 内置了 `node:assert` 模块：

```javascript
// 文件：code/ch06/06-defensive.js（续）
import assert from "node:assert/strict";

function divide(a, b) {
  assert(typeof a === "number", "a 必须是数字");
  assert(b !== 0, "除数不能为 0");
  return a / b;
}
divide(10, 2); // 5
divide(10, 0); // 抛错：除数不能为 0
```

**逐段拆解：** `assert(条件, 消息)` 在条件为假时抛出 `AssertionError`。它和“参数校验”的区别更多是**语气**：校验面向“外部可能给错的输入”，断言面向“程序内部本不该发生、一旦发生就是 bug”的情形。断言让 bug **在离案发现场最近的地方**炸出来，而不是让错误数据流到很远的地方才出问题。

不想引入模块时，自己写个轻量断言也很常见：

```javascript
// 文件：code/ch06/06-defensive.js（续）
function invariant(condition, message) {
  if (!condition) throw new Error("Invariant 违反：" + message);
}
invariant(list.length > 0, "列表不应为空");
```

**逐段拆解：** `invariant`（不变量）是很多知名库（如 React 早期）里的经典小工具，就是“条件为假就抛错”这么简单，却能极大提升代码的健壮性和可读性。

## 6.7 调试技巧

写代码有一半时间在**调试（debugging）**——找出“程序为什么没按预期工作”。这一节给你一套趁手的工具。

### console 家族

`console.log` 你已经很熟了，但 `console` 上还有一堆更专业的方法：

```javascript
// 文件：code/ch06/07-debugging.js
console.log("最常用的打印");
console.error("输出到 stderr（错误流），常用红色");
console.warn("警告");

// console.table：数组/对象以表格展示，一眼看清结构
console.table([
  { id: 1, name: "小明", age: 20 },
  { id: 2, name: "小红", age: 22 },
]);

// console.dir：查看对象“属性结构”，可控制深度
console.dir({ a: { b: { c: { d: 1 } } } }, { depth: 2 });
```

**逐段拆解：**

- `console.error` / `console.warn`：输出到**错误流（stderr）**而非标准流（stdout），语义上表示“这是错误/警告”，在终端里通常有颜色区分，也便于日志系统分流。
- `console.table`：把“对象数组”渲染成漂亮的表格，调试列表数据时神器。
- `console.dir(obj, { depth })`：专门查看**对象的嵌套结构**，`depth` 控制展开几层，避免深层对象刷屏。

### console.assert：只在“出问题时”出声

```javascript
// 文件：code/ch06/07-debugging.js（续）
console.assert(2 + 2 === 4, "数学坏了"); // 条件为真，静默
console.assert(2 + 2 === 5, "这条会打印"); // 条件为假，打印
```

**逐段拆解：** `console.assert(条件, 消息)` 只有在**条件为假**时才打印消息（且不中断程序）。它适合撒在代码里做“健康检查”——一切正常时完全安静，出问题时才冒出来提醒你。

### 读懂调用栈（stack trace）

调用栈是排错第一线索。你甚至不用等报错，随时可以打印当前栈：

```javascript
// 文件：code/ch06/07-debugging.js（续）
function level3() {
  console.log(new Error("仅用于查看栈").stack);
}
function level2() { level3(); }
function level1() { level2(); }
level1();
```

输出大致是：

```
Error: 仅用于查看栈
    at level3 (.../07-debugging.js:32:15)
    at level2 (.../07-debugging.js:35:3)
    at level1 (.../07-debugging.js:38:3)
    at ...
```

**怎么读：**

- 📌 **栈是“从上往下、由近及远”读的**：最上面一行是“**错误发生的确切位置**”（`level3`），往下每一行是“**是谁调用了它**”——`level3` 被 `level2` 调用，`level2` 被 `level1` 调用。
- 每行的 `文件:行:列` 让你直接跳到源码。看到真实报错时，**第一眼看最上面那行属于你自己代码的位置**，通常问题就在那儿（很多顶部行是库/Node 内部的，可以先跳过）。

### console.time：找性能瓶颈

```javascript
// 文件：code/ch06/07-debugging.js（续）
console.time("耗时统计");
let total = 0;
for (let i = 0; i < 1_000_000; i++) total += i;
console.timeEnd("耗时统计"); // 打印这段跑了多久
```

**逐段拆解：** `console.time(标签)` / `console.timeEnd(标签)` 成对使用，测量之间代码的耗时。标签要一致。这是最轻量的性能测量手段。

### debugger 语句与 Node --inspect

`console.log` 是“撒点”式调试；真正强大的是**断点调试**——让程序停在某一行，逐行执行、随时查看每个变量。

```javascript
// 文件：code/ch06/07-debugging.js（续）
function buggy(x) {
  debugger; // 打开调试器时会停在这一行
  return x * 2;
}
```

**逐段拆解：**

- `debugger;` 语句是一个“断点”。普通 `node 07-debugging.js` 运行时它**被忽略**，不影响程序。
- 但当你用调试器启动时，程序会**停在 `debugger` 这一行**，让你检查此刻 `x` 是多少、逐行往下走（step over / step into）、观察变量变化。

怎么开启调试器？两种常见方式：

1. **命令行 + 浏览器**：运行 `node --inspect-brk 07-debugging.js`（`--inspect-brk` 表示“在第一行就暂停等你”），然后在 Chrome 地址栏打开 `chrome://inspect`，点击 “inspect” 就进入了熟悉的 DevTools 调试界面。
2. **VS Code**：直接在编辑器行号左侧点一下打红点（图形化断点），按 F5 启动调试。这是最省事的方式，不需要写 `debugger` 语句。

💡 **技巧**：`console.log` 适合快速看一两个值；一旦要追踪“变量在循环里怎么一步步变错的”，断点调试的效率是 `console.log` 的好几倍。两者都要会。

## 6.8 优雅降级与错误恢复

好的程序不只是“出错时报错”，而是**出错时依然尽量提供可用的服务**——这叫**优雅降级（graceful degradation）**。核心思路是：区分“致命错误”和“可以兜底的错误”，对后者提供一个安全的替代方案。

### 兜底默认值

```javascript
// 文件：code/ch06/08-graceful.js
function getAvatar(user) {
  try {
    return user.profile.avatarUrl.trim();
  } catch {
    return "/images/default-avatar.png"; // 优雅降级
  }
}
getAvatar({ profile: { avatarUrl: "  /a.png " } }); // "/a.png"
getAvatar({}); // "/images/default-avatar.png"，缺字段也不崩
```

**逐段拆解：** 取头像时，如果 `user` 结构不完整（比如没有 `profile`），读属性会抛 `TypeError`。我们不让它崩，而是返回一张默认头像——用户看到的是“默认头像”而不是“白屏 + 报错”。这就是降级。

### 重试（retry）

有些错误是**临时性**的（网络抖动、服务瞬时繁忙），重试几次很可能就成功了：

```javascript
// 文件：code/ch06/08-graceful.js（续）
function withRetry(fn, maxAttempts = 3) {
  let lastError;
  for (let attempt = 1; attempt <= maxAttempts; attempt++) {
    try {
      return fn(); // 成功就直接返回，不再重试
    } catch (e) {
      lastError = e;
      console.log(`第 ${attempt} 次失败（${e.message}）`);
    }
  }
  throw new Error(`重试 ${maxAttempts} 次仍失败`, { cause: lastError });
}
```

**逐段拆解：**

- 循环最多试 `maxAttempts` 次。一旦 `fn()` 成功，`return` 立刻结束，不再重试。
- 每次失败记下 `lastError` 继续下一轮。
- 全部试完仍失败，就抛出一个汇总错误，并用 `cause` 带上最后一次的真凶——错误链再次派上用场。

⚠️ **陷阱**：不是所有错误都该重试。像“参数校验失败”“余额不足”这种**确定性错误**，重试一百次也一样失败，只会浪费资源。重试只适合**临时性故障**。

### 部分失败不拖垮整体

批量处理时，一条数据坏了不该让整批都失败。给每一项单独 `try/catch`，最后汇总成功与失败：

```javascript
// 文件：code/ch06/08-graceful.js（续）
function processAll(items, handler) {
  const results = { ok: [], failed: [] };
  for (const item of items) {
    try {
      results.ok.push(handler(item));
    } catch (e) {
      results.failed.push({ item, reason: e.message });
    }
  }
  return results;
}

const out = processAll([2, 0, 5, -1], (n) => {
  if (n <= 0) throw new Error("必须为正数");
  return 100 / n;
});
// out.ok = [50, 20]；out.failed = [{item:0,...}, {item:-1,...}]
```

**逐段拆解：** 每个 `item` 独立处理，坏的那条被记进 `failed`，好的照常进 `ok`。最终你得到一份“完整的成败报告”，而不是“因为第二条坏了，全部没处理”。这在导入数据、批量任务里极其实用。

📌 优雅降级的思维核心：**先问“这个错误是不是致命的？有没有可接受的替代方案？”** 能兜底就兜底，不能兜底再让它向上抛。

## 常见坑与最佳实践

- ⚠️ **永远抛 `Error` 对象，别抛字符串/数字**。只有 `Error` 才带 `stack`，才好排查。
- ⚠️ **绝不写空 `catch`（吞错误）**。要么真正处理，要么用 `throw ...; { cause }` 重新抛出。悄悄吞掉的错误会变成最难查的 bug。
- ⚠️ **别在 `finally` 里 `return` 或 `throw`**——它会覆盖 `try` 的结果，制造诡异行为。`finally` 只做清理。
- ⚠️ **`try/catch` 抓不住异步回调里的错误**。`setTimeout`、事件回调里的错误要用 Promise / `async+await+try/catch`（第 8 章详解）。
- ⚠️ **自定义错误类别忘了 `this.name = "类名"`**，否则日志里全显示成 `Error`，分不清类型。
- 📌 **用 `instanceof` 区分错误类型**，对不同错误做不同处理（校验错误提示用户、系统错误重试或上报）。
- 📌 **底层错误往上包装时用 `{ cause }`** 保留原始信息，形成可追溯的错误链。
- 📌 **函数开头用早返回做参数校验**，主逻辑保持在最浅缩进，可读性最佳。
- 💡 **调试先看调用栈最上面属于你代码的那一行**；追踪变量变化用断点调试而非满屏 `console.log`。
- 💡 **区分“可兜底错误”和“致命错误”**：前者优雅降级（默认值/重试/部分失败），后者向上抛并妥善上报。

## 本章小项目：健壮的 JSON 解析器

`JSON.parse` 是日常最常用的函数之一，但它有个“暴脾气”：**输入稍有不合法就抛 `SyntaxError`**，如果你没包 `try/catch`，整个程序就崩了。我们来封装一个**永不崩溃、结果统一、还能校验字段**的 `safeParseJSON`。

需求：

1. 内部用 `JSON.parse`，但捕获异常，**永远不抛错**。
2. 返回**统一结构**：成功 `{ ok: true, value }`，失败 `{ ok: false, error }`。
3. 支持 `defaultValue`：失败时把兜底值也放进结果，方便直接用。
4. 支持简单 **schema 校验**：可要求解析结果必须包含某些字段（`requiredFields`），缺字段就返回自定义的 `ValidationError`。

先看自定义错误类和核心函数：

```javascript
// 文件：code/ch06/safe-json.js
export class ValidationError extends Error {
  constructor(message, missingFields = []) {
    super(message);
    this.name = "ValidationError";
    this.missingFields = missingFields; // 到底缺了哪些字段
  }
}

export function safeParseJSON(text, options = {}) {
  const { defaultValue, requiredFields } = options;

  // 防御性校验：连字符串都不是，直接判失败
  if (typeof text !== "string") {
    const error = new TypeError(`期望字符串，收到 ${typeof text}`);
    return finalizeError(error, defaultValue);
  }

  let value;
  try {
    value = JSON.parse(text); // 可能抛 SyntaxError
  } catch (e) {
    // 把底层 SyntaxError 包进 cause，方便追踪
    const error = new SyntaxError(`JSON 解析失败：${e.message}`, { cause: e });
    return finalizeError(error, defaultValue);
  }

  // schema 校验：要求是对象且包含全部必需字段
  if (requiredFields && requiredFields.length > 0) {
    const missing = validateFields(value, requiredFields);
    if (missing.length > 0) {
      const error = new ValidationError(`缺少必需字段：${missing.join(", ")}`, missing);
      return finalizeError(error, defaultValue);
    }
  }

  return { ok: true, value };
}
```

**逐段拆解：**

- **`ValidationError`**：继承 `Error`，设置 `name`，并携带 `missingFields`——调用方能直接知道缺了哪几个字段。
- **入口防御**：先判断 `text` 是不是字符串。不是就返回 `TypeError` 结果，连 `JSON.parse` 都不用碰。这是早返回思想。
- **核心解析**：`JSON.parse` 包在 `try/catch` 里。捕获到的 `SyntaxError` 被**重新包装**成一条更友好的信息，并用 `{ cause: e }` 保留原始错误——错误链。
- **schema 校验**：如果调用方要求了 `requiredFields`，检查解析结果缺哪些字段，缺了就返回 `ValidationError`。
- **统一返回**：一路上出任何错都走 `finalizeError`（返回 `{ ok:false, error, value? }`），全部通过才返回 `{ ok:true, value }`。**调用方永远只需检查 `result.ok`**，不用写 `try/catch`。

再看两个辅助函数：

```javascript
// 文件：code/ch06/safe-json.js（续）
function finalizeError(error, defaultValue) {
  const result = { ok: false, error };
  if (defaultValue !== undefined) {
    result.value = defaultValue; // 有兜底值就带上
  }
  return result;
}

function validateFields(value, requiredFields) {
  const isPlainObject =
    value !== null && typeof value === "object" && !Array.isArray(value);
  if (!isPlainObject) {
    return [...requiredFields]; // 根本不是对象，全部算缺
  }
  return requiredFields.filter((field) => !(field in value));
}
```

**逐段拆解：**

- `finalizeError` 统一构造失败结果；如果调用方给了 `defaultValue`，就把它放进 `result.value`，这样即便失败也能 `const { value } = safeParseJSON(...)` 直接拿兜底值用。
- `validateFields` 返回“缺失字段”数组：如果解析结果压根不是普通对象（是数组、`null`、或基本类型），视为“所有必需字段都缺”；否则用 `filter` + `in` 运算符挑出不存在的字段。（回顾第 3 章：`"key" in obj` 判断对象是否有某属性。）

来看它处理各种输入的表现（运行 `node safe-json.js`）：

```javascript
// 文件：code/ch06/safe-json.js（演示片段）
safeParseJSON('{"name":"小明","age":20}');
// { ok: true, value: { name: '小明', age: 20 } }

safeParseJSON("{ 这不是 JSON }");
// { ok: false, error: SyntaxError: JSON 解析失败：... }  (error.cause 是原始 SyntaxError)

safeParseJSON("坏数据", { defaultValue: { name: "游客" } });
// { ok: false, error: ..., value: { name: '游客' } }   ← 失败也有兜底值

safeParseJSON('{"name":"小红"}', { requiredFields: ["name", "email"] });
// { ok: false, error: ValidationError, ... }  error.missingFields = ['email']

safeParseJSON('{"name":"小刚","email":"a@b.com"}', { requiredFields: ["name", "email"] });
// { ok: true, value: { name: '小刚', email: 'a@b.com' } }

safeParseJSON(null);
// { ok: false, error: TypeError: 期望字符串，收到 object }
```

**逐段拆解：** 无论你喂给它合法 JSON、非法 JSON、缺字段的对象，还是干脆不是字符串，`safeParseJSON` **从不抛错，永远返回统一结构**。调用方的代码因此极其干净——这正是“把易错的操作封装成安全接口”的价值。

最典型的实战用法（一行搞定“解析 + 兜底”）：

```javascript
// 文件：code/ch06/safe-json.js（续）
const { ok, value, error } = safeParseJSON(process.env.CONFIG ?? "{}", {
  defaultValue: {},
});
if (ok) console.log("配置读取成功：", value);
else console.log("配置读取失败，已用默认值：", value, "/", error.message);
```

**逐段拆解：** 读取环境变量里的配置——即使 `CONFIG` 是坏的 JSON，程序也不会崩，而是退回默认配置 `{}` 继续运行。这就是把前面所有知识点（`try/catch`、自定义错误、`cause`、防御性校验、优雅降级）揉进一个真实可用的小工具。

### 作为模块复用

因为用了 `export`，`safeParseJSON` 可以被别的文件 `import` 进来用：

```javascript
// 文件：code/ch06/demo.js
import { safeParseJSON, ValidationError } from "./safe-json.js";

const result = safeParseJSON('{"name":"小红"}', {
  requiredFields: ["name", "email"],
  defaultValue: { name: "游客", email: "guest@x.com" },
});

if (result.ok) {
  console.log("✅ 解析成功：", result.value);
} else if (result.error instanceof ValidationError) {
  console.log("⚠️ 校验失败，缺字段：", result.error.missingFields, "→ 用默认值：", result.value);
} else {
  console.log("❌ 解析失败：", result.error.message, "→ 用默认值：", result.value);
}
```

**逐段拆解：** 调用方用 `result.error instanceof ValidationError` 区分“校验错误”和“解析错误”，分别给不同提示——这正是 6.4 节自定义错误 + `instanceof` 的实战价值。运行 `node demo.js` 看效果。

### 扩展练习

1. **更强的 schema**：让 `requiredFields` 支持“字段 + 类型”，比如 `{ name: "string", age: "number" }`，类型不匹配也算校验失败（提示：`typeof`）。
2. **嵌套字段校验**：支持 `"profile.email"` 这样的路径，校验嵌套对象里的字段是否存在。
3. **safeStringify**：反过来写一个安全的 `JSON.stringify` 封装——处理循环引用（`JSON.stringify` 遇到循环引用会抛 `TypeError`），捕获后返回统一结构。
4. **收集所有错误**：目前 schema 校验只报“缺失字段”。改造成同时校验类型，把“所有问题”一次性收集进一个 `errors` 数组返回，而不是遇到第一个就停。
5. **加一个 `transform` 选项**：解析成功后允许调用方传入一个函数对 `value` 做后处理（如把字符串日期转成 `Date`），并在 transform 抛错时也纳入统一的错误处理。

## 小结与自测

这一章我们给程序装上了“安全带和气囊”：

- **错误的本质**是被 `throw` 的值；抛错会中断正常流程、向上传播，被 `catch` 接住则恢复。**永远抛 `Error` 对象**，因为它带 `message` / `name` / `stack`。
- **`try / catch / finally`**：`try` 放危险代码，`catch` 处理错误，`finally` 一定执行、只做清理。不关心错误可省略 catch 绑定；处理不了就用 `{ cause }` **重新抛出**，绝不空 `catch` 吞错。
- **内置错误类型**（TypeError / RangeError / ReferenceError / SyntaxError）各有语义，全部继承自 `Error`，可用 `instanceof` 区分。
- **自定义错误类** `extends Error` + 设置 `name` + 携带结构化字段；**`Error.cause`** 把底层错误串成可追溯的错误链。
- **同步 vs 异步**：`try/catch` 只抓同步错误；异步错误靠 Promise 的 `.catch()` 或 `async` 里的 `try/catch`（第 8 章深入）。
- **防御性编程**：参数校验 + 早返回（guard clause）让主逻辑清爽；断言让 bug 在最近处炸出。
- **调试**：`console.log/error/table/dir`、`console.assert`、`console.time`、读懂调用栈（从上往下、由近及远）、`debugger` 语句 + `node --inspect-brk` / VS Code 断点。
- **优雅降级**：默认值兜底、重试临时性故障、部分失败不拖垮整体。
- 小项目 **`safeParseJSON`** 把这些全串了起来：一个永不崩溃、结果统一、可校验字段的健壮解析器。
