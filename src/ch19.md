# 第 19 章 · 装饰器与元编程

## 本章目标

到这一章，你已经能熟练地写类、接口、泛型，也懂了类型收窄。现在我们要学一类"给代码本身加功能"的工具：**装饰器（decorator）**与**元编程（metaprogramming）**。

先建立直觉。假设你有几十个方法，都需要在调用前后打日志、统计耗时、做权限校验。这些需求跟每个方法的**业务逻辑无关**，却又要**散落在每个方法里**——这类"横切各处的关注点"，叫**横切关注点（cross-cutting concern）**，处理它们的编程范式叫**面向切面编程（AOP，Aspect-Oriented Programming）**。装饰器就是 JS/TS 里实现 AOP 的主力工具：你把"打日志"这件事写成一个装饰器，然后往方法上一贴 `@log`，逻辑就自动织入了，业务代码一个字都不用改。

元编程则更底层：它是"用代码去操作代码/对象的行为本身"。`Proxy` 能拦截"读属性、写属性、调用函数"这些最基本的操作；`Reflect` 把这些内部操作暴露成可调用的函数；`Object.defineProperty` 让你精细控制一个属性的读写规则。装饰器的底层实现，很多就是靠这些元编程原语。

学完本章，你将能够：

- 说清楚 **TS5 标准装饰器（Stage 3 / TC39 decorators）** 与旧的 **实验装饰器（experimental decorators）** 的区别，知道自己在用哪一套、为什么。
- 写出类装饰器、方法装饰器、getter/setter、字段、accessor 装饰器，理解**装饰器上下文对象（decorator context）**。
- 用**装饰器工厂（decorator factory）**写出带参数的装饰器。
- 说清楚装饰器的**执行顺序**与组合规则。
- 用 `Reflect`、`Proxy`、`Object.defineProperty` 做拦截、校验、响应式等元编程。
- 理解**元数据（metadata）**与**依赖注入（DI，Dependency Injection）**的思想，并亲手实现一个迷你 DI 容器。

📌 **一句话记住装饰器**：装饰器就是**"一个在定义类/成员时被自动调用、用来观察或改写它的函数"**。你在 `@` 后面写的名字，就是那个函数。

## 运行前提：两套装饰器，你在用哪一套？

这是本章最容易踩坑、也最需要讲清楚的地方，请务必先读完。

JavaScript 的装饰器提案走了很多年。历史上出现过**两套语法/语义完全不同**的装饰器：

| | 旧：实验装饰器 | 新：标准装饰器（本章采用） |
|---|---|---|
| 别名 | experimental / legacy decorators | Stage 3 / TC39 decorators |
| tsconfig 开关 | `"experimentalDecorators": true` | 不需要（是语言标准的一部分） |
| 方法装饰器签名 | `(target, propertyKey, descriptor)` | `(value, context)` |
| 第二参数 | 属性名字符串 / 属性描述符 | **上下文对象** `context`（含 `kind`/`name`/`addInitializer` 等） |
| 参数装饰器 | 支持 | **不支持**（标准装饰器没有参数装饰器） |
| 配套元数据 | `emitDecoratorMetadata` + `reflect-metadata` | `Symbol.metadata` / `context.metadata`（需运行时支持） |
| 现状 | Angular / NestJS / TypeORM 等老框架仍在用 | TS 5.0+ 默认、ES 提案定稿方向 |

⚠️ **两套装饰器的函数签名不兼容，写法几乎完全不同**。为一套写的装饰器，直接搬到另一套下会报错。这就是"迁移注意"的核心：迁移时装饰器函数几乎都要重写。

**本书选择讲标准装饰器（Stage 3）**，原因有三：它是 TS 5.0+ 的默认、是语言的未来方向、而且 `tsx`（底层是 esbuild）默认就支持它，无需任何额外依赖。

但这里有个**关键配置陷阱**，你必须知道：

> 本书的 `code/tsconfig.json` 为了兼容早期章节，开了 `"experimentalDecorators": true`。而 `tsx` 会读取它能找到的 `tsconfig.json` 来决定用哪套装饰器——一旦读到 `experimentalDecorators: true`，`tsx` 就会切回**旧的实验装饰器**编译，本章的标准装饰器代码就会报出 `context is undefined`、`__decorateClass` 之类的错误。

为解决这个冲突，本章在 `code/ch19/` 目录下放了一份**独立的 `tsconfig.json`**（`experimentalDecorators: false`），专门覆盖父级配置。因此本章的运行方式与前面章节略有不同，请二选一：

```bash
# 方式一（推荐）：先进入 ch19 目录，再运行。tsx 会就近读到本目录的 tsconfig。
cd code/ch19
npx tsx 01-decorator-basics.ts

# 方式二：从 code/ 目录运行，用 --tsconfig 显式指定本章配置。
npx tsx --tsconfig ch19/tsconfig.json ch19/01-decorator-basics.ts
```

💡 记住这个规律：**装饰器出问题，九成是 tsconfig 的 `experimentalDecorators` 没对上**。当代码"看起来没错但装饰器行为诡异"时，第一件事就是确认当前生效的到底是哪套装饰器。

`code/ch19/tsconfig.json` 的关键部分：

```jsonc
// 文件：code/ch19/tsconfig.json（节选）
{
  "compilerOptions": {
    "experimentalDecorators": false, // 关掉实验装饰器 → 走标准（Stage 3）装饰器
    "emitDecoratorMetadata": false
  }
}
```

关于 `reflect-metadata`：旧装饰器要"自动读取构造函数参数类型来做依赖注入"，得配合 `emitDecoratorMetadata` 和额外安装的 `reflect-metadata` 库。**本章不需要、也不安装它**——我们的 DI 容器改用"显式声明依赖"的方式，一样开箱即跑。标准装饰器有自己的元数据机制（`Symbol.metadata`），但目前 `tsx` 默认未内置 `Symbol.metadata` 的运行时 polyfill，所以本章也不依赖它。

## 你的第一个装饰器：方法装饰器

我们从最常见的方法装饰器开始——给方法加"调用日志"。

```typescript
// 文件：code/ch19/01-decorator-basics.ts

// 方法装饰器接收两个参数：
//   1. target：被装饰的原始方法（一个函数）。
//   2. context：装饰器上下文对象，描述"这是谁、装在哪、叫什么名字"。
// 它可以返回一个"新函数"来替换原方法。
function logged<This, Args extends any[], Return>(
  target: (this: This, ...args: Args) => Return,
  context: ClassMethodDecoratorContext<This, (this: This, ...args: Args) => Return>,
) {
  const name = String(context.name);

  // 返回的这个新函数将取代原方法被挂到类上。
  function replacement(this: This, ...args: Args): Return {
    console.log(`→ 调用 ${name}(${args.join(", ")})`);
    const result = target.call(this, ...args); // 用 call 保持正确的 this
    console.log(`← ${name} 返回 ${result}`);
    return result;
  }
  return replacement;
}

class Calculator {
  @logged
  add(a: number, b: number): number {
    return a + b;
  }

  @logged
  mul(a: number, b: number): number {
    return a * b;
  }
}

const calc = new Calculator();
console.log("结果:", calc.add(2, 3));
console.log("结果:", calc.mul(4, 5));
```

逐段拆解：

- **`logged` 的两个参数**。这是标准装饰器方法签名的核心：第一个参数 `target` 是**被装饰的原始方法本身**（一个函数），第二个参数 `context` 是**上下文对象**。请对比上一节表格——旧装饰器这里是 `(target, propertyKey, descriptor)`，完全不同。
- **泛型 `<This, Args, Return>`**。这三个类型参数让装饰器对"任意 this、任意参数、任意返回值"的方法都通用，同时保留类型。你不用死记，照抄这个模板即可。
- **`context.name`**。上下文对象告诉你被装饰成员的名字（这里是 `"add"` / `"mul"`）。它可能是字符串或 `symbol`，所以用 `String(...)` 转一下。
- **返回一个新函数 `replacement`**。方法装饰器的返回值如果是函数，就会**替换掉原方法**。于是每次调用 `calc.add(...)`，实际跑的是 `replacement`——它先打印"调用"，再用 `target.call(this, ...args)` 转发给真正的 `add`，最后打印"返回"。
- **`target.call(this, ...args)` 里的 `this`**。为什么不直接 `target(...args)`？因为方法里可能用到 `this`（虽然本例没用到）。用 `call` 把当前实例作为 `this` 传进去，才能保证被包裹的方法行为不变。这是写方法装饰器的**必备习惯**。

运行：

```bash
cd code/ch19 && npx tsx 01-decorator-basics.ts
```

输出会是：调用 `add(2, 3)` → 打印进入、返回、结果 5；`mul(4, 5)` 同理。**注意：业务方法 `add`/`mul` 里我们一行日志都没写，日志能力完全是 `@logged` "织入"的。** 这就是 AOP 的价值：横切关注点（日志）与业务逻辑彻底分离。

## 装饰器工厂：给装饰器传参数

上面的 `@logged` 是"光杆装饰器"，不带括号。但很多时候我们希望装饰器**可配置**，比如 `@measure("求和")` 自定义标签、`@retry(3)` 指定重试次数。这就要用**装饰器工厂（decorator factory）**：它本身是个函数，你先调用它传参，它**再返回真正的装饰器**。

```typescript
// 文件：code/ch19/02-decorator-factory.ts

// @measure() 用于给方法计时；参数 label 用来自定义日志前缀。
function measure(label?: string) {
  // 外层函数接收"配置参数"，返回真正的装饰器。
  return function <This, Args extends any[], Return>(
    target: (this: This, ...args: Args) => Return,
    context: ClassMethodDecoratorContext,
  ) {
    const name = label ?? String(context.name);
    return function (this: This, ...args: Args): Return {
      const start = performance.now();
      const result = target.call(this, ...args);
      const cost = (performance.now() - start).toFixed(3);
      console.log(`⏱ ${name} 耗时 ${cost}ms`);
      return result;
    };
  };
}

// @cached() 缓存"参数可序列化"的方法结果，避免重复计算。
function cached() {
  return function <This, Args extends any[], Return>(
    target: (this: This, ...args: Args) => Return,
    _context: ClassMethodDecoratorContext,
  ) {
    const store = new Map<string, Return>();
    return function (this: This, ...args: Args): Return {
      const key = JSON.stringify(args);
      if (store.has(key)) {
        console.log("✔ 命中缓存", key);
        return store.get(key)!;
      }
      const result = target.call(this, ...args);
      store.set(key, result);
      return result;
    };
  };
}

class Report {
  // @measure 计时；@cached 缓存结果。两个装饰器叠加，由内向外生效。
  @measure("统计求和")
  @cached()
  sumTo(n: number): number {
    let total = 0;
    for (let i = 1; i <= n; i++) total += i;
    return total;
  }
}

const r = new Report();
console.log("sumTo(1000000) =", r.sumTo(1_000_000));
console.log("sumTo(1000000) =", r.sumTo(1_000_000)); // 第二次直接命中缓存
```

逐段拆解：

- **两层函数结构**。`measure` 本身**不是**装饰器，它是"生产装饰器的工厂"。你写 `@measure("统计求和")` 时，JS 先执行 `measure("统计求和")`，拿到里面 `return` 的那个内层函数，那个内层函数才是真正贴到方法上的装饰器。所以带参数的装饰器**必须带括号** `@measure(...)`；不带括号的 `@measure` 会把函数本身当装饰器用，报错。
- **`cached` 的闭包缓存**。注意 `const store = new Map(...)` 定义在装饰器函数体内、返回的包裹函数外面。这样每个被 `@cached()` 装饰的方法都拥有**一个独立、长期存活的缓存 Map**（靠闭包保住）。`JSON.stringify(args)` 把参数序列化成缓存键——⚠️ 这招只适用于参数能被 JSON 序列化的场景（数字、字符串、简单对象），遇到函数、循环引用会失效。
- **两个装饰器叠加**。`sumTo` 上同时贴了 `@measure` 和 `@cached`。它们的组合顺序很重要，下一节专门讲。运行后你会看到：第一次调用真正跑了循环、耗时若干毫秒；第二次直接"命中缓存"，耗时几乎为 0。

运行：

```bash
npx tsx 02-decorator-factory.ts
```

📌 **区分三个概念**：`measure` 是**装饰器工厂**；`measure("统计求和")` 的返回值是**装饰器**；装饰器再返回的函数是**替换后的方法**。三层，别混。

## 五种位置：类、方法、getter、字段、accessor 装饰器

标准装饰器能贴在五种位置上，每种位置对应一个不同的**上下文类型**，`target` 的含义也不同。下面一次性演示全部。

```typescript
// 文件：code/ch19/03-all-decorator-kinds.ts

// 1) 类装饰器：接收类本身与 ClassDecoratorContext。
//    这里返回一个"子类"来替换原类，给每个实例自动打上创建时间戳。
function stamped<T extends new (...args: any[]) => object>(
  value: T,
  context: ClassDecoratorContext,
) {
  console.log("[类装饰器] 处理类:", context.name);
  return class extends value {
    createdAt = new Date().toISOString();
  };
}

// 2) 字段装饰器：value 恒为 undefined，可返回一个"初始化器"函数，
//    它接收字段的初始值，返回加工后的新值。
function trimmed(_value: undefined, context: ClassFieldDecoratorContext<unknown, string>) {
  return function (initial: string) {
    console.log("[字段装饰器] 初始化", String(context.name));
    return initial.trim();
  };
}

// 3) accessor 装饰器：装饰用 accessor 关键字声明的"自动访问器"。
//    value 是 { get, set }，可返回改写后的 { get, set, init }。
function loggedAccessor<This, V>(
  value: ClassAccessorDecoratorTarget<This, V>,
  context: ClassAccessorDecoratorContext<This, V>,
): ClassAccessorDecoratorResult<This, V> {
  const name = String(context.name);
  return {
    get(this: This) {
      return value.get.call(this);
    },
    set(this: This, newValue: V) {
      console.log(`[accessor 装饰器] 写入 ${name} =`, newValue);
      value.set.call(this, newValue);
    },
  };
}

// 4) getter 装饰器：装饰 get 访问器，target 是 getter 函数本身。
function frozen<This, Return>(
  target: (this: This) => Return,
  context: ClassGetterDecoratorContext<This, Return>,
) {
  return function (this: This): Return {
    const v = target.call(this);
    console.log(`[getter 装饰器] 读取 ${String(context.name)} =`, v);
    return v;
  };
}

@stamped
class Person {
  // ⚠️ 字段装饰器加工的是"字段初始值"，所以这里给一个带空格的初始值。
  //    如果 name 只在构造函数里赋值、没有初始值，@trimmed 的初始化器拿到的会是 undefined。
  @trimmed
  name = "  Alice  ";

  @loggedAccessor
  accessor score = 0;

  @frozen
  get summary(): string {
    return `${this.name}(${this.score})`;
  }
}

const p = new Person();
console.log("name = [" + p.name + "]"); // 已被 trim → [Alice]
p.score = 95; // 触发 accessor 的 set 日志
console.log("summary =", p.summary); // 触发 getter 日志
console.log("createdAt =", (p as any).createdAt); // 由 @stamped 子类注入
```

逐段拆解：

- **类装饰器 `@stamped`**。它的 `value` 是类本身，`context.kind` 为 `"class"`。这里我们返回了一个**匿名子类** `class extends value { createdAt = ... }`——于是被装饰后的 `Person` 实际上是这个子类，每个实例都自动多了 `createdAt` 字段。这就是类装饰器最强大的用法：**替换/增强整个类**。若你返回 `undefined`，则保持原类不变。
- **字段装饰器 `@trimmed`**。它的第一个参数 `value` **永远是 `undefined`**（字段装饰器拿不到字段值本身）。它可以返回一个**初始化器函数**，该函数接收字段的**初始值**、返回加工后的值。这里把 `"  Alice  "` 的两端空格去掉。⚠️ 划重点：字段装饰器加工的是**初始值**（`= "..."` 那部分），不是构造函数里赋的值。所以示例里我们特意给 `name` 一个带空格的初始值——注释里也提醒了，如果字段没有初始值、只在构造函数里赋值，`@trimmed` 拿到的会是 `undefined`。
- **accessor 装饰器 `@loggedAccessor`**。注意字段前的 `accessor` 关键字——这是标准装饰器**新引入**的语法，声明一个"自动访问器"：`accessor score = 0` 背后其实是一对 getter/setter 加一个私有存储。装饰器的 `value` 是 `{ get, set }`，我们返回改写后的 `{ get, set }`，在 `set` 里插入日志。这是**做响应式、校验属性写入的标准姿势**。
- **getter 装饰器 `@frozen`**。装饰 `get summary()`，`target` 是 getter 函数本身。我们包裹它，在读取时打日志。setter 装饰器与之对称，用 `ClassSetterDecoratorContext`。

运行后你会看到：`name` 被 trim 成 `[Alice]`；给 `score` 赋值触发写日志；读 `summary` 触发读日志；`createdAt` 由 `@stamped` 注入。

```bash
npx tsx 03-all-decorator-kinds.ts
```

💡 **速查**：类 → `ClassDecoratorContext`；方法 → `ClassMethodDecoratorContext`；getter/setter → `ClassGetter/SetterDecoratorContext`；字段 → `ClassFieldDecoratorContext`；accessor → `ClassAccessorDecoratorContext`。它们都有公共成员 `kind`、`name`、`static`、`private`、`addInitializer`。

## 执行顺序、组合与 addInitializer

多个装饰器叠在一起时，顺序容易让人困惑。规则其实很清晰，我们用实验直接看。

```typescript
// 文件：code/ch19/04-order-and-initializer.ts

function trace(tag: string) {
  console.log(`① 求值工厂 ${tag}`); // 工厂本身：自上而下先被"调用"
  return function (target: any, context: ClassMethodDecoratorContext | ClassDecoratorContext) {
    console.log(`② 应用装饰器 ${tag}`); // 装饰器应用：由内向外（自下而上）
    return target;
  };
}

@trace("类@外层")
@trace("类@内层")
class Demo {
  @trace("方法@上")
  @trace("方法@下")
  run() {}
}
```

逐段拆解与运行结果：

- **第一阶段：工厂求值，自上而下**。`@trace(...)` 是装饰器工厂调用，JS 从上到下依次执行这些"括号里的调用"来得到装饰器。所以先打印 `① 求值工厂 类@外层`、`类@内层`，再是方法的 `方法@上`、`方法@下`。
- **第二阶段：装饰器应用，由内向外**。真正把装饰器"贴上去"时，顺序**反过来**——离被装饰目标最近的（最内层/最下面的）先应用。所以打印 `② 应用装饰器 方法@下` 在 `方法@上` 之前，`类@内层` 在 `类@外层` 之前。
- **为什么"由内向外"很重要**？因为装饰器是层层包裹的。回看上一节的 `@measure` + `@cached`：`@cached`（下/内）先应用，先包住原方法；`@measure`（上/外）后应用，包在最外层。于是调用时先进 `measure` 计时、再进 `cached` 查缓存——`measure` 测到的耗时**包含**了缓存查找。顺序换一下，语义就变了。📌 **记法：装饰器像"洋葱"，写在上面的包在外层，先执行到。**

接着看 `addInitializer`——上下文对象提供的一个钩子，让你在**实例构造时**运行一段代码：

```typescript
// 文件：code/ch19/04-order-and-initializer.ts（续）

// addInitializer：在"实例被构造时"运行一段代码，常用于把方法/字段注册到某处。
function autoBind(target: (...args: any[]) => any, context: ClassMethodDecoratorContext) {
  // 把方法绑定到实例，避免 this 丢失。
  context.addInitializer(function (this: any) {
    this[context.name] = target.bind(this);
  });
}

class Button {
  label = "OK";
  @autoBind
  click() {
    console.log("点击了", this.label);
  }
}

console.log("--- 顺序演示结束，开始构造实例 ---");
const b = new Button();
const handler = b.click; // 把方法"拆下来"单独调用
handler(); // 因为 autoBind 已绑定 this，这里仍能拿到 label
```

逐段拆解：

- **`context.addInitializer(fn)`**。它注册的 `fn` 会在**每次 `new Button()` 构造实例时**执行，且 `this` 指向新实例。这是标准装饰器替代旧写法的关键能力之一。
- **`@autoBind` 解决的经典问题**。JS 里把方法"拆下来"单独调用（`const handler = b.click; handler()`）会丢失 `this`，`this.label` 就变 `undefined` 甚至报错——React 类组件时代人人踩过。`@autoBind` 在实例初始化时用 `target.bind(this)` 把方法固定绑到实例，再挂回 `this[context.name]`，于是 `handler()` 依然能正确访问 `this.label`。

## 元编程基石之一：Object.defineProperty 与 Reflect

装饰器的底层，本质是在操作对象和属性。要理解它们、也要能自己做元编程，得先认识两个原语。

```typescript
// 文件：code/ch19/05-reflect-and-defineproperty.ts

// 1) Object.defineProperty：精细控制一个属性的"特性"（descriptor）。
const config: Record<string, unknown> = {};
Object.defineProperty(config, "version", {
  value: "1.0.0",
  writable: false, // 只读
  enumerable: true, // 可被 for...in / Object.keys 枚举
  configurable: false, // 不可删除、不可再次重定义
});
console.log("version =", config.version);
try {
  // 模块顶层代码在 ESM 里默认是严格模式，给只读属性赋值会抛错。
  config.version = "9.9.9";
} catch (e) {
  console.log("修改只读属性被拒绝:", (e as Error).message);
}
console.log("尝试修改后 version =", config.version); // 仍是 1.0.0

// 定义一个"计算属性"（访问器属性）
const rect = { width: 3, height: 4 };
Object.defineProperty(rect, "area", {
  get() {
    return (this as typeof rect).width * (this as typeof rect).height;
  },
  enumerable: true,
});
console.log("area =", (rect as any).area); // 12
```

逐段拆解：

- **属性描述符（property descriptor）**。我们平时写 `obj.x = 1` 得到的属性，默认是可写、可枚举、可配置的。`Object.defineProperty` 让你逐项控制：`writable`（能不能改）、`enumerable`（会不会被 `Object.keys`/`for...in` 列出）、`configurable`（能不能删除或重定义）。这里把 `version` 设成只读，再改它就会（在严格模式下）抛错。
- **访问器属性（accessor property）**。用 `get`/`set` 定义的属性不存实际值，而是"读的时候现算"。`area` 每次读取都用当前的 `width * height` 算出来。这正是很多"响应式/计算属性"的底层机制。

第二部分是 `Reflect`——它把"语言内部操作"变成一组普通函数：

```typescript
// 文件：code/ch19/05-reflect-and-defineproperty.ts（续）

// 2) Reflect：把"语言内部操作"变成可调用的函数，与 Proxy 天生一对。
const user = { name: "Alice", age: 30 };

console.log("Reflect.get:", Reflect.get(user, "name")); // 等价于 user.name
Reflect.set(user, "age", 31); // 等价于 user.age = 31
console.log("修改后 age:", user.age);
console.log("Reflect.has:", Reflect.has(user, "name")); // 等价于 "name" in user
console.log("Reflect.ownKeys:", Reflect.ownKeys(user)); // 拿到自有键（含 Symbol）
Reflect.deleteProperty(user, "age"); // 等价于 delete user.age
console.log("删除后:", user);

// Reflect.get / set 支持第三个参数 receiver，用于在继承链上正确传递 this。
const base = {
  _x: 1,
  get x() {
    return this._x;
  },
};
const child = { _x: 42 };
console.log("带 receiver 的 get:", Reflect.get(base, "x", child)); // 42，用 child 当 this
```

逐段拆解：

- **`Reflect` 是干什么的**？它把 `obj.name`、`obj.x = 1`、`"k" in obj`、`delete obj.k`、`Object.keys` 这些"语法操作"改写成**函数调用**：`Reflect.get`、`Reflect.set`、`Reflect.has`、`Reflect.deleteProperty`、`Reflect.ownKeys`。乍看没意义——`Reflect.get(user, "name")` 不就是 `user.name` 吗？关键在于：**函数可以被转发、被组合、被作为默认行为调用**，而语法不行。这在写 `Proxy` 时至关重要（下一节马上用到）。
- **`receiver` 参数**。`Reflect.get(base, "x", child)` 的第三个参数指定 getter 里 `this` 是谁。这里读 `base.x` 但让 `this` 指向 `child`，于是返回 `child._x = 42`。在代理继承链时，正确传 `receiver` 才能保证 getter/setter 里的 `this` 指向对的对象。

## 元编程基石之二：Proxy

`Proxy` 是元编程的重头戏：它包裹一个对象，**拦截**对该对象的各种操作（读、写、删、`in`、函数调用……），让你在中间插入自己的逻辑。Vue 3 的响应式系统、各种 ORM、Mock 库，底层都是它。

```typescript
// 文件：code/ch19/06-proxy.ts

// 1) 日志代理：任何读写都打印出来。
const target = { a: 1, b: 2 };
const logged = new Proxy(target, {
  get(obj, key, receiver) {
    console.log("读取", String(key));
    return Reflect.get(obj, key, receiver); // 用 Reflect 转发默认行为
  },
  set(obj, key, value, receiver) {
    console.log("写入", String(key), "=", value);
    return Reflect.set(obj, key, value, receiver);
  },
});
logged.a; // 触发 get
logged.b = 20; // 触发 set
```

逐段拆解：

- **`new Proxy(target, handler)`**。第一个参数是被代理的原对象；第二个是**处理器对象（handler）**，里面每个方法叫一个**陷阱（trap）**。`get` trap 拦截读属性，`set` trap 拦截写属性。此外还有 `has`（拦截 `in`）、`deleteProperty`（拦截 `delete`）、`apply`（拦截函数调用）、`construct`（拦截 `new`）等十几种。
- **为什么 trap 里用 `Reflect`**？看 `get` trap：我们打印日志后，得**执行默认的读取行为**返回真实值——`Reflect.get(obj, key, receiver)` 正是"默认读取"的函数形式，还能把 `receiver` 正确传下去。这就是上一节说的"Reflect 与 Proxy 天生一对"：**trap 负责插入自定义逻辑，`Reflect` 负责补上默认行为。**

接下来两个更实用的例子——校验与响应式：

```typescript
// 文件：code/ch19/06-proxy.ts（续）

// 2) 校验代理：拦截 set，拒绝非法值。
interface Account {
  balance: number;
}
function createAccount(): Account {
  return new Proxy({ balance: 0 } as Account, {
    set(obj, key, value) {
      if (key === "balance" && (typeof value !== "number" || value < 0)) {
        throw new RangeError("余额必须是非负数");
      }
      return Reflect.set(obj, key, value);
    },
  });
}
const acc = createAccount();
acc.balance = 100;
console.log("余额:", acc.balance);
try {
  acc.balance = -5;
} catch (e) {
  console.log("被拦截:", (e as Error).message);
}

// 3) 迷你响应式：属性变化时自动触发回调（Vue 响应式的雏形）。
function reactive<T extends object>(obj: T, onChange: (key: string) => void): T {
  return new Proxy(obj, {
    set(o, key, value, receiver) {
      const ok = Reflect.set(o, key, value, receiver);
      onChange(String(key));
      return ok;
    },
  });
}
const state = reactive({ count: 0 }, (key) => {
  console.log(`state.${key} 变了，重新渲染！当前 count =`, state.count);
});
state.count++;
state.count++;
```

逐段拆解：

- **校验代理**。`set` trap 在写入前检查值：`balance` 必须是非负数，否则抛 `RangeError`。这样非法状态**根本写不进对象**——比在每个赋值点手写 `if` 检查优雅得多，且无法绕过。
- **迷你响应式**。`reactive` 包一层 Proxy，每次 `set` 成功后调用 `onChange` 回调。于是 `state.count++`（读一次、写一次）触发回调，你可以在回调里"重新渲染 UI"。这正是 Vue 3 响应式的雏形——真实实现更复杂（依赖收集、嵌套代理等），但内核就是这个 `set` trap。

```bash
npx tsx 06-proxy.ts
```

📌 **Proxy vs 装饰器**：装饰器在"定义时"一次性改写类/成员；Proxy 在"运行时"逐次拦截每个操作。前者静态、后者动态，常常搭配使用。

## 常见坑与最佳实践

- ⚠️ **最大的坑：搞错了装饰器套别。** 本章代码在 `experimentalDecorators: true` 的配置下会用旧装饰器编译，导致 `context` 为 `undefined`。务必按本章"运行前提"里的方式运行（进 `ch19` 目录或加 `--tsconfig`）。反过来，用旧装饰器的老项目（Angular/NestJS）也不能直接套本章写法。
- ⚠️ **方法装饰器返回的包裹函数一定要用 `target.call(this, ...args)`**，不能 `target(...args)`——否则被装饰方法里的 `this` 会丢。
- ⚠️ **字段装饰器加工的是"初始值"**，且第一个参数恒为 `undefined`。想响应"每次写入"，用 `accessor` 装饰器而不是字段装饰器。
- ⚠️ **标准装饰器没有参数装饰器**。如果你在教程里看到装饰方法参数的写法（`method(@Inject() dep)`），那是旧实验装饰器，本章语义下不可用。
- 💡 **带参数用工厂、不带参数用光杆**，但要注意：`@foo` 和 `@foo()` 是两回事。工厂一定要带括号调用。
- 💡 **组合装饰器记住"洋葱模型"**：写在上面的包在外层、先执行。顺序影响语义时要想清楚。
- 💡 **Proxy 的 trap 里坚持用 `Reflect` 转发默认行为**，尤其别忘了传 `receiver`，否则继承链上的 getter/setter 的 `this` 会错。
- 💡 **元数据别急着上 `reflect-metadata`**。标准装饰器时代它已不是必需；很多需求（如本章 DI）用"显式声明"就能干净解决，依赖更少、更好懂。

## 本章小项目：简易依赖注入容器

**依赖注入（DI）**的思想：一个类不自己 `new` 它需要的依赖，而是"声明我需要什么"，由一个**容器（container）**负责创建并"注入"进来。好处是解耦、易测试（测试时可注入假的依赖）、单例复用。Angular、NestJS 的核心就是 DI 容器。

我们用**类装饰器 `@Injectable()`** 标记可注入的类、声明它的依赖，再实现一个 `Container` 负责 `resolve`（递归实例化 + 缓存单例）。

> 为什么显式声明依赖，而不是自动读构造函数参数类型？自动读取要靠旧装饰器 + `reflect-metadata`。为了在标准装饰器 + `tsx` 下开箱即跑、零额外依赖，我们让 `@Injectable({ deps: [...] })` 显式列出依赖——这也是很多真实框架支持的写法，且更直观。

```typescript
// 文件：code/ch19/di-container.ts（核心部分）

type Ctor<T = unknown> = new (...args: any[]) => T;

interface InjectableOptions {
  deps?: Ctor[]; // 该类构造函数需要的依赖（按顺序）
}

// 全局注册表：记录"哪些类可注入"以及"各自的依赖列表"。
const INJECTABLE = new Set<Ctor>();
const DEPS = new WeakMap<Ctor, Ctor[]>();

// 类装饰器工厂：接收配置，返回真正的类装饰器。
// 用泛型 C 让"接收的类"和"返回的类"是同一个具体类型，
// 满足标准装饰器"返回值类型必须与被装饰类兼容"的要求。
export function Injectable(options: InjectableOptions = {}) {
  return function <C extends Ctor>(value: C, context: ClassDecoratorContext): C {
    if (context.kind !== "class") {
      throw new Error("@Injectable 只能用在类上");
    }
    INJECTABLE.add(value);
    DEPS.set(value, options.deps ?? []);
    return value; // 不改写类本身，原样返回
  };
}
```

逐段拆解：

- **`Ctor` 类型**。`new (...args: any[]) => T` 表示"任何可 `new` 的类"，我们用它作为容器里的"令牌（token）"——用类本身当键去查它对应的实例。
- **`@Injectable` 做的事**。它是个装饰器工厂：接收 `{ deps }` 配置，返回真正的类装饰器。装饰器里把类登记进 `INJECTABLE` 集合，把依赖列表存进 `DEPS`（用 `WeakMap`，类被回收时元数据自动释放，防内存泄漏）。它**不改写类**，只是"打标签 + 记账"，所以原样 `return value`。这是类装饰器的另一种常见用法——**注册（registration）**，不改行为、只登记信息。

接着是容器本身：

```typescript
// 文件：code/ch19/di-container.ts（Container 部分）

export class Container {
  private singletons = new Map<Ctor, unknown>(); // 类 → 实例（单例缓存）
  private resolving = new Set<Ctor>(); // 正在解析中的类，用来检测循环依赖

  register<T>(token: Ctor<T>): void {
    INJECTABLE.add(token);
    if (!DEPS.has(token)) DEPS.set(token, []);
  }

  resolve<T>(token: Ctor<T>): T {
    // 1) 命中缓存 → 直接返回同一个实例（单例语义）。
    if (this.singletons.has(token)) {
      return this.singletons.get(token) as T;
    }
    // 2) 没标记也没注册 → 报错，避免"凭空造对象"。
    if (!INJECTABLE.has(token)) {
      throw new Error(`无法解析 ${token.name}：它没有被 @Injectable 标记或 register`);
    }
    // 3) 循环依赖检测。
    if (this.resolving.has(token)) {
      throw new Error(`检测到循环依赖：${token.name}`);
    }
    this.resolving.add(token);

    // 4) 递归解析每个依赖，再把它们按顺序传给构造函数。
    const deps = DEPS.get(token) ?? [];
    const args = deps.map((dep) => this.resolve(dep));
    const instance = new token(...args);

    this.resolving.delete(token);
    this.singletons.set(token, instance); // 缓存为单例
    return instance as T;
  }
}
```

逐段拆解 `resolve` 的四步（这是整个容器的灵魂）：

1. **查缓存**。如果这个类已经造过实例，直接返回同一个——这就是"单例"：整个应用里 `Logger` 只有一个。
2. **校验**。没登记过的类不给造，抛出清晰的错误，避免误用。
3. **循环依赖检测**。用 `resolving` 集合记录"正在解析途中"的类。如果解析 A 时又要解析 A，说明 A→B→A 形成了环，直接报错——否则会无限递归爆栈。
4. **递归 + 注入**。取出依赖列表，对每个依赖**递归调用 `resolve`**（依赖自己可能还有依赖，层层往下），拿到所有依赖实例后 `new token(...args)`。这就是"注入"：构造函数需要的东西，全由容器备齐塞进去。造好后存进缓存。

演示（三层依赖 `UserService → Database → Logger`）：

```typescript
// 文件：code/ch19/di-container.ts（演示部分）

@Injectable()
class Logger {
  log(message: string): void {
    console.log(`[LOG] ${message}`);
  }
}

@Injectable({ deps: [Logger] })
class Database {
  constructor(private logger: Logger) {
    this.logger.log("数据库连接已建立");
  }
  query(sql: string): string[] {
    this.logger.log(`执行查询: ${sql}`);
    return ["row1", "row2"];
  }
}

@Injectable({ deps: [Logger, Database] })
class UserService {
  constructor(
    private logger: Logger,
    private db: Database,
  ) {}
  findAll(): string[] {
    this.logger.log("UserService.findAll 被调用");
    return this.db.query("SELECT * FROM users");
  }
}

const container = new Container();
const svc = container.resolve(UserService);
console.log("查询结果:", svc.findAll());

const svc2 = container.resolve(UserService);
console.log("UserService 是单例吗?", svc === svc2); // true

const logger = container.resolve(Logger);
const db = container.resolve(Database);
console.log("Logger 被复用吗?", logger === (db as any)["logger"]); // true
```

逐段拆解：

- **依赖声明与构造函数一一对应**。`UserService` 声明 `deps: [Logger, Database]`，构造函数就按这个顺序接收 `(logger, db)`。⚠️ 顺序必须对齐——`deps` 数组的顺序就是注入实参的顺序。
- **一行 `resolve(UserService)` 触发整条链**。容器发现 `UserService` 依赖 `Logger` 和 `Database`；`Database` 又依赖 `Logger`。容器递归把它们全造好、按序注入。你只写了一行，三个对象自动就位。
- **单例被验证**。`svc === svc2` 为 `true`，说明二次解析拿到同一实例。而 `UserService` 里的 `logger` 与容器直接 `resolve(Logger)` 得到的**是同一个对象**（`Logger 被复用吗? true`）——因为 `Logger` 在缓存里只造一次，被所有依赖它的类共享。这正是 DI 容器"共享单例"的价值。

**扩展练习**（自己动手，加深理解）：

1. **瞬态（transient）作用域**：给 `@Injectable({ scope: "transient" })` 加一个选项，让这类每次 `resolve` 都返回**新实例**（不缓存）。提示：在 `resolve` 里根据 scope 决定是否走 `singletons`。

4. **改造成方法拦截**：结合本章的 `@measure`，给容器解析出的实例的每个方法自动加上计时（提示：用 `Proxy` 包裹返回的实例，在 `get` trap 里把方法换成计时版本）。
