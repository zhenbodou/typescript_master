# 第 15 章 · 函数类型、类与枚举

## 本章目标

第 13、14 章我们给 TypeScript 打好了地基：变量与函数的类型注解、接口（interface）、类型别名（type alias）、联合（union）与交叉（intersection）类型。这一章我们把三块"日常写业务最离不开"的能力补齐：

- **把函数本身当成一等公民来标注类型**。函数类型表达式、调用签名（call signature）、构造签名（construct signature）、`this` 参数、类型谓词（type predicate）——尤其是让新手又爱又怕的**函数重载（overload）**，我们要把它一次讲清、且**不讲错**。
- **在 TS 里写类（class）**。第 4 章你已经会 JS 的 `class` 与原型了，这一章补上 TS 特有的那层"类型盔甲"：字段类型注解、访问修饰符 `public/private/protected`、`readonly`、**参数属性**这个省事的简写、`static`、getter/setter、**抽象类 `abstract`**、**类实现接口 `implements`**，以及一个高频面试题——`#` 私有字段和 `private` 到底差在哪。
- 分清**类作为"类型"和作为"值"的双重身份**。
- **枚举（enum）**：数字枚举、字符串枚举、`const enum`，重点讲它**编译成 JS 之后长什么样**（运行时形态）以及由此带来的坑；再讨论现代项目里越来越流行的替代方案——**联合字面量类型 + `as const` 对象**，以及两者怎么取舍。
- 最后用一个**类型安全的有限状态机（Finite State Machine）**把 class + 联合类型 + 可辨识联合串起来实战。

本章的 TypeScript 示例都用 `npx tsx ch15/文件.ts` 运行。我们从函数类型开始。

## 15.1 函数类型深入

### 15.1.1 函数类型表达式

第 13 章我们已经给函数写过类型，但那是"内联"在函数定义上的。真正把**函数的类型抽出来复用**，靠的是**函数类型表达式（function type expression）**，语法是 `(参数: 类型) => 返回类型`：

```typescript
// 文件：code/ch15/01-func-types.ts
type NumberMapper = (n: number) => number;

const double: NumberMapper = (n) => n * 2; // n 的类型由 NumberMapper 推出，无需再写

function mapNumbers(arr: number[], fn: NumberMapper): number[] {
  return arr.map(fn);
}
console.log(mapNumbers([1, 2, 3], double)); // [ 2, 4, 6 ]
```

**逐段拆解：**

- `type NumberMapper = (n: number) => number` 起了一个名字，表示"一个接收 `number`、返回 `number` 的函数"。注意箭头 `=>` 在**类型位置**表示"返回类型"，别和箭头函数的 `=>` 混淆——它们长得一样但一个在类型里、一个在值里。
- `const double: NumberMapper = (n) => n * 2`：因为左边已经写了类型，右边箭头函数的参数 `n` 就**不用再标类型**了，TS 会从 `NumberMapper` 反推出 `n: number`。这叫**上下文类型推断（contextual typing）**——类型信息"从左往右流"。
- `mapNumbers` 的第二个参数 `fn: NumberMapper` 说明它要一个"数字映射函数"。把 `double` 传进去，类型严丝合缝。

📌 函数类型表达式让你像传数据一样传"行为"，且行为的形状被类型钉死。回调、事件处理器、策略模式，全靠它。

### 15.1.2 调用签名：既能被调用，又带属性

JS 里函数也是对象，可以往上挂属性（比如 `fn.count`）。普通的函数类型表达式没法同时描述"能被调用"和"带哪些属性"。这时用**调用签名（call signature）**，写在对象类型里：

```typescript
// 文件：code/ch15/01-func-types.ts（续）
type Counter = {
  (): number; // 调用签名：像函数一样被调用，返回 number
  count: number; // 同时它还带一个 count 属性
  reset(): void;
};

function makeCounter(): Counter {
  const fn = (() => {
    fn.count += 1;
    return fn.count;
  }) as Counter;
  fn.count = 0;
  fn.reset = () => {
    fn.count = 0;
  };
  return fn;
}

const tick = makeCounter();
console.log(tick(), tick(), tick()); // 1 2 3
console.log("count =", tick.count); // 3
tick.reset();
console.log("reset 后 count =", tick.count); // 0
```

**逐段拆解：**

- 对象类型里那行 `(): number` **没有属性名**，只有参数列表和返回类型——这就是调用签名，意思是"这个对象可以被当函数调用"。
- 同一个类型里又写了 `count: number` 和 `reset(): void`，于是 `Counter` 描述的是"一个能被调用、还带 `count`/`reset` 的函数对象"。
- `makeCounter` 里用 `as Counter` 做了一次类型断言（第 13 章学过），因为我们要先创建函数、再给它挂属性，中间那一刻它还不满足 `Counter`，用断言告诉 TS"信我，最终它会齐全"。

💡 什么时候需要调用签名？当你要描述"带元数据的函数"——比如一个带 `.displayName` 的组件、一个带 `.cancel()` 的防抖函数。日常大多数函数用 15.1.1 的箭头写法就够了。

### 15.1.3 `this` 参数类型

在对象方法里，`this` 指向谁取决于**调用方式**（第 4 章讲过 JS 的 `this` 之坑）。TS 允许你在参数列表**最前面**放一个名为 `this` 的"假参数"，专门标注 `this` 的类型：

```typescript
// 文件：code/ch15/01-func-types.ts（续）
interface Button {
  label: string;
  onClick(this: Button, times: number): void;
}

const btn: Button = {
  label: "提交",
  onClick(this: Button, times: number) {
    console.log(`按钮「${this.label}」被点了 ${times} 次`);
  },
};
btn.onClick(3); // 按钮「提交」被点了 3 次
```

**逐段拆解：**

- `onClick(this: Button, times: number)` 里的 `this: Button` 是**纯类型标注**，编译成 JS 后**不会**变成真实参数。所以调用时你只传 `times`（即 `btn.onClick(3)`），`this` 那一格是给类型检查看的。
- 有了它，函数体里 `this.label` 就是类型安全的；而且如果你把 `onClick` 拆下来单独调用（`const f = btn.onClick; f(3)`），TS 会报错提醒你 `this` 上下文丢了。

⚠️ 别把 `this` 参数和普通参数搞混：`this` 必须是**第一个**参数，且它不占实际调用的参数位。

### 15.1.4 构造签名简介

调用签名描述"能被 `()` 调用"，那"能被 `new` 调用"呢？用**构造签名（construct signature）**，就是在前面加个 `new`：

```typescript
// 文件：code/ch15/01-func-types.ts（续）
class Point {
  constructor(public x: number, public y: number) {}
}
type PointCtor = new (x: number, y: number) => Point;

function createInstance(Ctor: PointCtor, x: number, y: number): Point {
  return new Ctor(x, y); // 因为 Ctor 有构造签名，才允许 new
}
const p = createInstance(Point, 3, 4);
console.log("point =", p.x, p.y); // 3 4
```

**逐段拆解：**

- `type PointCtor = new (x: number, y: number) => Point` 描述"一个可以 `new`、接收两个 `number`、`new` 出来是 `Point` 的东西"——也就是 `Point` 这个类本身（类作为值就是它的构造函数）。
- `createInstance` 接收一个构造器 `Ctor`，函数体里 `new Ctor(...)` 之所以合法，正是因为参数类型带构造签名。把类 `Point`（当值用）传进去即可。

📌 构造签名是"工厂函数""依赖注入"这类模式的类型基础。第 16 章泛型会让它更强大（`new () => T`）。

### 15.1.5 类型谓词函数简介

有时你想写一个"判断函数"，让 TS **在调用它之后自动收窄类型**。这靠**类型谓词（type predicate）**，返回类型写成 `参数名 is 某类型`：

```typescript
// 文件：code/ch15/01-func-types.ts（续）
function isString(value: unknown): value is string {
  return typeof value === "string";
}

const mixed: unknown[] = ["hi", 42, "world", true];
for (const item of mixed) {
  if (isString(item)) {
    // 在这个分支里，item 被收窄为 string，可以安全调用 .toUpperCase()
    console.log("字符串：", item.toUpperCase());
  }
}
```

**逐段拆解：**

- 普通函数返回 `boolean` 只是"给出真假"；而返回 `value is string` 的函数在**返回 `true` 的分支里**，会让 TS 把 `value` 收窄成 `string`。
- 所以 `if (isString(item))` 的花括号内，`item` 从 `unknown` 变成了 `string`，`item.toUpperCase()` 才不报错。

⚠️ 类型谓词是你对编译器的**承诺**：你说"返回 true 就代表它是 string"，TS 就信了。如果你函数体里判断逻辑写错（比如判成 `typeof value === "number"`），TS 不会发现，运行时就会出错。类型谓词第 18 章会深入。

### 15.1.6 函数重载：一个函数，多套签名

**函数重载（function overload）**解决这样的需求：同一个函数，**参数不同时返回类型也不同**，而且这种对应关系要能被类型系统精确表达。

TS 的重载由两部分组成：

1. 若干条**重载签名（overload signatures）**——只有签名、没有函数体，它们是对外**可见**的"菜单"。
2. 一条**实现签名（implementation signature）**——带函数体，负责真正干活，但它对外**完全不可见**。

看例子：

```typescript
// 文件：code/ch15/02-overload.ts
// 前面几行是「重载签名」（只有签名，没有函数体），对外可见。
export function reverse(value: string): string;
export function reverse<T>(value: T[]): T[];
// 最后一行是「实现签名」（带函数体），对外【不可见】，只负责真正干活。
export function reverse(value: string | unknown[]): string | unknown[] {
  if (typeof value === "string") {
    return value.split("").reverse().join("");
  }
  return value.slice().reverse();
}

const s = reverse("hello");
console.log(s.toUpperCase()); // OLLEH —— s 是 string，能调 toUpperCase

const a = reverse([1, 2, 3]);
console.log(a.map((n) => n + 1)); // [ 4, 3, 2 ] —— a 是 number[]，能调 map
```

**逐段拆解——这里最容易讲错，务必看仔细：**

- 前两行是重载签名：`reverse(string): string` 和 `reverse<T>(T[]): T[]`。它们只是"声明"，没有 `{}`。
- 第三个 `reverse(...)`（带 `{}` 的那个）是实现签名。它的参数和返回类型必须**足够宽**，能覆盖上面两条重载——所以写成 `string | unknown[]`。
- 调用时，TS **只在重载签名里挑**。`reverse("hello")` 匹配第一条，返回类型是 `string`，于是 `s.toUpperCase()` 合法；`reverse([1,2,3])` 匹配第二条，返回 `number[]`，于是 `a.map(...)` 合法。**返回类型精确跟着入参走**，这就是重载的价值。

📌 **重载最关键、最容易被讲错的一点：实现签名对外不可见。** 也就是说，虽然实现签名写了 `value: string | unknown[]`，但外部**不能**按这个宽签名去调用。能被调用的只有那两条重载签名：

```typescript
// 文件：code/ch15/02-overload.ts（续）
// ⚠️ 下面这行会报错：没有一个重载签名匹配 (number)。实现签名对外不可见！
// reverse(123);
```

`reverse(123)` 报错——因为 `123` 既不匹配 `string` 也不匹配 `T[]`，而那个"看起来能接受任何东西"的实现签名根本不参与对外匹配。

再看一个"用重载约束参数组合"的经典例子：

```typescript
// 文件：code/ch15/02-overload.ts（续）
function makeDate(timestamp: number): Date;
function makeDate(year: number, month: number, day: number): Date;
function makeDate(a: number, b?: number, c?: number): Date {
  if (b !== undefined && c !== undefined) {
    return new Date(a, b - 1, c); // 月份从 0 开始，所以 b-1
  }
  return new Date(a);
}

console.log(makeDate(0).getUTCFullYear()); // 1970
const d = makeDate(2026, 7, 2); // 用"年/月/日"三参重载
console.log(`${d.getFullYear()}-${d.getMonth() + 1}-${d.getDate()}`); // 2026-7-2
// makeDate(2026, 7); // ⚠️ 报错：没有接收 2 个参数的重载
```

**逐段拆解：**

- 两条重载分别声明"要么 1 个参数（时间戳），要么 3 个参数（年月日）"。
- 实现签名用可选参数 `b?`、`c?` 来兼容这两种形态，函数体里再靠 `b/c` 是否为 `undefined` 分流。
- 好处是 `makeDate(2026, 7)`（传 2 个）会被**编译期拒绝**——因为没有一条重载接收正好 2 个参数。这种"参数个数/组合级别的约束"，是重载相较于"一个参数写成联合类型"的独特能力。

⚠️ **重载的常见误区，逐条记牢：**

1. 实现签名**不算**一条对外重载。只写一条重载签名 + 一条实现签名时，很多人以为"有两条签名"，其实对外只有那一条重载签名。
2. 实现签名必须**兼容所有**重载签名，否则报错。
3. 重载签名要放在**最前面**，从上到下匹配，把**更具体**的放前面、更宽泛的放后面。
4. 很多场景其实**不需要重载**——如果返回类型不随入参变化，用联合类型或可选参数更简单。重载是"入参和返回类型有强关联"时才拿出来的重武器。

## 15.2 TypeScript 里的类

第 4 章你已经掌握了 JS 的 `class`、继承、原型。TS 在此之上加了一层**编译期的类型盔甲**。

### 15.2.1 字段类型、访问修饰符、readonly、static、getter/setter

一次性把这些放进一个"银行账户"类里看：

```typescript
// 文件：code/ch15/03-class-basics.ts
class BankAccount {
  public readonly owner: string; // public：默认，任何地方可读；readonly：初始化后不可改
  private _balance: number; // private：只有 BankAccount 内部能访问
  protected currency: string; // protected：本类 + 子类内部能访问
  static bankName = "TS 银行"; // static：属于"类"本身，而非某个实例

  constructor(owner: string, initial: number, currency = "CNY") {
    this.owner = owner;
    this._balance = initial;
    this.currency = currency;
  }

  deposit(amount: number): void {
    if (amount <= 0) throw new Error("存款必须为正数");
    this._balance += amount;
  }

  get balance(): number {
    return this._balance; // 读 account.balance 时执行
  }

  set balance(value: number) {
    if (value < 0) throw new Error("余额不能为负"); // 写 account.balance = x 时执行
    this._balance = value;
  }

  static describe(): string {
    return `欢迎使用 ${BankAccount.bankName}`;
  }
}
```

**逐段拆解访问修饰符——这块不能讲错：**

- **`public`**：默认修饰符（不写就是它）。类内、子类、类外都能访问。`owner` 就是 public。
- **`private`**：**只有该类自己内部**能访问，子类和外部都不行。`_balance` 是 private，外面写 `acc._balance` 会编译报错。
- **`protected`**：类内部**和子类内部**能访问，但类外部不行。`currency` 是 protected——本类和将来继承它的子类能用，外部代码不能。
- **`readonly`**：只读。可以在声明时或**构造函数里**赋值，此后不能再改。`owner` 是 `readonly`，`acc.owner = "别人"` 会报错。
- **`static`**：属于**类本身**而不是实例。`bankName` 和 `describe()` 都是静态成员，用 `BankAccount.bankName`、`BankAccount.describe()` 访问，**不能**通过实例 `acc.bankName` 访问。

⚠️ **一句话记牢可见性范围：`public`（谁都行）⊃ `protected`（本类 + 子类）⊃ `private`（仅本类）。**

再看 getter/setter 和 static 怎么用：

```typescript
// 文件：code/ch15/03-class-basics.ts（续）
const acc = new BankAccount("阿伟", 100);
acc.deposit(50);
console.log(acc.owner, "余额：", acc.balance); // 阿伟 余额： 150（走 getter）
acc.balance = 200; // 走 setter（含校验）
console.log("改后余额：", acc.balance); // 200
console.log(BankAccount.describe()); // 欢迎使用 TS 银行（static 方法，用类名调）

// acc.owner = "别人";   // ⚠️ 报错：owner 是 readonly
// acc._balance = 999;   // ⚠️ 报错：_balance 是 private，外部不可访问
// acc.balance = -1;     // 运行时抛错：余额不能为负
```

**逐段拆解：**

- `acc.balance`（读）表面像访问属性，实际执行了 `get balance()`；`acc.balance = 200`（写）实际执行了 `set balance(value)`，顺便做了"不能为负"的校验。getter/setter 让你**用属性的语法，行方法的逻辑**——校验、懒计算、日志都能藏进去。
- `BankAccount.describe()` 用**类名**调用，因为它是静态方法。

### 15.2.2 参数属性：构造函数里的简写

上面 `BankAccount` 的构造函数里写了一堆 `this.owner = owner` 这样的样板赋值，很啰嗦。TS 提供**参数属性（parameter properties）**：在构造函数参数前直接加修饰符，TS 就会**自动声明同名字段并赋值**：

```typescript
// 文件：code/ch15/03-class-basics.ts（续）
class Point3D {
  // 等价于：声明 x/y/z 三个字段 + 在构造体里 this.x = x 等
  constructor(
    public readonly x: number,
    public readonly y: number,
    public readonly z: number,
  ) {}

  length(): number {
    return Math.sqrt(this.x ** 2 + this.y ** 2 + this.z ** 2);
  }
}

const v = new Point3D(2, 3, 6);
console.log("向量长度：", v.length()); // 7
```

**逐段拆解：**

- `constructor(public readonly x: number, ...)` 一行干了三件事：声明字段 `x`、把参数 `x` 赋给 `this.x`、并加上 `public readonly` 修饰。省掉了"先声明字段、再在构造体里 `this.x = x`"的重复。
- 触发条件：参数前**必须有至少一个修饰符**（`public`/`private`/`protected`/`readonly`）。裸参数 `x: number`（没修饰符）就是普通参数，不会变成字段。

💡 参数属性是 TS 里最实用的语法糖之一，数据类/值对象几乎都靠它，代码能省一大截。

### 15.2.3 抽象类与 implements

**抽象类（`abstract class`）**是"只能被继承、不能被 `new` 的模板"，它可以留下**抽象成员**逼子类去实现，也可以提供公共实现。**`implements`** 则让一个类**承诺满足某个接口的形状**。

```typescript
// 文件：code/ch15/04-abstract-implements.ts
interface Shape {
  readonly name: string;
  area(): number;
}

abstract class BaseShape implements Shape {
  abstract readonly name: string; // 抽象字段：子类必须提供
  abstract area(): number; // 抽象方法：只有签名，子类必须实现

  // 非抽象方法：所有子类共享的公共逻辑
  describe(): string {
    return `${this.name} 的面积是 ${this.area().toFixed(2)}`;
  }
}

class Circle extends BaseShape {
  readonly name = "圆";
  constructor(private radius: number) {
    super(); // 子类构造函数必须先调 super()
  }
  area(): number {
    return Math.PI * this.radius ** 2;
  }
}

class Rectangle extends BaseShape {
  readonly name = "矩形";
  constructor(private w: number, private h: number) {
    super();
  }
  area(): number {
    return this.w * this.h;
  }
}
```

**逐段拆解：**

- `interface Shape` 只描述"形状"：有个只读 `name`、一个 `area()` 方法。`implements Shape` 是 `BaseShape` 对编译器的承诺——"我一定长成 Shape 的样子"，如果没实现齐，编译报错。
- `abstract class BaseShape` 用了 `abstract` 修饰 `name` 和 `area`：它们只有声明没有实现，**逼子类去实现**。但 `describe()` 是普通方法，是所有子类共享的公共逻辑——这正是抽象类比接口强的地方：**接口只能描述形状，抽象类还能塞进公共实现**。
- `Circle` / `Rectangle` `extends BaseShape` 并实现了 `name` 和 `area()`。子类构造函数里 `super()` 必须先调（这是 JS 类的规则，第 4 章讲过）。
- `implements` 和 `extends` 的区别：`implements` 只是**类型检查**（承诺形状），不会继承任何实现；`extends` 才是**真继承**（拿到父类的字段和方法）。一个类可以 `implements` 多个接口，但只能 `extends` 一个类。

```typescript
// 文件：code/ch15/04-abstract-implements.ts（续）
// const s = new BaseShape(); // ⚠️ 报错：不能创建抽象类的实例
```

⚠️ 抽象类**不能被 `new`**。它是模板，实例化交给具体子类。

### 15.2.4 类作为类型 vs 作为值

一个常被忽略但很重要的事实：**声明一个类，同时创造了两样东西**——

- 一个**值（value）**：类的构造函数，可以 `new`，可以赋给变量、当参数传。
- 一个**类型（type）**：它的**实例类型**，可以用在类型注解位置。

```typescript
// 文件：code/ch15/04-abstract-implements.ts（续）
// 用 BaseShape 作类型（它带公共方法 describe）；抽象类不能被 new，但能当类型用。
const shapes: BaseShape[] = [new Circle(2), new Rectangle(3, 4)];
for (const shape of shapes) {
  console.log(shape.describe());
}
// 圆 的面积是 12.57
// 矩形 的面积是 12.00

const CircleCtor = Circle; // Circle 用作【值】——把构造函数赋给变量
const c2 = new CircleCtor(1);
console.log(c2.area().toFixed(2)); // 3.14
```

**逐段拆解：**

- `shapes: BaseShape[]` 里的 `BaseShape` 用作**类型**——表示"元素都是 BaseShape 的实例"。注意抽象类**不能 `new`，但完全可以当类型用**（因为 Circle/Rectangle 的实例都是它的子类型）。
- `const CircleCtor = Circle` 里的 `Circle` 用作**值**——把构造函数本身赋给变量，之后 `new CircleCtor(1)` 照样能造实例。

📌 记住这句话：**在类型位置，类名指"实例的类型"；在值位置，类名指"构造函数"。** 前面 15.1.4 的构造签名 `PointCtor` 描述的，正是"类作为值"那一面的类型。

### 15.2.5 `#` 私有字段 vs `private`

这是高频面试题，也是初学者最容易含糊的点。两者都表示"私有"，但**机制完全不同**：

- `private` 是 **TypeScript 的编译期检查**。编译成 JS 后，这个"私有"就**消失了**——字段照样在对象上，运行时任何人都能读到。它是"君子协定"。
- `#` 是 **JavaScript 语言层面的真私有**（ECMAScript 标准语法）。它在运行时也是私有的，类外部用**任何手段**都读不到。

```typescript
// 文件：code/ch15/04-abstract-implements.ts（续）
class Secret {
  #realKey = "hard-#-private"; // # 是 JS 运行时真正的私有
  private softKey = "soft-private"; // private 只是 TS 编译期检查

  reveal(): string {
    return `${this.#realKey} / ${this.softKey}`;
  }
}

const sec = new Secret();
console.log(sec.reveal());
// console.log(sec.#realKey);          // ⚠️ 语法错误：# 私有在类外不可见
console.log("绕过 TS 读 private：", (sec as any).softKey); // soft-private —— 证明它运行时还在
console.log("# 私有的属性名：", Object.keys(sec)); // 只有 softKey，看不到 #realKey
```

**逐段拆解——运行结果会给你实锤：**

- `(sec as any).softKey` 输出 `"soft-private"`：证明 `private` 字段**运行时依然存在**，只要绕过 TS 的类型检查（用 `as any`）就能读到。所以 `private` 挡不住恶意/意外访问。
- `Object.keys(sec)` 只列出 `["softKey"]`，**看不到 `#realKey`**：证明 `#` 私有字段连"属性"都不算，运行时对外彻底隐形。
- `sec.#realKey`（若取消注释）是**语法错误**——`#` 私有在类外根本无法书写。

📌 **怎么选？** 想要"真正的封装、防外部访问"就用 `#`；只是"标记内部用、团队约定"或需要和老代码/装饰器/某些库配合，用 `private` 也行。新项目通常推荐 `#`，因为它是语言标准且是真私有。二者别混用同一个字段。

## 15.3 枚举（enum）

**枚举（enum）**用来给"一组命名的常量"起个类型。TS 的 enum 有点特殊——它**既是类型也会生成运行时代码**，这一点是所有坑的根源，务必理解。

### 15.3.1 数字枚举与它的反向映射

```typescript
// 文件：code/ch15/05-enum.ts
enum Direction {
  Up, // 0
  Down, // 1
  Left, // 2
  Right, // 3
}
console.log(Direction.Up, Direction.Right); // 0 3

// ⚠️ 数字枚举有「反向映射（reverse mapping）」：既能名→值，也能值→名。
console.log(Direction[0]); // "Up" —— 用数字反查名字
```

**逐段拆解：**

- 数字枚举不写值时，成员从 `0` 开始**自动递增**：`Up=0, Down=1, Left=2, Right=3`。
- `Direction[0]` 竟然能反查出 `"Up"`——这叫**反向映射（reverse mapping）**。数字枚举编译后会生成一个**双向填充**的对象（`{0:"Up", Up:0, ...}`），既能名查值也能值查名。这让它体积更大，也更容易被误用（比如不小心把一个随便的数字当成枚举传进去）。

### 15.3.2 字符串枚举

```typescript
// 文件：code/ch15/05-enum.ts（续）
enum LogLevel {
  Debug = "DEBUG",
  Info = "INFO",
  Warn = "WARN",
  Error = "ERROR",
}
console.log(LogLevel.Warn); // "WARN"
```

**逐段拆解：**

- 字符串枚举要求**每个成员都显式赋一个字符串**，没有自动递增。
- 字符串枚举**没有反向映射**（`LogLevel["WARN"]` 是 `undefined`）。这反而更安全：日志、存数据库、序列化到 JSON 时，值是可读的 `"WARN"` 而不是意义不明的 `2`，调试时一眼看懂。

💡 **优先用字符串枚举而非数字枚举**——避开反向映射的额外体积，也避开"隐式数值"带来的各种坑。

### 15.3.3 枚举的运行时形态

这是理解 enum 的关键：**普通 enum 不是"编译期擦除"的，它会生成一个真实存在的对象。**

```typescript
// 文件：code/ch15/05-enum.ts（续）
console.log("LogLevel 运行时对象：", LogLevel);
// { Debug: 'DEBUG', Info: 'INFO', Warn: 'WARN', Error: 'ERROR' }
```

**逐段拆解：**

- 直接打印 `LogLevel`，能看到它在运行时就是一个普通对象。这意味着你可以遍历它、把它当值传递。
- 代价：它**占运行时体积**，且和"接口/类型别名那种编译后完全消失"的东西不一样。这也是有人不喜欢 enum 的原因之一。

### 15.3.4 const enum：编译期内联

`const enum` 想解决"运行时体积"问题——它**不生成对象**，而是在编译时把每处引用**直接替换成字面量值**：

```typescript
// 文件：code/ch15/05-enum.ts（续）
const enum Planet {
  Mercury,
  Venus,
  Earth,
}
const home = Planet.Earth; // 编译后直接被替换成字面量 2，Planet 对象根本不生成
console.log("home =", home); // 2
```

**逐段拆解：**

- `Planet.Earth` 在编译后会被**内联**成 `2`，`Planet` 这个对象根本不会出现在产物里，省体积。
- ⚠️ **但 `const enum` 有坑**：它依赖编译期"看得到定义并内联"。在**跨模块**、开启 `isolatedModules`、或用某些打包器（Babel、esbuild、Vite 的部分配置）**逐文件转译**的场景下，内联可能失效甚至报错。**库作者尤其要避免对外暴露 `const enum`**。所以 `const enum` 虽好，用之前要确认你的构建链支持。

### 15.3.5 现代替代：联合字面量类型 + as const 对象

因为 enum 的这些"运行时痕迹"和坑，现代 TS 项目里越来越多人用**联合字面量类型 + `as const` 对象**来替代枚举：

```typescript
// 文件：code/ch15/05-enum.ts（续）
const Color = {
  Red: "red",
  Green: "green",
  Blue: "blue",
} as const;

// 从对象的值推出联合类型： "red" | "green" | "blue"
type Color = (typeof Color)[keyof typeof Color];

function paint(c: Color): void {
  console.log("涂上", c);
}
paint(Color.Red); // 涂上 red
paint("blue"); // 直接传字面量也行（enum 做不到这么自然）
// paint("purple");    // ⚠️ 报错：不在联合类型里

console.log("Color 对象的所有值：", Object.values(Color)); // [ 'red', 'green', 'blue' ]
```

**逐段拆解——这里信息量大，慢慢看：**

- `const Color = {...} as const`：`as const`（第 13 章的字面量类型 + 常量断言）把对象**冻结**成最窄的字面量类型，`Red` 的类型不是宽泛的 `string` 而是精确的 `"red"`。
- `type Color = (typeof Color)[keyof typeof Color]`：这行是从对象**反推**出联合类型的固定套路。`typeof Color` 拿到对象的类型，`keyof typeof Color` 拿到它所有键（`"Red" | "Green" | "Blue"`），再用索引访问 `[...]` 拿到所有值的类型——结果就是 `"red" | "green" | "blue"`。（这里 `Color` 同时是值和类型，属于合法的声明合并，互不冲突。）
- 于是 `paint` 既能收 `Color.Red`（有命名常量的可读性），也能直接收 `"blue"`（字面量），这是普通 enum 做不到的自然。
- 运行时 `Color` 只是一个普通对象，`Object.values(Color)` 就能遍历，**没有任何编译魔法**，打包器也不会有 `const enum` 那种坑。

📌 **enum vs 联合字面量，怎么取舍：**

| 需求 | 推荐 |
| --- | --- |
| 想要最小运行时、能直接传字符串字面量、无构建坑 | **`as const` 对象 + 联合类型** |
| 团队已大量用 enum、想要数字自增语义或命名空间感 | **enum**（且优先字符串枚举） |
| 极端在意产物体积且构建链确定支持 | `const enum`（谨慎，别对外暴露） |

无论选哪个，**尽量避开数字枚举的隐式数值和反向映射**。

## 15.4 泛型预告

你可能注意到，15.1.6 的 `reverse<T>(value: T[]): T[]` 和 15.1.4 里"能不能写成 `new () => T`"，都出现了一个尖括号里的 `T`。这就是**泛型（generics）**——"类型的参数"。

它让你写出"对多种类型都成立、且保持类型关系"的代码：比如一个 `identity<T>(x: T): T`，你传 `number` 进去就返回 `number`、传 `string` 就返回 `string`，而不是丢失类型信息的 `any`。函数、类、接口、类型别名都能泛型化。下一章我们专门讲透它。

## 常见坑与最佳实践

- ⚠️ **重载的实现签名对外不可见**。别以为写了实现签名就多了一条可调用签名——外部只能按重载签名调用。把更具体的重载放前面。
- ⚠️ **`private` 不是真私有**，编译成 JS 后运行时照样能读到（`as any` 就能绕过）。要真封装用 `#`。同一字段别混用二者。
- ⚠️ **参数属性要有修饰符才生效**。`constructor(x: number)` 是普通参数；`constructor(public x: number)` 才会自动变字段。
- ⚠️ **抽象类不能 `new`**，但可以当类型用。抽象成员必须被子类实现。
- ⚠️ **数字枚举有反向映射**、会隐式自增，容易误传。**优先字符串枚举**，或干脆用 `as const` 对象。
- ⚠️ **`const enum` 有构建兼容坑**，`isolatedModules` / Babel / esbuild 下可能失效；库的公共 API 不要用它。
- 💡 **本项目 tsconfig 下，每个示例文件最好带 `export {}` 或真实 `import/export`**，否则文件被当"全局脚本"，跨文件同名声明（如 `Point`、`Counter`）会冲突。本章代码文件已按此处理。
- 💡 **getter/setter 让你在"读写属性"时偷偷做校验/懒计算/日志**，对外接口却保持属性的简洁语法。
- 💡 **`this` 参数**能在编译期就抓住"方法被摘下来单独调用导致 `this` 丢失"的经典 bug。

## 本章小项目：有限状态机

**有限状态机（Finite State Machine，FSM）**是一个到处都在用的模型：订单流转、红绿灯、播放器、审批流……它的核心是"**在某个状态下，只有特定事件是合法的，且会把系统带到确定的下一个状态**"。我们用 TS 把这个"合法性"变成**类型安全 + 运行时都拦得住**的东西。

以**订单状态**为例：`pending`（待支付）→ `paid`（已支付）→ `shipped`（已发货）→ `delivered`（已送达），且在发货前可随时 `cancel`（取消）。

### 第 1 步：用联合字面量类型定义状态与事件

```typescript
// 文件：code/ch15/state-machine.ts
type OrderState =
  | "pending" // 待支付
  | "paid" // 已支付
  | "shipped" // 已发货
  | "delivered" // 已送达
  | "cancelled"; // 已取消

type OrderEvent =
  | "pay" // 支付
  | "ship" // 发货
  | "deliver" // 送达
  | "cancel"; // 取消
```

**逐段拆解：** 用**联合字面量类型**（第 14 章）而不是 enum 来表达状态和事件——它们零运行时开销，且在下面的映射类型里能被完美利用。

### 第 2 步：合法转移表

```typescript
// 文件：code/ch15/state-machine.ts（续）
type TransitionTable = {
  [S in OrderState]: {
    [E in OrderEvent]?: OrderState;
  };
};

const transitions: TransitionTable = {
  pending: { pay: "paid", cancel: "cancelled" },
  paid: { ship: "shipped", cancel: "cancelled" },
  shipped: { deliver: "delivered" }, // 已发货就不能再取消了
  delivered: {}, // 终态：无任何合法转移
  cancelled: {}, // 终态
};
```

**逐段拆解：**

- `TransitionTable` 用**映射类型（mapped type）**：外层 `[S in OrderState]` 表示"每个状态都要有一项"；内层 `[E in OrderEvent]?` 里那个 `?` 表示"事件是可选的"——**没列出的事件就代表在该状态下非法**。
- `transitions` 是这张表的具体值。`shipped` 下只有 `deliver`，没有 `cancel`——用类型把"已发货不能取消"这条业务规则固化下来。`delivered`/`cancelled` 是**终态**，值为空对象 `{}`，任何事件都不合法。
- 因为 `TransitionTable` 要求每个状态都有键，**如果你漏写了某个状态（比如忘了 `cancelled`），TS 会当场报错**——这就是类型安全的好处，转移表不会写漏。

### 第 3 步：转移结果类型（可辨识联合）

```typescript
// 文件：code/ch15/state-machine.ts（续）
type TransitionResult =
  | { ok: true; from: OrderState; to: OrderState; event: OrderEvent }
  | { ok: false; from: OrderState; event: OrderEvent; reason: string };

interface HistoryEntry {
  from: OrderState;
  event: OrderEvent;
  to: OrderState;
  at: number; // 时间戳
}
```

**逐段拆解：** `TransitionResult` 是一个**可辨识联合（discriminated union）**（第 14 章），靠 `ok` 这个字面量字段区分成功/失败。成功时带 `to`（新状态），失败时带 `reason`（原因）。调用方 `if (result.ok)` 一判断，TS 就能收窄到对应分支——成功分支能访问 `to`，失败分支能访问 `reason`，各取所需。

### 第 4 步：StateMachine 类

```typescript
// 文件：code/ch15/state-machine.ts（续）
class OrderStateMachine {
  #state: OrderState; // 用 # 私有，外部只能通过 getState() 读，不能乱改
  readonly #history: HistoryEntry[] = [];
  readonly #table: TransitionTable;

  constructor(initial: OrderState = "pending", table: TransitionTable = transitions) {
    this.#state = initial;
    this.#table = table;
  }

  getState(): OrderState {
    return this.#state;
  }

  getHistory(): readonly HistoryEntry[] {
    return this.#history.slice(); // 返回副本，防止外部篡改内部数组
  }

  can(event: OrderEvent): boolean {
    return this.#table[this.#state][event] !== undefined;
  }

  transition(event: OrderEvent): TransitionResult {
    const from = this.#state;
    const to = this.#table[from][event];

    if (to === undefined) {
      return {
        ok: false,
        from,
        event,
        reason: `状态「${from}」下不允许事件「${event}」`,
      };
    }

    this.#state = to;
    this.#history.push({ from, event, to, at: Date.now() });
    return { ok: true, from, to, event };
  }
}
```

**逐段拆解——这是项目的核心：**

- `#state`、`#history`、`#table` 全用 `#` **真私有**：外部无法直接改状态，只能通过 `transition` 走合法流程。这正是 15.2.5 学的封装。
- `getState()` 只读地暴露当前状态；`getHistory()` 返回 `#history.slice()` 的**副本**，且返回类型标成 `readonly HistoryEntry[]`，双重保险防止外部篡改内部历史。
- `can(event)` 查询"当前状态下这个事件合不合法"，靠 `this.#table[this.#state][event] !== undefined` 判断——转移表里有这项就合法。
- `transition(event)` 是关键：先查 `to = 表[from][event]`。**查不到（`undefined`）就返回失败、且完全不动状态**；查得到就更新 `#state`、记一笔历史、返回成功。所有非法转移在**运行时**被稳稳拦下，返回结构化的失败结果而非抛异常——调用方能优雅处理。

### 第 5 步：演示一系列转移

```typescript
// 文件：code/ch15/state-machine.ts（续）
const sm = new OrderStateMachine(); // 初始 pending
console.log("初始状态：", sm.getState()); // pending

const steps: OrderEvent[] = ["pay", "ship", "cancel", "deliver"];
for (const event of steps) {
  const result = sm.transition(event);
  if (result.ok) {
    console.log(`✅ ${result.from} --[${result.event}]--> ${result.to}`);
  } else {
    console.log(`❌ 拒绝：${result.reason}（状态保持 ${result.from}）`);
  }
}
```

运行 `npx tsx ch15/state-machine.ts`，实测输出：

```
初始状态： pending
✅ pending --[pay]--> paid
✅ paid --[ship]--> shipped
❌ 拒绝：状态「shipped」下不允许事件「cancel」（状态保持 shipped）
✅ shipped --[deliver]--> delivered
最终状态： delivered
成功转移次数（历史长度）： 3
delivered 还能 cancel 吗？ false
```

**逐段拆解：**

- `pay`、`ship` 依次合法，状态一路推进。
- 到 `shipped` 时试图 `cancel`——转移表里 `shipped` 没有 `cancel`，`transition` 返回 `ok:false`，状态**纹丝不动**保持 `shipped`。`if (result.ok)` 让 TS 在两个分支里分别看到 `to` 和 `reason`，这就是可辨识联合的威力。
- 接着 `deliver` 合法，走到终态 `delivered`。历史里只记了 **3 次成功转移**（失败的 `cancel` 不入历史）。`can("cancel")` 返回 `false`，因为 `delivered` 是终态。

### 扩展练习：加进入 / 离开回调

真实状态机常需要"进入某状态时做点事（发通知）、离开某状态时清理"。我们继承出一个带回调的版本：

```typescript
// 文件：code/ch15/state-machine.ts（续）
type StateHook = (state: OrderState) => void;

class HookedStateMachine extends OrderStateMachine {
  #onEnter = new Map<OrderState, StateHook>();
  #onLeave = new Map<OrderState, StateHook>();

  onEnter(state: OrderState, hook: StateHook): this {
    this.#onEnter.set(state, hook);
    return this; // 返回 this 支持链式调用
  }
  onLeave(state: OrderState, hook: StateHook): this {
    this.#onLeave.set(state, hook);
    return this;
  }

  override transition(event: OrderEvent): TransitionResult {
    const from = this.getState();
    const result = super.transition(event); // 复用父类的合法性判断与状态变更
    if (result.ok) {
      this.#onLeave.get(from)?.(from); // 离开旧状态
      this.#onEnter.get(result.to)?.(result.to); // 进入新状态
    }
    return result;
  }
}

const hooked = new HookedStateMachine()
  .onEnter("paid", () => console.log("🔔 进入 paid：给财务发通知"))
  .onLeave("pending", () => console.log("🔔 离开 pending：清理待支付提醒"));

hooked.transition("pay");
console.log("hooked 当前状态：", hooked.getState()); // paid
```

**逐段拆解：**

- `HookedStateMachine extends OrderStateMachine`：靠继承复用父类全部的合法性判断，不重复造轮子。
- `onEnter`/`onLeave` 把回调登记进 `Map`，并 `return this` 支持**链式调用**（`.onEnter(...).onLeave(...)`）。
- **重写（override）** `transition`：先记下旧状态 `from`，调 `super.transition(event)` 让父类干合法性判断和状态变更这些脏活；只有 `result.ok` 时，才用 `?.()`（可选链调用，第 14 章）触发"离开旧状态""进入新状态"的回调——没登记回调时 `?.` 会安全跳过。`override` 关键字让 TS 校验你确实在重写父类方法（拼错方法名会报错）。

运行时输出：

```
🔔 离开 pending：清理待支付提醒
🔔 进入 paid：给财务发通知
hooked 当前状态： paid
```

2. **泛型化**成 `StateMachine<S, E>`，把状态/事件类型作为类型参数传入，一套代码复用给红绿灯、播放器。（提示：这正是下一章泛型的用武之地。）
3. **加 `onTransition(from, event, to)` 全局回调**，记录到日志系统。

## 小结与自测

这一章我们补齐了 TS 里"函数、类、枚举"三块日常主力：

- **函数类型**：函数类型表达式 `(a: T) => R`；**调用签名**描述"可调用 + 带属性"的函数对象；**`this` 参数**是编译期专用、不占实参；**构造签名** `new (...) => T` 描述"可 `new`"；**类型谓词** `x is T` 让调用后自动收窄。**函数重载**——多条对外可见的重载签名 + 一条对外不可见的实现签名，返回类型随入参精确变化。
- **类**：字段类型注解；`public/protected/private` 三级可见性（谁都行 ⊃ 本类+子类 ⊃ 仅本类）；`readonly` 只读；**参数属性**在构造函数里一行搞定"声明+赋值+修饰"；`static` 属于类本身；getter/setter 用属性语法藏方法逻辑；**抽象类**是能带公共实现的模板、不能 `new`；`implements` 只做形状检查、`extends` 才真继承。
- **类的双重身份**：类型位置指实例类型，值位置指构造函数。
- **`#` vs `private`**：`#` 是运行时真私有；`private` 只是编译期君子协定，`as any` 就能绕过。
- **枚举**：数字枚举会自增且有反向映射（体积大、易误用）；字符串枚举更安全可读；**普通 enum 会生成运行时对象**；`const enum` 内联省体积但有构建坑。现代替代——**`as const` 对象 + 联合字面量类型**，零运行时魔法且能直接传字面量。
- 小项目**有限状态机**：用联合字面量定义状态/事件、映射类型定义转移表、可辨识联合表达成功/失败、`#` 私有封装状态、`transition` 运行时拦截非法转移，并用继承 + `override` 扩展了进入/离开回调。

**自测题（先自己答，再运行代码验证）：**

1. 函数重载里，为什么 `reverse(123)` 会报错，即便实现签名写的是 `value: string | unknown[]`？
2. `public`、`protected`、`private` 三者的可访问范围分别是什么？用一句话概括包含关系。
3. `#private` 和 TS 的 `private` 有什么本质区别？下面哪个能在运行时被 `Object.keys` 看到？
4. 参数属性 `constructor(public x: number)` 和普通参数 `constructor(x: number)` 有何不同？触发条件是什么？
5. 抽象类和接口都能描述"形状"，抽象类多了什么能力？抽象类能被 `new` 吗？
6. 数字枚举的"反向映射"是什么？为什么推荐优先用字符串枚举？普通 enum 编译后运行时长什么样？
7. 用 `as const` 对象替代 enum 有哪些好处？`type X = (typeof X)[keyof typeof X]` 这行在做什么？
8. 状态机的 `transition` 遇到非法事件时做了什么？为什么它返回 `TransitionResult` 而不是直接抛异常更好？

（参考答案：1. 因为实现签名对外**不可见**，能被调用的只有那两条重载签名 `string` / `T[]`，`123` 都不匹配。2. `public`：类内/子类/类外都能访问；`protected`：本类+子类内部；`private`：仅本类内部。包含关系：public ⊃ protected ⊃ private。3. `#` 是 JS 语言级真私有、运行时对外彻底隐形；`private` 只是 TS 编译期检查、运行时字段还在，`as any` 能读到——所以 `Object.keys` 能看到 `private` 字段，看不到 `#` 字段。4. 带修饰符的参数会自动声明同名字段并赋值（省样板），普通参数只是局部参数；触发条件是参数前有 `public/private/protected/readonly` 至少一个。5. 抽象类还能提供**公共实现**（非抽象方法），接口只能描述形状；抽象类**不能** `new`。6. 反向映射指数字枚举编译后生成"值→名"的双向对象，`Direction[0]` 能查出 `"Up"`；字符串枚举无反向映射、值可读、更安全；普通 enum 编译后是一个真实存在的运行时对象。7. 好处：零运行时魔法、无构建坑、能直接传字符串字面量；那行是从对象类型里取所有值的类型，合成联合类型。8. 查转移表查不到就返回 `{ok:false, reason}` 且**不改状态**；返回结果比抛异常好，因为调用方能用可辨识联合优雅地分支处理成功/失败，不必用 try/catch 包裹。）
