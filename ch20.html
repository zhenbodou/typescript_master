JavaScript & TypeScript 从入门到精通

第 20 章 · tsconfig、声明文件与 JS 互操作

本章目标

前面十九章，你写的每个 .ts 文件都是靠 npx tsx 直接跑起来的。tsx 帮你把 TypeScript 悄悄转成 JavaScript 再执行，你几乎没操心过“编译器到底怎么配置“。但真实项目里，一切都绕不开一份文件——tsconfig.json。它是 TypeScript 编译器（tsc）的“控制面板“：编到哪个 JS 版本、用哪种模块系统、检查严不严、怎么找模块、要不要顺带处理 .js……全在这里定。配错一个选项，可能整个项目红成一片，或者反过来——检查形同虚设、bug 一路溜进生产环境。

与之相伴的还有两个绕不开的话题。其一，你迟早要用一些只有 JS、没有类型的第三方库（npm 上一大半老库都是这样），或者要在自己项目里保留一部分旧 .js 文件。TypeScript 怎么“看懂“它们？答案是声明文件（declaration file，.d.ts）——一种“只有类型、没有实现“的文件，专门给 JS 补上类型。其二，很多时候你不想推倒重写，而是想让 TS 和 JS 共存、再逐步迁移，这就要用到 allowJs、checkJs 和 JSDoc。

学完本章，你将能够：

  • 看懂并写出一份生产级 tsconfig.json，理解常用编译选项分组的含义与取舍。
  • 说清楚 strict 及其子项分别在防什么错，以及 noUncheckedIndexedAccess、exactOptionalPropertyTypes 这些“严上加严“的选项。
  • 理解 TS 的**模块解析（module resolution）**是怎么找模块和类型的，会用 baseUrl + paths 配路径别名。
  • 掌握声明文件 .d.ts 的写法：declare、declare module、全局声明、各种 export 形态。
  • 会用 @types/*（DefinitelyTyped） 给没类型的库装类型。
  • 会用 allowJs / checkJs + JSDoc 做 JS→TS 的渐进迁移。
  • 认识 triple-slash 指令 /// <reference> 与 declare global。
  • 亲手为一个纯 JS 库手写一份 .d.ts，让它在 TS 里获得完整类型提示与检查。

本章的示例配置放在 code/ch20/tsconfig.example.jsonc（带注释讲解），小项目代码在 code/ch20/ 下，用 npx tsx ch20/use-lib.ts 运行。

📌 一句话记住 tsconfig：它回答两个问题——“编译成什么”（target/module/输出）和“检查多严“（strict 及各种检查项）；顺带回答”怎么找模块和类型“（moduleResolution/paths/types）。

tsconfig.json 是什么，从哪来

在项目根目录放一个 tsconfig.json，就等于告诉 TypeScript：“这里是一个 TS 项目，编译时按我写的选项来。” 你可以手写，也可以用 npx tsc --init 生成一份带大量注释的模板。

一份最小可用的配置长这样：

{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "strict": true
  },
  "include": ["src"]
}

逐段拆解：

  • compilerOptions：绝大多数配置都在这个对象里，控制编译器的行为。下面各小节会分组细讲。
  • include：告诉编译器“编译哪些文件“。"src" 表示 src 目录下所有 .ts/.tsx（以及开了 allowJs 后的 .js）。
  • 没写的选项都用默认值。TS 的默认值有些偏“宽松“（历史包袱），所以我们几乎总要显式打开 strict 等。

⚠️ tsconfig.json 允许写注释和尾逗号——它其实是 JSONC（JSON with Comments），不是严格 JSON。所以 tsc --init 生成的模板里全是 // 注释。但如果你用别的工具去 JSON.parse 它就会报错，注意区分。

本书的示例配置文件 code/ch20/tsconfig.example.jsonc 就是一份带满注释的“讲解版“，你可以对照着读。下面我们分组过一遍最常用的选项。

编译选项之一：输出（编成什么样）

这一组决定“TS 编译出来的 JS 长什么样、放到哪“。

// 文件：code/ch20/tsconfig.example.jsonc（节选）
{
  "compilerOptions": {
    "target": "ES2022",            // 编译到哪个 JS 版本
    "module": "ESNext",            // 生成哪种模块格式
    "moduleResolution": "Bundler", // 模块解析策略
    "lib": ["ES2022"],             // 允许使用的内置 API 类型
    "outDir": "./dist",            // 产物输出目录
    "rootDir": "./src",            // 源码根目录
    "declaration": true,           // 顺便生成 .d.ts
    "sourceMap": true              // 生成 .map
  }
}

逐段拆解：

  • target：把你的 TS/新语法降级到哪个 JS 版本。比如 target: "ES2015" 时，可选链 ?.、async/await 等会被编译成等价的老写法，以便在老环境里跑；target: "ES2022" 则保留大部分现代语法（因为现代 Node 和浏览器都支持）。选 target 的依据是“你的代码最终跑在哪“：现代 Node 用 ES2022/ES2023 即可，需要兼容老浏览器就调低（并交给打包器进一步处理）。
  • module：生成哪种模块系统的代码。ESNext/ES2022 生成 import/export（ES Modules）；CommonJS 生成 require/module.exports（老 Node 风格）；NodeNext 让 TS 按 Node 的规则在两者间自动切换。跑在现代 Node（package.json 里 "type": "module"）或交给打包器时，用 ESNext。
  • moduleResolution：编译器“怎么找模块“的算法，下一节专门讲。这里先记住三个常见值：
    • "node"（旧版 Node 算法，逐渐淘汰）；
    • "nodenext"（匹配现代 Node 的 ESM/CJS 双模式，跑在 Node 上时选它，通常与 module: "NodeNext" 配对）；
    • "bundler"（给 Vite/webpack/esbuild 这类打包器用的宽松算法，允许省略 import 里的 .js 后缀，本书选它，因为 tsx 与它兼容）。
  • lib：告诉 TS “你可以用哪些内置 API 的类型”。["ES2022"] 提供 Array、Promise、Map 等到 ES2022 为止的类型。写浏览器代码要加 "DOM"（才有 window、document、fetch 等类型）；纯 Node 后端一般不加 DOM。不写 lib 时，TS 会根据 target 推一个默认值。
  • outDir / rootDir：outDir 是编译产物（.js）的输出目录；rootDir 是源码根。rootDir 决定了 outDir 里的目录结构——src/a/b.ts 会输出到 dist/a/b.js。
  • declaration：额外为每个 .ts 生成一份 .d.ts。发布 npm 库时必开——别人 import 你的库时靠这些 .d.ts 获得类型。
  • sourceMap：生成 .js.map，把编译后的 JS 映射回原始 TS，这样调试器/浏览器报错时能定位到你的 .ts 源码行。

📌 本书主配置为什么没有 outDir？ 因为我们用 tsx 直接跑、并设了 noEmit: true（只做类型检查、不产出文件）。真实项目要发布或部署时才需要真正 emit，那时 outDir/declaration 就派上用场了。

编译选项之二：严格性（检查多严）

这是最影响“TS 能帮你挡多少 bug“的一组。

{
  "compilerOptions": {
    "strict": true,                     // 一键打开一整组严格检查
    "noUncheckedIndexedAccess": true,   // 下标访问结果带上 | undefined
    "exactOptionalPropertyTypes": true  // 区分「没有属性」和「属性=undefined」
  }
}

strict: true 是一个“开关组“——打开它，等于同时打开下面这些子项（都默认变成 true）：

  • noImplicitAny：禁止“隐式 any“。如果 TS 推断不出一个参数/变量的类型，它默认会给个 any（等于放弃检查）。开启后，这种情况会报错，逼你要么标类型、要么显式写 any。这是 strict 里最重要的一项。
  • strictNullChecks：把 null 和 undefined 当成独立类型看待。关闭时 null/undefined 能赋给任何类型（无数运行时 Cannot read property of undefined 的根源）；开启后，string 就不再能装 null，你必须写成 string | null 并显式处理。这是 TS 类型安全的基石，务必开。
  • strictFunctionTypes：更严格地检查函数参数类型（逆变检查），防止把“要求宽参数“的函数赋给“只接受窄参数“的位置。
  • 还有 strictBindCallApply（检查 bind/call/apply 的参数）、strictPropertyInitialization（类字段必须在构造时初始化）、alwaysStrict（输出 "use strict"）、useUnknownInCatchVariables（catch (e) 里 e 是 unknown 而非 any）等。

💡 无脑建议：新项目一律 "strict": true。 逐项开太麻烦也容易漏。老项目迁移时可以先关掉，再逐个子项打开逐步收紧。

再看两个 strict 之外、需要单独开的“严上加严“选项，它们能堵住 strict 覆盖不到的漏洞：

// 文件：code/ch20/strictness-demo.ts（讲解示例，未纳入运行）

// —— noUncheckedIndexedAccess 的效果 ——
const arr = [1, 2, 3];
const x = arr[10]; // 未开启时类型是 number（骗你！其实是 undefined）
                   // 开启后类型是 number | undefined，逼你先判空

const dict: Record<string, number> = { a: 1 };
const v = dict["z"]; // 同理：开启后是 number | undefined

逐段拆解：

  • noUncheckedIndexedAccess：默认情况下，arr[10] 的类型是 number——但数组只有 3 个元素，arr[10] 运行时其实是 undefined！这是 TS 一个众所周知的“善意的谎言“。开启此项后，任何下标/索引访问（arr[i]、obj[key]）的结果都会自动带上 | undefined，逼你在用之前先判空。它能挡掉大量“下标越界当成有值“的 bug，代价是要多写一些判空。
// —— exactOptionalPropertyTypes 的效果 ——
interface Options {
  timeout?: number; // 可选属性
}
// 未开启时：下面两者被视为「一样」
const a: Options = {};                     // 没有 timeout
const b: Options = { timeout: undefined }; // 显式设成 undefined
// 开启 exactOptionalPropertyTypes 后：b 会报错，
// 因为「可选（可以没有）」不等于「值可以是 undefined」。

逐段拆解：

  • exactOptionalPropertyTypes：timeout?: number 严格来说意思是“可以没有这个属性“，而不是“这个属性的值可以是 undefined”。默认 TS 把这两者混为一谈；开启此项后，显式写 { timeout: undefined } 会报错。这个区别在某些库（比如它用 "timeout" in options 判断“用户到底传没传“）里很关键。它比较“轴“，默认不开也很常见，但知道它的存在有助于理解可选属性的精确语义。

编译选项之三：额外检查

strict 管的是类型层面的严格；下面这组管的是代码质量层面的常见疏忽，需要单独开。

{
  "compilerOptions": {
    "noUnusedLocals": true,            // 未使用的局部变量报错
    "noUnusedParameters": true,        // 未使用的参数报错
    "noImplicitReturns": true,         // 有的分支 return 有的不 return 时报错
    "noFallthroughCasesInSwitch": true // switch 漏写 break 时报错
  }
}

逐段拆解：

  • noUnusedLocals：声明了却没用到的局部变量会报错。能帮你清掉死代码。（注意：本书主 tsconfig 把它设成了 false，因为教学示例里经常故意留一些“演示用但没使用“的变量。）
  • noUnusedParameters：没用到的函数参数报错。想保留参数占位（比如回调签名要求）又不想报错，把参数名以 _ 开头即可，如 (_event) => ...。
  • noImplicitReturns：如果一个函数有的代码路径 return 了值、有的路径却“走到底没 return“（隐式返回 undefined），报错。逼你把所有分支都显式处理。
// noImplicitReturns 会抓的错误：
function grade(score: number): string {
  if (score >= 60) return "及格";
  // ❌ 少了 else 分支的 return，某些路径隐式返回 undefined
}
  • noFallthroughCasesInSwitch：switch 的某个 case 里写了语句、却没 break/return，导致“贯穿“到下一个 case，报错。（空的 case（一行都没有，故意贯穿）不算。）这能堵掉经典的“忘写 break“bug。本书主 tsconfig 已开启此项。

💡 这几个选项不影响类型安全，但对团队协作和代码整洁很有价值。个人练习项目可以关掉 noUnusedLocals/noUnusedParameters 图个清静，正式项目建议全开。

编译选项之四：互操作与 JS

这组关系到“TS 怎么和 JS 生态、模块系统打交道“，也是本章后半部分的重点。

{
  "compilerOptions": {
    "esModuleInterop": true,       // 让默认导入 CommonJS 模块好用
    "allowJs": true,               // 允许编译 .js
    "checkJs": false,              // 顺带对 .js 做类型检查
    "resolveJsonModule": true,     // 允许 import ... from "./x.json"
    "skipLibCheck": true,          // 跳过对 .d.ts 的检查
    "isolatedModules": true,       // 保证每个文件可被单独转译
    "verbatimModuleSyntax": true   // import/export 原样保留
  }
}

逐段拆解：

  • esModuleInterop：解决 ESM 和 CommonJS 混用时的“默认导入“痛点。很多老库是 CommonJS 的（module.exports = ...），按 ESM 规范你本该写 import * as express from "express"，很别扭。开启此项后可以写更自然的 import express from "express"。几乎所有项目都开。
  • allowJs：允许 TS 项目里包含并编译 .js 文件。默认关闭——所以本书主 tsconfig 没开它，include 也只匹配 .ts。做 TS/JS 混编、或渐进迁移时才开。
  • checkJs：在 allowJs 基础上，顺带对 .js 也做类型检查（配合 JSDoc 注解，见后文）。相当于“给 JS 也上一层类型保护“。
  • resolveJsonModule：允许 import data from "./config.json"，并根据 JSON 内容推断类型。本书已开。
  • skipLibCheck：跳过对所有 .d.ts 文件（包括 node_modules 里第三方库的类型）的类型检查。几乎必开——它能大幅加快编译，并避开“两个第三方库的类型互相冲突“这种你无力修复的问题。代价是：你自己写的 .d.ts 里的错误也不会被检查到。
  • isolatedModules：要求每个文件都能被单独转译（不依赖跨文件的类型信息）。像 tsx、esbuild、swc、Babel 这类“逐文件转译“的工具需要它，因为它们不做全局类型分析。开启后 TS 会禁止一些“单文件转译搞不定“的写法（比如 export 一个 const enum）。
  • verbatimModuleSyntax：让 import/export 语句原样保留到输出（不由 TS 自作主张删掉它认为“只用于类型“的导入）。开启后，导入类型时必须显式写 import type 或 import { type Foo }，否则它会被当成值导入保留下来。这消除了“这个 import 到底会不会出现在运行时“的歧义。

⚠️ allowJs 与 .d.ts 的关系，一个常见误解：很多人以为“要 import 一个 .js 文件、必须开 allowJs“。不完全对。如果那个 .js 旁边有同名 .d.ts（如 foo.js + foo.d.ts），TS 会直接用 .d.ts 提供的类型来理解这个 import，根本不需要 allowJs——因为它把 .js 当成了“已经有类型声明的外部模块”，并不需要自己去编译那个 .js。本章小项目正是利用了这一点：主 tsconfig 没开 allowJs，但 use-lib.ts 依然能 import 旁边的 legacy-math.js，因为有 legacy-math.d.ts 顶着。只有当你想让 TS 去编译/检查 .js 本身（比如迁移场景）时，才需要 allowJs。

编译选项之五：类型来源与 include/exclude

{
  "compilerOptions": {
    // "types": ["node"],          // 只自动加载这些全局类型包
    // "typeRoots": ["./node_modules/@types", "./typings"]
  },
  "include": ["src/**/*.ts", "src/**/*.js"],
  "exclude": ["node_modules", "dist"]
}

逐段拆解：

  • typeRoots：TS 去哪些目录里找“全局类型包“。默认是 ["./node_modules/@types"]——所以你 npm i -D @types/node 后，@types/node 里的全局类型（如 Node 的 process、Buffer）会自动生效，无需 import。
  • types：在 typeRoots 找到的众多类型包里，只自动加载列出的这几个。不写 types 时，@types 下的所有包都被自动加载。本书主 tsconfig 写了 "types": ["node"]，意思是“只自动加载 @types/node“，避免无关的全局类型污染。⚠️ 注意：types 只限制”自动全局加载“的范围，不影响你用 import 显式引入的类型。
  • include：要编译的文件（支持 glob）。本书用 ["ch*/**/*.ts"]，只收 .ts。
  • exclude：从 include 里排除的文件。默认就会排除 node_modules，通常再加上 dist 等产物目录。
  • files（未在上面示例中）：精确列出少量入口文件，与 include 二选一，适合小项目或库入口。

extends：配置继承

一份 tsconfig 可以 extends（继承）另一份，自己只写“差异项“。这在 monorepo 或“一套基线、多处复用“时极有用。

// tsconfig.base.json —— 团队共享的基线
{ "compilerOptions": { "strict": true, "target": "ES2022", "module": "ESNext" } }

// 某子包的 tsconfig.json —— 继承基线，只改自己需要的
{
  "extends": "./tsconfig.base.json",
  "compilerOptions": { "outDir": "./dist" },
  "include": ["src"]
}

社区还维护了一批开箱即用的基线，比如 @tsconfig/node20、@tsconfig/strictest，装上后写 "extends": "@tsconfig/node20/tsconfig.json" 即可，省去自己调一堆选项。

project references 简介

大型项目（尤其 monorepo）可以拆成多个“子项目“，每个有自己的 tsconfig，再用 project references 把它们串起来：

{
  "references": [{ "path": "./packages/core" }, { "path": "./packages/web" }],
  "files": []
}

配合 tsc --build（或 tsc -b），TS 能增量编译：只重编改动过的子项目，并按依赖顺序构建。这是大仓库加速编译的标准手段。日常小项目用不到，知道有这么回事即可。

模块解析：TS 是怎么找模块和类型的

当你写 import { add } from "./legacy-math.js" 或 import express from "express" 时，TS 要先找到这个模块的类型才能检查你的代码。它遵循一套算法（由 moduleResolution 决定），大致是：

对相对路径（以 ./ 或 ../ 开头，如 "./legacy-math.js"）：

  1. 从当前文件所在目录出发，按路径找对应文件；
  2. 优先找类型来源——.ts、.d.ts（同名的 .d.ts 会被认作那个 .js 的类型），再考虑 .js（视 allowJs 而定）。

对“裸模块名“（不带路径，如 "express"、"lodash"）：

  1. 到 node_modules/express 里找，读它的 package.json（types/typings 字段指向 .d.ts，exports/main 指向入口）；
  2. 如果这个库自带类型（很多现代库会打包 .d.ts），直接用；
  3. 如果库不带类型，就去 node_modules/@types/express 找社区补的类型（下一节讲 @types）；
  4. 逐级往上层目录的 node_modules 找，直到找到或报 Cannot find module。

📌 moduleResolution 的选择会影响细节。"bundler"（本书用）比较宽松，允许你在 import 里省略后缀（写 "./foo" 而非 "./foo.js"）、也允许写 .js 后缀指向 .ts/.d.ts；"nodenext" 更严格，要求你按 Node ESM 规范写全后缀。

baseUrl 与 paths：路径别名

深层目录里满屏的 import x from "../../../../utils/x" 又丑又脆（移动文件就断）。用 baseUrl + paths 可以配路径别名：

{
  "compilerOptions": {
    "baseUrl": ".",              // 非相对 import 的解析基准
    "paths": {
      "@/*": ["src/*"],          // @/utils/x  ->  src/utils/x
      "@utils/*": ["src/utils/*"]
    }
  }
}

于是你可以写：

import { x } from "@/utils/x";     // 等价于 src/utils/x
import { y } from "@utils/y";      // 等价于 src/utils/y

逐段拆解：

  • baseUrl：非相对导入的“根“。"." 表示以 tsconfig 所在目录为基准。
  • paths：一组映射规则。"@/*": ["src/*"] 表示“凡是以 @/ 开头的导入，把 @/ 换成 src/ 去找“。* 是通配，代表后面的任意路径片段。

⚠️ paths 只在类型层面生效，编译器不会改写运行时的 import 路径！ 也就是说，光配 paths 后 tsc 类型检查能过，但真正运行时（Node/tsx/浏览器）并不认识 @/。你必须让运行时或打包器也知道这套别名：Vite/webpack 有各自的 resolve.alias 配置，tsx/Node 可用 tsconfig-paths 之类的加载器，或在 package.json 的 imports 字段里配。别名要“两头配“（TS 一份、运行时一份）才真正可用。 本书示例为避免这层复杂性没有使用 paths。

声明文件（.d.ts）：给 JS 补类型

声明文件是后缀为 .d.ts 的文件。它的特点是：只有类型声明，没有任何实现。它不会被编译成 JS（里面本来也没有可执行代码），它的唯一使命是“告诉 TS：某段 JS 代码/某个模块/某个全局变量，具有什么类型“。

核心关键字是 declare，意思是“我声明存在这么个东西，它的实现在别处（某个 JS 文件里）“。

// 文件：code/ch20/globals-demo.d.ts（讲解示例）

// 声明一个「运行时确实存在、但 TS 本来不知道」的全局变量。
// 比如某个 <script> 往 window 上挂了 APP_VERSION。
declare const APP_VERSION: string;

// 声明一个全局函数。
declare function track(event: string, payload?: Record<string, unknown>): void;

逐段拆解：

  • declare const APP_VERSION: string：告诉 TS “有一个全局常量 APP_VERSION，类型是 string，它的值哪来的不用你管（运行时由别的脚本注入）”。有了它，你在任意 .ts 里用 APP_VERSION 就有类型、不报错。
  • declare function track(...)：同理，声明一个全局函数的签名。declare 后面不能写函数体——因为它只是声明，实现在别处。

declare module：给整个模块声明类型

如果一个没有类型的第三方模块你要 import，可以用 declare module "模块名" 给它整体补类型：

// 文件：code/ch20/module-decl-demo.d.ts（讲解示例）

// 给一个假想的、没有类型的库 "tiny-emitter" 补类型。
declare module "tiny-emitter" {
  export interface Emitter {
    on(event: string, handler: (...args: unknown[]) => void): void;
    emit(event: string, ...args: unknown[]): void;
  }
  const createEmitter: () => Emitter;
  export default createEmitter;
}

// 也常用来让 TS 认识「非代码资源」的 import（配合打包器）：
declare module "*.svg" {
  const src: string;
  export default src;
}

逐段拆解：

  • declare module "tiny-emitter" { ... }：括号里描述“import ... from "tiny-emitter" 时能拿到什么“。这样即便这个库一行类型都没有，你 import 它时也有完整提示。
  • declare module "*.svg"：用通配符匹配一类 import。前端项目里 import logo from "./logo.svg" 本来会让 TS 报错（它不认识 .svg 模块），加上这段声明后，TS 就知道“import 一个 svg 会得到一个 string（图片 URL）“。

全局声明 vs 模块声明：一个关键区别

一个 .d.ts 文件是“全局脚本“还是“模块“，取决于它里面有没有顶层的 import/export：

  • 没有 import/export：整个文件是全局脚本，里面的 declare const/declare function 会成为全局可见的声明（无需 import 就能用，如上面的 APP_VERSION）。
  • 有 import/export：整个文件是一个模块，里面的声明默认是“模块作用域“的，别人要 import 才能用。

📌 这个规则常常“坑“到人：你本想写一个全局声明，结果文件里不小心带了个 export，全局声明就“消失“了。反过来，本章小项目的 legacy-math.d.ts 里全是 export——因为它就是要作为一个模块被 import。

triple-slash 指令与 declare global

  • /// <reference types="node" />：这种以三斜线开头的注释叫 triple-slash 指令，用于在一个 .d.ts 里“引入另一份类型声明的依赖“。现代项目多用 tsconfig 的 types/typeRoots 管理，triple-slash 主要出现在老的 .d.ts 或某些库的类型入口里，认得即可。
  • declare global：在一个模块文件（带 import/export 的）里，如果你想往全局添加声明，就得用 declare global { ... } 把它“提升“到全局作用域。
// 文件：code/ch20/declare-global-demo.d.ts（讲解示例）
export {}; // 有这行（或任意 import/export），本文件就是「模块」

declare global {
  // 给全局的 Window 接口补一个属性（前端常见：第三方脚本挂在 window 上）
  interface Window {
    dataLayer?: unknown[];
  }
  // 往全局加一个变量
  const BUILD_TIME: string;
}

逐段拆解：

  • export {} 是个惯用法——它什么都不导出，但足以把文件标记成“模块“（这样才能用 declare global，且避免文件里其他声明意外变成全局）。
  • declare global { interface Window { ... } }：接口是“开放“的，同名接口会自动合并（declaration merging）。这里给内置的 Window 补了一个 dataLayer 属性，于是 window.dataLayer 全项目可用且有类型。这是给第三方脚本、polyfill 补全局类型的标准手法。

@types 与 DefinitelyTyped

现实中，你要用的大多数没类型的库，不用自己写 .d.ts——社区早就替你写好了，放在一个叫 DefinitelyTyped 的巨型仓库里，以 @types/库名 的形式发布到 npm。

规则很简单：

  • 库自带类型（现代库、TS 写的库）：直接 npm i 库名 就有类型，什么都不用做。
  • 库不带类型（很多老的 JS 库，如早期的 lodash、express）：额外装 npm i -D @types/库名，类型就来了。
# 例：lodash 本身是 JS，没类型；装它的社区类型包
npm i lodash
npm i -D @types/lodash

装完 @types/lodash 后，import _ from "lodash" 就有完整提示了——因为 TS 的模块解析会自动去 node_modules/@types/lodash 找类型（回忆前面“模块解析“那节的第 3 步）。

判断一个库“要不要装 @types“的最快方法：先 import，如果 TS 报 Could not find a declaration file for module 'xxx'，就去搜 @types/xxx；如果不报错，说明库自带类型或已装过。

💡 @types/* 一律装成 devDependencies（-D）——它们只在开发/编译期需要，运行时用不到。

⚠️ @types 版本可能和库本身版本对不上（社区维护有滞后）。偶尔会遇到“类型说有这个方法，实际库版本没有“或反之。遇到诡异的类型错误时，检查一下两者版本是否匹配。

与 JS 互操作：allowJs、checkJs 与 JSDoc 迁移

假设你接手一个纯 JS 老项目，想渐进迁移到 TS——不可能一夜之间全改成 .ts。TS 为此提供了平滑路径。

第一步：让 TS “接纳” JS。 开 allowJs，TS 就允许项目里有 .js 并参与编译。此时 TS 不检查 .js 的类型（只当它是一堆 any），但你已经可以在 .ts 里 import .js、逐个文件把 .js 改名成 .ts。

第二步：让 TS “检查” JS。 再开 checkJs（或在单个 .js 文件顶部写 // @ts-check），TS 就会对 .js 也做类型检查。这时它会尽力推断，并且——关键——读取你写的 JSDoc 注释当类型用。

JSDoc 就是“写在注释里的类型“。它让你不改 .js 后缀、不引入编译步骤，就能获得几乎和 TS 一样的类型检查：

// 文件：code/ch20/jsdoc-demo.js（讲解示例，配合 checkJs 或 // @ts-check）
// @ts-check

/**
 * 把两个数相加。
 * @param {number} a 第一个加数
 * @param {number} b 第二个加数
 * @returns {number}
 */
function add(a, b) {
  return a + b;
}

/**
 * @typedef {Object} Point
 * @property {number} x
 * @property {number} y
 */

/** @type {Point} */
const p = { x: 1, y: 2 };

add(1, 2);       // ✓
// add(1, "x");  // 报错：参数 "x" 不是 number（和 .ts 一样的检查！）

逐段拆解：

  • // @ts-check：写在 .js 文件顶部，让 TS 只对这一个文件做类型检查（即使没开全局 checkJs）。反过来 // @ts-nocheck 是关闭某个文件的检查。
  • @param {number} a：用 JSDoc 标注参数 a 的类型是 number。{...} 里就是 TS 类型语法（可以写联合类型 {string|number}、数组 {number[]} 等）。
  • @returns {number}：标注返回值类型。
  • @typedef {Object} Point + @property：在 JS 里“定义一个类型/接口“，相当于 TS 的 interface Point { x: number; y: number }。
  • @type {Point}：给一个变量标类型。
  • 效果：add(1, "x") 会像在 TS 里一样报错。一行代码没改成 TS，却拿到了类型检查——这就是渐进迁移的第一站。

渐进迁移 JS→TS 的推荐顺序：

  1. 建 tsconfig，开 allowJs（让项目跑起来、TS 接纳现有 JS）。
  2. 开 checkJs（或给关键文件加 // @ts-check），给核心 JS 补 JSDoc，先把最容易出错的地方保护起来。
  3. 一个文件一个文件地把 .js 改名成 .ts，把 JSDoc 翻成真正的 TS 类型标注（TS 里类型更强大、更好写）。
  4. 全部迁完后，关掉 allowJs/checkJs，收紧 strict。

💡 JSDoc 类型的表达力接近但不等于 TS——泛型、复杂条件类型等写起来很啰嗦。它的定位是“迁移过渡“和“不方便引入编译步骤的纯 JS 项目（如某些 Node 脚本、构建配置）“，正式 TS 项目还是直接写 .ts 更舒服。

常见坑与最佳实践

  • ⚠️ paths 别名要“两头配“：只配 tsconfig 的 paths，类型检查过了但运行时会 Cannot find module。必须同时配运行时/打包器的别名（Vite/webpack 的 alias、Node 的 imports、tsconfig-paths）。

  • ⚠️ .d.ts 里有没有 export，决定它是“模块“还是“全局“。想写全局声明就别带 import/export（或用 declare global）；想作为模块被 import 就写 export。搞混了会出现“我明明声明了全局变量，却找不到“或“我的全局声明污染了所有文件“。

  • ⚠️ skipLibCheck: true 会跳过你自己 .d.ts 的检查。它带来的速度和“避开第三方类型冲突“的好处通常远大于代价，建议开；但要知道：你手写的 .d.ts 里如果有类型错误，不会在编译时被抓到——所以手写声明文件时要格外仔细，最好写个 use-lib.ts 实际 import 来验证（就像本章小项目做的）。

  • ⚠️ strict 是新项目的默认起点，不是可选项。省下的调试时间远超“多写几个判空“的成本。老项目迁移时逐子项打开。

  • 💡 target 跟着运行环境走：现代 Node 后端用 ES2022+ 即可，别无脑降到 ES5（会生成臃肿的降级代码）。前端交给打包器时，TS 的 target 也不必太低，让打包器（配 browserslist）负责最终兼容。

  • 💡 发布库时：开 declaration: true 生成 .d.ts，并在 package.json 里用 "types"（或 exports 的 types 条件）指向类型入口，别人才能用上你的类型。

  • 💡 手写 .d.ts 的名字与位置：给 foo.js 写声明，最省事的方式是同目录放一个同名的 foo.d.ts——TS 会自动关联，无需任何配置（本章小项目正是如此）。

  • ⚠️ allowJs 不是 import .js 的必要条件：只要 .js 旁有同名 .d.ts，不开 allowJs 也能 import（TS 用 .d.ts 当类型）。只有要让 TS 编译/检查 .js 本身时才需要 allowJs。

本章小项目：为 JS 库编写 .d.ts

现在把声明文件的知识用到实处。我们有一个“第三方“纯 JS 库 legacy-math.js（假装是从 npm 装来的、一行类型都没有的老库），要为它手写一份配套的 legacy-math.d.ts，然后在 use-lib.ts 里 import 它、享受完整类型提示与检查。

第一步：认识这个“没类型“的 JS 库

// 文件：code/ch20/legacy-math.js
export function add(a, b) {
  return a + b;
}

export function clamp(value, min, max) {
  if (value < min) return min;
  if (value > max) return max;
  return value;
}

export function randomBetween(min, max, integer) {
  const r = Math.random() * (max - min) + min;
  return integer ? Math.floor(r) : r;
}

export function formatCurrency(amount, options) {
  const opts = options || {};
  const currency = opts.currency || "CNY";
  const locale = opts.locale || "zh-CN";
  const decimals = opts.decimals == null ? 2 : opts.decimals;
  return new Intl.NumberFormat(locale, {
    style: "currency",
    currency,
    minimumFractionDigits: decimals,
    maximumFractionDigits: decimals,
  }).format(amount);
}

const legacyMath = { add, clamp, randomBetween, formatCurrency };
export default legacyMath;

观察这个库的“接口形态“（这决定了 .d.ts 怎么写）：

  • add(a, b)：两数相加。但 + 对字符串是拼接——我们希望类型上体现“数字+数字=数字、字符串+字符串=字符串“，这需要函数重载（overload）。
  • clamp(value, min, max)：三个数字参数，返回数字。普通签名。
  • randomBetween(min, max, integer)：第三个参数 integer 可选。
  • formatCurrency(amount, options)：第二个参数是选项对象，且整体可选；对象里的字段（currency/locale/decimals）也都可选。这要用一个接口描述。
  • 既有命名导出（add 等），又有一个默认导出（打包所有函数的对象）。

第二步：手写 legacy-math.d.ts

// 文件：code/ch20/legacy-math.d.ts
export interface CurrencyOptions {
  /** 货币代码，如 "CNY"、"USD"、"JPY"。默认 "CNY"。 */
  currency?: string;
  /** BCP 47 语言标签，如 "zh-CN"、"en-US"。默认 "zh-CN"。 */
  locale?: string;
  /** 小数位数。默认 2。 */
  decimals?: number;
}

// 函数重载：数字 + 数字 => 数字；字符串 + 字符串 => 字符串。
export function add(a: number, b: number): number;
export function add(a: string, b: string): string;

export function clamp(value: number, min: number, max: number): number;

export function randomBetween(min: number, max: number, integer?: boolean): number;

export function formatCurrency(amount: number, options?: CurrencyOptions): string;

declare const legacyMath: {
  add: typeof add;
  clamp: typeof clamp;
  randomBetween: typeof randomBetween;
  formatCurrency: typeof formatCurrency;
};
export default legacyMath;

逐段拆解——这是本章最核心的写法：

  • 文件名与位置：legacy-math.d.ts 和 legacy-math.js 同名、同目录。这样 TS 在解析 import ... from "./legacy-math.js" 时会自动把这份 .d.ts 当作那个 .js 的类型来源，无需任何 tsconfig 配置、也无需 allowJs。
  • export interface CurrencyOptions：把选项对象的形状抽成一个接口，并 export 出去（这样使用方也能引用这个类型）。三个字段都带 ?（可选），因为库里对它们都有默认值。/** ... */ 是 JSDoc 文档注释——它会显示在编辑器的悬浮提示里，是给声明文件加说明的好习惯。
  • 函数重载：add 写了两行声明——(number, number): number 和 (string, string): string。这就是声明文件里的函数重载：列出多个签名，调用方按实参匹配其中一个。注意 .d.ts 里的 function 声明天然没有函数体（它本就是“声明“）。这样 add(1, 2) 得 number、add("a", "b") 得 string、而 add(1, "x") 因不匹配任何重载而报错。
  • clamp：最普通的函数声明，三个 number 参数、返回 number。
  • randomBetween：integer?: boolean ——参数名后加 ? 表示可选参数，可传可不传。可选参数必须排在必填参数后面。
  • formatCurrency：options?: CurrencyOptions ——整个选项对象可选，且类型是我们上面定义的接口。于是使用方写 options 时，currency/locale/decimals 都会有提示和检查。
  • 默认导出：declare const legacyMath: { ... } 先声明一个“内部“常量，描述默认导出对象的形状——每个字段用 typeof add 复用上面已声明的函数类型（避免重复写签名，typeof 在类型位置表示“取某个值的类型“）。最后 export default legacyMath 把它作为默认导出。这样 import legacyMath from "./legacy-math.js" 也能拿到完整类型。

📌 .d.ts 里只写“形状“，不写“实现“。你不会在这里写 return a + b——那是 .js 的事。声明文件的职责是“描述契约“，实现与描述由你保证一致（TS 无法替你核对 .js 实现是否真的符合声明，这也是手写 .d.ts 要仔细的原因）。

第三步：在 use-lib.ts 里享受类型

// 文件：code/ch20/use-lib.ts
import legacyMath, {
  add,
  clamp,
  randomBetween,
  formatCurrency,
  type CurrencyOptions,
} from "./legacy-math.js";

const sum: number = add(2, 3);                 // 走 (number, number) => number
const joined: string = add("Type", "Script"); // 走 (string, string) => string
// add(1, "x"); // 报错：没有匹配 (number, string) 的重载

const clamped: number = clamp(15, 0, 10); // => 10

const rndFloat: number = randomBetween(0, 1);     // 不传 integer
const rndInt: number = randomBetween(1, 7, true); // 传 integer

const opts: CurrencyOptions = { currency: "USD", locale: "en-US", decimals: 2 };
const priceUSD: string = formatCurrency(1234.5, opts);
const priceCNY: string = formatCurrency(88.8);    // options 可省略
// formatCurrency(100, { currency: 123 }); // 报错：currency 应为 string

const viaDefault: number = legacyMath.clamp(99, 0, 100);

console.log("add(2,3) =", sum);
console.log('add("Type","Script") =', joined);
console.log("clamp(15,0,10) =", clamped);
console.log("randomBetween(0,1) =", rndFloat.toFixed(4));
console.log("randomBetween(1,7,true) =", rndInt);
console.log("formatCurrency(1234.5, USD) =", priceUSD);
console.log("formatCurrency(88.8) =", priceCNY);
console.log("legacyMath.clamp(99,0,100) =", viaDefault);

逐段拆解：

  • 导入方式：import legacyMath, { add, ..., type CurrencyOptions } from "./legacy-math.js" 一次性演示了默认导入（legacyMath）、命名导入（add 等）、以及类型导入 type CurrencyOptions。因为本书开了 verbatimModuleSyntax，导入类型时必须加 type 前缀，否则会被当成值导入。
  • import 路径写 .js：我们 import 的是 "./legacy-math.js"（真实存在的运行时文件），TS 会自动去找同名的 legacy-math.d.ts 拿类型。moduleResolution: "bundler" 允许这样写。
  • 重载生效：add(2, 3) 命中数字重载得 number，add("Type", "Script") 命中字符串重载得 string。注释里的 add(1, "x") 会报错——没有任何重载匹配“数字+字符串“。
  • 可选参数：randomBetween(0, 1) 省略了第三个参数（合法），randomBetween(1, 7, true) 传了。
  • 选项对象：opts 标成 CurrencyOptions 后，写错字段类型（如 currency: 123）会被抓。formatCurrency(88.8) 省略整个 options 也合法。
  • 默认导出：legacyMath.clamp(...) 同样有完整类型。

运行与验证

npx tsx ch20/use-lib.ts

预期输出：

add("Type","Script") = TypeScript
clamp(15,0,10) = 10
randomBetween(0,1) = 0.xxxx   （随机小数）
randomBetween(1,7,true) = 5   （1~6 的随机整数）
formatCurrency(1234.5, USD) = $1,234.50
formatCurrency(88.8) = ¥88.80
legacyMath.clamp(99,0,100) = 99

npx tsc --noEmit

关键点回顾：本项目的主 tsconfig 没有开 allowJs，include 也只匹配 .ts，但 use-lib.ts 依然能 import legacy-math.js 并获得完整类型——全靠那份同名 legacy-math.d.ts。这印证了前面反复强调的：有 .d.ts 时，import .js 不需要 allowJs。

扩展练习

  1. 用 JSDoc 达到类似效果：新建一个 legacy-math-jsdoc.js，把函数实现和 JSDoc 类型注解写在同一个文件里（@param/@returns/@typedef），文件顶部加 // @ts-check。对比“手写独立 .d.ts“和“就地 JSDoc“两种方式的取舍——JSDoc 无需额外文件、类型和实现在一处不易脱节，但表达复杂类型（如重载）时更啰嗦。（提示：JSDoc 里重载可用多个 @overload 标签。）

  2. 给 formatCurrency 的 currency 收窄类型：把 currency?: string 改成联合字面量 currency?: "CNY" | "USD" | "JPY" | "EUR"，让传错货币代码时编译报错。体会“声明文件比原 JS 更严“是完全可以的。

  3. 补一个 declare module：假设这个库在 npm 上叫 "legacy-math"（裸模块名），写一份 declare module "legacy-math" { ... } 形式的声明，放到项目的 typings/ 目录并配好 typeRoots，让 import ... from "legacy-math" 也能用。

  4. 故意写错声明验证：把 .d.ts 里 clamp 的返回改成 string，观察 use-lib.ts 里 const clamped: number = clamp(...) 报错。这说明声明文件的正确性由你负责——写完务必用真实调用验证。

小结与自测

本章我们把“配置与互操作“这一块讲透了：

  • tsconfig.json 是 TS 编译器的控制面板。分组记忆：
    • 输出：target（编到哪个 JS 版本）、module（模块格式）、moduleResolution（node/nodenext/bundler）、lib、outDir/rootDir、declaration、sourceMap。
    • 严格性：strict（一键开一组）及其子项（noImplicitAny、strictNullChecks、strictFunctionTypes…），以及 noUncheckedIndexedAccess、exactOptionalPropertyTypes。
    • 额外检查：noUnusedLocals、noImplicitReturns、noFallthroughCasesInSwitch。
    • 互操作/JS：esModuleInterop、allowJs、checkJs、resolveJsonModule、skipLibCheck、isolatedModules、verbatimModuleSyntax、types/typeRoots。
    • 组织：include/exclude/files、extends（继承）、project references（大仓库增量编译）。
  • 模块解析：相对路径按文件找、优先 .ts/.d.ts；裸模块名去 node_modules 及 @types 找。baseUrl+paths 配别名（记得两头配）。
  • 声明文件 .d.ts：只有类型没有实现，用 declare 声明“实现在别处“的东西。declare module 给整个模块补类型，全局声明（无 export 的文件）/declare global 加全局类型。@types/* 是社区维护的类型包（npm i -D @types/xxx）。
  • JS 互操作：allowJs 让 TS 接纳 .js，checkJs + JSDoc 给 .js 上类型检查，是 JS→TS 渐进迁移的阶梯。
  • 小项目：为纯 JS 库 legacy-math.js 手写 legacy-math.d.ts，演示了函数重载、可选参数、选项对象接口、命名导出与默认导出；并验证了“有 .d.ts 时 import .js 无需 allowJs“。

自测题（先自己想，再回代码/配置验证）：

  1. strict: true 打开后，至少列出 3 个被同时打开的子项，各防什么错？
  2. noUncheckedIndexedAccess 开启后，const x = arr[0] 里 x 是什么类型？为什么这样更安全？
  3. 一个 .d.ts 文件什么情况下是“全局脚本“、什么情况下是“模块“？这个区别导致什么后果？

  7. paths 别名为什么“只配 tsconfig 还不够“？运行时会发生什么？

答得上来，你就掌握了从“能跑“到“能配置、能对接 JS 生态“的关键一跃。

配好了 tsconfig、也能给 JS 补类型了，你的 TypeScript 项目已经“立得住“。但真实工程里，光有编译器还不够：代码风格要统一（不然每次 diff 全是空格之争）、低级错误要在写代码时就被点出、成百上千个模块最终还要打包成能部署的产物。这些活儿交给一套工具链。
