第 24 章 · 数据持久化与配置管理

第 23 章我们把任务管理 API 跑了起来，但它有一个致命短板：数据存在一个内存数组里，进程一重启，所有任务就消失了。这一章我们就来解决它——让数据「落地」，重启不丢；顺便学会用规范的方式管理配置（端口、数据库路径、密钥这些「因环境而异」的东西）。

本章目标

  • 理解**数据持久化（persistence）**的几个层次：内存 → JSON 文件 → 嵌入式数据库（SQLite）→ 独立数据库（Postgres/MySQL）→ ORM，以及各自的取舍。
  • 会用 fs/promises 把数据存成 JSON 文件，并理解它的并发写风险。
  • 掌握 SQLite，用 Node 内置的 node:sqlite 模块（无需安装依赖）完成建表、增删改查、参数化查询（防 SQL 注入）和事务（transaction）。
  • 速成一小段 SQL 基础：CREATE TABLE / INSERT / SELECT / UPDATE / DELETE。
  • 了解 ORM / Query Builder（Prisma、Drizzle、Knex）的定位（只讲概念，不要求安装）。
  • 学会配置管理：环境变量、.env 文件、区分 dev/prod、敏感信息不入库不入 git，并用第 13 章的类型化配置思想校验环境变量。
  • 小项目：把第 23 章任务 API 的内存 store 换成 SQLite 持久化，数据重启不丢。

📌 本章所有代码都在 code/ch24/api/ 这个独立子项目里，零第三方运行时依赖（node:sqlite 是 Node 自带的）。本机 Node 版本是 v24，node:sqlite 已可直接使用，无需任何命令行标志。

24.1 持久化的层次：从内存到数据库

「持久化」就一句话：让数据在程序结束后依然存在。实现方式有很多档次，越往下越强大，也越重：

层次代表优点缺点适合
内存一个数组 / Map最快、最简单重启即失，容量受内存限制缓存、临时状态、demo
JSON 文件fs + JSON.stringify零依赖、人可读全量读写慢、并发写易丢数据、无查询能力极小数据、原型、配置
嵌入式数据库SQLite一个文件、无需服务器、支持 SQL/事务/索引高并发写入受限（单写者）学习、桌面应用、中小型服务
独立数据库PostgreSQL / MySQL高并发、网络访问、功能完备要单独部署运维生产级、多实例应用
ORM / Query BuilderPrisma / Drizzle / Knex用对象/类型操作数据库、迁移管理有学习成本和抽象泄漏中大型项目的工程化

💡 不要一上来就上最重的方案。 选型的黄金法则是「够用就好」。一个人用的小工具，JSON 文件可能就够了；一个中小型 Web 服务，SQLite 往往完全胜任；当你需要多台服务器同时高并发写同一份数据时，才真的需要 Postgres/MySQL。本章主线用 SQLite——它是「学习成本低」和「真实可用」的最佳平衡点。

下面我们从最简单的 JSON 文件讲起，感受它的好与不好，再自然过渡到 SQLite。

24.2 文件持久化：用 JSON 文件当仓库

思路非常直白：内存里维护一个数组，每次改动后，把整个数组序列化成 JSON 写回文件；程序启动时再把文件读回内存。

我们用第 22 章学过的 fs/promises（Promise 版的文件 API）。看核心实现：

// 文件：code/ch24/api/src/store/jsonTaskStore.ts
import { readFile, writeFile, mkdir } from "node:fs/promises";
import path from "node:path";
import { randomUUID } from "node:crypto";
import type { Task, CreateTaskInput, UpdateTaskInput } from "../types.js";

export class JsonTaskStore {
  private tasks: Task[] = [];
  private loaded = false;
  // 写入队列：保证所有写盘操作【串行】执行，避免并发覆盖。
  private writeChain: Promise<void> = Promise.resolve();

  constructor(private file: string) {}

  /** 首次使用前从磁盘加载；文件不存在则视为空列表。 */
  private async ensureLoaded(): Promise<void> {
    if (this.loaded) return;
    try {
      const text = await readFile(this.file, "utf8");
      this.tasks = JSON.parse(text) as Task[];
    } catch (err: unknown) {
      // ENOENT = 文件不存在，属正常（第一次运行）；其它错误照常抛出。
      if ((err as NodeJS.ErrnoException).code !== "ENOENT") throw err;
      this.tasks = [];
    }
    this.loaded = true;
  }
  // ……（create / update / remove 见仓库文件）
}

逐段拆解：

  • private tasks: Task[]：内存里的「当前快照」。所有读操作都从它返回，所以读很快。
  • ensureLoaded()：懒加载。第一次真正用到数据时才读文件，用 loaded 标志保证只读一次。
  • try/catch 里那个 ENOENT 判断很关键：文件第一次运行时并不存在，readFile 会抛一个 code === "ENOENT" 的错误。我们把「文件不存在」当作「空列表」正常处理，其它错误（比如权限问题、JSON 损坏）则原样抛出——不要用 catch 把所有错误都吞掉。

写盘部分是这个方案最需要小心的地方：

// 文件：code/ch24/api/src/store/jsonTaskStore.ts
  /** 把当前内存快照写回磁盘。用 writeChain 把多次写串行化，避免竞态。 */
  private persist(): Promise<void> {
    const snapshot = JSON.stringify(this.tasks, null, 2);
    this.writeChain = this.writeChain.then(async () => {
      await mkdir(path.dirname(this.file), { recursive: true });
      await writeFile(this.file, snapshot, "utf8");
    });
    return this.writeChain;
  }

逐段拆解与陷阱：

  • ⚠️ 并发写风险（race condition）：JSON 文件方案最大的坑。想象两个请求几乎同时到达：请求 A 读到 [t1]，加了 t2，准备写 [t1, t2]；与此同时请求 B 也读到 [t1]，加了 t3，写了 [t1, t3]。如果 B 后写，A 的 t2 就被覆盖丢失了。这类「读—改—写」竞态在异步代码里非常隐蔽。
  • 我们用一个小技巧缓解：writeChain 是一条 Promise 链，每次 persist 都把新的写操作 .then 挂到上一次之后，从而保证所有写盘串行执行，不会交叉。注意我们先在链外算好 snapshot（当下的快照），避免闭包读到后来被改动的数组。
  • 即便如此，JSON 方案仍然是每次都覆盖整个文件——数据一多就慢，而且没有「按条件查询」的能力。这正是我们要转向数据库的原因。

📌 JSON 文件适合：配置、极小数据量、单进程原型。一旦你需要「并发安全 + 查询 + 只改一行」，就该上 SQLite 了。

24.3 SQLite：一个文件就是一个数据库

SQLite 是世界上部署最广的数据库引擎（你的手机里可能就有几十个）。它的特别之处：没有独立的服务器进程，整个数据库就是磁盘上的一个 .db 文件，你的程序直接读写这个文件。

它为什么适合学习和中小项目？

  • 零运维：不用安装、不用启动服务、不用配账号密码。
  • 完整的 SQL：支持表、索引、事务、约束——你学到的 SQL 知识可以平滑迁移到 Postgres/MySQL。
  • 足够快：对「读多写少 / 中等写入」的场景，性能绰绰有余。
  • ⚠️ 写入是单写者：同一时刻只允许一个写事务。高并发写的场景（成千上万用户同时下单）才需要换成 Postgres/MySQL；大多数应用达不到这个量级。

在 Node 里用 SQLite 有两条路：

  1. node:sqlite（本章首选）：Node 内置模块（Node 22 引入，Node 24 起可直接使用）。无需 npm install，import { DatabaseSync } from "node:sqlite" 即可。
  2. better-sqlite3：一个成熟的第三方库，API 与 node:sqlite 高度相似，需要 npm install 并编译原生模块。老版本 Node 上没有内置模块时用它。

📌 本机实测：Node v24.18.0 下 import { DatabaseSync } from "node:sqlite" 无需任何标志即可运行。如果你的 Node 是 22.x，可能需要加 --experimental-sqlite（例如 node --experimental-sqlite ...），届时命令行会有提示。若你的环境完全没有 node:sqlite，把 DatabaseSync 换成 better-sqlite3 的 Database，其余代码几乎一字不改。

24.4 SQL 基础速成

在写代码前，先花几分钟认识我们要用到的 SQL 语句。SQL（Structured Query Language）是操作关系型数据库的通用语言。

建表——定义一张表的结构（列名 + 类型 + 约束）：

CREATE TABLE IF NOT EXISTS tasks (
  id         TEXT    PRIMARY KEY,          -- 主键，唯一标识一行
  title      TEXT    NOT NULL,             -- 非空
  done       INTEGER NOT NULL DEFAULT 0,   -- SQLite 无布尔类型，用 0/1
  created_at TEXT    NOT NULL              -- ISO 8601 字符串
);
  • IF NOT EXISTS：表已存在就跳过，让「建表」这一步可以安全地重复执行。
  • PRIMARY KEY：主键，保证唯一、且自带索引，按它查很快。
  • ⚠️ SQLite 没有独立的布尔类型和日期类型。布尔我们用 INTEGER 存 0/1；日期我们用 TEXT 存 ISO 字符串。取出来时在代码里转换回 boolean。

插入 / 查询 / 更新 / 删除（CRUD）：

INSERT INTO tasks (id, title, done, created_at) VALUES (?, ?, ?, ?);

SELECT id, title, done, created_at FROM tasks
  WHERE done = 0                 -- 过滤条件
  ORDER BY created_at DESC       -- 排序（DESC 倒序）
  LIMIT 10;                      -- 只取前 10 条

UPDATE tasks SET title = ?, done = ? WHERE id = ?;

DELETE FROM tasks WHERE id = ?;

那些 ? 是占位符（placeholder），我们下一节会重点讲它——它是防 SQL 注入的关键。

24.5 用 node:sqlite 实现 TaskRepository

现在把 SQL 落到 TypeScript 里。我们写一个 TaskRepository 类，方法名和第 23 章的内存 store 完全对应（findAll / findById / create / update / remove），这样上层代码几乎不用改。

先看构造函数和建表：

// 文件：code/ch24/api/src/store/taskRepository.ts
import { DatabaseSync } from "node:sqlite";
import { randomUUID } from "node:crypto";
import type { Task, CreateTaskInput, UpdateTaskInput } from "../types.js";

export class TaskRepository {
  private db: DatabaseSync;

  constructor(dbFile: string) {
    // 打开（不存在则创建）数据库文件。传 ":memory:" 则用纯内存库。
    this.db = new DatabaseSync(dbFile);
    // 打开外键约束（SQLite 默认关闭），养成好习惯。
    this.db.exec("PRAGMA foreign_keys = ON;");
    this.migrate();
  }

  private migrate(): void {
    // exec 执行「不返回结果、也不带参数」的 SQL。
    this.db.exec(`
      CREATE TABLE IF NOT EXISTS tasks (
        id         TEXT    PRIMARY KEY,
        title      TEXT    NOT NULL,
        done       INTEGER NOT NULL DEFAULT 0,
        created_at TEXT    NOT NULL
      );
    `);
    this.db.exec(`CREATE INDEX IF NOT EXISTS idx_tasks_created_at ON tasks(created_at);`);
  }
}

逐段拆解：

  • new DatabaseSync(dbFile)：打开一个数据库连接。DatabaseSync 是同步的（方法直接返回结果，不用 await）——对「一个进程一个文件」的场景，这反而更简单直观，不用到处写 async。
  • db.exec(sql)：执行一段 SQL，不带参数、不取结果，适合建表、PRAGMA、BEGIN/COMMIT 这类语句。
  • migrate()：把「初始化表结构」收拢在一处。因为用了 IF NOT EXISTS，每次启动都调用它是安全的——这是最朴素的「迁移（migration）」雏形。
  • 那个 CREATE INDEX：给 created_at 建索引，让「按时间排序/查询」更快。数据量小时看不出差别，量大时是数量级的区别（本章扩展练习会展开）。

接着是读操作。⚠️ 数据库里的列是 snake_case、done 是 0/1，而我们的领域模型是 camelCase、done 是 boolean，所以需要一个映射函数：

// 文件：code/ch24/api/src/store/taskRepository.ts
  private rowToTask(row: Record<string, unknown>): Task {
    return {
      id: row.id as string,
      title: row.title as string,
      done: (row.done as number) === 1, // 0/1 → boolean
      createdAt: row.created_at as string,
    };
  }

  findAll(): Task[] {
    const rows = this.db
      .prepare("SELECT id, title, done, created_at FROM tasks ORDER BY created_at DESC")
      .all() as Record<string, unknown>[];
    return rows.map((r) => this.rowToTask(r));
  }

  findById(id: string): Task | undefined {
    const row = this.db
      .prepare("SELECT id, title, done, created_at FROM tasks WHERE id = ?")
      .get(id) as Record<string, unknown> | undefined;
    return row ? this.rowToTask(row) : undefined;
  }

逐段拆解：

  • db.prepare(sql)：把一条 SQL 预编译成一个可复用的「语句对象（statement）」。之后可以用不同参数反复执行它，既快又安全。
  • .all() 返回所有匹配行（数组）；.get(...) 返回第一行（或 undefined）。
  • rowToTask 做「数据库行 → 领域对象」的翻译。把这层转换收拢在仓储内部，上层拿到的永远是干净的 Task，不用关心数据库里存的是 0 还是 1。

参数化查询：防 SQL 注入的正解

注意 findById 里的 WHERE id = ?，我们把 id 通过 .get(id) 传进去，而不是拼字符串。这就是参数化查询（parameterized query），也叫预处理语句。

先看反例，感受一下危险在哪：

// ❌ 绝对不要这样做：把用户输入直接拼进 SQL
const badSql = `SELECT * FROM tasks WHERE id = '${userInput}'`;
this.db.exec(badSql);

如果 userInput 是普通 id，看起来没问题。但如果攻击者传入：

'; DROP TABLE tasks; --

拼出来的 SQL 就变成了：

SELECT * FROM tasks WHERE id = ''; DROP TABLE tasks; --'

数据库会先查询、再把整张表删掉——这就是 SQL 注入（SQL injection）。用户输入被当成了 SQL 代码执行。

而参数化查询从根上杜绝了它：

// ✅ 正确：? 是占位符，id 作为「数据」传入，永远不会被当成 SQL 代码
this.db.prepare("SELECT * FROM tasks WHERE id = ?").get(userInput);

📌 为什么参数化就安全？ 因为 SQL 语句的结构在 prepare 阶段就已经确定并编译好了，参数是在执行阶段才作为「纯数据」填进预留的槽位。数据库引擎从不会把参数值重新解析成 SQL 语法。所以哪怕你传一整段 DROP TABLE，它也只是被当作一个（找不到的）字符串 id。这是防注入唯一可靠的办法——不是靠「过滤特殊字符」，而是靠「代码与数据分离」。

我们在演示脚本里实测了这一点：把上面那个恶意字符串当 id 去查，结果是 undefined，表安然无恙。

写操作：create / update / remove

// 文件：code/ch24/api/src/store/taskRepository.ts
  create(input: CreateTaskInput): Task {
    const task: Task = {
      id: randomUUID(),
      title: input.title,
      done: input.done ?? false,
      createdAt: new Date().toISOString(),
    };
    this.db
      .prepare("INSERT INTO tasks (id, title, done, created_at) VALUES (?, ?, ?, ?)")
      .run(task.id, task.title, task.done ? 1 : 0, task.createdAt);
    return task;
  }

  update(id: string, patch: UpdateTaskInput): Task | undefined {
    const existing = this.findById(id);
    if (!existing) return undefined;
    const next: Task = {
      ...existing,
      title: patch.title ?? existing.title,
      done: patch.done ?? existing.done,
    };
    this.db
      .prepare("UPDATE tasks SET title = ?, done = ? WHERE id = ?")
      .run(next.title, next.done ? 1 : 0, next.id);
    return next;
  }

  remove(id: string): boolean {
    const result = this.db.prepare("DELETE FROM tasks WHERE id = ?").run(id);
    return result.changes > 0;
  }

逐段拆解：

  • .run(...)：执行「写」语句（INSERT/UPDATE/DELETE）。四个 ? 一一对应 run 的四个参数，顺序必须一致。布尔要手动转 0/1。
  • create：id 和 createdAt 由服务端生成（不信任客户端提供的 id），这和第 23 章一致。
  • update 采用 PATCH 语义：先查出已有值，用 patch.title ?? existing.title 只覆盖传入的字段，其余保持不变。
  • remove：run() 返回对象里的 changes 是受影响的行数。删到了返回 > 0，本来就不存在返回 0，据此返回 true/false。

事务：要么全成，要么全不做

有时我们要「一次性写入多条，且不允许写一半」。这时需要事务（transaction）：

// 文件：code/ch24/api/src/store/taskRepository.ts
  createMany(inputs: CreateTaskInput[]): Task[] {
    const created: Task[] = [];
    this.db.exec("BEGIN");           // 开启事务
    try {
      for (const input of inputs) {
        created.push(this.create(input));
      }
      this.db.exec("COMMIT");        // 全部成功，提交
      return created;
    } catch (err) {
      this.db.exec("ROLLBACK");      // 中途出错，撤销这一批的全部写入
      throw err;
    }
  }

逐段拆解与「不能讲错」的要点：

  • BEGIN 开启事务，COMMIT 提交，ROLLBACK 回滚。
  • 📌 事务的核心保证是「原子性」（atomicity）：事务里的一组写操作，要么全部生效（COMMIT 后），要么全部不生效（ROLLBACK 后）。绝不会出现「插了 3 条、剩 2 条没插」这种半截状态。
  • 所以：如果循环写到第 3 条时抛错，catch 里的 ROLLBACK 会把前 2 条也一并撤销，数据库回到事务开始前的样子。这就是为什么转账、下单这类操作必须用事务。
  • 💡 事务还有个附带好处：把大量写入包在一个事务里，比逐条自动提交快得多（少了很多次磁盘刷新）。批量导入时尤其明显。

24.6 ORM 与 Query Builder：概念定位

到目前为止我们都是手写 SQL 字符串。项目变大后，很多团队会引入 **ORM（Object-Relational Mapping，对象关系映射）**或 Query Builder（查询构造器），用更贴近编程语言的方式操作数据库。快速认识三个主流选手：

  • Prisma：最流行的 TypeScript ORM。你在一个 schema.prisma 文件里声明模型，它生成完全类型安全的客户端和数据库迁移。
  • Drizzle：更「贴近 SQL」的轻量方案，用 TypeScript 定义表结构，查询写法接近 SQL 但有类型提示，运行时开销小。
  • Knex：老牌 Query Builder（不是完整 ORM），帮你用链式调用拼 SQL（knex('tasks').where('done', 0)），并管理迁移。

感受一下 Prisma 的 schema 长什么样（仅作概念展示，本章不要求安装使用）：

// 概念示例：Prisma schema（schema.prisma）
model Task {
  id        String   @id @default(uuid())
  title     String
  done      Boolean  @default(false)
  createdAt DateTime @default(now())
}

有了它，查询就变成了类型安全的对象调用，而不是字符串：

// 概念示例：Prisma 客户端用法（有完整类型提示与补全）
const tasks = await prisma.task.findMany({
  where: { done: false },
  orderBy: { createdAt: "desc" },
});

该不该用 ORM？

  • 💡 好处：类型安全、自动补全、内建迁移工具、跨数据库（同一份代码换 SQLite/Postgres）。
  • ⚠️ 代价：多一层抽象和学习成本；复杂查询有时反而更难写、性能更难掌控（「抽象泄漏」）。
  • 📌 建议：先把手写 SQL 和参数化查询搞明白（也就是本章做的事），你才能真正读懂 ORM 生成了什么、出问题时如何排查。学习阶段用 node:sqlite 手写，是打地基；工程化项目里再按需引入 Prisma/Drizzle。

24.7 配置管理：环境变量与 .env

端口、数据库文件路径、第三方密钥……这些值因环境而异（本地开发 vs 线上生产），而且有些是敏感信息。把它们硬编码在源码里是大忌：既不灵活，又容易把密钥泄露到 git 历史里。

正确做法：通过环境变量（environment variables）注入配置。

.env 文件与 process.loadEnvFile

开发时逐个 export 环境变量很麻烦，社区惯例是把它们写在一个 .env 文件里：

# 文件：code/ch24/api/.env.example
PORT=3000
DB_FILE=data/tasks.db
NODE_ENV=development

过去我们用第三方库 dotenv 来加载它。但从 Node 20.12 / 22 起，Node 内置了这个能力，无需依赖：

// 文件：code/ch24/api/scripts/demo-config.ts
try {
  process.loadEnvFile(envFile);   // 把 .env 里的 KEY=VALUE 灌进 process.env
  console.log("已加载 .env：", envFile);
} catch {
  console.log("没有 .env 文件，仅使用当前环境变量。");
}

逐段拆解：

  • process.loadEnvFile(path)：读取指定 .env 文件，把里面的键值对写进 process.env。作用等价于 dotenv。
  • 用 try/catch 包住：.env 不存在时它会抛错，而生产环境常常没有 .env 文件（直接用真实环境变量），所以「没有就跳过」是合理的。
  • 💡 loadEnvFile 不会覆盖已存在的 process.env。所以命令行传入的 PORT=8080 node ... 优先级高于 .env 里的值——这正是我们想要的：临时覆盖很方便。

敏感信息不入库、不入 git

📌 三条铁律：

  1. .env 绝不提交到 git（它可能含密钥）。在 .gitignore 里忽略它：
# 文件：code/ch24/api/.gitignore
.env
data/*.db
  1. 提交一个 .env.example 作模板（不含真实密钥），告诉协作者「需要哪些变量」。别人 cp .env.example .env 后填自己的值。
  2. 密钥不入库，也不入数据库。数据库存业务数据，密钥属于「运行环境的配置」，两者分开。

用「类型化配置」校验环境变量

环境变量有个天然问题：它们全是字符串，而且可能缺失或非法（process.env.PORT 的类型是 string | undefined）。如果直接 Number(process.env.PORT) 拿去用，传了个 "abc" 就会得到 NaN，然后在运行时某个莫名其妙的地方崩掉。

我们沿用第 13 章的类型化配置思想：把「宽松的字符串输入」在启动时一次性解析、校验成一个强类型配置对象，非法就立刻报错（fail fast）。

// 文件：code/ch24/api/src/config/config.ts
export interface AppConfig {
  port: number;
  dbFile: string;
  nodeEnv: "development" | "production" | "test";
}

function parsePort(value: string | undefined, fallback: number): number {
  if (value === undefined || value.trim() === "") return fallback;
  const n = Number(value);
  if (!Number.isInteger(n) || n < 1 || n > 65535) {
    throw new Error(`配置错误：PORT 必须是 1~65535 的整数，收到的是 "${value}"`);
  }
  return n;
}

function parseNodeEnv(value: string | undefined): AppConfig["nodeEnv"] {
  const v = value ?? "development";
  if (v === "development" || v === "production" || v === "test") return v;
  throw new Error(`配置错误：NODE_ENV 只能是 development/production/test，收到的是 "${v}"`);
}

export function loadConfig(
  env: Record<string, string | undefined> = process.env,
): AppConfig {
  return {
    port: parsePort(env.PORT, 3000),
    dbFile: env.DB_FILE ? path.resolve(env.DB_FILE) : defaultDbFile,
    nodeEnv: parseNodeEnv(env.NODE_ENV),
  };
}

逐段拆解：

  • AppConfig 是我们想要的目标形状：port 是 number，nodeEnv 是三选一的联合类型。程序其它地方拿到的都是这种干净的强类型。
  • parsePort 不仅转数字，还校验范围（端口必须 1~65535）。非法就抛错——在启动的第一秒就失败，好过带着坏配置跑到半夜才崩。
  • parseNodeEnv 校验取值并把返回类型收窄成联合类型，编译器从此知道 config.nodeEnv 只可能是那三个值之一。
  • loadConfig(env = process.env)：默认读 process.env，但允许传入自定义对象——这让单元测试可以直接喂一份假 env 做断言，不用真的改环境。
  • dbFile：给了相对路径就 path.resolve 成绝对路径，没给就用默认的 data/tasks.db。💡 区分 dev/prod 很自然：本地 .env 指向一个测试库，生产环境的环境变量指向真实库，同一份代码，不同配置。

常见坑与最佳实践

  • ⚠️ JSON 文件的并发写会丢数据。多个异步操作「读—改—写」同一个文件时后写覆盖先写。要么串行化写入（如本章 writeChain），要么干脆换 SQLite。
  • ⚠️ 永远不要拼接 SQL 字符串。凡是把用户输入放进 SQL 的地方，一律用 ? 占位符 + 参数传值。这是防 SQL 注入的唯一可靠办法，「过滤特殊字符」不可靠。
  • ⚠️ SQLite 没有布尔和日期类型。布尔用 INTEGER（0/1）、日期用 TEXT（ISO 字符串）存，读出时在仓储层转换回来，别让这层细节泄漏到上层。
  • ⚠️ 事务别忘了 ROLLBACK。BEGIN 之后如果不 COMMIT 也不 ROLLBACK，事务会一直悬着、锁住数据库。务必用 try/catch 保证出错时回滚。
  • ⚠️ .env 千万别进 git。先写好 .gitignore 再写 .env。已经误提交的密钥要视为已泄露、立即轮换。
  • 💡 配置在启动时校验、快速失败。坏配置越早报错越好，别等到运行时才神秘崩溃。
  • 💡 数据库文件路径由配置决定。测试用 :memory: 或临时文件，开发用 data/tasks.db，生产用环境变量指定——同一套代码到处跑。
  • 💡 让仓储层的方法名与内存版保持一致（findAll/create/…），换存储时上层零改动。这就是「面向接口而非实现」。

扩展练习

  1. 迁移（migration）版本管理：真实项目里表结构会演进。建一张 schema_version 表记录当前版本号，migrate() 里按版本号依次执行「升级脚本」（比如「v2：给 tasks 加一列 priority」）。想想如何让升级幂等且可回退。
  2. 索引与查询性能：给 tasks 插入 10 万条数据，对比「有 idx_tasks_created_at 索引」和「删掉索引」时 ORDER BY created_at 查询的耗时（用 console.time）。再用 EXPLAIN QUERY PLAN SELECT ... 看 SQLite 是否用上了索引。
  3. 分页查询：给 findAll 加上 limit / offset 参数（SELECT ... LIMIT ? OFFSET ?），实现「每页 10 条」的翻页。
  4. 把 JSON 方案接上 API：用 JsonTaskStore 替换仓储，跑并发请求，观察它和 SQLite 在数据一致性上的差异。

小结与自测

这一章我们让数据「活」了过来——从「重启即失」走到了「重启不丢」：

  • 理清了持久化的层次（内存 → JSON → SQLite → 独立数据库 → ORM）和够用就好的选型原则。
  • 用 fs/promises 实现了 JSON 文件仓储，并认清了它的并发写风险。
  • 用 Node 内置的 node:sqlite 完成了建表、CRUD、参数化查询（防 SQL 注入）和事务（原子性）——这是本章的核心。
  • 补齐了 SQL 基础，了解了 ORM/Query Builder 的定位。
  • 建立了规范的配置管理：环境变量 + .env（process.loadEnvFile）+ 类型化校验 + 敏感信息不入库不入 git。
